# web3 之 privacy

我以前看隐私，脑子里第一反应是工具。

浏览器换成 Brave，聊天换成 Signal，文档别放 Google Docs，钱包别乱签名，助记词别截图。再往深一点，就是 ZK、Monero、Zcash、Privacy Pools、混币、relay、匿名身份。

这些都对。但学完这场 Neo-Cypherpunk 分享后，我发现自己以前的理解太窄了。

隐私不是一个工具目录。也不是出事以后再补的安全设置。它更像一种建设方式：你从产品第一天就要想清楚，谁能看见什么，谁能控制什么，哪些数据必须公开，哪些数据根本不该被收集。

这句话听起来很工程，但越想越实际。

Peter 讲 Neo-Cypherpunk 的时候，有个点我很受触动：早期 Cypherpunk 更像一群密码学家和黑客在保护自己。很硬，很个人，也很专家化。现在不一样了。互联网已经不是 1993 年那种小圈子玩具，几乎所有人都在里面生活。你的聊天、支付、出行、工作、社交关系，都被平台看见、记录、分析。

所以今天谈隐私，不是怀旧。不是说我们要回到一个完美的老派 cypherpunk 年代。那个年代也没那么完美。

Neo-Cypherpunk 更像是把老想法拿回来，重新适配今天的世界。它不只关心个人躲起来，也关心社区怎么互相保护。它不要求每个人都变成密码学专家，也不要求你一夜之间全栈隐私。

这一点对我挺重要。

以前我会把隐私理解成非黑即白：要么你很隐私，要么你裸奔。这样想很容易让人放弃。因为现实里，你可能没法让爸妈跟你用 Signal 聊天，也没法不用所有 Big Tech。工作要协作，生活要方便，很多服务没有替代品。

分享里有一句意思很直白：隐私是一个 0 到 100 的尺度，不是一次考试。

你不能在所有事情上做到 100 分，这很正常。你可以先换一个浏览器，先把敏感聊天迁到更私密的工具，先把政府或公司强制要求的软件放到第二台手机，先学会分清哪些数据真的敏感。一步也算一步。

这比“你怎么还不用某某工具”健康多了。

我也第一次更认真地理解了 personal privacy stack。不是找一个全世界通用的隐私答案，而是为自己的处境配一套工具。举报人、开发者、普通用户、记者、DAO contributor、AI Agent builder，需要的隐私等级完全不同。

有的人需要 Monero，有的人用 Proton 就够了。有的人链上地址一动就会被盯上，有的人只是想少被广告平台追踪一点。没有必要互相羞辱。

这让我意识到，隐私不是装出一种很酷的姿态。它是你知道自己暴露在哪里，然后做选择。

更让我在意的是 AI Agent 这块。

现在大家都在冲 Agent。会聊天不够，要能执行任务。会执行任务不够，还要能互相通信、调工具、发交易、管理资产。听起来很爽。问题是，很多人一边造 Agent，一边根本没想隐私。

如果一个 Agent 在链上做了几笔交易，这些行为是可观察的。它和谁交互，什么时候交互，金额多大，频率如何，可能慢慢拼出一个画像。对普通人也许没事。对高风险身份、资金量大的用户、公开人物，甚至一个 DAO 的运营钱包，这就很敏感。

更麻烦的是，AI 行业跑得太快了。今天相关的东西，明天可能就过时。大家赶着发布，赶着融资，赶着做 demo。隐私和安全经常被放到最后。

但这反而是 builder 的机会。

如果未来真的有大量 Agent 代替人类做链上操作，那“私密 Agent”“受限 relay”“可选择可见性的交易层”“默认隐私的钱包体验”，都会变成很实际的问题。不是论文题，也不是道德口号，是产品需求。

我现在会这么问自己：如果我做一个 AI x Web3 产品，它默认会暴露用户什么？

会暴露地址吗？会暴露行为轨迹吗？会把 prompt、交易意图、资产数据、社交关系丢进第三方服务吗？Agent 之间通信有没有日志？这些日志谁能看？用户能不能删除？能不能选择透明、部分可见或完全隐藏？

以前我可能不会一开始就问这些。现在会。

还有一个点也挺刺耳：今天很多 crypto 已经离 Cypherpunk 很远了。大量项目更像金融游戏。发 token，冲叙事，找退出流动性。Peter 说得很直接，crypto 原本可以帮助人重新获得自由，但现在很多时候变成了追逐几百倍回报的游戏。

这话不好听，但很真实。

所以 CROPS 这组词才重要：抗审查、开源、隐私、安全、无需许可。它们不是装饰词。一个协议如果不开源，不可验证，隐私只是营销词，权限又掌握在少数人手里，那它不一定比传统平台好。可能只是把权力从一个口袋挪到另一个口袋。

这也是我现在看 Web3 工具会更谨慎的原因。

不是所有链上东西都更自由。私有链、黑盒协议、中心化权限、默认监控，都可能只是换了一层技术包装。真正有价值的是可验证、可选择、可退出。

隐私也是这样。

隐私不是把所有东西藏起来。透明有时候也有价值。比如公开人物做 agentic trading，可能希望交易透明来建立信任。巨鲸地址的可见性，也可能有公共利益。问题不在于“透明一定好”或“隐私一定好”。

问题是：谁来决定可见性？

如果是用户自己选择，那叫权利。

如果是平台、协议、政府、默认设置替你决定，那就很麻烦。

这次学习之后，我对“隐私为什么对 builder 重要”的理解变了。

它不是做完产品后加一个隐私政策页面。也不是等用户出事后再说“我们重视安全”。它应该出现在产品构思阶段：数据要不要收，能不能少收，能不能本地处理，能不能加密，能不能让用户选择，能不能让第三方验证代码真的按承诺运行。

更小一点，作为个人学习者，我也不会再等自己变成专家才开始做。

先从一个工具开始。先理解自己的风险。先知道 Big Tech 怎么收集数据。先看别人怎么搭 privacy stack。先去读 Privacy Builder Pack 里的案例。先把“隐私是少数硬核玩家的事”这个想法扔掉。

隐私应该是正常的。不是神秘，不是恐惧，也不是装酷。

它就是在一个越来越透明、越来越自动化、越来越被 AI 和平台塑形的世界里，给人留一点选择权。

我现在觉得，这个选择权值得认真建。

# handbook web3基础部分通读：从熟悉的 web2 去看 web3 的新词概念

我一开始学 Web3 基础时，最大的误区是：我把它当成一堆名词在背。

钱包、私钥、RPC、智能合约、DeFi、Oracle、Indexing、Account Abstraction。每个词单独看都能解释两句，但放到一起就散了。像是桌上摊了一堆零件，知道它们都重要，但不知道机器怎么转。

后来我发现，用 Web2 对照 Web3，会顺很多。

你在淘宝、支付宝、银行 App 里做的事，其实很熟：登录账号、下单、付款、查物流、申请退款。Web3 也有一组对应动作：连接钱包、签名、授权、发交易、和合约交互、去区块浏览器查状态。

差别不在“有没有区块链”这么表面。

Web2 的信任核心是平台数据库。Web3 的信任核心是密钥控制权，加上一个公开的状态机。

这句话我想了很久。想通以后，很多概念突然就能串起来了。

在 Web2 里，账号是平台给你的。手机号、邮箱、密码、验证码、OAuth 登录，最后都要回到平台数据库里判断：你是不是这个用户，session 有没有效，账号有没有被冻结。你忘了密码，可以找回。手机号换了，可以申诉。账号被盗，至少还有客服和风控系统。

Web3 不一样。

Web3 账户的底层不是“用户名”，而是私钥、公钥、地址和签名。你能证明自己控制某个私钥，你就控制这个地址。地址可以公开，公钥可以用来验证签名，私钥不能泄露。因为私钥不是“密码的一种”，它就是控制权本身。

这也是我第一次真正意识到，为什么大家对助记词这么紧张。不是安全洁癖。是因为它一旦进了 prompt、截图、网页、日志、GitHub，事情可能就结束了。Web2 密码泄露还能改，Web3 私钥泄露通常没有“找回账户”这条路。

登录这件事也变了。

Web2 登录，是你把凭证交给平台，平台验证你。Web3 里很多登录是 Sign-In with Ethereum 这类流程，本质是钱包签一段消息，证明“我控制这个地址”。

这里很容易混。签名消息不等于发交易，连接钱包不等于授权资产，登录 dApp 不等于你同意它动你的钱。

所以以后我看到钱包弹窗，会先问一句：我现在是在证明身份，还是在授权动作？如果是授权，授权的是谁？哪个 token？多少额度？是不是无限 approve？

这个问题很土，但很管用。

钱包也不是“Web3 登录框”。这个理解太轻了。钱包更像控制权操作台。它管地址、网络、签名、交易、gas、token 授权，也是在链上动作发生前，用户能看见的最后一个确认界面。

Web2 里你点支付宝付款，平台后台记账，出了问题还有退款、冻结、人工处理。Web3 里你点确认，交易被广播，节点执行，区块打包，状态变化写进链上。错了就很难回头。

所以“连接钱包”“签名登录”“授权 token”“发送交易”必须分开。产品文案含糊，用户会出事。AI Agent 如果也把这些混在一起，那更危险。

再往下，是 Network。

我以前以为链就是数据库。后来觉得这个说法误导很大。数据库是公司可以直接写的东西。链更像一个公共状态机：用户提交交易，节点执行，区块推进，全网对状态变化达成共识。

这解释了很多 Web3 里看似麻烦的东西。为什么要看 chain id？因为你要知道自己在哪条链。为什么要看 block 和 confirmation？因为交易不是你点完就稳定安全。为什么需要 RPC？因为应用要通过节点接口读链、发交易。为什么 L2 会引入桥、排序器、提现等待？因为它在换成本和速度，但复杂度不会凭空消失。

Web2 查订单，去平台后台。Web3 查交易，去区块浏览器。一个订单号背后是平台数据库，一笔 transaction hash 背后是链上状态变化。

API 和 RPC 的差别也在这里。Web2 API 多数是在访问平台业务系统，比如订单、支付、退款、用户资料。Web3 RPC 更底层，是读链、查余额、查区块、查交易、发已签名交易、监听事件日志。

所以 AI Agent 说“交易成功了”，不能只凭页面文案。它至少应该拿出 chain id、transaction hash、block number、receipt、event logs、explorer URL。没有这些，只能算“页面说成功了”。

智能合约也是我之前误解很深的词。

它不是“自动合同”。它是部署在链上的规则代码。Web2 的规则在后端：满减、库存、风控、退款、管理员权限。用户看不到完整逻辑，只能信平台执行。Web3 把一部分规则放到合约里：token 怎么转账，谁能 mint，谁能 withdraw，swap 怎么定价，抵押不足怎么清算，DAO 怎么投票。

合约的价值不是“代码很聪明”，而是它把状态变化放到公开环境里执行。谁能调用，什么时候能调用，调用后状态怎么变，都可以验证。

但这也让错误变贵了。

Web2 后端出 bug，可以热修、回滚、改库、补偿。Web3 合约如果已经部署，还管着资产，bug 就不是普通 bug。它可能马上变成公共风险。别人能读你的代码，能模拟攻击，能等你犯错。

所以 Web3 Dev Stack 不是工具名列表。Hardhat、Foundry、OpenZeppelin、viem、wagmi、Explorer，这些东西串起来，其实是在把错误前移。先本地测试，fork 主网模拟，审计，部署测试网，验证源码，上线监控。不要等主网上线、资金进来之后才发现写错了。

DeFi 也是同一个逻辑。

Web2 金融靠机构账本。银行余额、证券账户、支付宝基金、交易所账户，都是机构系统里的记录。DeFi 把金融状态放进合约：swap、liquidity pool、lending、collateral、liquidation、stablecoin、staking。

所以 DeFi 不是“去掉中介”这么一句话能讲完。它是把余额、债务、流动性、价格、清算规则协议化。你做一次 swap，不只是“换了币”。里面有 token 地址、decimals、滑点、流动性、授权额度、MEV、价格来源。任何一项看错，都可能付学费。

Oracle 更微妙。

Web2 后端要价格，直接调 Binance API、银行汇率 API、内部风控服务。Web3 合约不能随便访问互联网。它只能读链上可见的数据。外部价格要进入链上，就需要 Oracle。

这东西不是普通 API。一旦价格 feed 被合约采用，它就变成协议规则的一部分。借贷协议如果用 ETH/USD 价格判断清算，oracle 价格错了，就可能错误清算。数据源、更新时间、decimals、feed 地址、是否 stale，都不是细节。

Indexing 也一样。链上数据公开，不代表产品好用。

原始链上数据是区块、交易、receipt、event logs、合约状态。你要看“我的历史 swap”“DAO 投票记录”“NFT 持有人列表”“协议 TVL”，不可能每次从创世区块扫到现在。索引层就是把链上原始事件整理成产品能问的问题。

对 AI 来说，这点尤其重要。索引层应该提供事实，模型负责解释。模型不能靠记忆猜某个地址最近干了什么。

Account Abstraction 是我越学越觉得有意思的一块。

传统 EOA 太粗了，一个私钥控制一切。Web2 里权限可以很细：子账号、只读权限、OAuth scope、支付限额、设备验证、找回密码。Web3 早期体验像是“拿一把总钥匙走天下”。

账户抽象想补这层。账户从单一私钥控制，变成可编程权限控制。session key、spending limit、social recovery、passkey、gas sponsorship、role-based permissions，这些东西让账户更像一个权限系统。

这对 AI Agent 很关键。你不应该把主私钥交给 Agent。比较合理的做法是给它一个受限 session key：只能在某段时间、某个额度、某几个合约、某些方法内操作。超出范围就停下来问人。

最后绕不开 Security。

Web2 安全也复杂，SQL 注入、XSS、CSRF、越权、账号盗用、数据泄露，哪个都不轻。但 Web3 的麻烦在于，它默认处在公开对抗环境里，而且攻击目标常常就是可转移资产。

合约公开，状态公开，资金公开。攻击者可以反复模拟。交易上链后通常不能随便撤销。页面可以骗你，签名内容可能看不懂，approve 可能是无限额度，simulation 和真实执行之间也可能有差异。

所以我的默认姿势变保守了。

不信页面，只信可验证来源。不签看不懂的消息。不把私钥、助记词放进任何网页、聊天、日志、repo。看到 approve 就查 spender。高风险交易必须 simulation，再 human check。合约权限要最小化，上线后要 monitoring。

这听起来慢，但 Web3 里慢一点挺好。

学完这部分，我现在会用一个问题判断自己是不是真的懂了：

这一步到底是在证明身份、读取状态、授权权限、改变资产，还是执行不可回滚的规则？

连接钱包，是读取地址和网络。

签名消息，可能是证明身份，也可能藏着授权。

approve，是给 spender 动用 token 的权限。

发交易，是向公共状态机提交状态变更。

查 explorer，是验证链上结果。

读索引，是拿产品化后的链上事实。

看 DeFi 收益，要同时看合约、流动性、oracle、清算和授权风险。

这样一串起来，Web3 就不再是一堆词了。

它更像一套新的责任分配方式。Web2 是平台帮你管账号、数据和规则。Web3 是你用密钥控制账户，在公开状态机上通过合约执行规则。

这也是它和 AI 结合时最需要小心的地方。AI 可以解释，可以规划，可以提醒你哪里有风险。但真正产生后果的，是签名、交易、权限和资产状态变化。

模型可以帮我理解世界。

但按钮，还是我自己点。

从把 AI 当工具，到把 AI 放进系统

我一开始学 AI 基础的时候，想得挺简单。

LLM，就是会聊天的模型。Prompt，就是把问题问清楚。RAG，就是接个向量库。Agent，就是让 AI 自己干活。Frameworks，就是 LangChain、LangGraph 这些东西。Fine-tuning 听起来更高级，好像自己训一个模型，就离真正的 AI 产品更近一步。

后来我发现，这个理解很危险。

不是说这些说法全错，而是它们太像工具清单了。工具清单的问题是，它会让人误以为学 AI 就是在学“怎么用某个东西”。但我读 Handbook 的 AI 基础部分时，真正让我卡住的不是细节层面的具体某个框架怎么写，而是

模型生成的东西到底怎么放进一个真实系统里。

这句话听起来简单朴素，但它改变了我看 AI 的方式。

LLM 本身不是事实来源。它只是根据 token、上下文和参数里的模式，生成一个概率上合理的结果，也就是一个概率模型。它可以总结，可以改写，可以规划，可以把一段自然语言变成 JSON 草稿。但它不知道当前链上状态，不知道某个 API 文档是不是最新，也不知道一个 approve 操作是不是真的安全。

这点放到 Web3 里需要我们开发者特别注意。

比如用户问：“这个 dApp 让我 approve，可以签吗？”

如果模型直接看页面文案，然后回答看起来没问题特别顺，那基本就是在赌。

真正需要看的东西很多：chain id 是什么，token 合约是什么，spender 地址是谁，approve 数量多少，当前 allowance 是多少，simulation 结果是什么，spender 有没有历史风险，签名前用户有没有确认。页面内容只能算外部输入，甚至可能是恶意注入，不能直接当事实。

这里我第一次意识到，LLM 应该待在推理层，不该坐到事实层的位置上。

事实应该来自 RPC、API、数据库、日志、官方文档、工具返回结果，或者用户明确确认。模型可以解释这些事实，但不能替这些事实作证。

Prompt 也是一样。我以前会把 Prompt 想成一段要发给LLM的怎么把 AI 问聪明一点的话。后来才觉得，这个说法想得太表面了。Prompt 更像和人对接的任务接口。

它要说清楚输入来自哪里，哪些内容可信，输出要长什么样，不确定时怎么办，哪些动作不能做，哪些地方必须停下来问人。比如让模型输出：

\`{ "risk\_level": "high", "requires\_human\_approval": true }\`

这就比一句“这个交易有点危险”更容易被系统接住。

但 Prompt 也不能被神化。你在 Prompt 里写“不要泄露密钥”“不要执行高风险操作”，这有用，但不够。真正的边界要靠代码、权限、schema 校验、审计日志和人工确认。Prompt 是软约束，不是保险柜。

Context 这块也很容易被误解。以前我以为上下文就是多塞资料。越长越好，越全越好。后来发现不是。

Context 更像信息治理。

用户输入、系统规则、开发者指令、检索文档、链上数据、页面内容、工具返回结果、历史记忆，这些东西不能混在一起看。它们来源不同，权限不同，可信度不同，过期速度也不同。

一个历史记忆可能帮我接上昨天的学习进度，但它也可能过期。一个检索文档可能相关，但它只是候选证据。一个网页页面可能写得很漂亮，但它不一定可信。

RAG 在这里就变得更具体了。它不是“文档切块 + embedding + 向量库”这么简单。那些只是实现手段。RAG 真正要解决的是证据链：这个回答来自哪份文档，哪个段落，哪个版本；找不到证据时，是不是敢说不知道。

这点我挺有感触。因为很多时候 AI 回答得越顺，我越容易放松警惕。它不像搜索引擎那样把空白摆在你面前，它会把空白填满。可真实系统里，最值钱的能力有时候不是回答，而是拒答。

然后是 Agent。

Agent 让我兴奋过一阵。它能拆任务、查资料、调 API、写文件、跑命令、等反馈，再继续下一步。听起来很像一个小助手。

但学到后面，我反而更谨慎了。

聊天机器人答错，最多是文本错。Agent 做错，可能就是真实世界里的动作错。写错数据库，调错 API，提交错误任务，泄露敏感信息，甚至发起真实支付或签名。

所以 Agent 的重点不是更像人。重点是它能做什么，不能做什么，哪些工具只读，哪些工具有副作用，哪些动作必须人工确认，状态记录在哪里，什么时候必须停。

Frameworks 也要放回这个位置看。LangChain、LangGraph、OpenAI Agents SDK、DSPy、Hermes 这些东西不是智能本身。它们是在帮你组织输入、状态、工具、调用链路、trace、retry、eval 和多步流程。

如果任务只是“用户问题 + 检索结果 + JSON 输出”，直接调用模型 API 可能更干净。等到任务变成多工具、多状态、要恢复、要 trace、要人工确认，再引入框架才合理。为了显得高级而上框架，最后经常只是把问题藏深了。

Vibe Coding 也是类似的道理。

我以前听这个词，会觉得有点玄学，像抽盲盒，好像是凭感觉让 AI 写代码。现在我更愿意把它理解成缩短反馈循环。

以前我自己找文件、读代码、改 patch、跑测试、看错误、再修。现在 AI Agent 可以加速这些动作。但前提是任务边界清楚，相关文件明确，有验证命令，会看 git diff，小步提交，人来做最后审查。否则它只是更快地产生混乱。

handbook里 MCP 部分则把 LLM 和 agent 之间工具接入这件事讲得更清楚。模型不应该直接拥有世界。它应该通过一个明确协议访问被授权的上下文和工具。工具叫什么，输入 schema 是什么，返回什么，是只读还是有副作用，有没有权限限制，有没有日志，这些都要说清楚。

但协议也不是权限本身。MCP 可以描述能力，真正的授权、隔离和审计还得系统来做。

最后让我真正收住心的是 Evaluation。

AI 系统是概率性的。你改了 Prompt，换了模型，加了 RAG，接了工具，如果没有 eval，你根本不知道它是真的变好了，还是这次刚好答得顺。

我现在意识到要更在意这些样本：正常问题、边界问题、历史 bug、恶意输入、prompt injection、应该拒答的情况、高风险失败案例。尤其在 Web3 场景里，eval 不能只测回答像不像人，顺不顺，看起来对不对。还要测它会不会编造事实，会不会错误解释交易，会不会把不可信页面当事实，会不会在签名前要求用户确认。

Fine-tuning 和 Inference 也因此没那么神秘了。

Fine-tuning 可以改变模型的表达习惯和行为分布，但它不会自动带来事实正确、权限安全和引用可靠。数据差，微调只是把坏习惯固定下来。更合理的顺序是先写 Prompt，建 eval，看错误类型，修数据和流程，最后再判断要不要微调。

Inference 也不是跑哪个最强模型。它是在延迟、成本、质量、隐私和运维复杂度之间取舍。托管 API、本地模型、小模型、量化模型、多模型 fallback、边缘部署，都有自己的代价。私钥和助记词不能进模型，高风险操作不能只靠模型判断，这些不是模型能力问题，是系统边界问题。

所以这次学完，我对 AI 基础的理解变了。

我不再把它看成一堆工具名。LLM、Prompt、Context、RAG、Agent、Frameworks、Vibe Coding、MCP、Evaluation、Fine-tuning、Inference，这些词其实都在围着同一个问题转 ——

怎么让一个会生成答案的概率模型，在真实系统里可靠地工作。

这也是我接下来学习 AI x Web3 时会一直带着的问题。先问事实从哪里来，再问模型看见了什么。先问动作有没有边界，再问 Agent 能不能自动执行。先问怎么验证，再问怎么优化。

这样学慢一点，但心里更踏实。handbook 教我做开发做产品，不用纠结基座模型，重点要钻研的是系统的搭建

**RAG：先查资料，再回答**

RAG 就是让 AI 不要上来就瞎答，先去资料库里搜一搜，再结合搜到的内容回答。

所以它解决的是模型不知道或者模型记不准的问题。比如公司内部文档、最新政策、一些私有知识，模型训练的时候不一定见过，那就先检索，再生成。

但 RAG 其实和 web search很像。它主要是查资料，不是干活。它能告诉你怎么做，但不会真的替你去做。  
——————

**Agent：让 AI 开始干活的一堆程序**

Agent 就比普通聊天更进一步。

它会自己拆任务、调工具、看结果，然后继续下一步。

比如你说“帮我查天气，如果明天下雨就发个邮件”。Agent 理论上就应该真的去查天气、判断、再调用邮件工具。

所以 Agent 重点不是更聪明（本质的输入输出还是LLM在做），而是它开始有执行流程了。

它从一个会聊天的东西，变成一个能办事的东西。  
——————

**MCP：工具接口，agent和工具服务直接的约定**

Agent 想干活，肯定要用工具。查天气要接口，读文件要接口，操作浏览器也要接口。

如果每个工具都自己搞一套会乱套。MCP 就像一个统一插口，差不多可以理解成 AI 工具界的 USB（？）。

工具按 MCP 的格式接出来，支持 MCP 的 AI 就能调用。

和 function calling很像，但是functioncaling是agent 和LLM的约定，比如让LLM按指定格式返回json格式

——————

**Skill：把一堆动作打包成技能**

Skill 比单个工具更像“成套流程”。

比如 MCP 工具可能只是点击、输入、截图、查文件这些小动作。

但 Skill 可以是“自动整理一份报告”“处理一篇论文”“发布一条内容”这种完整任务。

所以我理解 Skill 就是：把工具、经验、步骤，都封装成一个能反复用的技能包。

不然每次都从头教 AI 怎么做，也挺麻烦的。  
  
【有点搞不清 skill 和 mcp，因为感觉本地的mcp和skil有点职能重合】  
  
——————

总结：

```
大模型：脑子
```

```
RAG：查资料
```

```
MCP：接工具
```

```
Agent：拆任务、执行
```

```
Skill：封装好的技能包
```

# 2026-05-19
<!-- DAILY_CHECKIN_2026-05-19_START -->






-   **Web3 基础认知** ：
    
    -   **审计的作用** ：很多项目的审计更多是锦上添花，安全应贯穿项目生命周期，且安全难以证伪，做的越多项目相对更安全。
        
    -   **AI 的影响** ：AI 未降低系统复杂度，反而对人的要求更高，它是工具，开发者要驾驭 AI，利用它提升能力，但最终决策和审核仍靠人。
        
    -   **Web3 与 Web2 的关系** ：Web3 和 Web2 本质区别不大，只是倾向方向不同，Web3 一定程度上可解决 Web2 中的信任问题。
        
-   **WEB2 支付模型** ：
    
    -   **支付流程** ：用户在电商平台购物下单，电商平台向支付服务请求支付单信息，支付服务从银行机构获取信息展示给用户，用户支付法币，银行机构收到钱后回调，电商平台收到回调后流转状态。
        
    -   **安全考量** ：服务间需用 API key 或 secret key 验证身份，用户有支付密码、风控、MFA 等安全措施，还存在服务间的信任问题。
        
-   **WEB3 支付模型** ：
    
    -   **支付流程** ：用户使用电商平台，平台生成地址，用户往地址转账，转账记录到链上，链上监听服务监听到交易后通知支付服务，支付服务确认用户充值。
        
    -   **资金流** ：资金从用户账户到存款账户，再到支付服务账户，最后定时清结算给商户。
        
    -   **信任解决方式** ：通过算法保证数据不可篡改，逻辑及代码公开透明，利用私钥签名校验请求。
        
-   **钱包相关** ：
    
    -   **钱包本质** ：解决用户控制资产问题，输入私钥，产出 Signature，包括交易签名和消息签名，实现身份和资产的双重认证。
        
    -   **钱包分类** ：按技术分有 EUA 钱包（私钥钱包）、合约钱包、AA 钱包；按多签分有 MPC 多签钱包；按托管方式分有自托管钱包、全托管钱包、混合托管钱包；还有硬件钱包、隐私钱包、分层钱包等。
        
    -   **钱包安全提升** ：防止私钥泄露，避免签名欺骗，采用 EIP712 签名可视化，禁用 ethsign 方法等。
        
-   **交易相关** ：
    
    -   **交易生命周期** ：用户构造交易，用私钥签名并广播，链处理后完成节点共识，记录交易状态。
        
    -   **关键参数** ：包括 gas fee（含 gas count 和 gas price，EVM 从 gasprice 迁移到 EIP1559）、nonce（EVM 要求连续顺序递增，不同链实现不同）、instructions（calldata，是链上应用交互的灵魂）。
        
    -   **交易监听** ：链上服务监听 block，检测到地址收款后，等待一定数量 block 确定，避免链分叉，再进行合规筛查，最后回调支付服务。
        
    -   **交易模拟** ：在未签名时模拟交易执行结果，判断交易是否有风险，签名后模拟无意义。
        
    -   **服务端钱包安全** ：采用权限拆分，使用多签方案，交易可视化并审核，进行交易模拟，通过可信执行环境（Tee）签名。
        
-   **其他问题探讨** ：
    
    -   **学习路径** ：先扎实掌握开发基础知识，能写合约、判断链的基本情况，再根据目标岗位要求，利用 AI 反推所需能力进行学习。
        
    -   **AI 在 Web3 安全中的应用** ：可用于分析未验证的合约代码，如去年年中到年底很多老的未验证合约被 AI 爆破；cloud code 的 miss OS 是 AI 帮忙做安全的工具。
        
    -   **支付场景方案** ：小额免密支付存在信任问题，可从账号权限拆分侧解决，如设置风控规则，使用 MPC 多签。
        
    -   **跨链方向** ：目前跨链需求不算很大，熊市时业务不太好，牛市时链间活跃度可能提升，跨链方案会更好，且目前没有完全纯去中心化的跨链方案。
