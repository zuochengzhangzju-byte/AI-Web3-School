今天主要学习了模型上下文协议，也就是 MCP。通过今天的学习，我对 MCP 的理解不再停留在“给模型接工具”这个层面，而是更清楚地意识到，它真正解决的是模型、外部工具、数据源和应用上下文之间如何标准化连接的问题。

在实际的 AI 应用中，模型不可能只依靠 prompt 中的文本完成复杂任务。它往往需要读取文件、访问数据库、调用 API、使用外部工具，甚至结合用户当前的应用上下文来完成工作。如果每一个应用都用自己的方式连接工具，那么整个 Agent 生态就会变得非常碎片化：同一个工具可能需要为不同客户端重复开发适配层，权限控制、参数格式、错误处理和日志记录也很难统一。

MCP 的价值就在于把这层连接抽象成一种协议。它类似于 AI 应用中的“工具接口标准”，让客户端、模型和工具服务端之间形成更清晰的分工。Client 负责和模型交互，发现 server 暴露的能力，并把模型的调用请求转化为协议消息；Server 则负责提供 resources、tools、prompts 等能力，让模型能够在被授权的范围内读取信息或执行动作。

今天让我印象最深的是 MCP 的一个核心原则：模型不应该直接拥有世界，而应该通过明确协议访问被授权的上下文和工具。也就是说，真正重要的不是让模型“能接更多工具”，而是让工具接入变得可描述、可发现、可限制。一个工具应该有清晰的 schema，说明它叫什么、什么时候使用、需要哪些参数、会返回什么结果，以及是否会产生外部副作用。如果没有 schema，模型调用工具就容易变成自然语言式的猜测，参数填写和执行结果都会变得不可控。

同时，我也意识到权限设计是 MCP 真正进入产品时非常关键的一环。读取文档、查询 issue、创建 PR、删除文件、发起支付，这些动作的风险完全不同。MCP 可以标准化工具入口，但不能替代真正的授权、审计和隔离机制。尤其是在 AI x Web3 场景中，MCP 可以作为 Agent 连接链上工具、合约文档、RPC、区块浏览器和开发工具的接口层，但它本身并不是钱包安全方案。最终的账户权限、签名确认、交易模拟、session key 和审计日志，仍然需要由 Web3 账户系统和安全机制来完成。

今天还参加了 colearning。通过交流，我进一步明确了接下来黑客松的准备方向：我想围绕 AI Agent 展开。相比单纯做一个聊天应用，AI Agent 更强调模型如何结合工具、状态、权限和任务目标来完成真实工作流，而 MCP 正好提供了一个理解 Agent 工具调用体系的切入点。

今天的学习主要围绕 AI Agent 工作流、安全威胁建模、DAO 治理协作流程，以及隐私文化对于 Builder 的意义 展开。整体来看，今天的核心收获是：在设计 AI + Web3 产品时，不能只关注 Agent 能不能自动完成任务，更要提前定义它能访问什么、能执行什么、什么时候必须停下来让人确认，以及哪些行为必须进入治理或链上记录。

## 一、Agent Workflow Threat Model 与确认策略

今天首先围绕 Adaptive DAO Communication Strategy Agent 做了 Agent Workflow Threat Model 的梳理。相比单纯描述 Agent 的功能，这部分更像是在回答一个问题：当 Agent 被放进真实 DAO 协作环境中，它可能会误用哪些权限、泄露哪些数据、被怎样攻击，以及哪些关键节点必须由人类确认。

我从四个方面进行了整理：

第一是 资产清单。需要明确 Agent 在运行过程中会接触哪些资产，例如 DAO 成员画像、治理提案内容、会议纪要、投票数据、沟通渠道、钱包权限、外部 API、策略建议结果等。不同资产的敏感程度不同，因此不能把所有数据都默认交给 Agent 自由处理。

第二是 权限边界。Agent 可以自动完成信息整理、任务拆解、提醒生成、草案撰写等低风险工作，但涉及发出正式治理建议、代表用户行动、调用钱包、发布公开消息、触发链上交易等行为时，必须有清晰边界。尤其是与钱包、投票、资金、身份声誉相关的操作，不能让 Agent 默认拥有无限权限。

第三是 数据流威胁。Agent 在“输入—处理—调用工具—输出”的过程中，可能出现 prompt injection、敏感信息泄露、错误上下文引用、外部工具返回污染、成员身份画像被滥用等风险。因此，不能只在最终输出时检查结果，而应该在数据进入、工具调用、跨系统传递和最终发布几个环节都设置防护。

第四是 模拟攻击场景。我梳理了四类典型风险：

1.  恶意输入诱导 Agent 忽略原有规则；
    
2.  外部数据源污染导致 Agent 生成错误治理建议；
    
3.  Agent 误把私人讨论内容发布到公开治理渠道；
    
4.  钱包或治理权限被滥用，导致未经授权的投票、签名或链上操作。
    

基于这些威胁，人工确认策略就变得非常重要。我的理解是，Human-in-the-loop 不应该只是“最后看一眼”，而是要嵌入到关键节点中。例如：公开发布前确认、调用钱包前确认、涉及成员隐私时确认、策略影响治理方向时确认、超过额度或权限范围时确认。这样 Agent 才能既提高效率，又不会替代人的责任判断。

## 二、DAO Meeting-to-Action Pipeline 治理协作流程草图

今天还整理了一个 DAO Meeting-to-Action Pipeline，把 DAO 会议从讨论到行动的过程拆成 11 个步骤，并用四种边界标记不同责任：

-   🔵 AI 自动
    
-   🟡 人工确认
    
-   🔴 治理投票
    
-   🟢 链上记录
    

这个流程让我更清楚地看到，DAO 协作不是简单的“开会—总结—执行”，而是一个多阶段的治理流水线。AI 可以在其中承担大量辅助工作，比如会议转录、摘要生成、任务提取、责任人匹配、行动项提醒、历史提案关联、风险提示等。但并不是所有步骤都适合自动化。

比如，会议内容整理、行动项初步拆解可以由 AI 自动完成；但是否把某个讨论升级为正式提案，需要人工确认；涉及资源分配、预算、规则变更时，需要进入治理投票；而投票结果、资金流转、关键承诺等，则适合链上记录。

这让我意识到，设计 DAO Agent 的关键不是“让 AI 做完所有事”，而是 明确每个阶段的责任归属。AI 负责提高信息处理效率，人类负责判断和确认，治理机制负责集体决策，链上记录负责保证可追溯性。

## 三、GovReach 项目初步 Proposal

今天也继续推进了项目方向深挖，初步形成了 GovReach 的 Proposal。这个项目的想法是：基于 Snapshot 等链上/治理数据，构建一个帮助 DAO 优化治理参与率的 Agent。

GovReach 关注的问题是，很多 DAO 并不是没有治理机制，而是成员参与率低、提案触达弱、投票动机不足、不同类型成员的信息需求不一样。Agent 可以通过分析历史投票数据、成员参与行为、提案类型和社区沟通节奏，帮助 DAO 发现哪些成员容易流失、哪些提案参与率较低、什么时间发布更有效、哪些提醒方式更适合不同成员。

不过，今天的隐私学习也提醒我，这类项目必须特别注意边界。治理参与率优化不能变成对成员的过度画像或操控。GovReach 应该优先采用聚合数据、匿名化分析和可解释建议，而不是直接给成员贴标签或做过度个性化干预。比较理想的方向是：让 DAO 更好地理解治理参与状况，同时让成员保有选择权和隐私空间。

## 四、Handbook 学习收获

今天早上的 Handbook 学习主要包括三个部分：Agent Workflow、Agent Wallet 和几个新概念。

在 Agent Workflow 部分，我学习到 Agent 的运行可以理解为一个五步循环：任务分解、工具选择、执行、观察、纠错。这个框架帮助我把 Agent 从“一个会回答问题的 AI”理解为“一个可以持续执行任务的系统”。它不是一次性输出答案，而是在目标、工具、反馈和修正之间不断循环。

在 Agent Wallet 部分，我重点学习了最小权限、时间锁、额度上限和 session key。这里最大的收获是：Agent 拥有钱包能力之后，风险会明显上升，所以必须通过权限设计限制它的行动范围。比如，Agent 不应该长期持有高权限私钥，而应该使用临时 session key；不应该拥有无限额度，而应该有额度上限；不应该随时执行高风险操作，而应该通过时间锁和人工确认降低风险。

今天也理解了几个重要概念：

Workflow 是 Agent 执行任务的结构化流程，不只是简单对话。  
Human-in-the-loop 是指在关键决策或高风险操作中保留人工确认。  
Session Key 是一种临时授权机制，可以让 Agent 在有限时间和有限权限内执行任务，而不是直接暴露主钱包权限。

这些概念和上午的 Threat Model 结合起来之后，我更清楚地理解了 Agent 安全设计的基本原则：不是完全禁止 Agent 行动，而是让它在被限制、可观察、可撤销、可确认的环境中行动。

## 五、晚间会议学习：Co-learning 与 Neo-Cypherpunk

晚上参加了两场会议，其中最有启发的是 Neo-Cypherpunk & the Cultural Layers of Privacy: Why Privacy Matters for Builders。

这场分享让我重新理解了隐私。隐私不是少数技术专家才关心的极端议题，也不是只有在遇到风险之后才需要补救的功能。对于 AI 和 Web3 Builder 来说，隐私应该是产品设计早期就要考虑的基础能力。

Peter 在分享中提到，过去的 Cypherpunk 更强调技术专家、加密工具和个人防御，而 Neo-Cypherpunk 更强调开放、协作、社区、关怀和可访问性。也就是说，隐私不应该只是“全副武装的技术精英”才能做到的事情，而应该变成普通用户也能逐步采用的实践。

我印象最深的一点是：不必追求一开始就做到 100% 隐私。很多时候，用户可以先从一个工具、一个习惯、一个选择开始，例如使用更隐私友好的浏览器、通信工具或钱包策略。隐私不是非黑即白，而是一条从低到高的连续光谱。

这对我今天的 Agent 项目设计很有启发。比如在 GovReach 中，不应该默认把所有治理数据都用于精细化画像，而应该给 DAO 和成员提供不同层级的隐私选项。某些数据可以公开聚合展示，某些分析只能在本地或匿名环境中完成，某些建议则必须经过人工确认后才能进入治理流程。

## 六、今日总结

今天最大的收获是：AI Agent 的价值不只是自动化，更在于能否被安全地嵌入真实协作流程中。

一个好的 Agent 系统需要同时考虑四件事：  
一是任务流程是否清晰；  
二是权限边界是否明确；  
三是隐私和安全风险是否被提前建模；  
四是人类、治理机制和链上记录分别在什么时候介入。

对于 AI + Web3 Builder 来说，未来的产品设计不能只追求“更快、更自动、更智能”，还要追求“更可控、更透明、更尊重隐私、更适合社区协作”。这也是今天从 Agent Workflow、Agent Wallet、DAO Pipeline 到 Neo-Cypherpunk 分享中贯穿起来的一条主线。

今天主要围绕 Agent Wallet 的安全执行机制 和 Agent Profile 与能力声明设计 做了学习与梳理。整体感受是：Agent 未来真正有价值的地方，不只是“能自动完成任务”，而是能够在清晰边界内，被用户信任地执行任务。

## 1\. Agent Wallet：核心不是自动转账，而是受控执行

今天会议中最重要的启发是：Agent 操作链上资产时，问题不只是“AI 能不能帮用户转账”，而是 AI 是否能在用户明确授权的范围内安全地动用资金。

如果只是给 Agent 一个钱包，让它自己执行任务，风险会非常大。因为自然语言 prompt 只能表达意图，但不能形成强制约束。比如用户说“最多花 100 美金”，Agent 仍然可能因为幻觉、错误理解或执行路径异常，实际发起超过预算的交易。链上资金一旦转出，通常不可撤回，所以必须在执行层加入硬性限制。

会议里提到的 Pact 思路很有启发：用户不是一次性把钱包完全交给 Agent，而是通过一份授权协议，规定 Agent 能做什么、不能做什么、最多花多少钱、可以和哪些地址或合约交互、什么时候自动停止。这样 Agent 的行为就不再只是依赖模型“听话”，而是被系统规则约束。

我理解这对 Agent 产品非常重要：  
未来 Agent 的可信度，不只来自模型能力，而来自 权限边界、审计能力、撤销机制和异常拦截能力。

## 2\. 对黑客松方向的启发

结合会议内容，我觉得黑客松可以选择一个更小、更清晰、更容易落地的 Agent 支付场景，而不是一上来做很复杂的 DeFi Agent 或全自动交易 Agent。

一个比较合适的方向是：Agent 小额自动充值 / 自动续费工具。

比如 Agent 在使用 OpenAI、Claude、Zapier、Notion、数据库 API 或其他 SaaS 服务时，可能会遇到额度不足、token 不够、订阅即将到期等问题。这个时候，可以让 Agent 在用户预先授权的范围内自动完成小额支付。

这个场景的优势是：

用户需求明确：余额不足就充值。  
风险相对可控：金额小、对象固定、频次有限。  
产品逻辑清楚：预设规则，触发条件，自动支付，超限提醒。  
适合黑客松展示：可以做出完整 demo，包括授权、执行、风控、通知。

关键不是展示“Agent 会付款”，而是展示“Agent 只能在被允许的范围内付款”。例如用户可以设置：只允许给指定服务商充值；单次不超过 50 或 100 美金；每天最多充值一次；总预算不超过 1000 美金；超过预算必须人工确认。

这个方向本质上是在做一个 Agent Auto Top-up with Spending Guardrails，也就是带有资金护栏的 Agent 自动充值工具。

## 3\. Agent Profile 与能力声明草图

今天还做了 Agent Profile 与能力声明草图，案例是 Zapier Sales Lead Qualification Agent。

这个部分让我意识到，Agent 不能只是一个“黑盒自动化工具”，而应该像一个可以被识别、被理解、被授权的数字角色。Agent Profile 的作用，就是让用户、系统或其他 Agent 在交互前就知道：这个 Agent 是谁、能做什么、不能做什么、需要什么权限、输出什么结果、风险在哪里。

以 Zapier Sales Lead Qualification Agent 为例，它的核心任务是帮助销售团队自动判断线索质量。它可能会读取 CRM、表单、邮件、公司官网信息，然后根据用户设定的标准判断这个 lead 是否值得跟进。

这个 Agent 的能力声明可以包括：

它可以读取 lead 信息、公司名称、职位、邮箱、行业、预算线索等数据；  
它可以根据预设规则给 lead 打分；  
它可以把线索分成 high priority、medium priority、low priority；  
它可以把结果写回 CRM，或者触发 Zapier workflow；  
但它不能擅自发送销售邮件，不能删除 CRM 数据，也不能修改关键客户信息，除非用户额外授权。

这个设计和今天会议里的 Agent Wallet 很像：都强调 Agent 必须有明确边界。区别是，Agent Wallet 管的是资金权限，而 Agent Profile 管的是身份、能力和任务权限。

## 4\. 今日最大的收获

今天最大的收获是：  
Agent 的未来不是简单地让 AI 代替人操作，而是要让 AI 在可声明、可授权、可审计、可撤销的框架内操作。

无论是 Agent Wallet，还是 Zapier Sales Lead Qualification Agent，本质上都需要回答几个问题：

这个 Agent 是谁？  
它能做什么？  
它不能做什么？  
它需要哪些权限？  
它执行任务时如何被监督？  
如果出错，用户如何撤销或接管？

今天参加了 AI and Web3 School 的 Bonus Session，主题是 Open Agentic Economy。本次分享主要围绕 AI agents 与 Ethereum 的关系展开，帮助我理解了当 AI 不再只是聊天工具，而是逐渐成为经济参与者时，为什么需要开放、中立、可验证的基础设施。

一、本次分享的主要收获

Sophia 提到，过去的大语言模型更多像是一个可以回答问题的 chatbot，但现在 AI agent 已经可以写代码、调用工具、做决策、执行支付，甚至和其他 agent 协作。这意味着 AI 正在从工具变成一种新的经济参与者。

当 AI agent 开始参与经济活动时，就会出现很多问题：它们如何付款？如何证明身份？如何建立声誉？如何被限制权限？传统互联网基础设施主要是为人类设计的，并不完全适合机器之间的大规模协作。

因此，Ethereum 的作用就变得重要。它可以提供一个开放、中立、可编程的协调层，让 agent 的身份、支付、规则和行为可以被验证，而不是完全依赖某个中心化平台。

二、关于 Ethereum 和 CROPS 的理解

分享中 Sophia 提到，Ethereum for AI actually means Ethereum for humans。我理解这句话的意思是，Ethereum 不是单纯为 AI 服务，而是在 AI agent 越来越强大的时代，帮助人类继续保留控制权、所有权和选择权。

她还介绍了 Ethereum 的 CROPS 原则，包括抗审查、开源自由、隐私和安全。抗审查意味着没有单一主体可以随意阻止有效交易；开源自由意味着开发者可以查看、复制和继续构建；隐私保护可以帮助用户在不暴露底层数据的情况下完成验证；安全则保证系统和智能合约按照规则运行。

这些原则共同构成了 AI agent 所需要的基础设施，例如身份、声誉、支付和可执行协议。

三、关于 ERC-8004 / ERC-8183 和信任机制的理解

本次分享中，我对 ERC-8004 印象比较深。它主要关注 AI agent 的身份和声誉问题，也就是让用户或其他 agent 能够判断一个 agent 是否可信。

在 Q&A 中，有人问 reputation 代表 agent 本身还是 agent owner 的声誉。Sophia 回答说，它更偏向 agent 本身，也就是某个具体数字软件实体的声誉，而不是背后开发者个人的声誉。

这让我意识到，未来如果用户要使用 AI agent，不能只看它是否“智能”，还要看它是否有可验证的身份、历史记录和安全边界。

四、关于支付场景的收获

Sophia 提到，payment 可能是 AI agent 最早落地的重要场景之一。她用买披萨的例子解释了 programmable money 的意义：如果直接把信用卡交给 agent，风险很高；但如果通过智能合约限制预算和支付对象，就可以让 agent 在可控范围内完成任务。

这个例子让我更直观地理解了 Web3 支付和 AI agent 的结合。它不是简单地让 AI 去花钱，而是通过规则控制 AI 能花多少钱、能向谁付款、能完成什么任务。

五、我的思考

今天最大的收获是，我意识到 AI agent 和 Ethereum 的结合，不只是技术上的“AI 上链”，而是在解决一个更重要的问题：当 AI 开始代表人类执行任务和参与经济活动时，人类如何继续保持控制权？

从市场营销角度看，我也觉得未来这类项目不能只强调智能合约、链上声誉、去中心化这些技术词，而要把它们转化成普通用户能理解的价值。比如告诉用户：你可以限制 AI 的预算，可以查看 agent 的历史记录，也可以通过规则降低风险。

接下来，我会继续了解 ERC-8004 / ERC-8183 中关于 agent identity、reputation 和 verification 的内容。同时，我也会关注 AI agent 在支付、钱包和任务执行中的真实应用案例，思考这些技术如何被普通用户理解和接受。

总的来说，今天这场分享让我对 AI agent 和 Ethereum 的关系有了更清晰的认识，也让我看到未来 AI/Web3 项目在技术、信任和用户教育方面都有很大的探索空间。

上午推了两个 WCB 任务。一个是测试网交易，在 Sepolia 用 MetaMask 发了笔 0.00001 ETH 到销毁地址，Gas 21000，烧了大概 0.000031 SepoliaETH——第一次盯着 Etherscan 看 Gas 怎么算，感觉终于不是走过场了。另一个是做了个叫「签还是不签」的 Web 工具，模拟五个合约授权场景，接 DeepSeek API 实时分析风险。做这个是因为我发现很多新手（包括之前的我）看到钱包弹窗就直接点 confirm，根本不看提示，想训练一下"签名前停三秒"的意识。累计 17/37，305pt。

今天的 Prompt 学习最有收获的一句话是：Prompt 是软约束不是安全边界。学 Prompt Injection 的时候才真正理解这件事。一句"忽略之前所有规则"就能让模型切换模式，system prompt 根本挡不住，真正的防护得在应用层做。Few-shot 我之前理解也是错的，以为是在教模型新知识，其实只是给格式锚点。下午实时参加了 Hermes Agent 入门分享，工具调用、记忆、任务分解这几块串起来了，Agent 不再是魔法了。

项目拆解读了 Bittensor 和 Render Network。Bittensor 我花时间最多，直接进官方文档看子网、emission、Yuma Consensus 那几页。最有意思的发现是 2025 年 11 月他们把 emission 分配从价格驱动改成了流量驱动（Taoflow），原因是有人发现可以人为拉 token 价格来套奖励。他们自己承认了这个漏洞然后修掉了，这个让我觉得这个项目至少是认真在跑的。但 weight copying 问题（验证者互相抄评分）我还没想明白 Yuma Consensus 3 有没有真的解决。Render Network 相对好理解，本质就是闲置 GPU 接活，Web3 是外壳。让我意外的是节点接入居然还要填 Interest Form 申请，"去中心化"这个标签打了不少折扣。

晚上搞定了信息流关注清单，X 上关注了 13 个账号。进阶部分去 @GoPlusSecurity 看了 Echo Protocol 被黑的分析，攻击者凭空铸了 1000 eBTC 借走 86 万刀。看完之后对上午做的那个签还是不签工具更有感触了，用户层的安全意识和协议层的漏洞是两个战场，但都是真实的。

今天新记住三个词。Prompt Injection 上面说了。Yuma Consensus 我最开始以为是类似 PoW/PoS 的共识机制，后来才发现它解决的问题完全不同。不是"谁来记账"，是"谁的 AI 产出更好"。TEE是 CPU 里的加密黑箱，Phala 用来做 AI Agent 的安全执行，跑完能出可验证证明，但它和 ZK 证明的区别我还没搞清楚，下次要挖一下。

# 2026-05-18
<!-- DAILY_CHECKIN_2026-05-18_START -->













第一天正式开始，比想象中混乱很多，但还好没放弃。

上午在啃 Handbook 的 LLM 章节。说实话一开始看不太进去，全是定义。直到看到LLM 本质是猜下一个词这句话才突然开窍——之前觉得 AI 很玄乎，什么神经网络深度学习，原来核心逻辑就是在做超级自动补全。做了个小实验，同一个问题「ETH Gas 是什么」，一次直接问，一次加了角色、任务、格式要求，回答质量差了好几个档次。才意识到自己之前用 AI 都太随意了，prompt 不是在提问，是在给 AI 设定一个说话的框架。

下午跟着做交易解释器，Node.js + ethers.js，说实话大部分代码我看不懂，是 CC 帮我写的。但整个逻辑我是理解的：输入一个 tx hash，工具去 Sepolia 链上读原始数据，自动解码 ERC20 Transfer 事件，然后把数据喂给 DeepSeek 生成人话解释。跑通的那一刻有点小兴奋，第一次感觉自己真的读懂了一笔链上交易，不是通过钱包界面看余额，是直接读底层数据。做的过程里有一个细节让我印象很深：要把链上能验证的事实和模型在推断的东西分开。比如转了 155 个 USDT+是链上写着的，谁都能查。但这笔钱是用来支付服务费的是 AI 猜的，不一定对。以前用 AI 从来没想过这个边界，现在会开始区分了。

晚上连着看了两场活动，有点累，老师讲的好快全是干货，我这个小白真是有点云里雾里。Co-learning 那场聊到 Hermes 本地部署的优势，还有 AI 工具选型，信息量挺大。Web3 架构那场有一句话记下来了：安全源于设计，不是事后修补。Web2 出了漏洞可以推更新打补丁，Web3 合约一旦部署上链基本改不了，设计的时候就得想好所有情况。还有私钥那部分，听到私钥丢了真的就没了，没有客服没有找回，有点慌，会后悄悄检查了一下自己的钱包备份。

下午做交易解释器的时候顺手发现了一个问题——之前 Hermes 帮我提交的 proof 全不合格，一条条都是光秃秃的 repo URL，平台要求的格式完全没满足。当时有点崩，只能暂停手头的事逐条重交，对照要求一个个改。有点烦，但也是教训——以后提交前要自己看一眼，不能完全甩给 AI。
