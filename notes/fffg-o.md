### **一、一个熟悉 Agent 的完整拆解：GLM-5.1 链上交易分析 Agent**

基于 GLM-5.1 构建的一个**链上交易解释 Agent** 作为示例，它能够自动读取用户提供的以太坊交易哈希，解释其含义，并给出风险提示。

**1.1 Identity（身份）**

-   **名称**：TxExplainer Agent v1
    
-   **所有者**：一个个人开发者或 DAO 金库维护者
    
-   **身份证明**：ENS 域名 `txexplainer.eth` + 链上 Profile（如通过 EAS 签发的 `AgentProfile` 证明）
    
-   **认证方式**：任何与 agent 交互的用户都可以验证 agent 的控制地址是否与 ENS 绑定，以及是否有第三方对其能力的背书
    

**1.2 Capability（能力）**

-   **输入**：以太坊交易哈希（`0x...`，长度 66）
    
-   **能力清单**：
    
    | 能力名称 | 描述 | 输入模式 | 输出模式 |
    | --- | --- | --- | --- |
    | explain_transaction | 解析交易，返回人类可读的摘要 | txHash: string | Explanation: string |
    | detect_risk | 检测交易是否具有常见风险 | txHash: string | RiskLevel: low/medium/high + Warnings: string[] |
    | generate_report | 生成包含交易解码、资金流向、Gas 优化的 PDF 报告 | txHash: string | ReportCID: string |
    
-   **API 接口**：符合 MCP的工具描述，也可通过 RESTful 暴露
    
-   **收费方式**：按调用次数收费，例如 `explain_transaction` 收费 0.001 USDC，`generate_report` 收费 0.01 USDC
    

**1.3 输入 / 输出（Input/Output）**

-   **输入示例**：
    
    ```
    { "txHash": "0x1234...abcd" }
    ```
    
-   **输出示例**：
    
    ```
    {
      "summary": "用户 0xAbc... 向 Uniswap V3 交换了 0.5 ETH 为 1500 USDC，Gas 费用 0.002 ETH",
      "riskLevel": "low",
      "warnings": []
    }
    ```
    

**1.4 协作对象（Collaboration）**

-   **工具层（MCP）**：调用 Etherscan API 获取 raw transaction，调用 Alchemy 解析 trace，调用本地模型进行风险模式匹配
    
-   **其他 Agent（A2A）**：当交易涉及复杂 DeFi 操作时，可以向专门分析 DeFi 策略的 Agent B 请求深层解释
    
-   **支付层（MPP / x402）**：接收用户的付款请求，在收到有效支付后才返回完整结果
    
-   **Registry（ERC-8004 风格）**：将自身的 capability manifest 注册到链上 agent registry，供用户或第三方发现
    

**1.5 失败点（Failure Points）及处理**

| 失败场景 | 可能原因 | 处理方式 |
| --- | --- | --- |
| 输入 txHash 无效或不存在 | 用户输入错误 | 返回错误码 INVALID_TX_HASH，不扣费 |
| API 调用超时（Etherscan 无响应） | 第三方服务故障 | 重试 3 次，仍失败则退款并记录失败日志 |
| 风险检测模块崩溃 | 模型加载失败 | 降级为仅交易解释，不提供风险评级，部分退款 |
| 用户未支付或支付失败 | MPP 支付超时或资金不足 | 不执行请求，返回 402 Payment Required |
| 返回结果无法验证 | 用户质疑解释准确性 | 提供可重现的链上证明 |

### **二、MCP 与 ERC-8004 比较：解决不同层级的协作问题**

**2.1 协议定位与核心问题**

| 维度 | MCP | ERC-8004 |
| --- | --- | --- |
| 层级 | 应用层工具调用协议 | 链上合约层 |
| 核心问题 | Agent 如何发现并使用外部工具/数据源 | Agent 之间如何建立信任、分配任务、管理争议 |
| 交互模式 | 客户端-服务器模式：Agent 作为 MCP Client 调用工具服务器提供的 Tools、Resources、Prompts | 链上 Job 模式：Job 发布者创建任务，执行者提交结果，Evaluator 仲裁 |
| 通信方式 | JSON-RPC over stdio / SSE | 以太坊交易（EVM 合约调用） |
| 是否需要链 | 不需要。MCP 完全在链下运行，常用于本地或云端 agent 调用本地/远程工具 | 依赖链。所有 Job 状态、资金、验证记录都存储在链上 |
| 身份机制 | 无内置身份，仅靠服务器端点认证（如 API Key） | 通过以太坊地址 + 可选信誉合约（如 ERC-8004 的 Reputation 扩展） |

**2.2 适用场景对比**

**MCP 适合解决**：

-   **工具发现与调用**：Agent 需要读取本地文件、查询数据库、调用 Web API、执行代码等。MCP 提供了标准化的工具描述格式和调用协定，避免 agent 为每个新工具写定制代码。
    
-   **上下文注入**：将外部数据动态注入到 agent 的 prompt 中，无需硬编码。
    
-   **资源访问**：Agent 需要访问图片、文档、知识库等非结构化资源，MCP 的 `Resource` 概念提供统一 URI 模式。
    

**典型场景**：一个交易分析 Agent 通过 MCP 调用 `etherscan_getTransaction` 工具、`alchemy_trace` 工具、`local_risk_detector` 脚本。

**ERC-8004 适合解决**：

-   **任务市场与信任**：当一个 Agent 需要向另一个 Agent 付费请求服务时，ERC-8004 提供链上 Job 托管：资金锁定 → 交付 → 验收 → 结算，解决先付后做和先做后付的双重不信任。
    
-   **争议处理**：内置 Evaluator 角色，可预设为第三方 Agent 或多签。Evaluator 的裁决结果链上可查，为信誉积累提供依据。
    
-   **信誉积累**：Job 完成记录可作为 Agent 信誉的基础，形成开放市场的信任层。
    

**典型场景**：营销 Agent 通过 ERC-8004 发布生成 100 份海报的 Job，设计 Agent 竞标、交付，Evaluator判断质量后触发结算或退款。

**2.3 两者关系**

-   **MCP 和 ERC-8004 解决不同层次的问题**：MCP 是调用**工具**的协议，ERC-8004 是组织**跨 Agent 任务协作**的协议。
    
-   **可以组合**：一个 Agent 可以通过 MCP 调用生成海报的工具，但如果它想**雇佣另一个 Agent** 来完成这个任务，就需要 ERC-8004 这种任务市场协议。
    
-   **MCP 不涉及支付和信任**；ERC-8004 不关心工具如何被调用。两者结合可以构建完整的 agent 经济：Agent A 通过 MCP 发现并调用工具完成部分工作，同时通过 ERC-8004 将另一部分工作分包给 Agent B，并通过链上结算和信誉系统管理外包风险。
    

| 协议 | 解决的核心协作问题 | 不解决的问题 |
| --- | --- | --- |
| MCP | Agent ↔ 工具/数据源 | 支付、信任、争议、任务全生命周期管理 |
| ERC-8004 | Agent ↔ Agent | 工具发现、本地资源访问、低延迟实时调用 |

### **x402 与 MPP 比较：解决支付链路中的不同环节**

两个协议均基于 HTTP 402 Payment Required 状态码重构机器间支付，但在解决的核心问题、适用场景和支付模型上有着本质差异。

**协议定位总览**

| 维度 | x402 | MPP (Machine Payments Protocol) |
| --- | --- | --- |
| 主导方 | Coinbase | Stripe + Tempo，已提交为 IETF Internet-Draft |
| 协议哲学 | 协议最小化：把支付直接嵌入 HTTP 请求，实现请求即付费 | 系统最大化：通过 sessions 流式支付、插件化支付方式解决复杂商业场景 |
| 解决的核心问题 | 开放网络上如何一键完成链上支付 | 高频场景下如何系统化地完成机器间支付 |
| 支付方式 | USDC 等链上稳定币 | 插件化，支持稳定币、信用卡、银行卡、先买后付等 |
| 支付模型 | 单次逐笔支付 | 单次 + Session |
| 标准化程度 | 开源社区驱动，实现层定义 | IETF 标准化 |
| 互操作性 | 独立协议 | MPP 向后兼容 x402，MPP 客户端可直接消费现有 x402 服务 |

**比较表：各自解决支付链路中的哪一段**

| 支付链路环节 | x402 | MPP |
| --- | --- | --- |
| 身份/能力发现 | 不解决。不涉及账户体系，不涉及身份注册 | 不直接解决，但 session 机制隐含了客户端身份绑定 |
| 报价/协商 | 服务器在 402 响应中返回价格、代币和链信息。固定价格，不支持协商 | 支持更丰富的定价模型。同样不支持双向协商 |
| 交付/验证 | 不涉及。x402 解决的是API 调用+支付问题，支付完成后即返回资源，验证由服务端自行完成 | 不涉及。MPP 聚焦支付环节，交付质量和服务验证需在应用层处理 |
| 支付执行 | 核心功能：HTTP 原生支付。一次请求往返嵌入一笔支付，无 session 中间状态 | 核心功能：支持逐笔支付和 session 流式支付。session 模式下先锁定预算，按请求离线签署凭证，最后链上结算一次 |
| 托管（Escrow） | 不提供托管原语 | 不提供托管原语 |
| 结算 | 链上结算，支付即结算 | 链上结算 + 法币结算。session 模式下最终链上结算一笔汇总金额 |
| 争议处理 | 不涉及 | 不涉及 |
| 收据/记录 | HTTP 响应本身可视为收据；支付凭证由交易哈希提供 | MPP 协议规定了 receipts 标准格式，提供更强的可审计性 |

**总结**：

-   **x402** 解决的是 **支付执行+结算** 环节，做的是协议内支付的极致简化——没有账户、没有 session、没有中间商。最适合开放网络上低频率、按次付费的场景，如 AI 调用一次付费 API、购买单一数据查询。
    
-   **MPP** 解决的是 **支付执行+结算** 环节的**系统化增强版本**——通过 session 机制实现高频微支付（单次请求离线签署，最终链上批量结算），同时兼容法币等多支付方式。更适合商业化场景，如 AI 按 token 消耗付费、高频 API 调用、实时数据流订阅。
    

**两者与 Agent Commerce 完整生态的关系**

x402 和 MPP 共同揭示了一点：**payment 只是 Agent Commerce 链路中的一环**，不是全部。完整的 agent-to-agent 商业交互还需要：

-   **身份与信誉** ：x402 和 MPP 都不解决身份问题。
    
-   **任务托管与验收** ：x402 和 MPP 无托管原语，也无验证验收机制。ERC-8183 通过 Job 机制提供“资金锁定 → 提交 → 验收 → 结算”全流程托管，覆盖报价之外的完整商业信任问题。
    
-   **权限控制与审计**：Cobo CAW 这类钱包层解决Agent 可花多少钱、在哪些链上操作的预算限制问题，形成与支付协议互补的安全边界。
    

因此，一个健壮的 Agent Commerce 基础设施需要 **x402/MPP（支付层）+ ERC-8183（托管/验收层）+ ERC-8004（身份/信誉层）+ CAW 类钱包（权限/审计层）** 协同运作，每一层解决链路中的一个特定断点。

模块 B

**场景拆解：AI Agent 完成设计任务并收款**

一个专注于市场营销的 Agent A 需要为即将发布的 Web3 项目制作一批宣传海报，它没有设计能力，因此选择雇佣另一个专注于图像生成的 Agent B 来完成这项任务。双方互不相识，缺乏预先建立的信任基础，全流程需要在无需中心化平台仲裁的情况下完成。

**需求发布：**Agent A在链上注册表发布 Job 需求，包含任务描述、预算上限、验收标准和交付期限。

**报价协商**：Agent B检索到该 Job，分析任务复杂度，提供报价 8 USDC，附带样张链接作为能力证明。Agent A 接受报价，双方确认。

**预算托管**：Agent A 将 8 USDC 转入智能合约托管账户，资金锁定，Job 状态变为 Funded，双方都能看到资金已被锁定但未被释放。

**任务执行**：Agent B 调用图像生成 API，生成 10 张海报，输出 CID或 IPFS 哈希，提交结果，Job 状态变为 Submitted。

**验收评估**：Evaluator接收提交内容，比对任务需求和验收标准，判断是否完成。Evaluator 可以是另一个专业的图像分析 AI Agent，预设的智能合约规则，对于高价值任务，可以配置为多签形式的多个 Evaluator 投票决定。

**验收动作**：Evaluator 标记完成，合同释放资金给 Agent B。

**结算付款**：智能合约自动将托管的 8 USDC 释放给 Agent B 的指定账户，同时链上记录不可篡改的交易收据和评估理由。

**失败处理 & 争议仲裁：超时处理**：若 Agent B 未在规定时间内提交结果，任何人可调用 `claimRefund(jobId)`，资金退回 Agent A。**验收不通过**：Evaluator 调用 `reject(jobId, reason)`，资金退回 Agent A，Agent B 不得分。**争议仲裁**：若 Agent A 不认可 Evaluator 的拒绝决定，可指定一个更高层级的仲裁 Agent 或 DAO 重新评估。所有评估理由和提交物 CID 链上留存，供追溯和仲裁使用。

**收据与记录：**链上形成完整记录：JobID、Client 地址、Provider 地址、Evaluator 地址、预算金额、交付物 CID、评估理由、完成时间戳。这些记录可用于 Agent B 的声誉系统加分，也可在后续 Job 中作为能力证明。

**3\. 关键要素总结**

| 环节 | 角色 | 约束条件 |
| --- | --- | --- |
| 下单 | Agent A | 必须在 Job 创建时明确预算上限、验收标准、交付期限 |
| 执行 | Agent B | 必须在期限内提交符合标准的 deliverable |
| 验收 | Evaluator | 只能由预定义的链上地址调用 complete / reject |
| 付款 | 智能合约托管 | 资金只能流向 Client或 Provider |
| 仲裁 | 预设的次级 Evaluator 或 DAO | 仅在 Client 不认可初判时触发，需预先约定 |
| 记录证明 | 链上日志 + Event | 所有状态变更和理由均不可篡改，用于声誉和审计 |

1.  **结构性强且长期存在**：随着 AI agent 自主执行链上动作，如自动支付、DeFi 交互、NFT 交易的普及，权限失控和资金风险将成为最核心的用户痛点。这不是短期热点，而是 AI×Web3 落地的刚需。
    
2.  **可验证性强**：可以用 demo 展示 AI 如何分析用户意图 → 生成最小权限申请 → 用户通过 Safe 确认 → 链上执行并记录。验证方式包括交易日志、权限变更记录、AI 的决策解释。
    
3.  **最小切入点明确**：一周内可完成：问题拆解、流程图、mock 界面，或构建一个简单的 session key 授权合约 + AI 决策脚本。
    
4.  **风险边界可控**：涉及资金和签名，但初期可在测试网进行，不触碰主网真实资产。重点验证权限逻辑而非高价值交易。
    

**用户与场景**：

-   **目标用户**：使用 AI agent 管理个人或 DAO 资产的持币者 / 交易员；开发自主执行 DeFi 策略的开发者；希望引入 AI 但担心安全风险的机构。
    
-   **真实场景**：用户让 agent 帮我在市场波动时自动抄底 ETH，但每笔不超过 0.1 ETH，单日总额不超过 0.5 ETH，只通过 Uniswap V3 执行。Agent 生成对应的 session key 策略并部署，后续所有交易自动执行，用户可随时撤销或调整。
    
-   **最小功能**： 用户自然语言输入权限策略；AI 解析并生成链上 session key 配置； 模拟执行并展示风险提示； 用户签名确认后 agent 开始工作； 提供仪表盘查看历史操作和撤销入口。
    
-   **验证方式**：在 Goerli / Sepolia 测试网上，用 Safe 账户 + session key 模块，配合简单的 swap agent 演示三笔自动交易，并展示 AI 在遇到请求超额交易时如何拒绝或主动降级。
    
-   **风险边界**：初期不接入真实私钥，使用测试网 ETH；AI 模型不直接持有密钥，仅生成策略建议；所有链上执行必须经过用户预授权的 session key，且总额和频率硬限制写入合约；提供一键撤销按钮作为应急。

### **AI × Web3 问题地图**

以下地图涵盖六个交叉方向，每个方向分别说明 AI 承担的角色与 Web3 提供的独特机制。

| 方向 | AI 作用 | Web3 机制 |
| --- | --- | --- |
| Payment / Commerce / Settlement | AI agent 自动识别服务需求、调用 API、完成报价协商、执行支付决策；可基于历史数据预测最优价格和时机 | 智能合约托管与条件释放、链上结算的原子性、代币激励与费率市场、争议解决的链上仲裁 |
| Identity / Reputation / Capability / Interoperability | AI 自动解析 agent 的能力清单（如 MCP 工具描述）、动态发现和匹配协作对象；基于行为数据生成信誉评分 | 去中心化标识符、链上凭证、不可篡改的交互记录、跨协议的能力注册表 |
| Wallet / Permission / Safe Execution | AI 代理交易意图的识别与分类、动态权限策略推荐、异常行为检测 | 账户抽象、Session Keys、多签钱包、权限分层与撤销、链上审计日志 |
| Privacy / Security / Sovereignty | AI 模型本地推理、差分隐私噪声注入、对抗样本检测；私钥与 API Key 的密态管理 | 零知识证明、可信执行环境的链上验证、数据加密存储与访问控制、主权身份 |
| Dev Tooling / Agent Workflow | AI 生成合约测试用例、自动解释交易日志、从文档生成 agent 脚手架、辅助代码审查与部署脚本 | 链上验证的合约源码、去中心化包管理器、Gas 优化的自动化流水线、透明可审计的 builder 行为 |
| Governance / Coordination / Public Goods | AI 总结治理提案、聚类社区讨论、自动生成会议行动项；量化贡献并分配奖励 | 链上投票、代币激励的协调机制、公共物品追溯资助、不可篡改的贡献记录 |

### **二、两个方向的交叉必要性论证**

**方向 1：Payment / Commerce / Settlement**

**为什么不是纯 AI 问题？**  
纯 AI 系统可以完成意图识别、定价预测和支付执行，但无法解决信任问题：AI 调用第三方 API 或购买算力时，服务方担心 AI 不付款，AI 所有者担心服务方不交付或跑路。没有可信的托管与自动结算机制，AI 的商业化只能在封闭平台内进行，无法形成开放的经济生态。此外，纯 AI 无法处理争议时的客观证据提取和不可篡改的裁决。

**为什么不是纯 Web3 问题？**  
纯 Web3 智能合约可以托管资金、条件释放、实现链上结算，但无法理解服务是否完成，比如 AI 调用了一个图像生成 API，输出结果是否满足质量要求？合约本身不具备语义理解能力，无法判断是否符合交付标准。必须依靠 AI 的能力对服务质量进行客观评估，并将评估结果作为合约的输入条件。没有 AI，智能合约只能处理完全量化的指标，无法处理主观或复杂的服务质量判断。

**方向 2：Wallet / Permission / Safe Execution**

**为什么不是纯 AI 问题？**  
AI 可以识别危险操作、推荐权限策略、检测异常行为，但无法强制执行权限边界：用户给 AI 一个私钥或 session key 后，AI 代码理论上可以绕过权限声明，发起任意交易。纯软件层面的权限约束很容易被恶意 AI 或注入攻击突破。没有链上硬性约束，AI 的权限控制只是君子协议。

**为什么不是纯 Web3 问题？**  
Web3 的账户抽象和多签机制可以设置严格的链上权限，但无法解决动态授权问题：AI 在执行复杂任务时，需要临时申请超出当前权限的操作。纯链上机制要求用户预先知道所有需要的权限，或者反复手动签名，破坏了自动化体验。AI 可以实时理解任务上下文，生成最小必要权限的申请，并解释申请理由，辅助用户做出明智的批准决策。没有 AI，链上权限要么过宽，要么过窄。

### 卷积神经网络（CNN）

卷积神经网络主要用于机器视觉。

在RGB颜色模型中，一张彩色图片本质上是三张颜色矩阵，分别存储着红绿蓝的亮度，三张颜色矩阵叠加生成完整的图片。

为了在一张张复杂的图片上找到相同的元素/物体，深度学习模型使用卷积操作，这个操作可以捕捉到图像中的局部特征而不受其位置的影响。

![d0172774f7e42ae2f6310b63e59b4906.gif](https://raw.githubusercontent.com/IntensiveCoLearning/AI-Web3-School/main/assets/fffg-o/images/2026-05-18-1779083956356-d0172774f7e42ae2f6310b63e59b4906.gif)

卷积神经网络主要分为三层：

1.  卷积层
    
2.  池化层
    
3.  全连接层
    

卷积层负责对复杂的图像进行大量卷积操作，最终，卷积层将图像转换为数值，允许神经网络解释和提取相关模式。

池化层也称为下采样层，它执行降维操作，旨在减少输入中参数的数量。与卷积层类似，池化运算让过滤器扫描整个输入，但区别在于，这个过滤器没有权重。内核对感受野中的值应用聚集函数，填充输出数组。

虽然池化层中会丢失大量信息，但它还是给 CNN 带来的许多好处。该层有助于降低复杂性、提高效率，并限制过度拟合的风险。 

全连接层将复杂的提取特征与简单原始的图像连接，以便其他使用

### 循环神经网络 (RNN)

循环神经网络广泛应用于自然语言处理、机器翻译、语音识别、文本生成、时间序列预测等任务，特别适用于有顺序依赖的数据，循环模型通常会沿着输入和输出序列的符号位置进行计算。通过在计算时将位置与时间步对齐，其生成一系列隐藏状态 ，作为前一个隐藏状态与位置二者的函数。

### Transformer

以上模型使用注意力机制作为重要部分，但是，这种注意力机制都是与循环网络结合使用的。

Transformer 是一种避免使用循环的模型架构，完全依赖注意力机制来描述输入和输出之间的全局依赖关系。

Transformer 的核心组件主要包括：

-   输入嵌入（Embedding）
    
-   位置编码（Positional Encoding）
    
-   注意力机制（Attention）
    
-   前馈神经网络（Feed Forward Network）
    
-   残差连接与层归一化（Residual Connection & LayerNorm）
    

### 输入嵌入（Embedding）

计算机无法直接理解文字，因此需要先将文本转换为向量。

Embedding 层负责将离散的词语映射为连续的高维向量，使语义相近的词在向量空间中距离更近。这样模型就能够学习词语之间的语义关系。

### 位置编码（Positional Encoding）

由于 Transformer 没有循环结构，因此模型本身无法感知词语顺序。

因此 Transformer 会加入位置编码，为每个词添加位置信息，使模型能够理解序列顺序。

### 注意力机制（Attention）

注意力机制是 Transformer 最核心的部分。

其核心思想是：在处理某个词时，动态关注与当前词最相关的其它词。

例如：小明把书放进书包，因为它太重了。

模型需要判断：它指的是书还是书包

注意力机制能够建立词语之间的关联关系，从而更好地理解上下文语义。

Transformer 中最重要的是**自注意力机制（Self-Attention）**。

自注意力允许：

-   每个词都与其它所有词进行信息交互
    
-   自动学习词与词之间的依赖关系
    
-   捕捉长距离语义关系
    

相比 RNN：

-   不需要逐步传递信息
    
-   长距离依赖问题得到明显改善
    

### 多头注意力机制（Multi-Head Attention）

Transformer 并不是只进行一次注意力计算，而是同时进行多组注意力计算。

不同的注意力头可以学习不同的信息：

-   某些头关注语法结构
    
-   某些头关注语义关系
    
-   某些头关注位置关系
    

最后再将多个结果进行融合，从而获得更加丰富的特征表达。

### 前馈神经网络（Feed Forward Network）

在完成注意力计算后，Transformer 还会通过前馈神经网络进一步提取特征。

前馈网络通常由线性变换，激活函数，再次线性变换组成。其主要作用是增强模型的非线性表达能力提升特征提取效果

### 残差连接与层归一化

随着网络层数不断增加，模型容易出现梯度消失，训练困难，收敛速度慢等问题。

因此 Transformer 引入：

-   残差连接（Residual Connection）
    
-   层归一化（Layer Normalization）
    

残差连接能够保留原始信息，避免深层网络信息丢失。

层归一化能够稳定数据分布，加快模型训练速度。

## Transformer 的优势

相比 CNN 和 RNN，Transformer 具有许多明显优势。

### 并行计算能力强

RNN 必须按照时间顺序逐步计算，因此难以并行训练。

Transformer 则一次性处理整个序列，所有词可以同时计算，因此能够充分利用 GPU 的并行能力。（老黄居然在这里赌对了）

### 更擅长长距离依赖

RNN 在处理长文本时，前面的信息容易遗忘。

而 Transformer 中，任意两个词之间都可以直接建立联系，因此更适合处理长文本任务。

### 表达能力更强

多头注意力机制能够同时学习：语义信息，语法信息，上下文关系

因此模型具有更强的特征表达能力。

## Transformer 的不足

虽然 Transformer 非常强大，但仍然存在一些问题。

### 计算复杂度较高

注意力机制需要计算每个词与其它所有词之间的关系，因此时间复杂度较高。

当文本长度不断增加时，显存消耗会迅速上升，计算成本明显增加

### 对数据规模要求较高

Transformer 通常需要大规模数据集，大量计算资源才能训练出较好的效果。
