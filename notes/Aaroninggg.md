# 交易解释器 MVP 需求文档

## 1\. 项目目标

开发一个“交易解释器”的最小可用版本。用户输入一笔区块链交易哈希后，系统自动读取该交易的链上详情、事件日志和相关合约 ABI，并调用 LLM 生成一段面向普通用户的交易解释。

本版本只做“读取与解释”，不做钱包连接、不做签名、不发起交易、不提供投资建议。

* * *

## 2\. MVP 范围

### 2.1 支持范围

第一版优先支持：

-   Ethereum 主网或 Sepolia 测试网，二选一即可
    
-   输入交易哈希
    
-   读取交易详情
    
-   读取交易 receipt
    
-   读取事件日志 logs
    
-   获取相关合约 ABI
    
-   解码交易 input data
    
-   解码事件日志
    
-   调用 LLM 生成自然语言解释
    
-   输出 Markdown 格式结果
    

### 2.2 暂不支持

第一版不做：

-   钱包连接
    
-   用户签名
    
-   自动发交易
    
-   多链自动识别
    
-   复杂 DeFi 交易完整路径还原
    
-   风险评分
    
-   合约安全审计
    
-   资产价格估算
    
-   税务、投资、法律建议
    

* * *

## 3\. 用户流程

1.  用户输入一笔交易哈希。
    
2.  系统根据配置的链读取交易详情。
    
3.  系统读取交易 receipt 和事件日志。
    
4.  系统收集交易中涉及的地址，包括：
    
    -   交易发起地址 from
        
    -   交易接收地址 to
        
    -   事件日志中的合约地址
        
    -   ERC20 / ERC721 / ERC1155 Transfer 事件中涉及的地址
        
5.  系统尝试获取相关合约 ABI。
    
6.  系统使用 ABI 解码：
    
    -   交易调用的方法
        
    -   方法参数
        
    -   事件日志
        
7.  系统整理成结构化数据。
    
8.  系统将结构化数据发送给 LLM。
    
9.  LLM 生成交易解释。
    
10.  系统返回解释结果。
     

* * *

## 4\. 输入要求

### 4.1 最小输入

用户输入：

```text
0x 开头的交易哈希
```

示例：

```text
0xc2f3098906d406dfef5511711ef0c86ecfe233a54242af79060bf8807ee5b73c
```

### 4.2 配置项

系统通过环境变量配置：

```env
CHAIN=sepolia
RPC_URL=
BLOCK_EXPLORER_API_KEY=
LLM_BASE_URL=
LLM_API_KEY=
LLM_MODEL=
```

* * *

## 5\. 数据读取要求

### 5.1 交易详情

系统需要读取以下链上字段：

-   transaction hash
    
-   block number
    
-   block timestamp
    
-   from address
    
-   to address
    
-   value
    
-   input data
    
-   gas
    
-   gas price
    
-   max fee per gas，如存在
    
-   max priority fee per gas，如存在
    
-   nonce
    
-   chain id
    

### 5.2 Receipt 信息

系统需要读取：

-   transaction status
    
-   gas used
    
-   effective gas price
    
-   contract address，如为合约部署交易
    
-   logs
    
-   cumulative gas used
    

### 5.3 事件日志

系统需要读取每条 log 的：

-   log address
    
-   topics
    
-   data
    
-   log index
    
-   transaction index
    
-   removed 状态，如存在
    

* * *

## 6\. ABI 获取规则

系统需要按以下优先级获取 ABI：

1.  优先从区块浏览器 API 获取已验证合约 ABI。
    
2.  如果 ABI 获取失败，保留原始 input data 和 logs。
    
3.  对常见标准事件内置 ABI，至少包括：
    
    -   ERC20 Transfer
        
    -   ERC20 Approval
        
    -   ERC721 Transfer
        
    -   ERC721 Approval
        
    -   ERC1155 TransferSingle
        
    -   ERC1155 TransferBatch
        

当 ABI 无法获取时，系统不得编造方法名或事件含义，只能说明“该合约 ABI 未获取到，无法完全解码”。

* * *

## 7\. 解码要求

### 7.1 交易 input 解码

如果 to 地址对应的合约 ABI 可用，需要解析：

-   调用的合约方法名
    
-   方法参数名称
    
-   方法参数值
    

示例结构：

```json
{
  "methodName": "transfer",
  "params": {
    "to": "0x...",
    "amount": "1000000000000000000"
  }
}
```

### 7.2 事件日志解码

如果事件 ABI 可用，需要解析：

-   event name
    
-   event params
    
-   emitting contract address
    

示例结构：

```json
{
  "eventName": "Transfer",
  "contractAddress": "0x...",
  "params": {
    "from": "0x...",
    "to": "0x...",
    "value": "1000000000000000000"
  }
}
```

* * *

## 8\. LLM 输出要求

LLM 最终输出必须包含以下部分：

## 交易解释

### 1\. 用户发起了什么动作

用普通人能理解的语言说明该交易大概做了什么。

例如：

-   用户向某地址转账
    
-   用户调用了某个合约方法
    
-   用户授权某个合约使用自己的代币
    
-   用户与 DeFi 合约发生交互
    
-   用户领取、铸造、交换或销毁某种资产
    

如果无法确定，需要明确写出“不确定”。

### 2\. 涉及哪些资产和地址

列出：

-   发起地址
    
-   接收地址
    
-   被调用合约地址
    
-   涉及的代币合约地址
    
-   资产类型
    
-   数量
    
-   收款方
    
-   付款方
    

如果代币名称、符号或精度无法获取，需要保留原始链上数值，并说明未能确认。

### 3\. 哪些信息来自链上数据

明确列出来自链上数据的事实，例如：

-   交易哈希
    
-   from 地址
    
-   to 地址
    
-   value
    
-   status
    
-   gas used
    
-   block number
    
-   decoded method
    
-   decoded events
    
-   Transfer / Approval 事件
    

这一部分必须只写链上可验证信息，不允许模型自由推断。

### 4\. 哪些信息是模型推断

明确列出模型根据链上数据推断出的内容，例如：

-   这可能是一次 ERC20 转账
    
-   这可能是一次授权操作
    
-   这可能是一次 swap
    
-   这可能是一次 NFT 转移
    
-   这可能是某个协议交互
    

推断必须使用“可能”“看起来像”“根据事件日志推测”等措辞，不得说成确定事实。

### 5\. 模型不确定的地方

列出所有不确定信息，例如：

-   合约 ABI 不完整
    
-   事件无法解码
    
-   无法确认合约真实业务含义
    
-   无法确认地址归属
    
-   无法确认用户真实意图
    
-   无法判断交易是否安全
    
-   无法判断资产价格或实际收益
    

### 6\. 如果要签类似交易，用户应该检查什么

给出签名前检查清单，包括：

-   确认目标地址是否正确
    
-   确认调用的是哪个合约
    
-   确认授权额度是否过大
    
-   确认转出的资产类型和数量
    
-   确认交易是否包含 approve、permit、setApprovalForAll 等授权行为
    
-   确认 gas 费用
    
-   确认交易所在网络
    
-   确认是否在可信前端或可信钱包中发起
    
-   如果是复杂 DeFi 交易，确认滑点、最低收到数量、截止时间和交易路径
    

* * *

# 1. 密码学（Cryptography）

## 1.1 一句话解释

密码学是 Web3 里用来证明身份、控制资产、确认授权和验证数据是否被篡改的底层规则。

## 1.2 人话理解

在 Web3 里，你经常会看到这些词：账户、地址、私钥、签名、交易哈希、所有权。

这些东西看起来像传统互联网里的账号、密码、登录和确认按钮，但本质完全不同。

传统互联网里，如果你忘了密码，可以找客服、收验证码、重置密码。  
但在 Web3 里，很多时候没有客服，也没有“找回密码”。

谁掌握私钥，谁就控制这个账户和里面的资产。

所以，学密码学不是为了研究复杂公式，而是为了知道：

- 哪些动作是在证明身份；
- 哪些动作是在授权资产；
- 哪些信息一旦丢失就无法恢复；
- 哪些信息一旦泄露就可能导致资产被转走。

## 1.3 为什么要学

很多 Web3 新手会把：

- 钱包理解成账号；
- 地址理解成用户名；
- 签名理解成登录确认；
- 私钥理解成普通密码。

这样理解很危险。

因为在 Web3 里，很多“确认”按钮背后，可能不是简单登录，而是授权某个合约使用你的资产。

密码学在这里不是抽象理论，而是产品和资产安全的边界。

## 1.4 核心逻辑

Web3 的身份，不是平台发给你的，而是由你能否证明自己控制某个私钥决定的。

也就是说：

- 私钥代表控制权；
- 公钥和地址用于验证控制关系；
- 签名是授权证据；
- 哈希用于验证数据是否一致；
- Merkle Tree 用于高效验证大量数据中的某一条。

## 1.5 关键知识点

### 1.5.1 Hash：数据指纹

Hash 可以理解成给一段数据生成一个固定长度的“指纹”。

它的特点是：

- 原文只要改一点点，哈希结果就会明显不同；
- 通常不能通过哈希反推出原文；
- 主要用途是验证数据有没有被改过。

Hash 不是加密。  
加密通常是为了之后可以解密还原，Hash 更像是验证“是不是同一份数据”。

常见用途：

- 交易哈希：定位一笔交易；
- 区块哈希：定位一个区块；
- 合约字节码哈希：验证部署代码是否一致；
- 数据承诺：先公布哈希，之后公布原文，证明没有偷偷修改。

### 1.5.2 Public Key：公钥

公钥可以公开，用来验证某个签名是不是来自对应的私钥。

在以太坊里，用户更常看到的是地址，而不是完整公钥。地址可以理解成从公钥进一步处理得到的短标识。

需要记住：

- 地址可以公开；
- 公钥可以公开；
- 私钥绝对不能公开。

地址本身不代表可信。一个地址看起来像“官方地址”没有意义，真正重要的是它是否来自官网、官方文档、区块浏览器验证信息或其他可信来源。

### 1.5.3 Private Key：私钥

私钥是账户控制权本身。

谁拿到你的私钥，谁就可以像你本人一样操作你的钱包。

基本规则：

- 不截图；
- 不上传云盘；
- 不粘贴给网页；
- 不发给客服；
- 不发给 AI 工具；
- 不放进代码仓库；
- 不用主钱包测试陌生 dApp。

### 1.5.4 Signature：签名

签名不是简单的登录弹窗，而是对某段具体内容的授权证明。

签名前要知道：

- 这次签名是为了什么；
- 对应哪个网站；
- 在哪条链上；
- 涉及哪个合约地址；
- 有没有授权额度；
- 有没有过期时间；
- 是否涉及资产风险。

一句话：看不懂就不要签。

### 1.5.5 Merkle Tree：高效证明

Merkle Tree 可以用少量信息证明“某条数据属于一大批数据”。

比如空投名单里有几十万个地址，你不需要下载完整名单，只需要一段 Merkle proof，就可以证明自己的地址在名单里。

常见用途：

- 空投名单；
- 轻客户端；
- 状态证明；
- 可验证数据结构。

## 1.6 在 AI x Web3 中的位置

AI Agent 如果要解释交易、判断授权、生成链上操作或帮助用户管理权限，就必须理解签名和密钥边界。

AI 可以解释“这笔授权看起来在批准某个 token 额度”，但不能替用户保管私钥，也不能在用户没理解内容时推动签名。

## 1.7 小结

密码学不是高深数学，而是 Web3 中保护资产和确认身份的基本规则。

最重要的是：每次钱包弹窗出现时，都要问自己：我现在是在证明身份，还是在授权资产？

---

# 2. 钱包（Wallet）

## 2.1 一句话解释

钱包不是 Web3 的登录按钮，而是用户管理账户、签名授权、发送交易、切换网络和确认风险的入口。

## 2.2 人话理解

钱包更像是你的：

- 账户管理器；
- 资产控制器；
- 签名确认器；
- 交易发起器；
- 风险确认窗口。

在 Web3 应用里，连接钱包、读取地址、签名消息、发送交易、授权 token、切换网络、支付 gas，很多动作都发生在钱包边界上。

## 2.3 为什么要学

很多人以为连接钱包就等于登录成功，但连接钱包只是让应用知道：

- 你的钱包地址是什么；
- 你当前在哪条链上。

连接钱包不等于应用可以动用你的资产。

真正风险更高的动作通常发生在后面，比如签名、授权、发送交易。

## 2.4 核心逻辑

钱包管理的不是普通账号资料，而是链上账户的控制权。

传统互联网里，平台可以重置密码、恢复账号、修改数据库。  
Web3 里，控制权来自私钥或智能账户规则。

应用可以请求连接和签名，但不应该默认拥有用户账户权限。

## 2.5 钱包交互的三类动作

### 2.5.1 连接钱包

连接钱包通常只是读取地址和当前网络。

它不等于授权资产。

### 2.5.2 签名消息

签名消息是用户证明自己控制某个地址。

它可以用于登录、身份验证、创建订单或授权某些动作。风险在于用户可能看不懂自己签了什么。

### 2.5.3 发送交易

发送交易会真正改变链上状态。

比如：

- 转账；
- 授权 token；
- 调用合约；
- mint NFT；
- 质押资产；
- 领取空投。

交易一旦成功上链，通常不能像数据库记录一样随便撤回。

## 2.6 关键知识点

### 2.6.1 EOA

EOA 是由私钥控制的账户。

它可以：

- 签名消息；
- 发起交易；
- 支付 gas；
- 调用合约；
- 接收和发送资产。

优点是简单、通用、兼容性好。  
缺点是私钥丢失难恢复，权限难细分，自动化能力弱。

### 2.6.2 Mnemonic：助记词

助记词是一组用于恢复钱包私钥的单词。

它不是验证码，也不是普通密码。  
谁拿到你的助记词，谁就可能恢复你的钱包并转走资产。

任何网页、客服、AI、表单或脚本要求你输入助记词，都应该默认视为危险。

### 2.6.3 Transaction：交易

交易不是点击按钮本身，而是一整套链上流程：

1. 前端发起请求；
2. 钱包弹窗确认；
3. 用户签名；
4. 交易广播到网络；
5. 被打包进区块；
6. 执行成功或失败；
7. 前端和区块浏览器显示结果。

前端至少要处理：等待钱包确认、用户拒绝、交易已广播、等待确认、交易成功、交易失败、长时间 pending。

### 2.6.4 Gas

Gas 是链上执行成本。

用户发交易时通常要支付 gas fee。  
如果余额不足、网络拥堵、gas 估算不准，交易可能失败或卡住。

产品至少要告诉用户：

- 大概费用是多少；
- 用什么资产支付；
- 失败时是否仍可能消耗费用；
- 是否可以重试。

### 2.6.5 Explorer：区块浏览器

区块浏览器是查看链上事实的窗口。

看一笔交易时，重点检查：

- Status：成功还是失败；
- From / To：谁发起，调用了哪个地址；
- Method：调用了哪个方法；
- Value：是否转了原生资产；
- Token Transfers：是否发生 token 转移；
- Logs：合约发出了哪些事件；
- Gas Used：实际消耗了多少 gas。

## 2.7 在 AI x Web3 中的位置

AI 可以帮助用户解释交易、准备参数、检查风险、生成操作计划。  
但签名和权限不能随意交给模型。

更合理的模式是：AI 做理解和辅助，钱包负责确认和授权，智能账户或策略合约负责执行约束。

## 2.8 小结

钱包不是登录按钮，而是用户控制链上资产和权限的入口。  
每次钱包弹窗出现时，要先判断自己是在连接、签名，还是发送交易。

# 3. 智能合约（Smart Contract）

## 3.1 一句话解释

智能合约不是法律合同，而是部署在区块链上的程序。

## 3.2 人话理解

智能合约可以保存数据、执行规则、管理资产，并让所有人验证执行结果。

Web3 中很多核心逻辑都在智能合约里：

- Token；
- NFT；
- DEX；
- 借贷协议；
- 治理投票；
- 质押；
- 空投；
- 账户抽象。

## 3.3 为什么要学

普通互联网应用中，前端可以改，后端可以重构，数据库可以迁移。  
但智能合约一旦部署到链上，修改成本和风险通常更高。

如果合约写错了，可能不是简单修 bug，而是会直接影响资产安全。

学习智能合约，不一定是为了马上写复杂协议，而是为了知道：

- 哪些逻辑适合上链；
- 合约如何保存状态；
- 调用如何改变状态；
- 为什么测试和审计不能省；
- 权限和升级为什么重要。

## 3.4 核心逻辑

智能合约的价值来自可验证执行。

它让任何人都能：

- 检查规则；
- 调用接口；
- 验证状态变化。

这种开放性带来了可组合性，也带来了攻击面。

## 3.5 关键知识点

### 3.5.1 Solidity

Solidity 是 EVM 生态最常见的合约语言。

一个最小合约示例：

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

contract Counter {
    uint256 public count;

    event CountChanged(uint256 newCount);

    function increment() external {
        count += 1;
        emit CountChanged(count);
    }
}
```

这里：

- `count` 是链上状态；
- `increment()` 是改变状态的写操作；
- `event` 是外部系统可以索引的日志。

学习 Solidity 时，重点不只是语法，还要理解 storage、msg.sender、modifier、event、external call、revert 和权限控制。

### 3.5.2 EVM

EVM 是 Ethereum Virtual Machine，也就是运行智能合约的虚拟机。

Solidity 代码会被编译成 EVM 字节码，部署到链上后由节点执行。

理解 EVM 有助于理解：

- 为什么要 gas；
- 为什么 storage 很贵；
- 为什么外部调用有风险；
- 为什么同一套合约可以部署到多个 EVM 兼容链。

### 3.5.3 ABI

ABI 是应用和合约沟通的接口说明书。

它告诉前端或工具：

- 合约有哪些函数；
- 每个函数需要什么参数；
- 返回什么结果；
- 有哪些事件。

但 ABI 只告诉你“能调用什么”，不保证“调用是否安全”。

### 3.5.4 Event

Event 是合约执行时向外部系统留下的日志。

它常用于：

- 记录转账；
- 订单创建；
- 参数更新；
- 权限变更；
- 投票结果。

很多产品页面不是直接扫描所有 storage，而是通过索引 event 构建数据层。

### 3.5.5 Upgrade

合约升级是安全、治理和产品迭代之间的权衡。

不可升级合约规则更固定，但 bug 修复困难。  
可升级合约更灵活，但引入管理员权限和治理风险。

判断升级风险时要问：

- 谁有升级权限；
- 是否有多签或 timelock；
- 用户能不能提前看到升级内容；
- 升级能不能改变资产转移、提款或权限逻辑；
- 管理员密钥泄露的最坏结果是什么。

## 3.6 一次合约调用的流程

以用户点击投票按钮为例：

1. 前端读取钱包地址和当前网络；
2. 前端根据 ABI 编码调用数据；
3. 钱包展示交易信息；
4. 用户确认签名；
5. 交易广播到 RPC 节点；
6. 交易被打包进区块；
7. EVM 执行合约逻辑；
8. 成功则更新状态并发出 event，失败则 revert；
9. 前端和索引器更新页面或数据库。

## 3.7 常见错误

- 把 view 函数和写交易混在一起；
- 没有处理 token decimals；
- 只测试成功路径；
- 把 owner 当成永远可信；
- 忽略事件设计。

## 3.8 在 AI x Web3 中的位置

AI 可以帮助理解 ABI、生成调用参数、解释交易结果、写测试用例。  
但智能合约应该承担最终规则和约束，不能把所有判断都交给模型。

## 3.9 小结

智能合约是公开运行、可被验证、也可能管理真实资产的链上程序。  
它的核心不是“聪明”，而是可验证执行。

# 4. 开发栈（Dev Stack）

## 4.1 一句话解释

Web3 开发栈是一条从写合约、测合约、部署合约、连接钱包、调用合约到监控结果的工程链路。

## 4.2 人话理解

Web3 开发栈不是工具清单，而是完整流程：

写合约 → 编译合约 → 测试合约 → 部署合约 → 前端调用 → 区块浏览器验证 → 上线监控。

它的目标是让链上开发更可验证、更可复现、更少事故。

## 4.3 为什么要学

做一个 demo 不难，但真实项目会马上遇到：

- 合约编译；
- 测试；
- 部署脚本；
- RPC 配置；
- 环境变量；
- 前端 SDK；
- chain id 切换；
- 合约验证；
- 权限管理；
- 交易监控。

开发栈的价值是把这些流程变成可重复系统。

## 4.4 最小开发链路

一个 Web3 项目至少要经历：

1. 在本地或浏览器 IDE 写合约；
2. 编译合约，得到 bytecode 和 ABI；
3. 在本地链或测试网部署合约；
4. 写测试覆盖状态变化和权限边界；
5. 前端用合约地址和 ABI 调用合约；
6. 在区块浏览器验证源码、交易和事件。

## 4.5 核心原则

### 4.5.1 本地优先复现

合约逻辑、部署脚本和前端调用都应该先在本地或测试网跑通。

### 4.5.2 地址和 ABI 要版本化

前端调用的是哪份合约，必须可追踪。

要记录：

- 网络；
- chain id；
- 合约地址；
- ABI 版本；
- 部署交易；
- 部署时间；
- 对应代码 commit。

### 4.5.3 安全库不能替代审计

OpenZeppelin 等安全库能降低基础风险，但不能替代业务逻辑审查。

## 4.6 关键工具

### 4.6.1 Remix

Remix 是浏览器里的 Solidity IDE。

适合：

- 入门学习；
- 快速编译小合约；
- 部署到 JavaScript VM 或测试网；
- 理解 read 和 write 调用区别。

它像一个合约实验台，但不适合长期替代工程化仓库。

### 4.6.2 Hardhat

Hardhat 适合 JavaScript / TypeScript 项目。

它可以支持：

- 本地开发网络；
- 合约编译；
- 自动化测试；
- 部署脚本；
- 插件生态；
- 合约验证。

典型目录包括：

- `contracts/`：合约源码；
- `test/`：测试；
- `scripts/` 或 `ignition/`：部署脚本；
- `hardhat.config.ts`：配置；
- `artifacts/`：编译产物。

### 4.6.3 Foundry

Foundry 更偏命令行和 Solidity-native 测试。

常用工具：

- `forge build`：编译；
- `forge test`：测试；
- `anvil`：启动本地链；
- `cast call`：读取链上合约；
- `cast send`：发送交易。

它适合高强度合约开发、安全测试、fuzz testing 和 fork testing。

### 4.6.4 OpenZeppelin

OpenZeppelin 提供常用合约标准和安全组件。

常见模块：

- ERC-20；
- ERC-721；
- Ownable；
- AccessControl；
- Pausable；
- ReentrancyGuard；
- Upgradeable 合约工具。

使用成熟库可以减少基础实现错误，但不能替你决定业务权限。

### 4.6.5 viem / wagmi

viem 和 wagmi 主要解决前端与链交互的问题。

viem 偏底层 TypeScript 接口库。  
wagmi 偏 React hooks，适合管理钱包连接、账户状态、合约读写和交易状态。

前端至少要分清：

- 钱包是否已连接；
- 用户当前在哪条链；
- 合约读取是否加载中；
- 写交易处于等待签名、已广播、等待确认、成功或失败的哪一步。

## 4.7 在 AI x Web3 中的位置

AI 可以帮助生成部署脚本、补测试、解释 ABI、排查交易失败。  
但如果 AI Agent 能运行命令或发送交易，必须受到 repo workflow、测试、权限和 secret 管理约束。

高风险命令必须人工确认。

## 4.8 小结

Web3 开发的目标不是“能跑就行”，而是让每一次链上执行尽量可测试、可复现、可验证。

# 5. 网络（Network）

## 5.1 一句话解释

Web3 网络是交易能否被打包、状态能否同步、费用如何产生、确认需要多久的基础环境。

## 5.2 人话理解

网络不是背景，而是链上应用真正运行的地方。

它决定：

- 交易能不能发出去；
- 什么时候被打包；
- 手续费是多少；
- 前端状态准不准；
- 钱包是否切到了正确的链；
- L2 和主网之间资产如何流转。

## 5.3 为什么要学

很多问题表面像钱包或前端问题，底层其实是网络问题。

比如：

- 用户切错链；
- RPC 延迟；
- 交易 pending；
- 测试网没有测试币；
- L2 提现等待；
- 区块浏览器和前端状态不一致。

## 5.4 核心逻辑

区块链网络的核心不是简单存数据，而是让互不信任的参与者对状态变化达成一致。

普通应用是服务器直接更新数据库。  
链上系统是交易被节点传播、执行、打包、验证和确认。

这带来公开可验证性，也带来延迟、费用和最终性问题。

## 5.5 关键知识点

### 5.5.1 Block：区块

区块是交易被批量提交和排序的单位。

重点理解三件事：

- 交易有顺序，顺序会影响结果；
- 区块有 gas limit，吞吐不是无限的；
- 新区块引用前一区块，形成可验证历史。

### 5.5.2 Consensus：共识

共识决定哪段链上历史有效。

它影响：

- 交易需要等几个 confirmation 才算安全；
- 区块可能短暂重组；
- 前端不能过早假设最终结果；
- RPC 异常时读取状态可能延迟。

### 5.5.3 PoS

PoS 是 Proof of Stake，权益证明。

以太坊当前使用 PoS。验证者质押 ETH 参与区块提议和证明，行为错误可能被惩罚。

普通用户不一定要运行验证者，但要理解网络安全不是免费的，它来自质押、惩罚、客户端和节点参与。

### 5.5.4 Testnet：测试网

测试网用于在接近真实链的环境里测试合约、前端和交易流程。

它适合验证：

- 部署脚本；
- 钱包连接；
- RPC 配置；
- 合约调用；
- 区块浏览器验证；
- 前端状态处理。

但测试网不能证明主网经济机制一定安全。

### 5.5.5 L2

L2 是在主网安全和更低成本之间做扩展的网络层。

优势：

- 费用更低；
- 确认更快；
- 适合高频交互。

复杂点：

- 桥接；
- 提现等待；
- 排序器；
- 不同 chain id；
- 不同合约地址；
- 不同区块浏览器。

### 5.5.6 Rollup

Rollup 是主流 L2 扩展路线。

它把大量交易放到 L2 执行，再把数据、结果或证明提交回 L1。

Rollup 降低了单笔交易成本，但没有消除链上系统复杂度。

## 5.6 一笔交易在网络中的流程

1. 用户在前端点击按钮；
2. 钱包弹窗确认；
3. 用户签名；
4. 交易通过 RPC 发到网络；
5. 交易进入 mempool；
6. 被打包进区块；
7. 网络确认区块；
8. 区块浏览器显示交易状态；
9. 索引器更新数据；
10. 前端刷新页面。

## 5.7 在 AI x Web3 中的位置

AI Agent 读取链上状态或执行交易时，必须明确网络信息：

- chain id；
- 网络名称；
- RPC 来源；
- 区块高度；
- 交易哈希；
- 确认数；
- explorer 链接；
- 合约地址。

不能靠模型猜当前在哪条链。

## 5.8 小结

每次链上操作都要确认自己在哪条网络上。网络不同，资产、合约、费用、速度和风险都可能不同。

# 6. 账户抽象（Account Abstraction）

## 6.1 一句话解释

账户抽象让钱包账户从“一个私钥控制”，升级为“可编程规则控制”。

## 6.2 人话理解

普通钱包账户通常是 EOA，由一个私钥控制。

它的逻辑很简单：谁有私钥，谁就能控制账户。

账户抽象想让账户变得更像一个智能账户系统，可以设置权限、恢复方式、gas 支付方式和自动执行规则。

## 6.3 为什么要学

普通 EOA 钱包限制明显：

- 用户必须保管私钥；
- 必须用原生代币支付 gas；
- 签名权限很粗；
- 自动化很难；
- 恢复体验不好。

账户抽象可以支持：

- 多签；
- 社交恢复；
- gas sponsorship；
- session key；
- 限额；
- 批量交易；
- 更细粒度权限。

## 6.4 核心逻辑

账户抽象的核心不是“免 gas”，而是把账户控制权从单一私钥扩展成可编程规则。

权限从“有私钥 / 没私钥”变成：

- 在什么条件下；
- 允许谁；
- 执行什么动作；
- 使用多少额度；
- 持续多长时间；
- 是否可撤销和审计。

## 6.5 关键知识点

### 6.5.1 ERC-4337

ERC-4337 是以太坊生态重要的账户抽象标准。

传统钱包直接发送交易。  
ERC-4337 里，用户创建的是 UserOperation。

简化流程：

1. 用户或应用生成 UserOperation；
2. 智能账户验证签名、nonce、余额或策略；
3. Bundler 打包并提交操作；
4. EntryPoint 调用账户执行目标动作；
5. Paymaster 可以选择赞助 gas。

关键角色：

- UserOperation：用户想执行的操作；
- Bundler：收集并提交操作；
- EntryPoint：链上统一入口；
- Smart Account：定义账户规则；
- Paymaster：支付或赞助 gas。

### 6.5.2 Smart Account

Smart Account 是由合约控制的账户。

它可以规定：

- 大额转账需要多签；
- 小额交易自动通过；
- 某些 dApp 可在一定额度内调用；
- 钱包丢失后通过恢复人或设备找回；
- 多个交易批量执行。

风险是智能账户本身也是合约，合约 bug、模块权限、升级逻辑都会成为账户风险。

### 6.5.3 Bundler

Bundler 负责收集 UserOperation，模拟验证并提交到 EntryPoint。

Bundler 不稳定，用户操作可能卡住；模拟不充分，也可能导致失败交易和成本问题。

### 6.5.4 Paymaster

Paymaster 可以帮用户支付 gas，或者让用户用非原生资产承担费用。

常见用途：

- 新用户 onboarding；
- 活动补贴；
- 白名单交易；
- 订阅制产品；
- 应用内 gas 抽象。

但 Paymaster 必须有风控：赞助哪些方法、额度多少、是否限制目标合约、如何防止 spam。

### 6.5.5 Session Key

Session Key 是给应用或 Agent 的临时权限。

它可以被限制为：

- 某段时间有效；
- 只能调用某个合约；
- 只能使用某些方法；
- 只能花费某个额度；
- 只能在特定链上执行。

它不是用户主私钥。

## 6.6 在 AI x Web3 中的位置

账户抽象是 AI Agent 上链执行的重要底座。

AI 不应该拿用户主私钥，也不应该拥有无限交易权限。  
更合理的方式是给 Agent 一个可限制、可过期、可撤销、可审计的行动空间。

## 6.7 小结

账户抽象不是让 AI 或应用无限操作钱包，而是让权限更细、更安全、更可控。

# 7. 去中心化金融（DeFi）

## 7.1 一句话解释

DeFi 是建立在智能合约上的开放金融系统。

## 7.2 人话理解

DeFi 把交易、借贷、稳定币、流动性、收益等金融功能写进智能合约，让它们在链上运行。

比如：

- 在 DEX 上交易 token；
- 存入资产赚利息；
- 抵押资产借出稳定币；
- 给流动性池提供资产赚手续费；
- 组合多个协议生成策略。

## 7.3 为什么要学

很多 AI x Web3 项目都会碰到 DeFi：

- 交易解释；
- 收益策略；
- 风险监控；
- 清算预警；
- 自动再平衡；
- 资产管理；
- 协议数据分析。

即使不做金融产品，也需要理解 token、流动性、价格、借贷和预言机。

## 7.4 核心逻辑

DeFi 的核心不是简单去掉中介，而是把金融规则变成可组合、可验证、也可被攻击的链上协议。

它管理的是资产状态，而不只是应用界面。

透明度提高了，但机制错误、价格冲击和外部依赖也会更快暴露。

## 7.5 关键知识点

### 7.5.1 Token

Token 是 DeFi 的资产单位。

看 token 时不要只看名称和图标，至少检查：

- 合约地址是否正确；
- decimals 是多少；
- 总供应量和发行权限；
- 是否可暂停、冻结、增发或升级；
- 是否有转账税或黑名单逻辑。

### 7.5.2 AMM

AMM 用流动性池和定价公式替代传统订单簿。

用户向池子提供两种资产，交易者直接和池子交换资产。

关键概念：

- 滑点：预期价格和实际成交价格的差距；
- 流动性提供者：提供资产赚手续费，也承担无常损失；
- 价格影响：大额交易会改变池子比例；
- MEV 风险：交易排序可能被利用，比如 sandwich attack。

### 7.5.3 Lending

借贷协议把存款、借款、抵押率、利率和清算规则写进合约。

关注点：

- 抵押品价值；
- 债务价值；
- 清算阈值；
- 健康因子；
- 预言机价格；
- 流动性是否足够。

AI 如果给借贷建议，不能只说收益更高，必须解释抵押率、健康因子、清算价格和价格来源。

### 7.5.4 Stablecoin

稳定币是 DeFi 的计价和结算基础，但“稳定”不是天然保证。

看稳定币时要问：

- 它靠什么维持锚定；
- 谁能发行和销毁；
- 储备或抵押品在哪里；
- 脱锚时谁承担损失；
- 主要池子流动性多深。

### 7.5.5 Liquidity

流动性决定资产能不能以合理价格买入、卖出、借出、清算或退出。

有价格不等于能成交，能成交也不等于低成本成交。

流动性不足会导致：

- swap 滑点变大；
- 清算困难；
- 预言机价格容易被操纵；
- LP 退出导致协议风险上升；
- 策略无法及时撤出。

## 7.6 在 AI x Web3 中的位置

AI 适合先做信息整理和风险辅助：

- 解释交易；
- 总结仓位；
- 监控清算；
- 识别异常价格；
- 生成策略草案。

真正高风险的是自动执行 swap、借贷、加杠杆、授权、跨链和清算。

如果 Agent 要执行 DeFi 操作，至少要有：额度限制、协议白名单、滑点上限、模拟结果、预言机检查、人工确认和交易后审计记录。

## 7.7 小结

DeFi 不是“收益高就好”，而是要同时看资产、价格、流动性、权限、清算和协议依赖风险。

# 8. 预言机（Oracle）

## 8.1 一句话解释

预言机是把链外数据带到链上，让智能合约可以使用外部信息的机制。

## 8.2 人话理解

区块链自己不能天然知道链外世界发生了什么。

它不知道：

- ETH 现在多少钱；
- 今天有没有下雨；
- 某场比赛谁赢了；
- 某个航班是否延误；
- 某个公司是否有真实储备；
- 某个 AI 模型判断结果是什么。

Oracle 的作用就是把这些外部信息，以合约可以使用的方式带到链上。

## 8.3 为什么要学

很多链上协议依赖外部数据：

- 借贷协议需要资产价格；
- 衍生品需要指数价格；
- 保险协议需要事件结果；
- RWA 协议需要储备状态；
- AI x Web3 可能需要模型输出结果。

预言机是桥。桥带来能力，也带来风险。

## 8.4 核心逻辑

预言机不是“真实世界 API”，而是链上合约愿意信任的一套数据提交和验证机制。

当外部数据进入链上，就会成为协议规则的一部分。

比如借贷协议用某个 price feed 判断清算，那么这个 feed 的更新频率、数据源、聚合方式和异常处理，都会影响用户资产安全。

## 8.5 关键知识点

### 8.5.1 Price Feed

Price Feed 是最常见的预言机形式，为合约提供资产价格。

读取价格时至少检查：

- feed 对应哪个资产对；
- decimals 是多少；
- 更新时间是否过旧；
- 返回值是否异常；
- 当前链上的 feed 地址是否正确。

### 8.5.2 Data Feed

Data Feed 不只包括价格，也可以包括：

- 储备证明；
- 利率；
- 宏观数据；
- 体育结果；
- 天气；
- 航班状态；
- 随机数；
- AI 判断结果。

只要合约依赖链外数据，就要问：数据从哪里来、谁能提交、是否可验证、出错后怎么处理。

### 8.5.3 Oracle Risk

预言机风险包括：

- 数据源被操纵；
- feed 更新延迟；
- 聚合节点离线；
- decimals 或单位理解错误；
- 低流动性资产价格被攻击；
- 合约没有检查 stale price；
- 预言机升级或权限变更不透明。

预言机错误可能导致错误清算、坏账、套利和资产池损失。

### 8.5.4 AI Oracle

AI Oracle 是把模型推理、评分、分类或生成结果提交给链上系统。

但 AI 输出不一定是简单客观事实。

设计 AI Oracle 时至少要考虑：

- 输入数据是否可追溯；
- 模型版本和 prompt 是否记录；
- 输出是否可复核；
- 是否允许挑战或仲裁；
- 错误输出会造成什么链上后果。

## 8.6 在 AI x Web3 中的位置

Oracle 是 AI x Web3 的关键桥梁之一。

AI Agent 需要读取链上和链外数据；合约如果要使用 AI 结果，也需要把模型输出转成链上可验证或可争议的数据。

但不要设计成“模型说什么，合约就执行什么”。

高风险场景应引入 human-in-the-loop、挑战期、多人验证或经济惩罚。

## 8.7 小结

预言机不是绝对真实的数据源，而是链上合约选择信任的一套数据提交和验证机制。

# 9. 索引（Indexing）

## 9.1 一句话解释

索引是把公开但难用的链上原始数据，整理成产品、分析工具和 AI Agent 能快速查询的结构化数据。

## 9.2 人话理解

链上数据公开，但公开不等于好用。

链上有大量：

- 区块；
- 交易；
- 合约调用；
- event；
- 地址余额；
- 协议状态。

如果每次都从原始区块里查，会很慢、很麻烦。

索引层把这些数据整理成更适合产品查询的数据。

## 9.3 为什么要学

很多产品功能都依赖索引层：

- 用户历史；
- 排行榜；
- 协议看板；
- 风控监控；
- Agent 上下文；
- 钱包交易记录；
- NFT 订单列表。

索引解决的不是有没有数据，而是能不能按产品需要快速、准确、可回放地查询数据。

## 9.4 核心逻辑

链上是事实来源，索引层是可用数据层。

区块链按区块和交易组织数据，但产品关心的是：

- 这个地址的仓位；
- 这个协议的 TVL；
- 这个订单是否成交；
- 这个 Agent 执行过哪些动作。

索引层负责把底层事实转换成这些查询对象。

## 9.5 关键知识点

### 9.5.1 Event Indexing

Event Indexing 是监听合约日志，把链上动作整理成可查询记录。

例如合约发出：

- Transfer；
- OrderCreated；
- Deposit；
- Withdraw；
- VoteCast。

索引器可以把这些 event 转成数据库表或搜索索引。

设计 event 时要考虑：

- 是否包含关键地址；
- 是否需要 indexed 参数；
- 是否能从 event 还原业务状态；
- 失败交易不会产生成功 event；
- 合约升级后 event 是否兼容。

### 9.5.2 Subgraph

Subgraph 是用声明式方式描述如何索引合约事件，并通过 GraphQL 暴露查询接口。

通常包括：

- 要监听的合约和事件；
- event 到实体的 mapping；
- GraphQL schema。

它适合构建协议数据 API，比如 token、pool、swap、position、vote。

### 9.5.3 RPC

RPC 是应用和节点交互的接口。

它可以用于：

- 读取链状态；
- 查询日志；
- 估算 gas；
- 发送交易。

但 RPC 不是数据库。频繁扫描大量历史日志会遇到限流、节点不同步、archive 数据不可用、WebSocket 不稳定等问题。

### 9.5.4 Data Pipeline

Data Pipeline 把链上数据、链下数据、索引结果和业务事件组合成可分析、可监控、可供 AI 使用的数据系统。

可能包括：

- RPC 或节点数据源；
- event listener；
- ABI 解码；
- 数据库写入；
- reorg 处理；
- 数据校验；
- API / GraphQL；
- dashboard；
- alert；
- Agent context。

## 9.6 为什么索引要能重放

索引系统要能从某个区块重新处理数据。

原因包括：

- 索引逻辑写错；
- 数据库不一致；
- 合约升级；
- event 结构变化；
- 链发生 reorg；
- 需要修复历史数据。

不能重放的索引层，很难修复旧数据。

## 9.7 在 AI x Web3 中的位置

AI Agent 需要上下文，而链上上下文通常来自索引层。

好的索引层应该给 Agent 提供：

- 结构化数据；
- 数据来源；
- 时间戳；
- 区块高度；
- 交易哈希；
- 合约地址；
- 可回溯记录。

模型负责解释和推理，索引层负责提供事实。

## 9.8 小结

区块链数据虽然公开，但只有经过索引整理，才能真正变成产品和 AI Agent 好用的数据。

# 10. 安全（Security）

## 10.1 一句话解释

Web3 安全不是上线前审一次代码，而是从设计、权限、测试、模拟、监控到应急响应的一整套工程流程。

## 10.2 人话理解

Web3 的错误成本比普通应用高。

合约一旦部署，资产可能已经进入协议。攻击者可以直接和公开接口交互，交易结果也通常不能随意回滚。

所以安全不是最后补一层保险，而是从设计开始就要考虑的底层边界。

## 10.3 为什么要学

安全不是只属于 auditor 的工作。

这些都可能成为事故入口：

- 错误授权按钮；
- 无限 approve；
- 权限过大的 owner；
- 未限制的 Agent tool；
- 没有模拟的交易；
- 错误的合约地址；
- 未监控的升级操作。

Web3 安全的核心不是幻想没有 bug，而是把可预见风险尽量挡在执行前，并在执行后快速发现和止损。

## 10.4 核心逻辑

链上系统默认暴露在公开对抗环境里。

代码公开、状态公开、资金公开，攻击者可以反复模拟和抢跑。

所以任何可调用路径，都要按攻击面看待。

## 10.5 三个安全原则

### 10.5.1 权限最小化

owner、admin、upgrade、pause、withdraw 等权限都要有清楚边界。

要问：

- 谁能执行敏感动作；
- 是否需要多签；
- 是否有 timelock；
- 权限变更是否有 event；
- 私钥泄露时最坏结果是什么。

### 10.5.2 执行前模拟

签名前尽量模拟交易。

看清楚：

- 调用哪个合约；
- 会转出哪些 token；
- 会得到什么；
- gas 大概多少；
- 是否 revert；
- 是否触发授权变化。

### 10.5.3 上线后监控

上线后持续观察：

- 大额转账或提款；
- 管理员权限变更；
- 合约升级；
- 预言机价格异常；
- 大量失败交易；
- TVL 快速流出；
- 未预期 event；
- Agent 连续触发高风险工具。

## 10.6 关键知识点

### 10.6.1 Reentrancy：重入攻击

重入攻击是合约在外部调用未完成前被再次调用，导致状态被重复利用。

常见危险模式是：先转账，再更新内部余额。

防护思路：

- 使用 Checks-Effects-Interactions：先检查，再更新状态，最后外部调用；
- 对高风险函数使用 reentrancy guard；
- 避免在状态未更新前调用不可信合约；
- 测试多合约交互。

### 10.6.2 Access Control：权限控制

权限控制决定谁能执行敏感动作。

敏感动作包括：

- mint；
- burn；
- pause；
- upgrade；
- withdraw；
- setOracle；
- setFee；
- setRouter；
- changeOwner。

检查权限时，不只看有没有 `onlyOwner`，还要看 owner 是谁、权限多大、是否可监控。

### 10.6.3 Audit：审计

审计是外部安全审查，不是安全保证书。

读审计报告时要看：

- 覆盖哪些 commit 和合约；
- 哪些问题已修复；
- 哪些风险被项目方接受；
- 是否覆盖经济机制和外部依赖；
- 上线代码是否和审计代码一致。

### 10.6.4 Simulation：交易模拟

交易模拟是在真正签名前预演交易结果。

它可以帮助发现：

- 链 ID 错；
- 合约地址错；
- 授权额度异常；
- 滑点过大；
- 余额不足；
- 调用方法不符合预期。

模拟不能替代审计，但能挡住很多明显错误。

### 10.6.5 Monitoring：监控

监控是上线后的安全感知层。

但有效安全不是只有监控，而是“监控 + 响应”。

要明确：

- 谁收到告警；
- 谁确认误报；
- 谁能 pause；
- 恢复流程是什么；
- 事故如何复盘。

## 10.7 在 AI x Web3 中的位置

AI 会让安全边界更复杂。

Agent 可以解释合约、生成交易、调用工具、执行策略，但也可能：

- 读错上下文；
- 被 prompt injection 影响；
- 调用错误工具；
- 生成危险交易；
- 混淆主网和测试网。

合理设计应该是：

- 模型负责建议；
- 工具返回事实；
- policy 限制权限；
- simulation 预演结果；
- human check 确认高风险动作；
- monitoring 记录执行后果。

## 10.8 小结

Web3 安全的目标不是幻想没有风险，而是把风险尽量挡在执行前，并在事故发生后快速发现、快速止损。

这 10 个概念可以按下面顺序理解：

## **一、为什么 Web3 要懂一点密码学**

传统互联网：忘了密码，点“找回密码”，平台帮你重置。  
Web3 没有这种客服。你的账户就是一把私钥，丢了就永远没了。

所以不是要你变成密码学专家，而是要你**看懂每个操作在证明什么**——是在证明“我是我”，还是在说“我同意转账”，还是在确认“数据没被改过”。这样你才知道什么时候不能乱点确认。

* * *

## **二、Web3 身份的本质**

Web3 的身份不是平台给你的，是**你能不能用私钥证明自己**。  
好处：不需要任何人批准你就可以用。  
坏处：私钥丢了就是丢了，没人能帮你找回。

-   **私钥** = 控制权（谁拿到谁就能控制资产）
    
-   **签名** = 授权证据（你同意了某件事）
    
-   **哈希** = 数据指纹（用来检查数据有没有被改）
    

* * *

## **三、几个核心概念的“人话解释”**

### **1\. 哈希（Hash）—— 数据的指纹**

不管多长的数据，哈希都会变成一个固定长度的“指纹”。  
哪怕只改一个标点符号，指纹也会完全不一样。

> 用途举例：

-   交易哈希：像快递单号，用来查这笔交易
    
-   数据承诺：我先公布指纹，以后再把原始数据给你，你能自己验证我没撒谎
    

⚠️ 哈希不是加密，不能“解密”回原文。

* * *

### **2\. 公钥（Public Key）—— 公开的验证工具**

公钥和私钥是一对。别人用公钥可以验证“这个签名是不是你干的”，但**不能反过来推出你的私钥**。

以太坊里的“地址”就是公钥经过处理后的短号。

> 地址 ≠ 身份，地址只是“我能证明我控制它”的一个代号。

* * *

### **3\. 私钥（Private Key）—— 账户的控制权本体**

这是最敏感的东西。  
私钥 = 你能动这个账户里一切资产的凭证。

> 铁律：

-   不截图
    
-   不上传网盘
    
-   不给任何网页、客服、AI 工具
    
-   不放在代码仓库里
    
-   不要用主钱包去试陌生应用
    

* * *

### **4\. 签名（Signature）—— 你亲口同意的证据**

签名不是“登录”，而是**对具体某条消息或交易的授权**。  
出问题最多的情况：你根本没看懂自己签了什么。

好的产品应该让你看到：

-   签什么目的
    
-   哪个网站
    
-   哪个合约
    
-   转多少钱
    
-   有没有过期时间
    

> 对 AI Agent 来说也一样：不能让 AI 随便替你签名，必须有明确的权限边界。

* * *

### **5\. Merkle Tree（默克尔树）—— 高效证明“我的数据在里面”**

假设有一万个人的空投名单。  
你不需要下载全部名单，只需要一个很小的 **Merkle 证明**，就能向别人证明“我确实在这个名单里”。

> 类似“零知识证明的轻量版”，常用于空投、轻钱包、链上状态验证。

* * *

## **四、和 AI 有什么关系**

AI 如果帮用户操作链上资产，必须明白：

-   什么操作只是查询
    
-   什么操作需要签名授权
    
-   绝不能替用户保管私钥
    
-   不能催用户在不理解的情况下点确认
    

更进一步：AI 的建议本身也可以被签名或哈希，这样以后可以审计——当时 AI 说了什么，用户确认了什么，最终执行了什么。

* * *

## **一句话总结（帮你记住）**

> **私钥 = 你是谁，签名 = 你同意，哈希 = 没被改过，Merkle 树 = 不用全看也能证明。**

你需要的就是：看到每个弹窗，能判断它是在要“身份证明”还是“转账许可”。前者可以小心做，后者必须看懂再做。

\- 可以承载支付、身份、信誉、协议和治理

\- 不能通过人类渠道建立信誉

\### 人话解释

传统互联网和金融系统主要是为人设计的。

人可以记密码、开户、签合同、找法院维权、通过社交关系建立信誉。

但是 AI Agent 是机器，它们需要的是：

\- 自动化的身份系统

\- 自动化的支付系统

\- 可验证的信誉系统

\- 可执行的协议规则

\- 更快的结算方式

这就是为什么 Ethereum 这类开放网络会被认为适合 AI Agent。

\## 7. Ethereum 的核心定位：中立协调层

课件中的核心 thesis 是：

\> Ethereum is the decentralized settlement and coordination backbone for machines in the machine economy, with humans at the center.

可以理解为：

\> Ethereum 是机器经济中的去中心化结算与协调骨干网络，但人类仍然处在中心位置。

也就是说，分享者并不是主张“机器完全取代人类”，而是强调：

\- 机器可以在 Ethereum 上大规模协作

课件中提到，Ethereum 可以成为一个 Neutral Layer，即中立层。

1\. 结算价值

2\. 建立身份、信誉和协议

Agent 的身份、历史行为、信誉记录可以被公开验证。

3\. 接受人类治理

\- 对方是谁

\- 对方是否可信

\- 对方是否完成过任务

\- 对方是否真的付款

\- 协议规则是否公开

\- 出问题时谁可以修改规则

Ethereum 提供的是一个公开、可验证、可组合的基础环境。

\-–

\## 9. 设计原则：CROPS

课件中提出了几个设计原则，缩写为 CROPS：

\### 9.1 Censorship Resistance：抗审查

AI Agent 的协作网络不应该轻易被单一平台、公司或国家中断。

\### 9.2 Open Source and Free：开源与自由

基础协议应该尽可能开放，让更多开发者可以参与建设和审查。

\### 9.3 Privacy：隐私

Agent 的交易、身份和行为不应完全暴露，未来需要隐私支付和隐私验证能力。

\### 9.4 Security：安全

Agent 如果要处理资产、数据和协议，就必须有足够强的安全机制。

\## 10. EIP / ERC-8004：Agent 的身份与信任标准

课件中重点提到：

\> EIP 8004 - Identity and Trust for Agents

它被描述为一个为 AI Agent 提供可验证身份和信誉的标准。

课件中提到它包含三个 registry：

1\. Identity Registry

用于记录 Agent 的身份。

2\. Reputation Registry

用于记录 Agent 的信誉。

3\. Validation Registry

用于记录 Agent 的验证信息。

\### 人话理解

可以把 ERC-8004 理解成给 AI Agent 准备的一套链上“身份与信用系统”。

类似于：

\- Agent 的身份证

\- Agent 的信用记录

\- Agent 的任务验证记录

\- Agent 的可信度证明

\- 让 API 和 Agent 可以声明价格

\- 让 Agent 在链上完成支付证明

\- 让服务调用和付款流程自动化

\### 人话理解

1\. 请求某个服务

2\. 看到这个服务需要付款

3\. 自动完成链上支付

4\. 把支付证明发给服务方

5\. 获得服务结果

这就是“机器到机器支付”的一个典型场景。

## 1\. Intro to AI Agents

### 核心结论

> **AI Agent 是让 LLM 真正「做事」的系统 — 通过赋予工具和知识，让模型不只是回答问题，而是能执行动作。**

### Agent 五大组件

| 组件 | 作用 | 类比 |
| --- | --- | --- |
| LLM（大脑） | 理解自然语言、推理上下文、把模糊请求变成具体计划 | 大脑 |
| Sensors（感知） | 读取环境状态（查酒店空房、机票价格） | 眼睛 |
| Actuators（执行） | 执行动作（订房、发确认、取消预订） | 手 |
| Tools（工具） | 搜索数据库、调 API、发消息 — 取决于环境和开发者配置 | 工具箱 |
| Memory（记忆） | 短期（当前对话）+ 长期（用户偏好、历史交互） | 记忆 |

### Agent 七种类型

| 类型 | 特点 | 旅行 Agent 例子 |
| --- | --- | --- |
| Simple Reflex | 硬编码规则，无记忆无规划 | 看到投诉邮件 → 转给客服 |
| Model-Based Reflex | 维护世界模型，随变化更新 | 跟踪历史票价，标记异常涨价 |
| Goal-Based | 有目标，逐步推理达成路径 | 从当前位置规划完整行程 |
| Utility-Based | 不只找解，找最优解 | 权衡成本 vs 舒适度 |
| Learning | 从反馈中学习改进 | 根据用户评价调整推荐 |
| Hierarchical | 高层拆任务，委派给子 Agent | 取消行程 → 拆为取消机票/酒店/租车 |
| Multi-Agent | 多 Agent 协作或竞争 | 酒店/机票/娱乐分头处理 |

### 什么时候用 Agent？

✅ **应该用 Agent 的场景：**

-   **开放式问题** — 解决步骤无法预先编程，需要 LLM 动态找路径
    
-   **多步骤流程** — 跨多个回合使用工具，不是单次查询
    
-   **持续改进** — 希望系统根据反馈和环境信号越变越好
    

❌ **不应该用 Agent 的场景：**

-   单次问答就能搞定的事
    
-   不需要外部工具或动作的纯文本生成
    

### 我的理解

> 这七种 Agent 类型可以和 Day 1 学的四层架构对应：Simple Reflex ≈ 无编排层，Goal-Based / Utility-Based ≈ 编排层 + 安全层，Multi-Agent ≈ 编排层的进化版。微软的框架区分比 Day 1 更细，但从实用角度，大部分实际场景用的是 Goal-Based + Tool Use 的组合。

## 2\. Agentic Frameworks

### 核心结论

> **Agent 框架提供预制组件和抽象，让开发者专注业务逻辑，而不是每次从头造轮子。**

### 框架解决什么问题？

-   工具接入和编排
    
-   可观测性（Agent 在做什么？出错了怎么调试？）
    
-   多 Agent 协作
    

### 微软生态

| 工具 | 定位 |
| --- | --- |
| Microsoft Agent Framework (MAF) | 统一框架，支持顺序/并发/群聊编排 |
| Azure AI Foundry Agent Service | 托管平台，支持 OpenAI/Mistral/Meta 等模型 |

### 我的理解

> 框架选择的核心不是功能对比，而是你需要在哪个生态里干活。框架帮你搞定基础设施（工具接入、观测、编排），你的精力应该放在 Agent 的业务逻辑和安全边界上。

## 3\. Agentic Design Patterns

### 核心结论

> **设计 Agent 不是设计架构，而是设计用户体验。三个核心原则：透明 (Transparency)、可控 (Control)、一致 (Consistency)。**

### 三大设计原则

| 原则 | 含义 | 实践 |
| --- | --- | --- |
| Transparency | 让用户知道这是 AI Agent | Hello 消息、示例提示、清楚标注能力边界 |
| Control | 用户能修改 Agent 行为 | 可调整详细程度、写作风格、查看/删除历史 |
| Consistency | 交互模式统一可预测 | 标准图标（纸夹=上传文件）、一致的操作逻辑 |

### Agent 应该做什么？

-   扩展人类能力（头脑风暴、问题解决、自动化）
    
-   填补知识空白（快速了解新领域、翻译）
    
-   支持协作（按个人偏好的方式工作）
    
-   让我们成为更好的自己（生活教练、情绪调节）
    

* * *

## 4\. Tool Use

### 核心结论

> **Tool Use 是从「会回答」到「能做事」的分水岭。LLM 返回的是工具调用请求，不是最终答案 — Agent 系统负责执行工具并把结果喂回给模型。**

### Tool Use 工作流

```
用户请求 → LLM 选择工具+参数 → Agent 执行工具 → 结果返回 LLM → 生成最终回答
```

### 关键细节

-   LLM 返回 `tool_call`（函数名 + 参数），**不是**最终答案
    
-   工具定义用 JSON Schema：函数名、描述、参数类型和必填项
    
-   `tool_choice: "auto"` 让模型自己决定用不用工具
    

### 与 Day 1 的呼应

> Day 1 学了「Tool Use 是分水岭，能调用工具的 Agent 风险也升级」。这节课补充了具体实现：函数 Schema 定义、tool\_call 返回格式、「LLM 返回工具请求 → 系统执行 → 喂回结果」的循环。

* * *

## 5\. Agentic RAG

### 核心结论

> **Agentic RAG ≠ 静态的「检索→回答」。它是 LLM 自主规划下一步 + 迭代调用工具的循环：评估结果 → 优化查询 → 调用新工具 → 循环直到满意。**

### 对比

|   | 传统 RAG | Agentic RAG |
| --- | --- | --- |
| 流程 | 检索 → 生成（一次） | 检索 → 评估 → 优化 → 再检索（循环） |
| 决策 | 预设检索策略 | LLM 动态决定下一步 |
| 适用 | 简单问答 | 复杂数据库交互、长工作流 |
| 纠错 | 无 | 自我纠错、处理异常查询 |

### 核心模式：Maker-Checker

> 迭代循环：LLM 调用工具 → 获取结果 → 检查质量 → 决定是结束还是再调一轮。

* * *

## 6\. Building Trustworthy Agents

### 核心结论

> **Agent 的安全不只是「别做坏事」，而是「做对了该做的事」。三层防御：System Message 框架 → 安全措施 → 隐私保护。**

### System Message 三步法

1.  **Meta System Message** — 让 LLM 生成 Agent 的 system prompt 模板
    
2.  **Basic Prompt** — 定义 Agent 的角色、任务、职责
    
3.  **LLM 优化** — 用 meta prompt 优化 basic prompt，产出结构化的 system message
    

### 安全与隐私

-   识别和缓解 Agent 风险
    
-   数据访问权限管理
    
-   用户隐私保护（不只是合规，是体验质量）
    

### 我的理解

> 这和 Day 1 的 Prompt Injection 五层防护一脉相承。微软补充了 System Message 框架的具体操作：不是手写 system prompt，而是用 LLM 生成+优化 system prompt。这是一个「用 AI 管理 AI」的思路。

## 7\. Planning Design

### 核心结论

> **复杂任务不能一步完成。Planning = 定义总目标 → 拆解子任务 → 分配工具 → 评估结果 → 迭代改进。**

### Planning 流程

```
总目标 → 拆解子任务 → 编排工具 → 评估结果 → 迭代
```

### 关键技术

| 技术 | 作用 |
| --- | --- |
| Structured Output | 让 Agent 输出机器可读的 JSON（含 main_task + subtasks 列表） |
| Event-Driven | 处理动态任务和意外输入 |
| Agent Registry | 维护可用 Agent 列表及各自工具，路由子任务 |

### 例子：旅行规划

```json
{
  "main_task": "Plan family trip from Singapore to Melbourne",
  "subtasks": [
    {"assigned_agent": "flight_booking", "task_details": "Book round-trip flights"},
    {"assigned_agent": "hotel_booking", "task_details": "Find family-friendly hotels"}
  ]
}
```

## 8\. Multi-Agent

### 核心结论

> **Multi-Agent 是多 Agent 协作完成共同目标的模式。不是任何时候都需要 — 当单一 Agent 的复杂度超过阈值时才拆。**

### 什么时候用 Multi-Agent？

-   任务涉及不同领域需要不同「专家」
    
-   任务天然可并行
    
-   需要竞争/博弈机制（如竞价）
    

### 三种协作模式

| 模式 | 描述 | 场景 |
| --- | --- | --- |
| Group Chat | 多 Agent 群聊通信 | 团队协作、客服 |
| Hand-off | Agent 之间交接任务 | 工作流自动化 |
| Collaborative Filtering | 多 Agent 各有所长，协作推荐 | 股票分析（行业/技术/基本面） |

### 我的理解

> Multi-Agent 不是银弹。每增加一个 Agent，就增加通信开销和不可预测性。Day 1 的 Hierarchical Agents 其实就是 Hand-off 模式的变体。关键问题是：增加 Agent 带来的收益是否超过复杂度成本？

## 9\. Metacognition

### 核心结论

> **Metacognition = 「思考关于思考」。Agent 不只是执行任务，还要监控、评估、调整自己的行为策略。**

### 四大能力

| 能力 | 含义 | 例子 |
| --- | --- | --- |
| Self-Reflection | 评估自己的表现 | 「上次推荐错了，因为过度依赖历史偏好」 |
| Adaptability | 根据经验调整策略 | 用户说过不喜欢拥挤 → 以后避开热门景点 |
| Error Correction | 自主发现并修正错误 | 推荐了满房酒店 → 下次先检查可用性 |
| Resource Management | 优化时间/算力 | 缓存常用的航班查询 |

### 真 Metacognition vs 假 Metacognition

> **真 metacognition**：Agent 明确推理自己的推理过程。  
> 例子：「我优先推荐了便宜航班，因为…我可能漏掉了直飞选项，让我重新检查。」
> 
> **假 metacognition**：只是加了反馈循环，没有真正的自我审视。

### Agent 三要素

| 要素 | 含义 |
| --- | --- |
| Persona | Agent 的个性和交互风格 |
| Tools | 能执行的能力和功能 |
| Skills | 拥有的知识和专长 |

### 我的理解

> Metacognition 是 Agent 从「工具」进化到「同事」的分界线。一个会自我反思的 Agent 不只是执行指令，而是能说「我觉得我上次的做法有问题，这次换个方式」。这和 Day 1 的核心心法一致：「AI 只能开脑洞，不能下定论」— 但加上 metacognition，Agent 至少可以自己质疑自己的开脑洞。

## 10\. AI Agents in Production

### 核心结论

> **Agent 从原型到生产的关键：可观测性 (Observability) 和系统性评估 (Evaluation)。黑盒 Agent 不可维护。**

### 核心概念

| 概念 | 含义 |
| --- | --- |
| Trace | 一个完整 Agent 任务的全流程（如处理一个用户查询） |
| Span | Trace 中的单个步骤（如一次 LLM 调用、一次工具调用） |
| Observability | 用 Langfuse / Microsoft Foundry 可视化 Traces 和 Spans |
| Evaluation | 系统评估 Agent 的输出质量和行为正确性 |

### 关注点

-   性能优化
    
-   成本控制
    
-   系统性评估指标
    
-   Agent 行为可解释性

## 11\. Agentic Protocols (MCP / A2A / NLWeb)

### 核心结论

> **三个协议解决三个不同层面的标准化问题。**

| 协议 | 全称 | 解决的问题 |
| --- | --- | --- |
| MCP | Model Context Protocol | LLM 怎么统一接入外部工具和数据？ |
| A2A | Agent-to-Agent | 不同 Agent 之间怎么通信协作？ |
| NLWeb | Natural Language Web | 网站怎么用自然语言暴露接口给 Agent？ |

### MCP 核心价值

> 提供「通用适配器」：不同数据源和工具通过统一标准被 Agent 调用，而不是每种工具写一个桥接代码。

## 12\. Context Engineering

### 核心结论

> **Context Engineering ≠ Prompt Engineering。Prompt 告诉模型「怎么做」，Context 决定模型「依据什么做」。Context Engineering 的核心是信息治理。**

### 四大策略

| 策略 | 含义 |
| --- | --- |
| Write | 写清楚上下文，标注来源可信度 |
| Select | 选择放什么进上下文窗口 |
| Compress | 压缩冗余信息，保留关键信号 |
| Isolate | 隔离不同类型信息（指令/事实/知识/记忆） |

### 四种 Context 失败模式

| 失败模式 | 症状 |
| --- | --- |
| Poisoning | 外部内容被当成系统指令 |
| Distraction | 无关信息太多，关键信息被淹没 |
| Confusion | 来源不清，模型引用了过期信息 |
| Clash | 矛盾信息共存，模型无法判断优先 |

### 与 Day 1 的呼应

> 这节完美呼应了 Day 1 的 Context 五层模型（指令/任务/事实/知识/记忆）。微软的补充在于：不只讲「分什么层」，还讲了「怎么分」— Write/Select/Compress/Isolate 是操作层面，四种失败模式是诊断工具。

* * *

## 13\. Agent Memory

### 核心结论

> **Memory 是 Agent 自我改进的基础。没有记忆，Agent 每次都从零开始。**

### 记忆类型

| 类型 | 含义 | 存储方式 |
| --- | --- | --- |
| Short-term | 当前会话上下文 | 上下文窗口 |
| Long-term | 跨会话的用户偏好和历史 | Mem0 / Azure AI Search / Cognee |
| Structured Memory | 知识图谱形式的结构化记忆 | Cognee（自动构建 KG + 向量检索） |

### Memory ≠ 授权

> 和 Day 1 一致的结论：Memory 提升体验，但不能替代实时授权。涉及身份、权限、资产的记忆必须重新确认。

* * *

## 15\. Browser Use (CUA)

### 核心结论

> **Computer Use Agent 像人一样操作浏览器：打开页面、查看内容、点击操作。适用于 API 无法覆盖的场景。**

### 技术栈

-   **Browser-Use** — AI 驱动的浏览器导航
    
-   **Playwright + CDP** — 浏览器控制和生命周期管理
    
-   **Azure OpenAI Vision** — 视觉理解页面内容
    
-   **Pydantic** — 结构化提取页面数据
    

### 三种模式

| 模式 | 适用场景 |
| --- | --- |
| Agent-first | 页面结构不可预测，需要 AI 决策每一步 |
| Actor-first | 操作流程可预测，用固定脚本 |
| Hybrid | 部分可预测 + 部分需要 AI 决策 |

* * *

## 18\. Securing AI Agents

### 核心结论

> **加密收据 (Cryptographic Receipts) 为 Agent 的每一次工具调用提供不可篡改的审计追踪。**

### 加密收据能证明什么？

| 能证明 | 不能证明 |
| --- | --- |
| ✅ 谁发起了动作（归属） | ❌ 动作本身是否正确 |
| ✅ 内容未被篡改（完整性） | ❌ 策略本身是否合理 |
| ✅ 调用顺序（时间链） |   |

### 核心技术

-   **Ed25519 签名** — 对规范 JSON 做签名
    
-   **离线验证** — 只需公钥即可验证
    
-   **Hash Chain** — 链式收据，删除/重排任何一条都会断裂

## 1\. LLM 基础原理

### 核心结论

> **LLM 是 AI 应用的能力底座，但它不是数据库，无法获取外界最新信息，无法做事实校验。LLM 生成的是概率上合理的输出，不是天然可信的事实。**

### 关键知识点

-   LLM 的输出是**候选结果**，不是事实本身。本质上是在猜"最可能、最合理的下一句话是什么"
    
-   模型能力是**推理入口**，不是最终验证
    
-   **人话版**：模型负责推理，也很擅长推理，但是最终确认定稿还需要人类介入
    
-   **实践原则**：把输出变成可检查对象 — 摘要、分类、计划、操作建议都应该落成 schema、参数、引用或日志，不能只停留在自然语言
    
-   输出"听起来对"的话是无法被程序拆解与验证的。需要让 LLM 输出系统和规则能看得懂的结构性结果（schema、参数、引用、日志）
    

### 我的理解

> 用 Codex 生成原型图时，有些光声的知识是不对的要剔除，但脑洞很不错。模型可以帮你开脑洞，你自己要有判断力去做筛选。

* * *

## 2\. Embedding & Token

### 核心结论

> **Embedding 是把文字变成数字坐标。意思越像，数字离得越近；意思差得远，数字离得越远。但它只能看语义相似程度，无法判定文字的正确性。**

### 关键知识点

| 概念 | 解释 |
| --- | --- |
| Token | 文本的最小处理单元（不一定是完整单词） |
| Embedding 本质 | 把文字/代码转成一长串浮点数列表（向量） |
| 核心逻辑 | 向量距离越近 → 语义越相似；距离越远 → 语义越无关 |
| 输出形式 | 如 [-0.0069, -0.0053, ...] 这样的一长串数字 |

### Embedding 七大主流用途（OpenAI 官方）

1.  **语义搜索** — 按意思匹配，不是只匹配关键词
    
2.  **文本聚类** — 意思相近的文章自动分组
    
3.  **内容推荐** — 推荐相似文章、商品
    
4.  **异常检测** — 找出格格不入的内容
    
5.  **多样性分析** — 判断一批内容是否同质化
    
6.  **文本分类** — 自动打标签
    
7.  **代码搜索、情感分析、评分预测、零样本分类**
    

### 参考

-   [OpenAI Embeddings 官方文档](https://developers.openai.com/api/docs/guides/embeddings)

## 3\. Transformer & Attention

### 核心结论

> **Transformer 是当前主流 LLM 使用的框架，相当于人类的骨架，是 LLM 的核心底层结构。**

### Attention 机制

-   来源论文：_Attention Is All You Need_
    
-   **Attention 做什么**：自动盯着前文中有用的内容看
    
    -   写文章后半段 → 看前半段有没有呼应
        
    -   看代码 → 把整体逻辑串起来
        
-   **本质**：Attention 会找规律、找逻辑、拼接上下文，但**没有自己的知识库**
    

### 优缺点

| 优点 | 缺点 |
| --- | --- |
| 能串通所有人类提供的资料和需求，通顺表达 | 一旦关键信息缺失就开始瞎编 |
|   | 无法分辨对错，没有判断能力 |

## 4\. Hallucination 幻觉

### 核心结论

> **LLM 返回的结果看着说得特别通顺合理，实际全是假的、不存在的、不对的。**

### 危害

在工作中使用 LLM，如果提供的都是错的，会出大问题。

### 五层防御方案

| # | 方案 | 说明 |
| --- | --- | --- |
| 1 | 标注资料来源 | 让 AI 回答必须标注来源 |
| 2 | 固定格式校验 | 用固定格式严格校验输出内容 |
| 3 | 硬性规则核对 | 定死硬性规则 |
| 4 | 人工复核 | 重要内容必须人工确认 |
| 5 | 全程留记录 | 方便查错追溯 |

## 5\. Multimodal 多模态

### 核心结论

> **LLM 大语言模型喂的是文字。多模态 AI 啥都能看懂：文字、图片、声音、视频、截图全都能处理。**

### 优缺点

| 优点 | 缺点 |
| --- | --- |
| 满足日常办公中处理多模态数据的场景 | 准确性局限，无法识别或不支持的数据就出现幻觉 |

### 我的理解

> **AI 只能开脑洞，不能下定论。** 这句话贯穿整个学习过程，是 Day 1 最核心的认知。

## 6\. AI x Web3 四层架构

### 核心结论

> **LLM 在 AI x Web3 中的位置：处于"动脑、理解、出主意"，但不负责动手。**

### 四层架构图

```
┌─────────────────────────────────────────┐
│  4. 安全层（把关风控）                      │
│  风险拦截、交易模拟、权限管控、人工审核、留痕  │
├─────────────────────────────────────────┤
│  3. 执行层（动手干活）                      │

| 安全层 | 全程防犯错、防出事 | 风险拦截、交易模拟、权限管控、人工审核、留痕记录 |

## 7\. Prompt Engineering

### 核心结论

> **Prompt 不只是随口提问，是与 AI 定好的正式沟通规矩。作用：将模糊任务拆成模型可以稳定执行的工作说明。**
> 
> **好的 prompt 不是让模型更自信，而是让模型在合适的时候停下来。**

### Prompt 五大要素

| 要素 | 作用 |
| --- | --- |
| 任务目标 | 要完成什么 |
| 输入边界 | 输入什么、不输入什么 |
| 输出格式 | 输出的结构和约束 |
| 不确定处理 | 遇到不确定的事怎么处理 |
| 安全规则 | 什么不能做 |

### 三个核心组件

1\. Instruction（任务规则）

-   角色 Role
    
-   要完成的任务
    
-   不能做的事情
    
-   不确定事情的处理方式
    
-   最终输出是什么样的
    

2\. Few-Shot（少量举例）

-   示意模型应该怎么做
    
-   让模型看着例子学规则
    

3\. Structured Output（结构化输出）

-   返回值是可验证的、符合要求结构的结果
    
-   更倾向于 JSON、Schema
    
-   **结构化输出不是为了好看，而是为了让后续代码能检查、拒绝、记录和回归测试**
    

### Prompt Injection 提示词注入攻击

核心结论

> **通过输入内容诱导模型忽略原本规则，执行危险行为。一句话"不要被攻击"无法抵挡 Prompt Injection。**

攻击来源

用户输入、网页、文档、邮件、检索结果、外部数据库等

危害

Agent 被注入后可读取私有信息、调用工具等高危行为

五层防护

| # | 防护措施 |
| --- | --- |
| 1 | 把外部内容标记为不可信数据 |
| 2 | 工具调用前做参数校验 |
| 3 | 敏感动作必须经过 allowlist |
| 4 | 高风险操作必须 human approval |
| 5 | 不把密钥、主权限、不可撤销动作交给模型 |

### 写出好 Prompt 的标准模板（GPT 方法论）

```
Role 角色 → Task 任务 → Context 背景信息 → Input 输入信息
→ Constraints 约束条件 → Safety Rules 安全规则
→ Output Format 输出格式 → Examples 示例
→ Failure Handling 失败/不确定时如何处理
```

### 工具推荐

-   GPT Prompt 生成器：[https://chatgpt.com/g/g-NggrHHU0O-prompt-engineer-assistant（能力中等）](https://chatgpt.com/g/g-NggrHHU0O-prompt-engineer-assistant（能力中等）)
    

### Prompt 在 AI x Web3 中的六步流水线

```
Prompt 定义任务和输出格式
  → Context 提供可信数据和来源边界
    → Model 生成解释或候选动作
      → Code 校验 schema 和业务规则
        → Guard / Simulation 检查链上影响
          → Human Check 确认高风险动作
```

## 8\. Context 工程（上下文工程）

### 核心结论

> **Context 是模型这一次能看到、能使用、并据此做出判断的信息空间。很多时候不是模型不聪明，是 Context 设计混乱。**

### Context 包含什么

-   当前会话内容
    
-   系统规则与开发规则
    
-   长期记忆（用户偏好、项目背景等）
    
-   调用的工具
    

### Context 设计三大原则

| 原则 | 说明 |
| --- | --- |
| 可信来源要显式标注 | 系统状态、用户输入、检索文档、工具结果要分区放置，不能混成一段文本 |
| 上下文要可刷新 | 状态、配置、权限、价格、版本、任务进度等信息可能会变，不能长期缓存 |
| 记忆要可撤销 | 用户偏好和历史任务不能变成隐藏权限，也不能变成永久身份假设 |

### Context Window（上下文窗口）

> **模型一次请求中能处理的最大上下文长度。窗口越大，能放的内容越多。**

长上下文的常见问题

-   模型看见了很多内容，但没有抓住重点
    
-   无关信息太多，干扰判断
    
-   模型引用了旧文档或过期段落
    
-   关键信息被淹没在大量文本中
    

正确使用方式

> **Context Window 不是越大越好。** 实际产品中要配合：
> 
> -   检索（RAG）
>     
> -   摘要
>     
> -   结构化数据
>     
> -   来源标注
>     
> -   可信度分层
>     

> **Context Window 决定模型最多能看多少，但 Context 设计决定模型能不能看对、用对。**

### Context Engineering 本质

> 不是简单地把资料全部塞给模型，而是要决定：

-   哪些数据源可以用
    
-   哪些信息应该排在前面
    
-   哪些内容需要裁剪
    
-   哪些内容必须标注来源
    
-   哪些外部内容是不可信的
    
-   哪些状态必须每次重新查询
    
-   哪些信息可以作为历史参考
    

> **Context Engineering 的核心是信息治理：让模型知道什么是规则、什么是事实、什么是参考、什么是不可信内容。**

### 一个稳定 Agent 的 Context 应该包含

-   当前任务状态
    
-   用户原始意图
    
-   工具返回结果
    
-   相关日志和证据
    
-   可信数据来源
    
-   外部检查结果
    
-   系统禁止事项
    
-   输出 schema
    

### AI x Web3 场景的 Context 五层模型

```
┌──────────────────────────────────────┐
│  1. 指令层 — 必须遵守什么              │
│  系统规则、工具规则、禁止事项、安全边界、输出格式 │
├──────────────────────────────────────┤
│  2. 任务层 — 这次要做什么              │
│  用户目标、当前网络、任务参数和范围       │
├──────────────────────────────────────┤
│  3. 事实层 — 依据什么事实判断           │
│  链上查询结果、余额、合约状态、             │
│  工具返回、simulation、gas/slippage     │
├──────────────────────────────────────┤
│  4. 知识层 — 背景知识（可信度较低）        │
│  协议文档、SDK 文档、EIP/标准、历史案例     │
├──────────────────────────────────────┤
│  5. 记忆层 — 长期信息（不替代授权）        │
│  用户偏好、常用项目配置、历史分析习惯       │
└──────────────────────────────────────┘
```

### 为什么要分层？

| 目的 | 说明 |
| --- | --- |
| 权限控制 | 模型知道哪些信息只能参考，哪些可用于判断，哪些动作必须人工确认 |
| Prompt Injection 防护 | 外部文档、网页、论坛内容不会被误当成系统指令 |
| 审计与追溯 | 可追踪模型依据的是链上事实、工具结果、文档还是历史记忆 |

## 9\. Memory & Knowledge Base

### Memory（跨请求保留的信息）

核心结论

> **Memory 是让 Agent 更顺手，不需要用户每次都重新说明背景。但 Memory 不能替代实时授权 — 所有涉及身份、权限、资产或外部副作用的记忆，都必须重新绑定当前会话和当前授权。**

Memory 包含

用户偏好、历史任务、常用钱包、项目配置、上次分析结果、常用输出格式

关键原则

> **Memory 可以提升体验，但不能变成隐藏权限，更不能替代用户的实时确认。**

* * *

### Knowledge Base（外部知识库）

核心结论

> **Knowledge Base 是系统可以检索的外部知识集合，作用是解决模型知识过期的问题。但 Knowledge Base 不是天然可靠的，需要定期维护。否则 RAG 只会把旧的、错的、不适用的信息更稳定地喂给模型。**

关键原则

> **Knowledge Base 可以补充模型知识，但必须管理版本、来源和有效性，否则知识库本身也会变成错误来源。**

* * *

### Prompt vs Context 的一句话区分

> **Prompt 告诉模型怎么做，Context 决定模型依据什么做。**

* * *

## 10\. RAG 检索增强生成

### 核心结论

> **RAG 是让模型基于外部证据回答问题的一套机制，不只是接一个向量数据库。**
> 
> **RAG 解决两个问题：模型知识会过期、Context Window 有限制。**

### RAG 四步流程

```
1. 检索相关资料（从知识库中取回相关材料）
  → 2. 筛选证据
    → 3. 把证据交给模型
      → 4. 让模型基于证据回答
```

### 关键词拆解

| 词 | 含义 |
| --- | --- |
| Retrieval（检索） | 在用户提问时，从知识库里取回相关材料 |
| Augmented（增强） | 把材料注入 Context |
| Generation（生成） | 让模型基于这些材料回答，而不是只靠模型记忆 |

# Prompt

任务目标、输入边界、输出格式、不确定处理、安全规则

**好的 prompt 不是让模型更自信，而是让模型在合适的时候停下来。**

prompt中的指令层要讲清楚规则、目标；输出层要让机器可验证；遇到高风险问题要停下来。

### instruction 任务规则，

```

### Few-Shot 少量举例，示意模型应该怎么做。让模型看着例子学规则。

### Structured Output 结构化输出

```
就是返回值是可验证的，符合我要求结构的结果。
我更倾向于JSON、Schema
结构化输出不是为了好看，而是为了让后续代码能检查、拒绝、记录和回归测试。
```

### Prompt injection 提示词注入攻击

通过输入内容诱导模型忽略原本规则，执行危险行为，

# Context 上下文

定义：是模型这一次能看到、能使用、并据此做出判断的**信息空间**  
上下文包含：当前会话内容，系统规则与开发规则、长期记忆（用户偏好、项目背景等）  
、调用的工具。  
Context 不是简单地把很多资料塞给模型，而是要把不同类型的信息分清楚：  
\- 哪些是系统规则；  
\- 哪些是用户目标；  
\- 哪些是历史状态；  
\- 哪些是工具返回结果；  
\- 哪些是外部文档或网页内容；  
\- 哪些信息可信，哪些信息只是参考。

LLM 本身不会自动知道你的数据库、文件、API、项目文档、用户历史或链上状态。  
它只能根据当前上下文回答问题。  
所以，如果上下文：  
\- 缺失；  
\- 过期；  
\- 来源不清；  
\- 可信度不明；  
\- 权限边界混乱；  
模型就很容易给出错误答案。  
很多是够不是模型不聪明，是Context设计混乱。

### Context 设计关键原则

Context 设计要注意三点：

1.  **可信来源要显式标注**  
    系统状态、用户输入、检索文档、工具结果要分区放置，不能混成一段文本。
    
2.  **上下文要可刷新**  
    状态、配置、权限、价格、版本、任务进度等信息可能会变，不能长期缓存后继续使用。
    
3.  **记忆要可撤销**  
    用户偏好和历史任务可以帮助模型更懂用户，但不能变成隐藏权限，也不能变成永久身份假设。  
    Prompt 告诉模型怎么做，Context 决定模型依据什么做；
    

### Context Window：模型一次能看的最大范围

模型一次请求中能处理的最大上下文长度，窗口越大，能放进去的内容越多  
长上下文常见问题是：

-   模型看见了很多内容，但没有抓住重点；
    
-   无关信息太多，干扰判断；
    
-   模型引用了旧文档或过期段落；
    
-   关键信息被淹没在大量文本中。
    

所以 Context Window 不是越大越好。实际产品里，要配合：

-   检索；
    
-   摘要；
    
-   结构化数据；
    
-   来源标注；
    
-   可信度分层。  
    Context Window 决定模型最多能看多少，但 Context 设计决定模型能不能看对、用对
    

### Context Engineering：上下文工程

它不是简单地把资料全部塞给模型，而是要决定：

-   哪些数据源可以用；
    
-   哪些信息应该排在前面；
    
-   哪些内容需要裁剪；
    
-   哪些内容必须标注来源；
    
-   哪些外部内容是不可信的；
    
-   哪些状态必须每次重新查询；
    
-   哪些信息可以作为历史参考。
    

一个稳定的工具型 Agent，上下文通常应该包括：

-   当前任务状态；
    
-   用户原始意图；
    
-   工具返回结果；
    
-   相关日志和证据；
    
-   可信数据来源；
    
-   外部检查结果；
    
-   系统禁止事项；
    
-   输出 schema。
    

Context Engineering 的核心是信息治理：让模型知道什么是规则、什么是事实、什么是参考、什么是不可信内容。

### Memory：跨请求保留的信息、记忆

Memory 是模型或系统在不同请求之间保留下来的信息。  
\- 用户偏好；- 历史任务；- 常用钱包；- 项目配置；- 上次分析结果；- 常用输出格式。  
\- 优点：是让 Agent 更顺手，不需要用户每次都重新说明背景。

Memory 不能替代实时授权。**所有涉及身份、权限、资产或外部副作用的记忆，都必须重新绑定当前会话和当前授权。**

Memory 可以提升体验，但不能变成隐藏权限，更不能替代用户的实时确认。

## Knowledge Base：外部知识库

Knowledge Base 是系统可以检索的外部知识集合。作用是解决模型知识过期的问题。  
Knowledge Base 不是天然可靠的，需要定期维护。否则 RAG 只会把旧的、错的、不适用的信息更稳定地喂给模型。  
**Knowledge Base 可以补充模型知识，但必须管理版本、来源和有效性，否则知识库本身也会变成错误来源。**

### Sumup 涉及较多的Web3知识，需要消化

Context 在 AI × Web3 里，是模型连接真实链上世界的入口。  
AI 本身负责推理、解释和生成判断；Web3 系统负责提供真实状态、资产变化和最终结算。

### 一个靠谱的 Agent Context 应该分层

在 AI × Web3 场景里，Context 最好不要混在一起，而是按层级管理。

1\. 指令层

这一层放最高优先级规则，例如：

-   系统规则；
    
-   工具使用规则；
    
-   禁止事项；
    
-   安全边界；
    
-   输出格式要求。
    

它决定模型**必须遵守什么**。

2\. 任务层

这一层放本次用户的具体目标，例如：

-   用户想分析哪笔交易；
    
-   用户想查询哪个钱包；
    
-   当前网络是 Ethereum、Base、Arbitrum 还是 Solana；
    
-   本次任务的参数和范围。
    

它决定模型**这次要做什么**。

3\. 事实层

这一层放最关键的真实状态，例如：

-   链上查询结果；
    
-   钱包余额；
    
-   合约状态；
    
-   工具返回结果；
    
-   transaction simulation；
    
-   gas、slippage、approval 变化；
    
-   实际资产流入流出。
    

它决定模型**依据什么事实判断**。

4\. 知识层

这一层放背景知识，例如：

-   协议文档；
    
-   SDK 文档；
    
-   EIP / 标准；
    
-   官方说明；
    
-   论坛讨论；
    
-   历史案例；
    
-   风险知识库。
    

它帮助模型理解背景，但可信度通常低于实时工具结果。

5\. 记忆层

这一层放用户或项目的长期信息，例如：

-   用户偏好；
    
-   常用项目配置；
    
-   常用网络；
    
-   常见输出格式；
    
-   历史分析习惯。
    

但记忆层不能替代当前授权。尤其是钱包、资产、交易、权限相关内容，必须重新确认。

### 为什么要分层

Context 分层之后，系统才更容易做三件事：

1.  **权限控制**  
    模型知道哪些信息只能参考，哪些信息可以用于判断，哪些动作必须人工确认。
    
2.  **Prompt Injection 防护**  
    外部文档、网页、论坛内容不会被误当成系统指令。
    
3.  **审计与追溯**  
    当模型给出一个判断时，可以追踪它依据的是链上事实、工具结果、文档，还是历史记忆。
    

## [最小实践](https://aiweb3.school/zh/handbook/ai/context/#%E6%9C%80%E5%B0%8F%E5%AE%9E%E8%B7%B5)

为“钱包授权检查 Agent”设计一份 context spec。

选择一个场景：用户问“这个 dApp 要我 approve，可以签吗？”你需要列出模型回答前必须拿到的上下文：

-   chain id 和当前区块
    
-   token 合约、spender 地址、approve 数量
    
-   用户当前 allowance 和余额
    
-   spender 是否在可信列表
    
-   simulation 或静态检查结果
    
-   dApp 页面提供的说明，标记为不可信外部内容
    
-   用户本次意图
    

再写清楚哪些字段必须实时查询，哪些可以来自缓存，哪些不能被模型当成事实。
