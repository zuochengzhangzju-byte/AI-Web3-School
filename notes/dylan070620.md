\## 🧠 权限策略基础

今天最重要的一点是：agent 的链上权限不能一上来就给满。

我把 agent 链上动作分成几层：

| 层级 | 例子 | 风险 |

| --- | --- | --- |

| 只读层 | 查询 tx、receipt、余额、合约状态 | 低 |

| 准备层 | 生成 proof、解释交易、准备 calldata | 中低 |

| 低风险执行层 | 小额、白名单、测试网、限额内操作 | 中 |

| 高风险执行层 | approve、transfer、swap、mint、合约写入 | 高 |

| 权限变更层 | 改 owner、改 signer、提高限额、升级合约 | 极高 |

当前的 `Bootcamp Proof Generator` 只在 **只读层 + 准备层**，所以安全边界比较清楚：

\- 不连接钱包

\- 不接触私钥

\- 不签名

\- 不发交易

\- 不修改链上状态

如果未来真的要让 agent 执行链上动作，不能直接给主私钥，而应该考虑：

\- session key

\- spending limit

\- contract allowlist

\- function allowlist

\- human-in-the-loop

\- multi-sig / smart account

\## 🔁 Multi-Agent Handoff

今天我还记录了一个多 agent handoff 设计。

初步拆成这些角色：

| Agent | 负责什么 |

| --- | --- |

| Tx Reader Agent | 读取 transaction / receipt / logs |

| Tx Explainer Agent | 把链上字段翻译成大白话 |

| Proof Writer Agent | 生成 bootcamp / README / WCB proof |

| Risk Checker Agent | 检查解释是否越界、是否包含敏感信息 |

| Human Review | 最终确认和提交 |

我的原则是：

\> 多 agent 可以自动协作处理信息，但不能自动跨过权限边界。

也就是说，从读取数据到解释、生成 proof、风险检查，这些可以自动 handoff；但只要涉及签名、授权、转账、合约写入、提交 WCB 或修改权限，就必须停下来让人确认。

\## 💡 今天学到的点

\- EOA 简单直接，但风险高度集中在私钥和助记词。

\- 智能账户的重点不是“更高级的钱包”，而是可以把权限规则写进合约。

\## 🛡️ 安全理解

我今天最重要的理解是：

\> 普通低风险执行可以在预设边界内自动化，但权限边界的修改必须人工确认。

比如未来如果有链上交易机器人，不一定每笔低风险交易都手动确认；但这些操作一定要人工确认：

\- 授权 token

\- 合约升级

\- 提高限额

\- 修改交易策略

\- 添加或删除 signer / owner

\- 给 AI agent 新权限

这些操作不是普通交易，而是在改变系统的安全边界。
