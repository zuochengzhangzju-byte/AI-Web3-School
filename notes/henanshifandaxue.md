今天学习了两个内容：

1.**AI 工具功能特点** ：

-   **Hamilton 功能** ：作为学习助手，可根据个人时间安排学习计划，还能进行打卡提醒；能自动提取任务流程封装成 skills 并持久化记忆，持久化优化比小龙虾好。
    
-   **Hermes 特点** ：通过 API 方式调用大模型，使用门槛低，交互方式多样，可通过 TG、飞书、DC 等交互；能自动成长，将行为封装成 skill，自动纠正反馈。
    

-   **工具部署建议** ：
    
    -   **部署选择** ：Hamilton 部署推荐本地 Mac，可随叫随用；若部署在外网服务器，交互受限，功能受服务器性能限制，个人需求大、需处理大量资料的推荐本地部署。
        
    -   **Windows 使用** ：Windows 版处于早期 Beta 版本，功能不稳定，优先推荐用子系统安装，若安装不了再考虑 Windows 安装方式，遇到 bug 可查看 issue 或提 issue。
        
-   **工具配置相关** ：
    
    -   **配置步骤** ：Hermes 配置简单，执行指令后按引导操作，主要配置对接模型和消息推送渠道；如对接 TG 还需Hermesagent 沟通设置白名单、提示词等。
        
-   **应用场景探讨** ：
    
    -   **个人提效场景** ：可用于量化操作研究、代码脑暴、获取资讯等；还可作为个人知识库，配合 RAG 检索学习笔记内容。
        
    -   **多智能体编排** ：对于量化应用的多智能体编排，可让其生成 skill 调用对应模型，但智能体间自主探讨策略因子目前 Hermes 不支持，可参考 CCB 项目。
        
    -   **WEB3 与 AI 结合** ：WEB3 方面，可实现用户意图交易、根据用户习惯做规划、AI 支付等业务场景。

# 2.**AI 与人的协作及能力要求**：

-   **协作分工**：人负责设计方案、评估方案合理性、验收代码执行结果和分析问题；AI 协助设计方案、细化分析、梳理数据流等，并按方案编码。
    
-   **能力要求**：AI 未降低系统复杂度，反而对人的基础知识和架构能力要求更高，人要能判断 AI 方案和代码的正确性。
    

-   **Web3 与 Web2 的关系**：
    
    -   **能力占比**：Web3 项目中 Web2 能力占比不低，如 CEX、衍生品交易所等都需结合 Web2 技术。
        
    -   **安全特性**：Web3 资金交互特性决定安全源于设计，应贯穿项目生命周期，审计只是锦上添花。
        
-   **Web2 支付系统**：
    
    -   **支付模型**：用户在电商平台购物下单，支付服务获取支付单信息展示给用户，用户支付法币，银行机构收到钱后回调电商平台，完成支付流转。
        
    -   **安全考量**：服务间需用 API key 或 secret key 验证身份；用户有支付密码、风控、MFA 等安全措施；存在信任问题，各环节需相互信任。
        
-   **Web3 支付系统**：
    
    -   **支付流程**：电商平台生成地址，用户转账到地址，链上记录交易，链上监听服务通知支付服务，完成充值。
        
    -   **信任机制**：通过算法保证数据不可篡改，逻辑及代码公开透明，采用私钥签名校验请求。
        
-   **钱包相关知识**：
    
    -   **钱包概念**：输入私钥，通过算法产出 Signature，实现身份和资产的双重认证。
        
    -   **钱包分类**：按技术分有 EUA 钱包、合约钱包、AA 钱包；按多签分有 MPC 多签；按托管方式分有自托管、全托管、混合托管钱包；还有硬件钱包、隐私钱包、分层钱包等。
        
    -   **安全提升**：防止私钥泄露和签名欺骗，业界有 EIP712 签名可视化，禁用 ethsign 方法。
        
-   **交易相关内容**：
    
    -   **生命周期**：APP 构造交易，用私钥签名后广播，链处理达成节点共识，记录交易状态。
        
    -   **关键参数**：gas fee 由 gas count 和 gas price 决定交易上链速度；EVM 有 nonce 概念确保交易幂等，不同链实现方式不同；instructions（code data）是链上应用交互灵魂。
        
    -   **监听逻辑**：链上服务监听 block，检测地址资金变化，等待 reorg 避免链分叉，进行合规筛查后回调支付服务。
        
    -   **交易模拟**：在未签名时模拟交易执行结果，避免签有风险的交易。
        
-   **服务端钱包安全**：
    
    -   **权限拆分**：大资金账号用多链和多签方案，拆分资金量，避免单点风险。
        
    -   **安全措施**：交易可视化审核，模拟未签名交易，通过 Tee 可信执行环境签名。
        
-   **问答环节**：
    
    -   **学习路径**：先扎实开发基础知识，可参考慢雾招聘要求，让 AI 反推所需能力。
        
    -   **钱包私钥**：浏览器插件钱包私钥在本地；交易所钱包地址用于充值，资金归集到热钱包，提现由热钱包操作。
        
    -   **AI 应用**：AI 可分析链上未 verify 合约，Cloud Code 的 Misos 可辅助安全工作。
        
    -   **量化数据**：量化策略多为内部知识，大数据在 Web3 的应用需求和机会有待观察。
        
    -   **小额免密支付**：存在信任问题，可从账号权限拆分侧解决，如采用 NPC 方案。
        
    -   **钱包监听**：可通过轮询或 Websocket 监听交易，开源项目 Epusdt 可参考。
        
    -   **其他问题**：AA 钱包会更成熟；跨链需求有但不算大，目前无纯去中心化跨链方案；面试基础技术岗会问基础知识和 AI 应用；区块链监听可付费订阅；智能合约审计可参考接地；确定 AI Coding 交易辅助工具放权边界需考虑风控；WEB3 招聘可直投项目方；uniswap V4 hook 发展前景未明确；agentic payment 是信任问题，X402 应用少；风控需求与牌照有关；SQL 跨链可研究 CCTP；debug 能力提升建议需进一步研究。
