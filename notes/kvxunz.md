**一、 AI 时代的进化与 Agent 概念**

-   **AI 的演进阶段：**
    
    -   **早期：** 智能搜索引擎/聊天机器人（如初期的 ChatGPT），主要用于问答。
        
    -   **进阶：** 任务执行者（2023年后），能解释代码、总结文章、写文案。
        
    -   **现状：** **AI Agent（智能体）**。
        
-   **核心运作机制：** 用户提示词 -> 理解任务目标 -> 调用外部工具 -> 观察执行结果 -> **自我修复与迭代循环**。
    
-   **本质区别：** 聊天机器人侧重问答；Agent 侧重于理解自然语言指令并**主动完成任务**。
    

**二、 AI 工具生态的五大阵营**

1.  **聊天型 AI：** 如 ChatGPT、Claude、Dify。
    
2.  **AI 编程助手（集成 IDE）：** 如 Cursor、Trae (Antigravity)、Copilot（基于 VS Code 二次开发）。
    
3.  **终端型 Agent：** 如 Codex、Claude Code（运行在终端，适合 Web Coding）。
    
4.  **模型平台：** 如 OpenRouter、魔搭社区（汇集大模型供调用）。
    
5.  **通用助手底座（AI 基建）：** 如 **Hermes Agent**、OpenClaude（小龙虾）。这类工具连接了大模型、本地工具、长期记忆和消息平台，类似个人“贾维斯”。
    

**三、 Hermes Agent 核心解析**

-   **定位：** 通用助手的底座，连接大模型与本地环境。
    
-   **核心能力：**
    
    -   **多模型接入：** 支持主流大模型切换。
        
    -   **Memory（长期记忆）：** 能够持久化存储（如写入 `.md` 文件）。
        
    -   **Skills（技能）：** 将固定任务流程沉淀为模块化的“技能”，自动调用。
        
    -   **自我纠正：** 对产生的结果进行自检并修复错误。
        
-   **优势：** 比起纯终端工具（如 Codex），更适合小白用户，可通过微信或飞书等即时通讯工具交互。
    

**四、 实战部署与配置流程**

1.  **环境准备：** 推荐 Windows 用户使用 **WSL2 (Ubuntu)** 环境。
    
2.  **安装：** 通过 GitHub 仓库的一件安装脚本，自动配置 Node.js、Python 等依赖。
    
3.  **初始化：** 使用 `quickstart` 模式。
    
    -   设置 **Provider**（模型供应商）和 API Key。
        
    -   配置上下文长度（Context Length）和显示名称。
        
4.  **进阶设置：**
    
    -   **消息网关：** 扫码接入微信或 Telegram。
        
    -   **环境安全：** 建议通过环境变量设置 API Key，不要在聊天框中直接发送。
        

**五、 实际应用场景示例**

-   **学习助手：** 导入课程手册，根据用户画像（技术水平、目标）和时间安排，自动生成每日学习计划。
    
-   **信息简报：** 每天定时抓取热点新闻并整理成摘要。
    
-   **办公自动化：** 协助整理分享会大纲，生成可下载的 PPT。
    
-   **专业角色：** 在不同群组设置不同身份（如“量化工程师”），利用独立上下文进行脑暴。
    

**六、 Q&A 关键信息点**

-   **编程建议：** 纯编程开发更推荐用 Codex 或 Cursor；Hermes 适合作为协作辅助。
    
-   **隐私：** 具有白名单、唤醒词（如“小黑”）等控制机制。
    
-   **量化交易：** Jon 老师建议将其作为助手（记录盘感、生成报告），而非完全依赖其执行交易决策。
    
-   **API 消耗：** 大部分高级模型需付费 API，也有部分免费中转（但可能存在兼容性限制）。

**一、 AI 与开发者的协作模型**

-   **AI 没有降低系统的复杂度**：AI 只是开发者的“手和脚”，起到放大器（放大人效）的作用，但系统的架构设计、底层判断和最终责任仍然完全取决于人类的大脑。
    
-   **基础知识决定了驾驭 AI 的能力**：如果开发者的基础知识和架构能力不足，将无法判断 AI 生成的代码质量（“无法判断 AI 端上桌的是屎山还是珍馐”）。开发者遇到看不懂的 AI 方案时，应当让 AI 辅助教学，直到完全掌握，而不是盲目信任。
    
-   **协作流程**：人类负责设计方案、评估合理性并验收执行结果；AI 协助人类梳理数据流、细化方案并高效编码。其中，**AI 编码质量的关键在于其自我验证能力**。
    

**二、 Web3 与 Web2 的关联与核心差异**

-   **技术占比**：Web3 项目并非完全独立于 Web2，大部分 Web3 项目（如交易所、DeFi 前端页面）中，Web2 的技术能力占比相当高，两者是结合的系统。
    
-   **安全理念（最大差异）**：Web2 通常“先跑起来再考虑安全”，且资金出错后可以通过中心化机构（如银行）冻结或退回。而 **Web3 的安全必须“源于设计”并贯穿全生命周期**，因为一旦由于漏洞导致资产受损或私钥泄露，资产将直接归零且完全不可逆。
    

**三、 支付架构与信任转移**

-   Web2 支付依赖中心化机构（如银行）进行账户划扣，所有系统间的交互充满对权威机构的信任。
    
-   Web3 支付将中心化机构替换为区块链（Blockchain），通过**共识机制保证数据不可篡改，通过公开透明的智能合约代码保证执行的确定性**，从而解决信任问题。
    
-   在身份校验上，Web3 不再依赖传统的 API Key 或 Secret Key，而是**完全依靠私钥签名（Private Key Signature）来进行身份认证和交易意图确认**。
    

**四、 钱包基础理论与安全**

-   **核心运作逻辑**：私钥（本质是一串随机数）作为输入，通过特定加密算法输出签名（Signature）。该签名既可以证明交易意图（资产管理），也可以证明用户身份（消息签名），过程中无需向第三方暴露私钥即可完成数学验证。
    
-   **钱包分类**：
    
    -   **按底层技术**：EOA 钱包（纯私钥控制）、合钱包（链上程序，天然支持多签）、AA 钱包（账户抽象，将前两者融合的未来趋势）。
        
    -   **按托管方式**：自托管（安全性最高，完全由用户控制）、全托管（如交易所，用户不接触私钥，存在平台跑路风险）、混合托管（如结合 MPC 多签）。
        
    -   **按签名人数**：单签钱包、多签钱包（如智能合约多签、MPC 多方安全计算将私钥切片管理的钱包）。
        
-   **钱包安全防范**：为了防止盲签钓鱼导致的资产丢失，业界推出了 EIP-712（签名可视化）和**交易模拟（Transaction Simulation）**。**交易模拟必须在未签名之前进行**，以提前预知账户状态变化，避免恶意交易或签名在系统中暴露。
    

**五、 交易生命周期与核心参数**

-   **生命周期**：构造交易 -> 使用私钥签名 -> 广播 -> 节点共识处理 -> 记录到不可篡改的状态数据库中。
    
-   **三大关键参数**：
    
    -   **Gas (费用)**：决定交易上链的速度，分为固定消耗（Count）和愿付单价（Price/EIP-1559 模型）。
        
    -   **Nonce (幂等性)**：在 EVM 中用于确保交易顺序执行并防止重放攻击（避免同一笔交易被重复提交）。
        
    -   **Data (指令)**：**Data 是链上应用交互的灵魂**，它让 EVM 能够像解析器一样执行复杂逻辑代码，使区块链不仅限于转账，还能建立繁荣的生态。
        
-   **链上交易监听**：服务端在监听到区块上的交易后，**必须等待一定数量的区块确认（防范 Reorg 分叉重组风险）**，经过风控和合规筛选后，才能进行账目回调。
