# 2026-05-19
<!-- DAILY_CHECKIN_2026-05-19_START -->
我今天对 Prompt 的理解更具体了：Prompt 不是简单地向 AI 提问，而是在给 AI 设定任务边界。一个更清晰的 Prompt 通常需要说明目标、背景、输入材料、限制条件和输出格式。目标决定 AI 要解决什么问题，背景帮助 AI 理解语境，限制条件控制回答范围，输出格式则决定结果是否方便后续使用。

今天最大的收获是：Prompt 的质量会直接影响 AI 输出是否可用。比如只说“帮我写一个 Web3 介绍”，AI 很容易给出比较泛的内容；但如果补充目标用户、使用场景、知识水平、风险边界和希望输出的结构，结果就会更接近真实产品和内容工作的需要。

结合 AI x Web3，我觉得 Prompt 设计尤其需要注意安全边界。Web3 任务经常涉及钱包、签名、交易、Gas、合约和资产安全。AI 可以帮助解释交易、整理概念、翻译术语、拆解流程和生成风险提示，但涉及签名、授权、转账、私钥、助记词和合约交互时，不能让 AI 代替用户做最终决定，必须保留明确的人工确认。

我今天的一个产品角度收获是：Prompt 设计其实也是产品设计。好的 AI 产品不应该只是给用户一个聊天框，而应该提前帮用户拆好任务、准备上下文、设计确认步骤，并输出可直接使用的结构化结果。

我还不清楚的问题是：一个面向普通用户的 AI x Web3 工具，应该如何收集足够上下文，同时又不让用户觉得输入成本太高？

明天我计划继续学习 Context，把 Prompt 和上下文管理连接起来，思考 AI 在 Web3 场景里如何使用可靠信息来源。

# 2026-05-18
<!-- DAILY_CHECKIN_2026-05-18_START -->

今天是 AI x Web3 学习 Day 1，我参加了残酷共学和分享会，重点学习了 Web3 系统设计、钱包、交易生命周期和服务端钱包安全。

今天最大的收获是：Web3 不是简单地把支付系统换成链上版本，而是把传统支付里的部分信任角色，从银行/支付机构转移到区块链、智能合约和密码学签名上。钱包也不只是资产入口，而是身份、私钥和签名授权的关键界面。

我对交易生命周期的理解更清楚了：一笔交易通常会经历构造、签名、广播和确认，关键参数包括 Gas、Nonce 和 Data，服务端还需要通过交易确认和 Event 监听来判断业务状态。今天特别记住了“交易模拟”的重要性，它可以在真正广播前帮助用户理解交易后果。

结合 AI x Web3，我现在更关注安全边界：AI 可以帮助用户解释交易、整理信息、做风险提示和模拟结果，但签名、授权和资产相关动作必须保留明确的人类确认。

我还不清楚的问题是：一个面向普通用户的 AI x Web3 产品，应该如何把签名风险讲清楚，同时又不让用户被复杂术语吓退？

明天我计划继续学习 LLM / Prompt / Context，把今天的 Web3 系统设计内容和 AI 辅助用户教育的场景连接起来。
