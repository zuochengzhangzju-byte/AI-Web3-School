一、今天我在学什么？ 今天我在理解一个核心问题：

> **隐私为什么不只是“把数据藏起来”，而是一个和文化、权力结构、协作方式都相关的“系统设计问题”？**

Neo-Cypherpunk 的视角提醒我： • 隐私不是“反监管”或“神秘主义”，而是**数字社会中个体自主权的基础设施**； • 对 builder 来说，隐私不是可选项，而是产品可信度、可持续增长和抗风险能力的一部分； • 真正有价值的隐私设计，必须同时考虑：**技术层、交互层、治理层、文化层**。 ─── 二、我提炼出的核心问题地图（Week 2 可直接用） 我把这个方向拆成了 5 个可研究的问题簇： 1) 数据暴露问题（Data Exposure） • 用户在链上、AI 工具、社交平台上的行为是否被过度关联？ • 哪些“非敏感数据”组合后会变成敏感画像？ • builder 是否在“默认公开”中无意伤害了用户？ 2) 身份与可验证性冲突（Identity vs Verifiability） • 如何证明“我是合格用户”，但不泄露完整身份？ • 能不能做到“只证明必要事实”（如年龄区间、KYC 状态、持仓条件）？ 3) AI 训练与推理隐私（AI Privacy） • 用户数据用于训练时如何授权、追踪、撤回？ • Agent 访问钱包、交易、聊天记录时如何最小权限？ • 模型输出是否会反推出用户私密信息？ 4) 协作与治理隐私（Coordination Privacy） • DAO、社区投票如何避免“可追踪投票报复”？ • 团队协作中如何保留审计能力但减少个人暴露？ 5) UX 与教育问题（Privacy UX） • 用户真的理解自己在授权什么吗？ • 隐私产品是不是“安全但难用”，导致 adoption 失败？ • 如何让隐私行为变成“默认好习惯”而不是专家技能？ ─── 三、真实使用场景（最能启发 proposal 的部分） 场景 A：AI Agent + 钱包自动化 用户授权 Agent 管理日常 DeFi 操作。 **风险**：过度权限、行为画像泄露、交易意图被前置分析。 **机会**：做“可撤销细粒度授权 + 隐私执行日志”。 场景 B：链上信用与就业 创作者/开发者想证明能力与信誉，但不想暴露全部历史。 **风险**：永久公开记录导致“身份锁死”。 **机会**：做“选择性披露的技能证明系统”。 场景 C：隐私投票与社区治理 DAO 成员希望真实表达观点，又担心被标签化。 **风险**：投票可追踪导致寒蝉效应。 **机会**：做“匿名投票 + 可验证计票”的治理工具。 场景 D：AI 学习平台中的隐私学习分析 平台要做个性化学习推荐。 **风险**：学习轨迹和心理画像过度收集。 **机会**：本地推理/最小化上传/差分统计。 ─── 四、可参考的项目 / 工具 / 方法（按“能用于选题”筛）

技术方法 • **ZK（零知识证明）**：证明“我满足条件”而不泄露原始数据 • **MPC / TEE**：多方协作计算但不直接暴露私有数据 • **DID + VC**：去中心化身份 + 可验证凭证，支持选择性披露 • **FHE（全同态加密，前沿）**：密文上计算，适合长期探索型项目 • **Differential Privacy**：做群体统计时保护个体信息 产品设计方法 • **Privacy by Design**：从需求阶段就定义最小数据收集 • **Least Privilege for Agents**：Agent 按任务临时授权、自动过期 • **Consent Ledger**：用户授权可追踪、可撤回、可审计 • **Threat Modeling**：在 proposal 中明确攻击者模型和滥用路径 研究与验证方法（Week 2 很实用） • 访谈 5–8 位潜在用户，问“你最怕被暴露什么” • 做一张“数据流图”：数据从哪来、到哪去、谁能看到 • 用“可解释风险评分”评估方案前后改进（如数据最小化程度） ─── 五、它如何启发 AI × Web3 Hackathon 选题？ 我总结了 4 个高潜方向（可以直接改成 proposal 标题）： 1. **隐私优先的 AI 钱包助手** • 核心：可编程权限、任务级授权、撤销机制、私密执行记录 2. **ZK 驱动的学习/技能证明系统** • 核心：证明“我完成了某能力路径”，但不暴露完整学习历史 3. **面向 DAO 的匿名提案反馈与投票层** • 核心：保护表达自由，同时保证结果可验证 4. **AI 数据授权与收益分配协议（Data Consent + Revenue）** • 核心：用户能控制数据使用范围、期限，并分享价值回流 ─── 六、我今天的收获（反思） • 我以前把隐私当“安全模块”，今天更像把它看成**产品哲学 + 社会契约**。 • 对 builder 来说，隐私不是“多加一个加密库”，而是决定谁有权、谁可见、谁可撤回。 • 一个好的 AI × Web3 项目，不只要“能跑”，还要回答： **“用户是否在不牺牲尊严与自主权的前提下使用它？”** ───

# 一、今天学了什么

今天重点理解了一个核心方向：

> Agent 的关键瓶颈不是“会不会回答”，而是“能否长期保持一致的记忆、目标和风格”。

围绕这个方向，我建立了一个初步框架：**采集 → 存储 → 检索 → 治理**。 ───

# 二、核心问题总结

1\. **会话割裂**：跨会话后上下文丢失

2\. **记忆污染**：无关信息过多，召回噪声高

3\. **一致性不足**：人格、偏好、决策逻辑不稳定 这说明：如果没有长期记忆层，Agent 很难真的“像一个持续协作的人”。

# 三、问题地图（初稿）

1) Memory Capture（记忆采集）

• 记什么：偏好、任务进度、历史决策、失败经验 • 何时记：显式指令 + 关键事件触发 • 如何记：摘要、去重、重要性评分

2) Memory Storage（记忆存储） • 短期：工作记忆（会话内） • 长期：向量库 + 结构化数据 + 时间线日志 • 可验证：关键摘要哈希上链（Web3结合点）

3) Memory Retrieval（记忆检索） • 语义召回 + 规则过滤（时间/权限/来源） • 先粗召回，再重排 • 输出附来源，减少幻觉

4) Memory Governance（记忆治理） • 读写删权限控制 • 冲突与过期处理 • 一致性评估（persona consistency）

# 四、和 AI × Web3 Hackathon 的连接

今天明确了这个方向为什么适合 Web3： • **上链存证**：记忆不可篡改、可审计 • **DID身份**：跨平台迁移同一Agent人格 • **签名授权**：用户控制谁可读写记忆 这让“长期记忆”从功能问题升级为“可信基础设施问题”。

# 2026-05-20
<!-- DAILY_CHECKIN_2026-05-20_START -->







今天跟着布老师又学习了 Web3 的基本概念

1\. **Web1 / Web2 / Web3 的区别** Web1 是“可读”，Web2 是“可读可写”，Web3 更强调“可拥有”。 用户不只是使用平台，也可以拥有资产和身份。

2\. **去中心化的核心价值** 数据和资产不再完全依赖单一平台，规则更透明，参与者之间的信任更多来自代码与共识。

3\. **钱包与私钥的重要性** 钱包不仅是“存币工具”，更是进入 Web3 的身份入口。 私钥即资产控制权，安全意识必须放在第一位。

4\. **智能合约的作用** 智能合约可以把规则写进链上，自动执行，减少人为干预，提升透明度和可验证性。
