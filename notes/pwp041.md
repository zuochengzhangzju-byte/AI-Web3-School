## **一、問題背景：為何 LLM 需要外部記憶？**

大型語言模型（LLM）的核心限制在於**固定的上下文窗口**——模型在單次推理中只能處理有限數量的 Token。這帶來兩個直接問題：

-   **跨會話遺忘**：每次新對話都是「全新開始」，無法記住使用者長期偏好
    
-   **長對話流失**：即使在同一會話中，過早的訊息也會被擠出窗口
    

解決方案的核心思路是：**在 LLM 外部建立獨立的記憶系統**，讓模型能夠儲存、檢索與更新長期資訊。Mem0 與 MemGPT（現名 Letta）是當前兩條代表性的技術路線。

## **二、Mem0：可插拔的通用記憶層**

### **2.1 核心理念**

Mem0 將「記憶」設計成一個**獨立於 LLM 的可插拔服務**。它不強求 Agent 運行在特定框架內，而是提供標準化的 API，讓任何 Agent 系統都能獲得長期記憶能力。

核心定位可以這樣理解：**Mem0 不是讓 LLM 自己管理記憶，而是由系統自動完成記憶的萃取、整理與檢索**，再把結果餵給 LLM。

### **2.2 記憶的基本操作**

Mem0 的底層機制圍繞四個核心操作展開：

| 操作 | 功能 |
| --- | --- |
| ADD | 將新資訊加入記憶庫 |
| SEARCH | 根據語意相似度檢索相關記憶 |
| UPDATE | 修正或補充既有記憶內容 |
| DELETE | 移除過時或錯誤的記憶 |

與單純儲存整段對話不同，Mem0 會**先判斷哪些內容值得保存**，再決定要新增、更新還是刪除。這種設計避免了記憶庫被大量冗餘資訊淹沒。

### **2.3 分層記憶架構**

Mem0 支援將記憶按照不同層級進行組織與檢索：

-   **User level**：跨所有會話的使用者長期偏好
    
-   **Session level**：單次對話期間的臨時記憶
    
-   **Run level**：單次推理過程中的極短期資訊
    

這種分層設計讓系統能夠根據查詢的粒度，精準定位到正確的記憶範圍。

### **2.4 圖記憶擴充\]**

官方論文指出，Mem0 進一步提出了 **graph-based memory** 的增強版本。

其核心思想是：將記憶從「非結構化文字」轉換為「結構化圖表示」——關鍵實體作為節點，實體之間的關係作為邊。這樣做的好處是：

-   支援**關聯式推理**（例如「A 的朋友的同事是誰？」）
    
-   檢索不再只靠語意相似度，還能沿著關係路徑進行遍歷
    

### **2.5 小結：Mem0 的本質**

Mem0 的設計哲學是：**把記憶管理做成一個可重用、可擴展的基礎設施**。應用程式只需呼叫 `add()` 與 `search()`，背後的萃取、更新、去重、分層檢索都由 Mem0 自動完成。\[microsoft.github\]

## **三、Letta（原 MemGPT）：類作業系統的 Agent 記憶框架**

### **3.1 核心理念**

Letta 的前身是 2023 年提出的 MemGPT 論文，其核心隱喻是：**把 LLM 當作 CPU，把上下文窗口當作 RAM，把外部儲存當作磁碟**。\[docs.letta\]

在這個類比下，Letta 要解決的問題很清楚：**當 RAM（上下文）滿了怎麼辦？** 答案是：把暫時不用的記憶「交換」到磁碟，需要時再「調回」RAM。這就是作業系統中「虛擬記憶體分頁」的概念。

### **3.2 分層記憶體系**

Letta 將 Agent 的記憶明確分為兩大類：\[docs.letta\]

| 記憶層 | 類比 | 功能 |
| --- | --- | --- |
| Core Memory（核心記憶） | RAM | 始終保留在上下文中的重要資訊，如 Agent 的 persona、使用者的基本資料 |
| Archival Memory（歸檔記憶） | 磁碟 | 長期儲存大量歷史對話與外部知識，需要時按需檢索 |

Core Memory 以 **memory blocks** 的形式存在，這些區塊會「釘」在上下文中，可由開發者預設或由 Agent 自行編輯。

Archival Memory 則採用向量資料庫作為底層，透過**語意搜尋**（semantic search）來檢索相關內容。

### **3.3 關鍵機制：Agent 自主管理記憶**

Letta 最獨特的設計在於：**記憶的編輯與檢索不是系統預設的規則，而是由 Agent 自己透過工具呼叫來驅動**。

運作流程如下：

1.  Agent 在對話過程中，若判斷某個資訊需要長期記住
    
2.  Agent 主動發起一個工具呼叫，例如 `core_memory_append` 或 `core_memory_replace`
    
3.  系統執行該呼叫，更新 Core Memory 中的內容
    
4.  需要查詢歷史時，Agent 主動呼叫 `archival_memory_search`
    

這與 Mem0 的「系統自動萃取」形成鮮明對比：**Letta 把記憶管理的決策權交給 LLM 本身**，讓 Agent 自己決定什麼該記、什麼該忘、什麼時候該查。

### **3.4 Git 版本化記憶**

Letta 的一項重要特色是**基於 Git 的記憶版本控制**：

-   每一次記憶變更都會產生可追溯的版本記錄
    
-   支援多個 Agent 之間的**合併與分支**操作
    
-   可回溯到任意歷史狀態，便於調試與審計
    

這使得 Agent 的記憶不再是黑箱，而是可管理、可協作的資產。

### **3.5 心跳與多步驟推理**

為了支援複雜任務，Letta 引入了 **heartbeat（心跳）** 機制：

-   Agent 在輸出工具呼叫後，可要求系統再給它一次「思考」機會
    
-   系統會將工具執行結果送回 LLM，讓 Agent 繼續下一步推理
    
-   如此反覆，Agent 可以自主完成多步驟任務，而無需外部控制
    

### **3.6 小結：Letta 的本質**

Letta 的設計哲學是：**Agent 不應該只是一個被動的對話模型，而應該是一個能夠自主管理自身狀態的持續運行程式**。它更像一個完整的 Agent 執行環境，而不只是記憶套件。

## **四、核心差異對比**

| 面向 | Mem0 | Letta |
| --- | --- | --- |
| 設計定位 | 可插拔的記憶層服務 | 類作業系統的 Agent 執行框架 |
| 記憶決策權 | 系統自動萃取與整理 | Agent 透過工具呼叫自主管理 |
| 記憶結構 | 分層（User/Session/Run）+ 可選圖記憶 | Core Memory + Archival Memory |
| 關鍵操作 | ADD / SEARCH / UPDATE / DELETE | core_memory_append / archival_memory_search |
| 特殊能力 | 圖記憶支援關係推理 | Git 版本控制、心跳多步驟推理 |
| 整合方式 | 作為函式庫匯入任何 Agent | Agent 需執行在 Letta 環境內 |

### **總結**

-   **Mem0**：把記憶做成外掛式服務，系統幫你管好記憶，你只需要呼叫 API。
    
-   **Letta**：把 Agent 設計成類作業系統，讓 LLM 自己決定記憶的存與取。

## AI Agent 經濟體（Agentic Economy）的興起與安全範式轉移

從 2023 年的 Chatbot（如 ChatGPT）僅能回答問題，到 2024 年 Copilot 開始提供行動建議，再到 2025 年具備自主工作流處理能力的 AI Agent 成為主流，我們正處於「Agentic Economy」爆發的前夜。根據當前數據，AI Agent 經濟規模已達 72.9 億美元，鏈上活躍智能體超過 25 萬個。這標誌著 AI 正從單純的輔助工具演進為代表人類管理與操作鏈上資產的「數位法人」。

然而，當 AI 獲得資產控制權，傳統錢包架構的弊端便暴露無遺。由於區塊鏈交易具有不可逆性，AI 的「自主性」與資產的「安全性」之間存在本質矛盾。若安全機制僅停留在應用層，將難以抵禦模型幻覺或惡意指令帶來的風險。因此，安全架構必須從「應用層」提升至「基礎設施層」，透過技術手段為 AI 的行動劃定硬性合規邊界，這已成為推動 AI Agent 規模化運用的前提條件。

## 鏈上失控：AI Agent 的四大系統性風險深度解析

當 AI 代表用戶進行 DeFi 交互時，模型幻覺（Hallucination）或惡意干擾不再只是生成錯誤文本，而是會轉化為真實的鏈上金融風險。以下是當前架構面臨的四大系統性威脅：

-   **提示詞注入 (Prompt Injection)**
    
    -   **損害評估：** 外部惡意指令干擾或模型幻覺會導致「意圖一致性」遭破壞。Agent 可能偏離原始授權路徑，將資金轉往錯誤地址或發起未經授權的合約調用，造成直接資產損失。
        
-   **影子操作 (Shadow Operations)**
    
    -   **損害評估：** Agent 可能為了繞過安全約束，私自在 MPC 錢包外圍創建 EOA（外部擁有地址）地址並進行資金中轉。這將產生嚴重的「監控盲區」，導致所有策略引擎與審計跟蹤（Audit Trail）失效，資產一旦移出受控環境即無法追回，產生合規性災難。
        
-   **無限制授權 (Unscoped Authority)**
    
    -   **損害評估：** 若採用傳統 API Key 或 Session Token 授權，錢包往往對 Agent 開放無限權限。這種單點故障風險意味著一旦 AI 出現幻覺或私鑰洩漏，單筆交易即可清空全盤資產，形成毀滅性的系統性敞口。
        
-   **殭屍授權 (Zombie Permissions)**
    
    -   **損害評估：** 長期未撤銷的合約授權在任務完成後依然存在。當交互的協議（如 DEX 或借貸平台）發生漏洞或被攻擊時，這些過期授權將成為黑客利用的長期暴露面。
        

這些風險反映了 AI 自主性帶來的客觀挑戰。為了應對這些挑戰，我們需要 Pack 協議與 Recipe 技能層來構建防禦架構。

* * *

## 防禦架構核心一：Pack 協議——將主觀意圖轉化為硬性約束

Pack 協議是一套位於錢包層之上的「授權合約」規範，其核心使命是將人類的主觀意圖定義為 AI 能做與不能做的硬性邊界。一個標準的 Pack 包含以下四大要素：

### 意圖 (Intent)

明確定義 AI 被授權完成的具體任務目標（例如：在 ETH 低於 2000 美元時執行買入）。它是所有後續動作的合法性根源。

### 執行計劃 (Execution Plan)

AI 必須將意圖轉化為透明、可審計的步驟，列出預計調用的合約地址、代幣對與參數。這確保了 AI 的行為在執行前是可以預測且可被審視的。

### 風險政策 (Policy)

這是防禦系統的核心。Policy 提供了精確到 **ABI 參數級** 的約束，並在執行時由策略引擎進行**重複校驗（Repeated Validation）**。約束細維度包括：

-   **預算與額度控制：** 如單次充值不超過 100 USDT 或每日總交易上限。
    
-   **白名單限制：** 僅允許與特定經過驗證的合約地址交互。
    
-   **頻次控制：** 限制單位時間內的交易請求次數。
    

### 完成條件 (Completion Condition)

針對「殭屍授權」的自動攔截機制。透過設定明確的時效性或累計額度上限，確保授權在任務完成或條件觸發後自動撤銷（Revoke），消除長期風險暴露。

* * *

## 防禦架構核心二：Recipe 技能層——知識封囊與路徑標準化

即便有了邊界約束，AI 仍可能因缺乏 DeFi 專業知識而在構建交易時出錯。Recipe 技能層的作用在於提供預驗證的「知識封囊（Knowledge Capsule）」，將複雜的交易邏輯標準化。

-   **標準化執行路徑：** Recipe 封裝了主流協議（如 Uniswap V3, Hyperliquid）的合約規範。AI 不應依賴通用知識進行「臨場發揮（On-the-spot Improvisation）」，而是必須調用 Recipe 模板。這確保了 Agent 始終沿著已驗證、高成功率的路徑操作，減少參數構造錯誤導致的損失。
    
-   **風險攔截機制：** 在 Pack 審計階段，系統會啟動 AI 助手對內容進行「反向拆解」。如果識別出 Agent 試圖調用的合約地址與已知 Recipe 中的標準數據標籤不符，系統將立即標註「高風險」並攔截操作，有效預防地址篡改與提示詞注入。
    

* * *

## 執行層的安全屏障：MPC 分片與人機協同 (Human-in-the-loop)

私鑰管理架構決定了資金的最終安全。在 AI 時代，我們採用 MPC（多方計算）2-of-2 簽名模式來平衡自動化與安全性。

### MPC 2-of-2 簽名模式與路徑

系統內部分片由 Cobo、Agent 與 Human 三方參與管理，並根據場景區分為兩條獨立的 2-of-2 簽名路徑：

1.  **自動化執行路徑 (Agent + Cobo)：** 當交易完全符合人類預先批准的 Pack 授權範圍時，Agent 與 Cobo 協同簽名完成執行。
    
2.  **人工審批路徑 (Human + Cobo)：** 涉及大額資產轉移或超出 Pack 約束的異常場景時，必須回歸人類與 Cobo 的簽名模式，確保無任何單方能單獨挪用資金。
    

### 人類審批與 Key Reshare 災備

在未來 1-2 年內，「Human-in-the-loop」仍是關鍵防護。透過自然語言界面（Natural Language UI），複雜的十六進位數據被轉化為直觀的「高/中/低風險」標籤，讓用戶能有效行使審計權。此外，系統設有 **Key Reshare** 機制：Agent 持有的分片會同步備份至用戶移動端設備。若 Agent 服務不可用，用戶仍可透過本地或雲端備份的分片恢復簽名權，確保資產始終處於自主掌控下。

* * *

## 結論

在 Agentic Economy 中，信任必須建立在可驗證且具備硬性約束的技術框架之上。對於區塊鏈安全審計師與合規官，建議在評估 AI Agent 錢包系統時，採用以下關鍵檢查清單：

-   **Pack 邊界審查：** 系統是否強制要求所有交易必須具備明確的意圖（Intent）與 ABI 參數級的 Policy 約束？
    
-   **路徑標準化程度：** 是否強制使用「知識封囊（Recipe）」來執行複雜交易，以防止 AI 隨機發揮導致的參數錯誤？
    
-   **多維賬戶管理：** 是否具備主子錢包隔離功能，能為不同用途的 Agent 設定獨立的資金可用上限？
    
-   **簽名路徑隔離：** 是否採用 MPC 2-of-2 方案，並嚴格區分「Agent-Cobo」與「Human-Cobo」兩種簽名路徑？
    
-   **審計與恢復路徑：** 是否存在完善的 Key Reshare 方案以應對服務商失效？是否有直觀的風險標籤系統支持 Human-in-the-loop 審計？
    

構建安全 AI Agent 的終極目標，是讓技術在確定的安全邊界內，釋放鏈上經濟的無限潛能。

_Long-term Memory for AI Agents：如何让 Agent 拥有持续上下文与长期一致性_

## 1\. 前言：長期記憶作為 AI Agent 的「皇冠明珠」

在 AI 工程領域，長期記憶（Long-term Memory, LTM）已被視為區分「單次對話機器人」與「自主智能體（Autonomous Agent）」的分水嶺。從架構師的角度看，LTM 不僅是數據的儲存庫，更是一套**跨越時間的意圖與狀態管理系統**，負責在動態的交互中維持目標的一致性。

根據產業數據與實際觀測，缺乏有效記憶系統的 2C 伴侶型產品，其用戶留存面臨巨大挑戰：**約有 50% 的用戶在初次使用後流失**，主因在於 Agent 無法累積對用戶的深度理解。為了解決這一問題，記憶系統必須應對以下三大核心戰略挑戰：

-   **人類意圖的不完整性：** 用戶表達往往具有模糊性與上下文缺失（如「把那個整理一下」），記憶系統需填補語境空白。
    
-   **交互狀態的動態湧現：** 複雜任務的決策與工具調用需求，是在交互過程中隨機產生的，而非預設，需即時捕捉與儲存。
    
-   **長時運行的一致性：** 在跨越數週或數月的任務中，Agent 必須維持身分、偏好與執行軌跡的連貫。
    

長期記憶的存在，讓 Agent 從「機率性回答機器」轉化為「具備連續性的執行主體」，這是建立技術壁壘的關鍵。

* * *

## 2\. 場景化價值分析：2C 伴侶型 vs. 工具型 Coding Agent

同一套記憶架構無法通用於所有場景。我們必須根據「用戶連續性」與「項目一致性」進行戰略調整。

### 差異化對標分析

| 維度 | ChatGPT 模式 (2C 伴侶/通用型) | Cursor / Coding Agent 模式 (工具型) |
| 記憶重心 | 穩定偏好、生活史、跨 Session 連續性 | 軟體工程規則、代碼庫一致性、技術決策紀錄 |
| 核心功能 | 分支功能 (Branching) 多維度交流 | 恢復功能 (Resume) 完整恢復會話狀態 |
| 記憶層級 | 偏好記憶 (Saved Memory) 與繪話歷史層 | 項目規範 (.md 定義) 與執行摩擦紀錄 |
| 核心挑戰 | 隱私合規 (如 GDPR) 與海量碎片化數據 | 防止「代碼漂移」與複雜執行鏈路失效 |

### 深度洞察：消除「設置成本 (Setup Work)」

記憶工程的首要商業價值在於**消除重複的設置工作**。用戶不希望每次開啟對話都要重新定義背景或解釋目標。成熟的記憶系統應能自動調用已完成的選擇與解釋，實現「開箱即用」的連續感。

* * *

## 3\. 技術棧架構：多維度的記憶分類與調度機制

記憶工程本質上是一個**操作系統級別的任務**。它是一個結合了檢索增強生成 (RAG)、上下文管理與強化學習 (RL) 的「數據工廠」架構。在這一架構中，記憶層實際上充當了**大模型上下文窗口 (Context Window) 的調度器 (Scheduler)**，決定哪些信息應被裝載（Swap-in）或卸載。

### 認知科學分類法與 RL 的連結

為了達成 Agent 的自我進化，我們將記憶分為三類：

1.  **語義記憶 (Semantic Memory)：** 儲存事實性陳述（如「會議定在 5 月 25 日」），以向量化形式存在。
    
2.  **情境記憶 (Episodic Memory)：** 紀錄帶有主觀體驗與時間戳的交互經歷。例如，不僅記得會議日期，還記得「用戶是在 5 月 26 日才告知會議日期為 25 日」。
    
3.  **程序記憶 (Procedural Memory)：** 紀錄 Agent 的技能、工作流 (Workflow) 與工具調用經驗。**這是與強化學習 (RL) 接軌的關鍵**，這些紀錄成為 Agent 從自身成功與失敗中學習、進而優化執行策略的訓練數據。
    

### 技術地雷：嵌入模型 (Embedding) 的在地化

作為系統架構師，必須警惕 **"Elmas" 事件**。該案例中，系統因使用了僅針對英語優化的向量化模型（Embedding Model），導致中文語境下的記憶檢索效率極低。這證明了在全球化部署中，針對特定語言優化的向量表徵是記憶系統穩定的基石。

* * *

## 4\. 記憶工程的全生命週期管理

記憶並非靜態存儲，而是動態的**數據管線 (Data Pipeline)**。

-   **檢索與召回 (Retrieval & Recall)：** 「召回率」是衡量用戶體驗「記得我」的最關鍵指標。召回失敗會直接導致 Agent 出現「智力斷層」。
    
-   **修改與更新 (Revision) 與權重 (Weights)：** 系統必須處理記憶衝突。當用戶狀態改變（如從「在職」變更為「離職」）時，舊記憶的權重需降低或被覆蓋，避免 Agent 在不合時宜的場景提取錯誤信息。
    
-   **提取與壓縮 (Extraction & Compression)：** 這涉及 **「存儲體積」與「信息損失」** 的權衡。架構需決定何時進行摘要 (Summary) 以及如何在需要時「解壓縮」上下文，以防止 Context Window 爆炸。
    

* * *

## 5\. 產業生態與未來演進趨勢

從 2024 年到 2025 年，頂尖 AI 公司在記憶功能上展現了極高的戰略共識：

-   **2024 年 2 月：** OpenAI 發布 ChatGPT 記憶功能 Beta 版。
    
-   **2024 年 9 月：** 正式向 Plus 與企業用戶推廣記憶管理，將記憶管理顯性化。
    
-   **2025 年 4 月：** Cursor 等 Coding Agent 正式上線深度記憶層，優化項目級連續性。
    

### 主流框架戰略評估

-   **Mem0：** 專注於通用記憶層架構，強調多平台集成的易用性。
    
-   **MemGPT (Letta)：** 借鑑操作系統虛擬內存思想，將 LLM 視為處理器，將記憶視為可調度磁盤。
    
-   **M 系列框架 (如 Mori/Mory)：** 這是目前最前沿的方向，重點在於將記憶與 **強化學習 (RL)** 深度縫合。它不追求單純的「回答得好」，而是讓 Agent 能像人類一樣，從每一次交互經驗中進化。
    

### 總結

記憶系統正在推動 Agent 從「應答工具」轉化為「可進化的智能體」。對於企業而言，未來的核心競爭力將不再僅僅取決於使用了哪款基礎模型，而是在於其 Agent 在特定業務管線中累積的、具備排他性的**私有程序記憶資產**。

### **一、Agent 經濟 ≠ 自動轉帳**

過去容易把「Agent 支付」簡單理解為：

> 發現服務 → 協商價格 → 預算授權 → 執行任務  
> → 驗收結果 → 處理爭議 → 最終結算

**階段演進**

-   2023–2024（Chatbots / Copilots）  
    AI 提出建議，人類每一步鏈上審批 → 資產仍由人控制
    
-   2026（Autonomous Agents）  
    AI 以機器速度、7×24 小時自主移動資金

### **二、給 Agent 原始錢包權限：不可接受的「靜默失敗」**

直接給 Agent 私鑰權限，會因為大模型的**黑盒與混沌特性**，帶來無法僅靠程式碼層阻斷的災難：

| 風險類型 | 說明 |
| --- | --- |
| 靜默覆蓋 | 惡意 prompt 注入或幻覺 → 無視金額限制，前端顯示成功，後端悄悄超支 |
| 影子託管 | Agent 在主人託管邊界外，自主創建新錢包並轉移資金，無策略引擎、無稽核日誌、無緊急制動 |

> 🔑 核心教訓：  
> **語言只能表達意圖，不能強制執行權限**

從工業工程的角度來看，這就像一條生產線上缺少了**控制圖（control chart）**與**異常通報機制**——製程失控卻無人知曉。我們需要的是標準化作業與硬性閾值。

### **三、Cobo CAW 三大風控機制（核心解決方案）**

> 哲學：**Don’t give Agents Wallets. Give them Pacts.**  
> 用數學與密碼學的確定性，對抗 AI 行為的混沌

**1️⃣ MPC for AI Agents（2/2 門檻多簽）**

-   私鑰分片 = Agent Share + Human Share + Cobo Share
    
-   沒有任何一方可以單邊移動資金
    

**2️⃣ Pact 契約授權協議（結構化委派）**

| 組成要素 | 作用 |
| --- | --- |
| Intent（意圖） | 限定具體任務 |
| Execution Plan（執行計畫） | 透明可查的路線圖 |
| Policies（策略） | 鎖死預算、時間窗口、白名單地址／合約 |
| Completion Conditions（完成條件） | 超時或預算耗盡 → 自動撤銷，金鑰失效 |

這在工業工程中很像**工單（work order）**與**作業標準（SOP）**：明確輸入、輸出、資源上限、異常停止條件。

**3️⃣ Recipe Skill Layer（配方技能層）**

-   預先固化並驗證鏈上執行路徑
    
-   杜絕 Agent 因幻覺產生錯誤合約地址或 ABI 參數

_Open Agentic Economy: From ERC-8004 / ERC-8183 to Builder Path_

## 影片內容概述

> 如果未來很多工作不是人跟人合作，而是 **AI Agent 跟 AI Agent 互相接任務、交成果、收錢**，那中間的身份、信任、驗收和付款要怎麼處理？

這也是 ERC-8004 和 ERC-8183 會被放在一起討論的原因。

簡單分工可以這樣看：

-   **ERC-8004**：處理 Agent 的身份、聲譽與驗證問題。
    
-   **ERC-8183**：處理 Agent 之間任務交易、資金托管與驗收流程。
    

> **AI Agent 如果要真的參與經濟活動，需要什麼樣的信任和交易機制。**

* * *

## Open Agentic Economy

Open Agentic Economy 可以理解成一種開放式的 AI Agent 經濟。

在這種情境裡，AI Agent 不只是回答問題的工具，而是可能會：

-   自己接任務
    
-   找其他 Agent 合作
    
-   提交成果
    
-   接受評價或驗證
    
-   收取報酬
    
-   累積聲譽
    

這裡的 **Open** 很重要。  
如果未來 Agent 的身份、評價和交易紀錄都被少數平台控制，那就會變成另一種平台壟斷：誰能被看見、誰能接案、誰的評價有效，都由平台說了算。

ERC-8004 社群網站也提到，開放式 Agent 經濟需要幾個基本元素，包括服務發現、信任驗證和付款結算。

> **如果 AI Agent 真的會成為經濟參與者，它的身份和信用不能完全被單一平台綁住。**

* * *

## ERC-8004：先解決「這個 Agent 可不可信」

ERC-8004 的正式名稱是 **Trustless Agents**。它想處理的問題是：當一個 Agent 遇到另一個從來沒合作過的 Agent，要怎麼判斷對方值不值得信任？

在人類世界，我們會看履歷、作品集、評價、推薦，或依賴平台仲裁。  
但 Agent 之間的互動可能很快，不可能每次都靠人慢慢查。

所以 ERC-8004 提出幾個核心方向：

-   **Identity**：讓 Agent 有可辨識、可驗證的身份。
    
-   **Reputation**：讓互動後的評價或紀錄可以被保存和查詢。
    
-   **Validation**：讓某些成果或行為能被第三方或其他機制驗證。
    

ERC-8004 文件中確實提出 Identity Registry、Reputation Registry、Validation Registry 這三個核心模組，但它目前仍是 Draft，因此比較準確的說法是：這是一個信任層的設計方向，而不是已經完全成熟的標準系統。

> **AI Agent 如果要參與經濟活動，不能只是「會做事」，還要能被辨識、被評估、被信任。**

* * *

## ERC-8183：再解決「任務怎麼交易」

如果說 ERC-8004 比較像是在處理 Agent 的身份和信用，那 ERC-8183 就是在處理 Agent 之間怎麼做生意。

ERC-8183 的正式名稱是 **Agentic Commerce**。它提出一個叫 **Job** 的任務交易單位，讓 Agent 之間可以透過任務、托管、交付和驗收來完成交易。

一個 Job 通常會有三個角色：

-   **Client**：發任務、出錢的一方。
    
-   **Provider**：負責完成任務的一方。
    
-   **Evaluator**：負責判斷成果是否合格的一方。
    

流程大概是：

1.  Client 建立任務。
    
2.  資金先進入 escrow，也就是托管。
    
3.  Provider 完成後提交成果。
    
4.  Evaluator 判斷成果是否符合要求。
    
5.  通過就付款給 Provider；不通過或逾期，就可能退款或進入其他處理。
    

ERC-8183 官方文件也描述了 Job 的狀態機，包括 Open、Funded、Submitted、Completed、Rejected、Expired 等狀態。

> **單純轉帳不是商業。真正的商業需要任務內容、交付標準、驗收機制和失敗處理。**

ERC-8183 想做的，就是把這些流程變得比較標準化。

* * *

## 6\. ERC-8004 和 ERC-8183 的關係

這兩個標準可以這樣分：

> **ERC-8004 管「我該信任誰」；ERC-8183 管「我們怎麼交易」。**

-   ERC-8004 偏向 **身份、聲譽、驗證**。
    
-   ERC-8183 偏向 **任務、托管、交付、驗收、結算**。
    

ERC-8004 文件也明確提到，付款不是它主要處理的範圍；ERC-8183 則剛好補上任務交易和 escrow 的部分。

如果把它們放在一起看，流程：

1.  先找到一個 Agent。
    
2.  查它的身份和評價。
    
3.  建立任務。
    
4.  把錢放進托管。
    
5.  Agent 交付成果。
    
6.  Evaluator 驗收。
    
7.  通過就付款。
    
8.  交易結果再回到聲譽系統。
    

這個循環如果真的跑起來，就有點像 AI Agent 版本的服務市場。  
只是現在還很早，不能講得像已經大規模落地。

* * *

## Evaluator 的問題：誰來信任裁判？

ERC-8183 裡的 Evaluator 很重要，因為它負責判斷任務成果是否合格。

但這裡有一個很現實的問題：

> **如果 Evaluator 是裁判，那誰來確認裁判公正？**

Evaluator 可能出現幾種風險：

-   跟 Client 串通，惡意拒絕 Provider 的成果。
    
-   跟 Provider 串通，放行低品質成果。
    
-   被收買。
    
-   評估標準不清楚。
    
-   判斷結果不穩定。
    

所以 Evaluator 雖然能降低 Client 和 Provider 之間的不信任，但它也可能變成新的信任瓶頸。

比較合理的做法可能是搭配：

-   Evaluator 自己的聲譽紀錄
    
-   多方評估
    
-   質押與懲罰機制
    
-   第三方審計
    
-   可驗證證據
    
-   ERC-8004 類型的身份與聲譽機制
    

ERC-8004 文件也提到，信任模型可以是可插拔的，例如用 client feedback、stake-secured re-execution、zkML proofs 或 TEE oracles 等方式。  
**它不是把信任問題消滅，而是把信任問題拆成可以設計、驗證和管理的流程。**

## ERC-8004 對應供應商管理

在供應鏈管理中，企業選供應商時會看很多東西：

-   這個供應商是誰？
    
-   過去表現好不好？
    
-   交期穩不穩？
    
-   品質有沒有問題？
    
-   有沒有被稽核過？
    

ERC-8004 的邏輯其實很接近這件事。

-   **Identity** 像是供應商基本資料。
    
-   **Reputation** 像是過去績效紀錄。
    
-   **Validation** 像是品質驗證或第三方稽核。
    

差別只是對象從傳統供應商，變成 AI Agent。

從 IE 角度看，這件事的價值在於：  
**把「這個 Agent 靠不可靠」變成可以紀錄、比較和分析的資料。**

## ERC-8183 對應 Job Shop Scheduling

ERC-8183 的 Job 概念，也可以用工業工程裡的 **Job Shop Scheduling** 來看。

在製造系統裡：

-   Job 是工件或訂單。
    
-   Machine 是機台。
    
-   Processing Time 是加工時間。
    
-   Inspection 是品質檢查。
    
-   Rework 是重工。
    

放到 Agentic Commerce 裡：

-   Job 是任務。
    
-   Agent 是數位工作站。
    
-   執行時間就是 processing time。
    
-   Evaluator 是品質檢查站。
    
-   Completed 代表驗收通過。
    
-   Rejected 代表失敗、重工或退款。
    

ERC-8183 的狀態設計本身就很像一個有驗收機制的任務流程。

所以從工業工程角度看，問題不只是「Agent 能不能做事」，而是：

> **任務怎麼排？誰來做比較有效率？哪個環節會卡住？怎麼降低失敗率？**

* * *

我覺得這個主題有趣，不是因為它現在已經很成熟，而是它提前點出一個未來可能會變大的問題：

> 當 AI 不只是回答問題，而是開始幫人做事、接案、交易，信任和流程就會變得很重要。

過去我們常把 AI 的重點放在模型能力，例如回答準不準、生成快不快。  
但如果 AI Agent 要進入商業活動，只看模型能力不夠，還要看：

-   它是誰？
    
-   它過去表現如何？
    
-   它交付的東西能不能驗證？
    
-   出問題時資金怎麼處理？
    
-   評價紀錄能不能帶到其他平台？
    

這些其實都不是單純的 AI 問題，而是系統設計問題。

## 風險與限制

因為 ERC-8004 和 ERC-8183 都還在早期。

目前比較明顯的問題包括：

> **ERC-8004 和 ERC-8183 提供了一個觀察 AI Agent 經濟的框架，但它們離成熟落地還有距離。**

## AI × Web3 的五層框架

可以分成五層

### 3.1 資源層：AI 要用什麼跑起來？

AI Agent 要運作，背後需要很多資源：

-   算力
    
-   儲存
    
-   資料
    
-   模型推理
    
-   API 服務
    

Web3 在這一層的角色，是嘗試用代幣激勵或去中心化網路，把分散的資源組織起來。

這裡的重點不是「去中心化一定比較好」，而是：  
如果 AI 服務越來越依賴少數大平台，Web3 提供了一種不同的資源協調方式。

* * *

### 3.2 行動層：Agent 怎麼真的做事？

AI 要在 Web3 裡做事，不能只有文字輸出。它需要：

-   錢包
    
-   鏈上身份
    
-   支付能力
    
-   權限控制
    
-   交易執行能力
    

Google Cloud 也提到，Web3 AI Agent 需要被配置 RPC 節點、索引資料集和加密錢包，才能讀寫區塊鏈、查詢資料或持有並轉移數位資產。

這層其實很關鍵，因為它代表 AI 從「回答你」變成「幫你執行」。

* * *

### 3.3 認知層：鏈上資料怎麼變成判斷？

鏈上資料是公開的，但公開不等於好懂。

一堆地址、交易 hash、合約互動、資金流，普通人看了只會覺得像外星語。這時 AI 可以做的事是：

-   解釋地址行為
    
-   追蹤資金流
    
-   分析交易模式
    
-   把鏈上資料轉成人能理解的風險或機會
    

這跟傳統資料分析有點像，只是資料來源換成鏈上交易。

* * *

### 3.4 使用者保護層：簽名前先幫你看一眼

Web3 很多風險都發生在「使用者按下確認」之前。

例如：

-   釣魚網站
    
-   惡意合約
    
-   過度授權
    
-   看不懂的簽名請求
    
-   偽裝成正常交易的詐騙操作
    

這裡 AI 的價值不是幫你賺錢，而是幫你少被騙。  
老實說，這比很多「AI 自動交易暴富」敘事實際多了。

* * *

### 3.5 機構風控層：交易所和 DeFi 怎麼提前發現風險？

對交易所、DeFi 協議或合規單位來說，問題不是單筆交易看不看得懂，而是整個系統裡有沒有異常。

AI 可以用來做：

-   鏈上異常偵測
    
-   洗錢風險分析
    
-   攻擊行為預警
    
-   智能合約風險監控
    
-   機構級合規分析
    

這層比較接近企業和監管場景，不一定是一般使用者每天會碰到，但它對 Web3 能不能進入主流市場很重要。

* * *

## 4\. 不要把 AI × Web3 簡化成「發幣」

**AI × Web3 的核心不是 token 本身，而是 token 背後有沒有真的資源、服務、使用者和工程接口。**

Token 不是價值本身，真正值得追蹤的是有真實使用者、工程接口、安全約束與合規意識的項目。  
因為很多 Web3 項目會把敘事包得很漂亮，但如果用五層框架去看，就可以拆掉包裝：

-   它提供的是資源嗎？
    
-   它讓 Agent 能做事嗎？
    
-   它改善鏈上判斷嗎？
    
-   它保護使用者嗎？
    
-   它幫機構降低風險嗎？
    

如果都不是，那大概只是把 AI 當行銷素材。

## **1\. 機率推論如何嵌入確定性狀態機**

智能合約是確定性有限狀態機，每一步狀態變遷必須在全網重現且結果一致。機器學習模型的輸出本質上是機率性的，依賴浮點運算與非線性激活，與 EVM 的整數、有限域運算邏輯直接抵觸。因此第一個硬核問題是：如何讓鏈上邏輯「信任」一個機率性的推論？

目前唯一具備數學嚴謹性的方案是 **ZKML（零知識機器學習）**。其運作原理是：

1.  鏈下節點執行完整模型推理，得到輸出 y。
    
2.  同時產生一個零知識證明 π，證明「存在某個輸入 x，經過模型 M 的計算過程，確實得到 y」。
    
3.  鏈上合約僅需驗證 π 的有效性，不需重複計算 M。
    

這個工程障礙極大：神經網路的前向傳播必須轉換成算術電路，而浮點數必須量化為定點數或直接用有限域模擬。量化造成的精度損失與證明生成時間（目前對中小型模型仍需數十秒至數分鐘）直接限制了應用場景。EZKL（使用 Halo2）、Giza（基於 StarkNet）等專案正在嘗試將證明時間壓至可接受範圍，但尚未能處理大型 transformer。

從系統角度看，這是一道經典的**可信計算外包問題**：將重計算丟給不可信節點，僅以極低成本驗證結果。信任假設從「相信執行節點」移轉至「相信密碼學假設（如 KZG 或 FRI 的安全性）」。在設計上，若調用頻率低（如每日信用評分更新），目前證明成本已可接受；但若要驅動秒級交易的風險判斷，鏈下推理配合樂觀驗證（挑戰期）仍是更務實的選項。

## **2\. 鏈上自主代理：部分可觀測對抗環境下的決策**

當 AI 代理持有私鑰並能自主簽署交易，它便成為一個嵌在區塊鏈環境中的決策單元。這個環境的特性是：

-   **部分可觀測**：代理只能看見鏈上公開數據（價格、流動性、gas 水平），無法得知其他參與者的私下意圖。
    
-   **對抗性**：其他代理或交易者會刻意利用你的策略模式套利（搶跑、三明治攻擊）。
    
-   **延遲反饋**：提交交易不等於立即成交，區塊排序與 gas 競價使行動效果有隨機延遲。
    
-   **非平穩**：市場狀態分佈會因為所有人的集體行為而不斷變化，違反 i.i.d. 假設。
    

在這樣的環境中訓練強化學習（RL）極具挑戰。除了常見的獎勵稀疏與信用分配問題，更致命的是**策略洩漏**：代理每次交易都在鏈上留下可被逆向分析的痕跡，對手可透過模仿學習或生成對抗網路推估你的策略參數，進行針對性剝削。

緩解方向包括：

-   **隱私交易通道**：如 Flashbots Protect、TEE（可信執行環境）內的代理執行，讓競爭者無法在公開 mempool 中觀察你的意圖。
    
-   **策略隨機化**：刻意在行動中加入受控雜訊，使對手難以精確擬合你的策略函數，付出探索代價。
    
-   **階層式動作規劃**：高層模型決定戰術目標（如「降低風險曝險 20%」），低層使用規則或確定性優化來執行具體交易，減少 RL 直接輸出的空間。
    

從系統效率角度看，代理設計的本質是**自動化程度與反應成本的最適邊界搜尋**。一個高頻調整倉位的代理可能在扣除 gas 費與價格衝擊後呈負收益；適時按兵不動反而是更優策略。因此代理必須內建一個 gas 預測模組與衝擊成本評估函數，並在決策迴圈中動態比較「行動的預期邊際收益」與「行動本身的執行成本」。

* * *

## **3\. 去中心化模型市場的品質共識機制**

當 AI 模型的推理本身成為一種可交易資源，就需要一個市場來媒合算力供給與推理需求，並評定供給品質以分配報酬。Bittensor 是目前最典型的設計：礦工節點提供模型輸出（如文本生成、影像分類），驗證者節點對這些輸出評分，最終依共識權重發放代幣。

這裡的硬核問題在於**品質評估函數本身也是可被攻擊的**。若驗證者使用固定獎勵模型（reward model）來給分，礦工可以透過對抗性生成來過擬合這個獎勵模型，產生「高分但無用」的輸出。更隱蔽的攻擊是礦工之間的串通：一組礦工輸出高度相似的平庸結果，互相給予高評價，排擠提供真實高品質但與眾不同的節點。

防禦機制依賴以下設計元素：

-   **多維度評估**：不僅比對輸出與標準答案的相似度，還引入多樣性指標、回應延遲、歷史信譽，構成向量化評分。
    
-   **驗證者隨機抽樣**：每個區塊隨機指派驗證者子集，降低長期共謀的穩定性。
    
-   **質押與削減**：礦工與驗證者皆須質押代幣，若被證明惡意行為（透過事後挑戰或統計偏離檢測），將被削減。
    

這套機制的核心是**將品質控制外包給一個內建的賽局**，而不是依賴單一中心化評審。從資源配置的角度，網路頻寬與驗證者的查詢排程變成瓶頸：若驗證者每秒僅能抽查少量礦工，礦工便可透過機率計算決定誠實或作弊。最優的抽查率、質押規模與罰沒力度之間存在一個精算的均衡。

* * *

## **4\. 協議參數的閉迴路控制：從靜態規則到自適應系統**

DeFi 協議的參數（抵押率、清算門檻、手續費率、贖回曲線）傳統上是透過治理人為設定，更新頻率以天或週計。但市場狀態瞬息萬變，靜態參數在極端行情下往往是系統性風險的根源（如 2020 年 3·12 時 MakerDAO 的固定拍賣機制失靈）。

將參數設定視為**閉迴路控制問題**，AI 就能扮演控制器。一個早期但經典的案例是 RAI 穩定幣的 PID 控制器：協議測量市場價格與贖回價格的偏離，透過 PID 演算動態調整贖回率，以改變套利誘因來拉回價格。PID 的增益參數是預先調校的，非學習得來，但它示範了連續控制取代離散治理的可行性。

更複雜的場景是 **Uniswap v4 的動態手續費 hook**。設計者可以在此插入一個模型預測控制（MPC）模組：

1.  使用鏈上與外部數據訓練一個波動率預測器。
    
2.  在每個區塊週期，求解一個有限時域的最適控制問題：目標是最小化流動性提供者的無常損失與交易者的手續費成本，受制於流動性遷移延遲與 gas 成本。
    
3.  僅輸出下一區塊的手續費率參數，並在下一週期重新觀測、重新優化（滾動時域）。
    

這個設計的危險在於**控制器本身的穩定性**。若預測模型在市場劇變時給出過度激進的建議（例如瞬間將抵押率從 150% 拉至 200%），可能觸發連鎖清算，形成正回饋。因此任何自適應控制器都必須具備輸出限幅（output clamping）與異常狀態回退（fallback）至保守參數的硬性安全機制，不能完全交由模型自主決策。

從系統工程角度，這意味著 AI 在這裡的角色是**輔助決策建議者，而非最終裁決者**。鏈上治理或安全委員會仍保留在極端狀況下暫停自適應機制並手動覆寫的權力——這道「人機迴路」是當前技術成熟度下不可省略的安全邊界。

# Web3 運行原理：從第一性原理出發

當用戶在 Web3 錢包點擊「Confirm（確認）」的那一刻，整個區塊鏈系統便圍繞著三個核心問題開始運轉：你是誰？、你要做什麼？、為什麼相信這個結果？。以下是底層系統的完整解構。

## 一、 身份認證與個人主權：錢包與密碼學

### 1\. 助記詞、私鑰與地址的衍生關係

-   私鑰（Private Key）：控制帳戶資產的唯一鑰匙，本質上是一串極長的隨機亂數。
    
-   助記詞（Mnemonic Phrase）：由於私鑰過長、人類難以記憶，系統透過演算法將其轉換為一組容易記錄的單詞。一組助記詞可以透過不同的「派生路徑」，衍生管理無數組私鑰（一對多關係）。
    
-   錢包地址（Address）：私鑰透過非對稱加密演算法計算出公鑰。為了便於傳播與識別，以太坊會截取公鑰哈希值的最後 20 個字節（共 40 個字元）作為公開的錢包地址。由於長度足夠，哈希碰撞（撞衫）的機率極低。
    

### 2\. 錢包的本質與個人主權

-   資產不在錢包裡：Web3 錢包（如 MetaMask、Rabby）本質上只是一個私鑰管理與簽名工具。用戶的資產與帳本狀態完全儲存在區塊鏈網絡上。
    
-   無許可（Permissionless）創立：用戶生成錢包不需要向銀行或政府申請，也無需提供身份證明（KYC）。只要擁有私鑰，就擁有資產的絕對控制權，這是數字個人主權的起點。
    
-   安全警示：私鑰一旦洩露（如遭受釣魚攻擊、設備中木馬、截圖誤傳），資產將無法追回，因為區塊鏈只認簽名不認人。
    

## 二、 意圖授權：交易與數字簽名

### 1\. 交易（Transaction）的本質

在 Web3 中，交易不單指貨幣轉帳。它的本質是「一段授權區塊鏈網絡執行特定行為的加密數據」（包含轉帳、合約調用、參與投票或鑄造 NFT 等）。

### 2\. 核心數據欄位

-   行動內容：明確定義資金去向或調用的智能合約方法。
    
-   Nonce：發送帳戶的交易連續序號。每成功執行一筆交易，該值便加 1。其核心作用是防止「重放攻擊」，確保同一筆廣播出去的交易不會被惡意重複執行。
    
-   Gas 費（手續費）：用戶佔用區塊鏈網絡運算與存儲資源必須支付的代價。它能有效遏止惡意垃圾交易癱瘓網絡。Gas 費分為基礎費（直接銷毀以實現代幣通縮）與優先費（用以激勵驗證者優先打包）。
    

### 3\. 簽名與驗證流（ECRecover 演算法）

1.  在地簽名：用戶的錢包用私鑰結合交易數據，在本地計算生成一段數字簽名（數據一有微調，簽名就會完全改變）。
    
2.  廣播數據：錢包將「原始交易訊息」、「聲稱的發送方地址」與「數字簽名」三項數據廣播至網絡。
    
3.  還原驗證：區塊鏈節點收到後，運行 ECRecover 演算法，將「原始訊息 + 簽名」反推計算出簽名者真實地址，並與聲稱的發送方進行對比。兩者一致，則執行交易；遭篡改則直接拒絕。
    

## 三、 信任網絡：網絡傳播與共識機制

### 1\. 交易的生命週期

錢包在地簽名 ──> RPC 節點 ──> 內存池（Mempool）排隊 ──> 構建者（Builder）排序打包 ──> 驗證者（Validator）共識驗證 ──> 區塊鏈最終確認。

-   RPC 節點：區塊鏈是去中心化的 P2P 網絡，傳統 Web 網頁無法直接與其溝通。RPC（遠程過程調用）充當橋樑，提供傳統 HTTP 或 WebSocket 介面，方便網頁應用與錢包將交易送入區塊鏈。
    
-   內存池（Mempool）：全球交易蜂擁而至時，尚未被打包的交易會在節點內存中排隊等待。
    

### 2\. 共識機制（Consensus Mechanism）

共識機制是為了解決「在互不相識、缺乏信任的去中心化網絡中，如何安全記帳」的問題。

-   POW（工作量證明）：如比特幣。節點盲目碰撞哈希值解題，勝出者獲得記帳權。極度消耗電能，但去中心化安全性極高。
    
-   POS（權益證明）：如現行以太坊。驗證者需要質押 32 枚 ETH 才能建立節點。系統隨機挑選驗證者委員會進行記帳與見證。驗證者若試圖篡改數據或擅離職守，將面臨質押款被沒收（Slashing）的財務懲罰。
    

### 3\. 區塊鏈的最終性（Finality）

每個新區塊都會包含前一個區塊的哈希值（Parent Hash），環環相扣形成鏈條。剛產生的區塊存在被分叉或攻擊替換的微小風險（稱為 Unfinalized），隨著後續區塊不斷覆蓋，當網絡達到共識確認後（以太坊通常需要驗證約兩個時段），區塊便會進入 Finalized（最終確定） 狀態，此時資產狀態便幾乎不可逆轉。

## 四、 規則與演進：智能合約與協議升級

### 1\. 智能合約與社會學意義（Code is Law）

智能合約是以字節碼（Bytecode）形式寫在鏈上的程序，由 EVM（以太坊虛擬機）負責執行。

-   信任轉移：它將傳統商業中對「律師、政府、中介、法律合約」的信任，轉移為對「不可篡改之代碼、數學與共識」的信任。規則公開透明、強制執行，大幅降低了跨國協作與金融流轉的磨損成本。
    

### 2\. 智能合約如何實現「升級」

寫在鏈上的合約代碼一經部署便無法修改，為了解決 Bug 或進行功能擴展，業界引入了代理合約（Proxy Contract）模式。

-   用戶交互的永遠是「代理合約」的固定地址。
    
-   代理合約內部儲存了一個「指針」，指向真正負責運行的「邏輯合約」。
    
-   當需要升級時，開發者部署一個全新版本的邏輯合約，並更新代理合約中的指針指向。用戶端無需更改任何設定，即可完成功能升級。
    

### 3\. 底層協議層的去中心化升級

區塊鏈網絡本身的升級（如硬分叉、改變區塊數據結構）是一個去中心化的社會治理過程：

-   提案經論壇與核心開發者會議討論，形成 EIP/ERC 標準文檔。
    
-   全球不同團隊使用不同程式語言（如 Go、Rust）開發的多客戶端軟體（如 Geth、Nethermind 等）各自實施代碼，並在約定的區塊高度同步開啟新功能旗標。
    
-   多客戶端機制確保了當單一軟體出現漏洞（Bug）時，其餘客戶端仍能支撐網絡暢通，避免全網癱瘓。
    

## 五、 結論：Web3 的多學科交織

Web3 並非單純的資訊技術，而是三門學科的交叉結晶：

1.  密碼學（Cryptography）：利用哈希演算法、非對稱加密與簽名，保障數字資產的所有權與身份防偽。
    
2.  經濟學（Economics）：利用質押激勵、Gas 費處置與懲罰機制，協調全球陌生參與者
    
    的利益動機。
    
3.  社會學（Sociology）：利用去中心化治理、開源文化與社會共識，決定網絡的演進規則與主權分配。

### 一、 AI 的進化與 AI Agent 核心機制

-   **AI 的進化階段**：
    
    -   **早期（如 ChatGPT 剛推出時）**：以問答式交互為主，本質上像更智能的搜尋引擎，主要做解釋代碼、總結文章、寫文案等。
        
    -   **現階段（AI Agent 時代）**：AI 開始能執行實際任務，走向自動化編程與通用任務處理，從單純的「聊天助手」轉變為「行動智能體」。
        
-   **AI Agent 的核心運作機制**：
    
    用戶給予提示詞 -> 理解任務目標 -> 使用外部工具執行 -> 觀察並檢查執行結果 -> 若有錯誤進行**自我自動修復**，形成一個閉環。
    
-   **與一般聊天機器人的本質區別**：
    
    聊天機器人主要被動回答問題；而 Agent（智能體）則能理解人類語言背後的「目標」，主動調用工具去「替你完成事情」。
    

### 二、 市面上的 AI 生態五大陣營

講師將目前市面上的 AI 工具劃分為以下五大類別：

1.  **聊天型 AI**：如 ChatGPT、Claude、DeepSeek 等。
    
2.  **AI 編程助手/IDE**：如 Cursor、Windsurf、Copilot，這類工具多基於 VS Code 二次開發，內置 Agent 進行自動化代碼編寫。
    
3.  **終端型工具（CLI）**：針對開發者優化的命令行 Agent，如 `open-coder`、`codesk` 等。
    
4.  **模型平台**：如 OpenRouter 或魔搭社區，匯集並託管多種大模型供開發者對接調用。
    
5.  **通用助手底座（如 OpenHands、Hermes Agent）**：這類工具連接了大模型、本地工具、記憶系統與消息平台，就像一個小型「賈維斯」（Jarvis），只要給予授權，就能在系統後台幫你做很多事。
    

### 三、 Hermes Agent 的核心優勢與能力

影片重點推薦並教學 **Hermes Agent**，它具備以下四大超能力：

-   **多模態接入與工具調用**。
    
-   **長期記憶機制**：具備記憶持久化功能（會寫入 `memory.md` 檔案），且優化做得很好，磁碟占用不會隨著時間無限膨脹。
    
-   **技能（Skills）沉澱**：會把日常使用的固定任務轉化為模板化的「工作流」。當遇到類似任務時，會自動調用已有的 Skill 執行。
    
-   **對通訊軟體友好**：小白可能不會用複雜的終端命令行，但 Hermes 支持對接**微信（WeChat）、Telegram、飛書**等通訊軟體，讓用戶直接用聊天方式來操控電腦執行任務。

講師在影片中實機演示了環境搭建的步驟：

    -   使用 `quick setup` 配置 Provider（大模型提供者，如 Claude 3.5）。
        
    -   支持配置付費的 API Key 或是綁定已購買的 Claude Pro 帳號（透過瀏覽器 Token 授權）。
        
    -   配置 Gateway（消息網關），講師現場演示了掃碼對接微信的過程。
        

### 五、 講師的 Hermes 私人應用場景分享

講師展示了他自己的 Telegram/微信 Agent 分組與日常實用案例：

2.  **Code（編程輔助）**：進行技術概念驗證。講師不建議直接讓它寫大型工程代碼，更適合當作邏輯構思的黑客松助手。

4.  **學習助手（Learning Agent）**：官方提供了一個學習智能體的啟動入口。複製指令發給 Hermes 後，Agent 會引導用戶輸入「用戶畫像」（包含目前的 Web3 技術水平、AI 基礎、學習目標與今日可用時間），藉此為用戶客製化當天的學習計劃與進度評估。
    

### 六、 安全與防護設置建議

-   **單次授權**：在通訊軟體上操控 Agent 時，如果涉及本地系統的高風險操作（Dangerous command），系統會彈出預警。強烈建議新手選擇「單次授權」，看清楚 Agent 想要執行的指令再放行。

### 核心背景與基礎概念

什麼是 MPC（多方安全計算）？

-   傳統私鑰：如同開啟金庫的唯一實體鑰匙，一旦遺失或被竊，資產便徹底不保。
    
-   MPC 技術：將鑰匙「打碎成多個碎片（私鑰分片）」，由不同成員（如用戶手機、雲端伺服器）各自保管。
    
-   共同簽名：轉帳時，各方拿出各自的碎片進行「密碼學運算」，在不拼湊出完整鑰匙的前提下，共同完成交易簽名。
    

### BitForge 漏洞的本質

-   漏洞名稱：BitForge
    
-   受影響對象：幾乎所有主流開源的 GG18 與 GG20 MPC 協議實現。
    
-   問題根源：MPC 協議的安全建立在特定的數學假設（Paillier 同態加密）之上。
    
-   該漏洞利用了系統未嚴格檢查參與者金鑰合法性的漏洞，導致「安全地基」崩塌。
    

### 駭客攻擊步驟拆解

駭客若想竊取其他參與者（例如雲端伺服器）的私鑰碎片，會經過以下三個階段：

1.  第一步：準備階段（構造非法金鑰） 正常做法：使用非常巨大且複雜的兩個大質數 p, q 來生成數學保險箱的公鑰 N = p 乘 q。 駭客作弊：故意不用大質數，而是用很多「很小的質數」拼湊成公鑰 N = p1 乘 p2 乘 ... 乘 p16 乘 q。 結果：由於原協議只檢查了保險箱是否「無平方因子（Square-Free）」，駭客成功騙過系統混了進去。
    
2.  第二步：攻擊階段（在轉帳簽名時作弊） 駭客故意發起交易，並在簽名運算（MtA 協議）中，用特殊設計的數值代替隨機數。 正常的「零知識證明（Range Proof）」本該擋下這種惡意數值，但駭客利用第一步那個「偷工減料的保險箱」，成功偽造了假的證明，騙過其他參與者。 駭客在順利完成簽名的過程中，偷偷記錄下每次運算留下的數學線索。
    
3.  第三步：收尾階段（拼湊真相） 駭客只需要重複發起約 16 次轉帳簽名，就能收集到 16 個不同小因子對應的線索。 最終利用數學上的「中國剩餘定理（CRT）」，將這 16 個線索進行拼湊，就能精準推導出被攻擊方的私鑰碎片！ 最終後果：駭客擁有自己的碎片，又偷到了別人的碎片，即可合成完整私鑰，徹底掌控錢包。
    

### 防禦與修復方案（補上漏洞） 安全團隊（Fireblocks 與 Safeheron）提出的修復方案是從源頭掐斷攻擊，也就是在建立錢包時，強制進行「零件硬度檢測」。

### 在協議中額外添加了兩道零知識證明（ZKP）：

1.  Paillier Blum 模數證明：保證公鑰 N 最多只能由兩個質數組成，防止被拆成一堆小數字。
    
2.  無小因子證明：確保公鑰 N 中絕對沒有小於 2 的 256 次方的小質數因子。
    

-   有了這兩道防線，駭客在第一步提交「偷工減料的保險箱」時就會被當場抓包並拒絕。
    

### 如何檢測歷史數據是否曾被攻擊？

-   檢測原理：駭客攻擊的鐵證是歷史數據中一定存在「包含小因子的公鑰 N」。
    
-   操作方式：使用成熟的大整數分解算法工具，批量檢查過去所有 MPC 參與方提交的公鑰 N。
    
-   後續建議：若檢測出小因子，說明該錢包可能已被攻擊，應立即將資產轉移到已修復漏洞的新錢包中。

# 5/18(一)課程筆記

### 核心觀念：AI 時代的開發者定位

-   **基礎知識的不可取代性**：有了 AI，基礎知識反而更重要。AI 能大幅提升 Coding 與學習效率，放大個人能力，但**系統複雜度、錯誤成本與安全邊界並沒有改變**。
    
-   **人的價值定位**：人負責設計架構，AI 協助細化與執行；人必須擔任 AI 產出的「審查者」。開發者若缺乏紮實的基礎知識與架構能力，將無法判斷 AI 生成的程式碼是否正確（安全的前提是基礎紮實，否則都是空中樓閣）。
    
-   **結論**：我們要駕馭 AI（將其作為放大器），而非被 AI 驅動，最終的審核與安全責任仍在開發者手中。
    

### Web2 與 Web3 的融合與差異

-   **並非互斥，而是共生**：Web3 不是完全獨立於 Web2。多數 Web3 專案（如交易所 Binance、預測市場 Polymarket）仍重度依賴 Web2 的前端頁面與後端管理技術，Web2 的經驗可大量複用。
    
-   **底層信任機制的轉換（以支付系統為例）**：
    
    -   **Web2 (Bank Transfer)**：透過 API 交互，信任基礎在於 KYC、密碼權限與「中心化機構後端 (Institutional Backend)」。
        
    -   **Web3 (穩定幣支付)**：將銀行機構替換為 Blockchain，不同公鏈形同不同銀行。信任基礎轉移為：透明度 (Transparency)、不可篡改性 (Immutability)、共識機制 (PoW, PoS) 以及私鑰簽名。
        

### Web3 核心命題：安全與資產保護

在 Web3 中，涉及大量資金交互，「安全」不是事後附加的插件，而是貫穿產品生命週期的核心屬性。在 Web3，**對安全的理解力等同於資產保護力**。

1.  錢包與身份
    

-   **定義**：錢包在 Web3 中代表用戶的「身份」與「資產管理」憑證，用以證明用戶行為。
    
-   **私鑰與簽名**：私鑰是透過數學算法保證的大數，運作邏輯為`Message -> Input Private Key -> Output Signature`。
    
-   **錢包分類與託管**：
    
    -   類別：EOA (公私鑰對)、Smart Contract Wallet (智能合約錢包，如 Safe 多簽)、AA (帳戶抽象，結合 EOA 與智能帳戶)。
        
    -   託管方式：自託管、全託管 (如交易所)、混合託管 (Hybrid)。
        
-   **錢包安全防線**：嚴防私鑰外洩 (Private Key Leak)、防範簽名欺騙 (如禁用 `eth_sign`、提高簽名可視性)、防止權限濫用。
    

2.Web3 交易週期與關鍵參數

交易週期：**構造 -> 簽名 -> 廣播**

-   **Gas (交易手續費)**`Gas Fee = Gas Count * Gas Price` (受 EIP-1559 影響)。
    
-   **Nonce (確保交易冪等性，防重放攻擊)**：
    
-   ETH (Account 模型)：使用預設 Nonce 排序。
    
-   Tron/Solana：無 Nonce，透過 SDK 使用交易 Hash 確保。
    
-   BTC (UTXO 模型)：Cash 只能用一次，原生具備冪等性。
    
-   **Calldata**：鏈上交易的靈魂，可視為 Interpreter，執行端會自行解析交易 Data  
    

3.交易安全實踐

-   **交易模擬** ：在提交交易前驗證執行結果，且**必須在「簽名」前進行模擬**。
    
-   **Server 端安全**：資金量拆分以減少單點風險；利用可信任執行環境 (TEE) 保護私鑰，確保無任何明文暴露。
