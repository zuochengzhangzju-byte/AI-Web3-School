_Open Agentic Economy: From ERC-8004 / ERC-8183 to Builder Path_

## 影片內容概述

> 如果未來很多工作不是人跟人合作，而是 **AI Agent 跟 AI Agent 互相接任務、交成果、收錢**，那中間的身份、信任、驗收和付款要怎麼處理？

這也是 ERC-8004 和 ERC-8183 會被放在一起討論的原因。

簡單分工可以這樣看：

-   **ERC-8004**：處理 Agent 的身份、聲譽與驗證問題。
    
-   **ERC-8183**：處理 Agent 之間任務交易、資金托管與驗收流程。
    

> **AI Agent 如果要真的參與經濟活動，需要什麼樣的信任和交易機制。**

## Open Agentic Economy

Open Agentic Economy 可以理解成一種開放式的 AI Agent 經濟。

在這種情境裡，AI Agent 不只是回答問題的工具，而是可能會：

-   自己接任務
    
-   找其他 Agent 合作
    
-   提交成果
    
-   接受評價或驗證
    
-   收取報酬
    
-   累積聲譽
    

這裡的 **Open** 很重要。  
如果未來 Agent 的身份、評價和交易紀錄都被少數平台控制，那就會變成另一種平台壟斷：誰能被看見、誰能接案、誰的評價有效，都由平台說了算。

ERC-8004 社群網站也提到，開放式 Agent 經濟需要幾個基本元素，包括服務發現、信任驗證和付款結算。

> **如果 AI Agent 真的會成為經濟參與者，它的身份和信用不能完全被單一平台綁住。**

* * *

## ERC-8004：先解決「這個 Agent 可不可信」

ERC-8004 的正式名稱是 **Trustless Agents**。它想處理的問題是：當一個 Agent 遇到另一個從來沒合作過的 Agent，要怎麼判斷對方值不值得信任？

在人類世界，我們會看履歷、作品集、評價、推薦，或依賴平台仲裁。  
但 Agent 之間的互動可能很快，不可能每次都靠人慢慢查。

所以 ERC-8004 提出幾個核心方向：

-   **Identity**：讓 Agent 有可辨識、可驗證的身份。
    
-   **Reputation**：讓互動後的評價或紀錄可以被保存和查詢。
    
-   **Validation**：讓某些成果或行為能被第三方或其他機制驗證。
    

ERC-8004 文件中確實提出 Identity Registry、Reputation Registry、Validation Registry 這三個核心模組，但它目前仍是 Draft，因此比較準確的說法是：這是一個信任層的設計方向，而不是已經完全成熟的標準系統。

> **AI Agent 如果要參與經濟活動，不能只是「會做事」，還要能被辨識、被評估、被信任。**

* * *

## ERC-8183：再解決「任務怎麼交易」

如果說 ERC-8004 比較像是在處理 Agent 的身份和信用，那 ERC-8183 就是在處理 Agent 之間怎麼做生意。

ERC-8183 的正式名稱是 **Agentic Commerce**。它提出一個叫 **Job** 的任務交易單位，讓 Agent 之間可以透過任務、托管、交付和驗收來完成交易。

一個 Job 通常會有三個角色：

-   **Client**：發任務、出錢的一方。
    
-   **Provider**：負責完成任務的一方。
    
-   **Evaluator**：負責判斷成果是否合格的一方。
    

流程大概是：

1.  Client 建立任務。
    
2.  資金先進入 escrow，也就是托管。
    
3.  Provider 完成後提交成果。
    
4.  Evaluator 判斷成果是否符合要求。
    
5.  通過就付款給 Provider；不通過或逾期，就可能退款或進入其他處理。
    

ERC-8183 官方文件也描述了 Job 的狀態機，包括 Open、Funded、Submitted、Completed、Rejected、Expired 等狀態。

> **單純轉帳不是商業。真正的商業需要任務內容、交付標準、驗收機制和失敗處理。**

ERC-8183 想做的，就是把這些流程變得比較標準化。

* * *

## 6\. ERC-8004 和 ERC-8183 的關係

這兩個標準可以這樣分：

> **ERC-8004 管「我該信任誰」；ERC-8183 管「我們怎麼交易」。**

-   ERC-8004 偏向 **身份、聲譽、驗證**。
    
-   ERC-8183 偏向 **任務、托管、交付、驗收、結算**。
    

ERC-8004 文件也明確提到，付款不是它主要處理的範圍；ERC-8183 則剛好補上任務交易和 escrow 的部分。

如果把它們放在一起看，流程：

1.  先找到一個 Agent。
    
2.  查它的身份和評價。
    
3.  建立任務。
    
4.  把錢放進托管。
    
5.  Agent 交付成果。
    
6.  Evaluator 驗收。
    
7.  通過就付款。
    
8.  交易結果再回到聲譽系統。
    

這個循環如果真的跑起來，就有點像 AI Agent 版本的服務市場。  
只是現在還很早，不能講得像已經大規模落地。

## Evaluator 的問題：誰來信任裁判？

ERC-8183 裡的 Evaluator 很重要，因為它負責判斷任務成果是否合格。

但這裡有一個很現實的問題：

> **如果 Evaluator 是裁判，那誰來確認裁判公正？**

Evaluator 可能出現幾種風險：

-   跟 Client 串通，惡意拒絕 Provider 的成果。
    
-   跟 Provider 串通，放行低品質成果。
    
-   被收買。
    
-   評估標準不清楚。
    
-   判斷結果不穩定。
    

所以 Evaluator 雖然能降低 Client 和 Provider 之間的不信任，但它也可能變成新的信任瓶頸。

比較合理的做法可能是搭配：

-   Evaluator 自己的聲譽紀錄
    
-   多方評估
    
-   質押與懲罰機制
    
-   第三方審計
    
-   可驗證證據
    
-   ERC-8004 類型的身份與聲譽機制
    

ERC-8004 文件也提到，信任模型可以是可插拔的，例如用 client feedback、stake-secured re-execution、zkML proofs 或 TEE oracles 等方式。  
**它不是把信任問題消滅，而是把信任問題拆成可以設計、驗證和管理的流程。**

## ERC-8183 對應 Job Shop Scheduling

ERC-8183 的 Job 概念，也可以用工業工程裡的 **Job Shop Scheduling** 來看。

在製造系統裡：

-   Job 是工件或訂單。
    
-   Machine 是機台。
    
-   Processing Time 是加工時間。
    
-   Inspection 是品質檢查。
    
-   Rework 是重工。
    

放到 Agentic Commerce 裡：

-   Job 是任務。
    
-   Agent 是數位工作站。
    
-   執行時間就是 processing time。
    
-   Evaluator 是品質檢查站。
    
-   Completed 代表驗收通過。
    
-   Rejected 代表失敗、重工或退款。
    

ERC-8183 的狀態設計本身就很像一個有驗收機制的任務流程。

所以從工業工程角度看，問題不只是「Agent 能不能做事」，而是：

> **任務怎麼排？誰來做比較有效率？哪個環節會卡住？怎麼降低失敗率？**

* * *

我覺得這個主題有趣，不是因為它現在已經很成熟，而是它提前點出一個未來可能會變大的問題：

> 當 AI 不只是回答問題，而是開始幫人做事、接案、交易，信任和流程就會變得很重要。

過去我們常把 AI 的重點放在模型能力，例如回答準不準、生成快不快。  
但如果 AI Agent 要進入商業活動，只看模型能力不夠，還要看：

-   它是誰？
    
-   它過去表現如何？
    
-   它交付的東西能不能驗證？
    
-   出問題時資金怎麼處理？
    
-   評價紀錄能不能帶到其他平台？
    

這些其實都不是單純的 AI 問題，而是系統設計問題。

## 風險與限制

因為 ERC-8004 和 ERC-8183 都還在早期。

目前比較明顯的問題包括：

-   **標準成熟度不足**：還不是已被大規模驗證的成熟標準。
    
-   **聲譽可能被操縱**：評價系統可能被刷分或攻擊。
    
-   **Evaluator 也需要被信任**：裁判本身可能成為新的風險點。
    
-   **任務描述可能不清楚**：規格不明會造成爭議或重工。
    
-   **鏈上流程可能有效率問題**：交易成本、確認時間都可能影響實際使用。
    
-   **高風險任務不一定適合全自動化**：例如醫療、金融、法律等場景，不能只靠流程自動執行。
    

所以比較合理的看法是：

> **ERC-8004 和 ERC-8183 提供了一個觀察 AI Agent 經濟的框架，但它們離成熟落地還有距離。**

## 影片內容概述

> **AI 如果只是在聊天，Web3 不一定必要；AI 一旦開始做事，就會遇到帳戶、支付、授權、記錄和追責問題。**

AI + Web3 的核心不是發幣，而是經濟基礎設施，具體包括去中心化 AI 基礎設施、AI Agent 錢包、鏈上分析、安全助手，以及 DeFi / 交易所風控等方向。

所以我對這部影片的理解是：

> **Web3 在 AI 裡真正有價值的地方，不是讓 AI 更會聊天，而是讓 AI 可以更安全地參與經濟活動。**

## AI × Web3 的五層框架

可以分成五層

### 3.1 資源層：AI 要用什麼跑起來？

AI Agent 要運作，背後需要很多資源：

-   算力
    
-   儲存
    
-   資料
    
-   模型推理
    
-   API 服務
    

Web3 在這一層的角色，是嘗試用代幣激勵或去中心化網路，把分散的資源組織起來。

這裡的重點不是「去中心化一定比較好」，而是：  
如果 AI 服務越來越依賴少數大平台，Web3 提供了一種不同的資源協調方式。

* * *

### 3.2 行動層：Agent 怎麼真的做事？

AI 要在 Web3 裡做事，不能只有文字輸出。它需要：

-   錢包
    
-   鏈上身份
    
-   支付能力
    
-   權限控制
    
-   交易執行能力
    

Google Cloud 也提到，Web3 AI Agent 需要被配置 RPC 節點、索引資料集和加密錢包，才能讀寫區塊鏈、查詢資料或持有並轉移數位資產。

這層其實很關鍵，因為它代表 AI 從「回答你」變成「幫你執行」。

* * *

### 3.3 認知層：鏈上資料怎麼變成判斷？

鏈上資料是公開的，但公開不等於好懂。

一堆地址、交易 hash、合約互動、資金流，普通人看了只會覺得像外星語。這時 AI 可以做的事是：

-   解釋地址行為
    
-   追蹤資金流
    
-   分析交易模式
    
-   把鏈上資料轉成人能理解的風險或機會
    

這跟傳統資料分析有點像，只是資料來源換成鏈上交易。

* * *

### 3.4 使用者保護層：簽名前先幫你看一眼

Web3 很多風險都發生在「使用者按下確認」之前。

例如：

-   釣魚網站
    
-   惡意合約
    
-   過度授權
    
-   看不懂的簽名請求
    
-   偽裝成正常交易的詐騙操作
    

這裡 AI 的價值不是幫你賺錢，而是幫你少被騙。  
老實說，這比很多「AI 自動交易暴富」敘事實際多了。

* * *

### 3.5 機構風控層：交易所和 DeFi 怎麼提前發現風險？

對交易所、DeFi 協議或合規單位來說，問題不是單筆交易看不看得懂，而是整個系統裡有沒有異常。

AI 可以用來做：

-   鏈上異常偵測
    
-   洗錢風險分析
    
-   攻擊行為預警
    
-   智能合約風險監控
    
-   機構級合規分析
    

這層比較接近企業和監管場景，不一定是一般使用者每天會碰到，但它對 Web3 能不能進入主流市場很重要。

* * *

## 4\. 不要把 AI × Web3 簡化成「發幣」

**AI × Web3 的核心不是 token 本身，而是 token 背後有沒有真的資源、服務、使用者和工程接口。**

Token 不是價值本身，真正值得追蹤的是有真實使用者、工程接口、安全約束與合規意識的項目。  
因為很多 Web3 項目會把敘事包得很漂亮，但如果用五層框架去看，就可以拆掉包裝：

-   它提供的是資源嗎？
    
-   它讓 Agent 能做事嗎？
    
-   它改善鏈上判斷嗎？
    
-   它保護使用者嗎？
    
-   它幫機構降低風險嗎？
    

如果都不是，那大概只是把 AI 當行銷素材。

# 風險和限制

## 1\. 很多應用還在早期

AI × Web3 聽起來很有想像空間，但不能講得像已經完全成熟。研究論文把 Web3 × AI Agents 描述為快速發展的前沿領域，也指出它仍面臨擴展性、安全、倫理與信任機制等挑戰。

所以比較穩的說法是：

> **這是一個值得觀察的方向，但還不是穩定成熟的產業答案。**

* * *

## 2\. Agent 有錢包，就有更高風險

一個會聊天的 AI 出錯，最多是回答錯。  
但一個有錢包、有交易權限的 Agent 出錯，可能就是真的損失資產。

所以 Agent 需要：

-   權限限制
    
-   預算上限
    
-   人工確認
    
-   交易模擬
    
-   異常偵測
    
-   可撤銷授權
    

Google Developers 在介紹 Agent 協議時，也用供應鏈代理示範了工具存取、Agent 溝通、商務協議和付款授權如何拆層處理，這反映出 Agent 真正執行任務時，不能只靠模型自己判斷。

* * *

## 3\. 鏈上透明不等於安全

Web3 的好處是很多紀錄可查，但「可查」不代表一般人看得懂，也不代表不會被騙。  
這也是為什麼影片會提到認知層、使用者保護層和機構風控層：資料公開只是第一步，真正難的是把資料轉成可用判斷。

* * *

# 總結

> **AI 在 Web3 的價值，不是讓聊天機器人上鏈，而是讓會做事的 AI Agent 擁有資源、身份、支付、授權、紀錄和風控能力。**

我覺得最有用的框架是五層：

-   **資源層**：AI 要用的算力、資料、模型從哪來
    
-   **行動層**：Agent 怎麼有錢包、身份和執行能力
    
-   **認知層**：鏈上資料怎麼變成人能理解的判斷
    
-   **使用者保護層**：簽名、授權、交易前怎麼降低風險
    
-   **機構風控層**：交易所、DeFi、合規單位怎麼提前發現異常
    

從工業工程角度看，這不是單純的 AI 或區塊鏈題目，而是一個新的服務系統設計問題。真正值得研究的是：**資源怎麼配置、流程怎麼設計、風險怎麼控制、決策品質怎麼衡量。**

比較保守但有深度的結論是：

> **AI × Web3 目前還在早期，但它提供了一個很好的觀察框架：當 AI Agent 從「回答問題」走向「執行任務」時，Web3 可能會成為它的經濟層和信任層。**

## **1\. 機率推論如何嵌入確定性狀態機**

智能合約是確定性有限狀態機，每一步狀態變遷必須在全網重現且結果一致。機器學習模型的輸出本質上是機率性的，依賴浮點運算與非線性激活，與 EVM 的整數、有限域運算邏輯直接抵觸。因此第一個硬核問題是：如何讓鏈上邏輯「信任」一個機率性的推論？

目前唯一具備數學嚴謹性的方案是 **ZKML（零知識機器學習）**。其運作原理是：

1.  鏈下節點執行完整模型推理，得到輸出 y。
    
2.  同時產生一個零知識證明 π，證明「存在某個輸入 x，經過模型 M 的計算過程，確實得到 y」。
    
3.  鏈上合約僅需驗證 π 的有效性，不需重複計算 M。
    

這個工程障礙極大：神經網路的前向傳播必須轉換成算術電路，而浮點數必須量化為定點數或直接用有限域模擬。量化造成的精度損失與證明生成時間（目前對中小型模型仍需數十秒至數分鐘）直接限制了應用場景。EZKL（使用 Halo2）、Giza（基於 StarkNet）等專案正在嘗試將證明時間壓至可接受範圍，但尚未能處理大型 transformer。

從系統角度看，這是一道經典的**可信計算外包問題**：將重計算丟給不可信節點，僅以極低成本驗證結果。信任假設從「相信執行節點」移轉至「相信密碼學假設（如 KZG 或 FRI 的安全性）」。在設計上，若調用頻率低（如每日信用評分更新），目前證明成本已可接受；但若要驅動秒級交易的風險判斷，鏈下推理配合樂觀驗證（挑戰期）仍是更務實的選項。

* * *

## **2\. 鏈上自主代理：部分可觀測對抗環境下的決策**

當 AI 代理持有私鑰並能自主簽署交易，它便成為一個嵌在區塊鏈環境中的決策單元。這個環境的特性是：

-   **部分可觀測**：代理只能看見鏈上公開數據（價格、流動性、gas 水平），無法得知其他參與者的私下意圖。
    
-   **對抗性**：其他代理或交易者會刻意利用你的策略模式套利（搶跑、三明治攻擊）。
    
-   **延遲反饋**：提交交易不等於立即成交，區塊排序與 gas 競價使行動效果有隨機延遲。
    
-   **非平穩**：市場狀態分佈會因為所有人的集體行為而不斷變化，違反 i.i.d. 假設。
    

在這樣的環境中訓練強化學習（RL）極具挑戰。除了常見的獎勵稀疏與信用分配問題，更致命的是**策略洩漏**：代理每次交易都在鏈上留下可被逆向分析的痕跡，對手可透過模仿學習或生成對抗網路推估你的策略參數，進行針對性剝削。

緩解方向包括：

-   **隱私交易通道**：如 Flashbots Protect、TEE（可信執行環境）內的代理執行，讓競爭者無法在公開 mempool 中觀察你的意圖。
    
-   **策略隨機化**：刻意在行動中加入受控雜訊，使對手難以精確擬合你的策略函數，付出探索代價。
    
-   **階層式動作規劃**：高層模型決定戰術目標（如「降低風險曝險 20%」），低層使用規則或確定性優化來執行具體交易，減少 RL 直接輸出的空間。
    

從系統效率角度看，代理設計的本質是**自動化程度與反應成本的最適邊界搜尋**。一個高頻調整倉位的代理可能在扣除 gas 費與價格衝擊後呈負收益；適時按兵不動反而是更優策略。因此代理必須內建一個 gas 預測模組與衝擊成本評估函數，並在決策迴圈中動態比較「行動的預期邊際收益」與「行動本身的執行成本」。

## **3\. 去中心化模型市場的品質共識機制**

當 AI 模型的推理本身成為一種可交易資源，就需要一個市場來媒合算力供給與推理需求，並評定供給品質以分配報酬。Bittensor 是目前最典型的設計：礦工節點提供模型輸出（如文本生成、影像分類），驗證者節點對這些輸出評分，最終依共識權重發放代幣。

這裡的硬核問題在於**品質評估函數本身也是可被攻擊的**。若驗證者使用固定獎勵模型（reward model）來給分，礦工可以透過對抗性生成來過擬合這個獎勵模型，產生「高分但無用」的輸出。更隱蔽的攻擊是礦工之間的串通：一組礦工輸出高度相似的平庸結果，互相給予高評價，排擠提供真實高品質但與眾不同的節點。

防禦機制依賴以下設計元素：

-   **多維度評估**：不僅比對輸出與標準答案的相似度，還引入多樣性指標、回應延遲、歷史信譽，構成向量化評分。
    
-   **驗證者隨機抽樣**：每個區塊隨機指派驗證者子集，降低長期共謀的穩定性。
    
-   **質押與削減**：礦工與驗證者皆須質押代幣，若被證明惡意行為（透過事後挑戰或統計偏離檢測），將被削減。
    

這套機制的核心是**將品質控制外包給一個內建的賽局**，而不是依賴單一中心化評審。從資源配置的角度，網路頻寬與驗證者的查詢排程變成瓶頸：若驗證者每秒僅能抽查少量礦工，礦工便可透過機率計算決定誠實或作弊。最優的抽查率、質押規模與罰沒力度之間存在一個精算的均衡。

* * *

## **4\. 協議參數的閉迴路控制：從靜態規則到自適應系統**

DeFi 協議的參數（抵押率、清算門檻、手續費率、贖回曲線）傳統上是透過治理人為設定，更新頻率以天或週計。但市場狀態瞬息萬變，靜態參數在極端行情下往往是系統性風險的根源（如 2020 年 3·12 時 MakerDAO 的固定拍賣機制失靈）。

將參數設定視為**閉迴路控制問題**，AI 就能扮演控制器。一個早期但經典的案例是 RAI 穩定幣的 PID 控制器：協議測量市場價格與贖回價格的偏離，透過 PID 演算動態調整贖回率，以改變套利誘因來拉回價格。PID 的增益參數是預先調校的，非學習得來，但它示範了連續控制取代離散治理的可行性。

更複雜的場景是 **Uniswap v4 的動態手續費 hook**。設計者可以在此插入一個模型預測控制（MPC）模組：

1.  使用鏈上與外部數據訓練一個波動率預測器。
    
2.  在每個區塊週期，求解一個有限時域的最適控制問題：目標是最小化流動性提供者的無常損失與交易者的手續費成本，受制於流動性遷移延遲與 gas 成本。
    
3.  僅輸出下一區塊的手續費率參數，並在下一週期重新觀測、重新優化（滾動時域）。
    

這個設計的危險在於**控制器本身的穩定性**。若預測模型在市場劇變時給出過度激進的建議（例如瞬間將抵押率從 150% 拉至 200%），可能觸發連鎖清算，形成正回饋。因此任何自適應控制器都必須具備輸出限幅（output clamping）與異常狀態回退（fallback）至保守參數的硬性安全機制，不能完全交由模型自主決策。

從系統工程角度，這意味著 AI 在這裡的角色是**輔助決策建議者，而非最終裁決者**。鏈上治理或安全委員會仍保留在極端狀況下暫停自適應機制並手動覆寫的權力——這道「人機迴路」是當前技術成熟度下不可省略的安全邊界。

# Web3 運行原理：從第一性原理出發

當用戶在 Web3 錢包點擊「Confirm（確認）」的那一刻，整個區塊鏈系統便圍繞著三個核心問題開始運轉：你是誰？、你要做什麼？、為什麼相信這個結果？。以下是底層系統的完整解構。

## 一、 身份認證與個人主權：錢包與密碼學

### 1\. 助記詞、私鑰與地址的衍生關係

-   私鑰（Private Key）：控制帳戶資產的唯一鑰匙，本質上是一串極長的隨機亂數。
    
-   助記詞（Mnemonic Phrase）：由於私鑰過長、人類難以記憶，系統透過演算法將其轉換為一組容易記錄的單詞。一組助記詞可以透過不同的「派生路徑」，衍生管理無數組私鑰（一對多關係）。
    
-   錢包地址（Address）：私鑰透過非對稱加密演算法計算出公鑰。為了便於傳播與識別，以太坊會截取公鑰哈希值的最後 20 個字節（共 40 個字元）作為公開的錢包地址。由於長度足夠，哈希碰撞（撞衫）的機率極低。
    

### 2\. 錢包的本質與個人主權

-   資產不在錢包裡：Web3 錢包（如 MetaMask、Rabby）本質上只是一個私鑰管理與簽名工具。用戶的資產與帳本狀態完全儲存在區塊鏈網絡上。
    
-   無許可（Permissionless）創立：用戶生成錢包不需要向銀行或政府申請，也無需提供身份證明（KYC）。只要擁有私鑰，就擁有資產的絕對控制權，這是數字個人主權的起點。
    
-   安全警示：私鑰一旦洩露（如遭受釣魚攻擊、設備中木馬、截圖誤傳），資產將無法追回，因為區塊鏈只認簽名不認人。
    

## 二、 意圖授權：交易與數字簽名

### 1\. 交易（Transaction）的本質

在 Web3 中，交易不單指貨幣轉帳。它的本質是「一段授權區塊鏈網絡執行特定行為的加密數據」（包含轉帳、合約調用、參與投票或鑄造 NFT 等）。

### 2\. 核心數據欄位

-   行動內容：明確定義資金去向或調用的智能合約方法。
    
-   Nonce：發送帳戶的交易連續序號。每成功執行一筆交易，該值便加 1。其核心作用是防止「重放攻擊」，確保同一筆廣播出去的交易不會被惡意重複執行。
    
-   Gas 費（手續費）：用戶佔用區塊鏈網絡運算與存儲資源必須支付的代價。它能有效遏止惡意垃圾交易癱瘓網絡。Gas 費分為基礎費（直接銷毀以實現代幣通縮）與優先費（用以激勵驗證者優先打包）。
    

### 3\. 簽名與驗證流（ECRecover 演算法）

1.  在地簽名：用戶的錢包用私鑰結合交易數據，在本地計算生成一段數字簽名（數據一有微調，簽名就會完全改變）。
    
2.  廣播數據：錢包將「原始交易訊息」、「聲稱的發送方地址」與「數字簽名」三項數據廣播至網絡。
    
3.  還原驗證：區塊鏈節點收到後，運行 ECRecover 演算法，將「原始訊息 + 簽名」反推計算出簽名者真實地址，並與聲稱的發送方進行對比。兩者一致，則執行交易；遭篡改則直接拒絕。
    

## 三、 信任網絡：網絡傳播與共識機制

### 1\. 交易的生命週期

錢包在地簽名 ──> RPC 節點 ──> 內存池（Mempool）排隊 ──> 構建者（Builder）排序打包 ──> 驗證者（Validator）共識驗證 ──> 區塊鏈最終確認。

-   RPC 節點：區塊鏈是去中心化的 P2P 網絡，傳統 Web 網頁無法直接與其溝通。RPC（遠程過程調用）充當橋樑，提供傳統 HTTP 或 WebSocket 介面，方便網頁應用與錢包將交易送入區塊鏈。
    
-   內存池（Mempool）：全球交易蜂擁而至時，尚未被打包的交易會在節點內存中排隊等待。
    

### 2\. 共識機制（Consensus Mechanism）

共識機制是為了解決「在互不相識、缺乏信任的去中心化網絡中，如何安全記帳」的問題。

-   POW（工作量證明）：如比特幣。節點盲目碰撞哈希值解題，勝出者獲得記帳權。極度消耗電能，但去中心化安全性極高。
    
-   POS（權益證明）：如現行以太坊。驗證者需要質押 32 枚 ETH 才能建立節點。系統隨機挑選驗證者委員會進行記帳與見證。驗證者若試圖篡改數據或擅離職守，將面臨質押款被沒收（Slashing）的財務懲罰。
    

### 3\. 區塊鏈的最終性（Finality）

每個新區塊都會包含前一個區塊的哈希值（Parent Hash），環環相扣形成鏈條。剛產生的區塊存在被分叉或攻擊替換的微小風險（稱為 Unfinalized），隨著後續區塊不斷覆蓋，當網絡達到共識確認後（以太坊通常需要驗證約兩個時段），區塊便會進入 Finalized（最終確定） 狀態，此時資產狀態便幾乎不可逆轉。

## 四、 規則與演進：智能合約與協議升級

### 1\. 智能合約與社會學意義（Code is Law）

智能合約是以字節碼（Bytecode）形式寫在鏈上的程序，由 EVM（以太坊虛擬機）負責執行。

-   信任轉移：它將傳統商業中對「律師、政府、中介、法律合約」的信任，轉移為對「不可篡改之代碼、數學與共識」的信任。規則公開透明、強制執行，大幅降低了跨國協作與金融流轉的磨損成本。
    

### 2\. 智能合約如何實現「升級」

寫在鏈上的合約代碼一經部署便無法修改，為了解決 Bug 或進行功能擴展，業界引入了代理合約（Proxy Contract）模式。

-   用戶交互的永遠是「代理合約」的固定地址。
    
-   代理合約內部儲存了一個「指針」，指向真正負責運行的「邏輯合約」。
    
-   當需要升級時，開發者部署一個全新版本的邏輯合約，並更新代理合約中的指針指向。用戶端無需更改任何設定，即可完成功能升級。
    

### 3\. 底層協議層的去中心化升級

區塊鏈網絡本身的升級（如硬分叉、改變區塊數據結構）是一個去中心化的社會治理過程：

-   提案經論壇與核心開發者會議討論，形成 EIP/ERC 標準文檔。
    
-   全球不同團隊使用不同程式語言（如 Go、Rust）開發的多客戶端軟體（如 Geth、Nethermind 等）各自實施代碼，並在約定的區塊高度同步開啟新功能旗標。
    
-   多客戶端機制確保了當單一軟體出現漏洞（Bug）時，其餘客戶端仍能支撐網絡暢通，避免全網癱瘓。
    

## 五、 結論：Web3 的多學科交織

Web3 並非單純的資訊技術，而是三門學科的交叉結晶：

1.  密碼學（Cryptography）：利用哈希演算法、非對稱加密與簽名，保障數字資產的所有權與身份防偽。
    
2.  經濟學（Economics）：利用質押激勵、Gas 費處置與懲罰機制，協調全球陌生參與者
    
    的利益動機。
    
3.  社會學（Sociology）：利用去中心化治理、開源文化與社會共識，決定網絡的演進規則與主權分配。

### 一、 AI 的進化與 AI Agent 核心機制

-   **AI 的進化階段**：
    
    -   **早期（如 ChatGPT 剛推出時）**：以問答式交互為主，本質上像更智能的搜尋引擎，主要做解釋代碼、總結文章、寫文案等。
        
    -   **現階段（AI Agent 時代）**：AI 開始能執行實際任務，走向自動化編程與通用任務處理，從單純的「聊天助手」轉變為「行動智能體」。
        
-   **AI Agent 的核心運作機制**：
    
    用戶給予提示詞 -> 理解任務目標 -> 使用外部工具執行 -> 觀察並檢查執行結果 -> 若有錯誤進行**自我自動修復**，形成一個閉環。
    
-   **與一般聊天機器人的本質區別**：
    
    聊天機器人主要被動回答問題；而 Agent（智能體）則能理解人類語言背後的「目標」，主動調用工具去「替你完成事情」。
    

### 二、 市面上的 AI 生態五大陣營

講師將目前市面上的 AI 工具劃分為以下五大類別：

1.  **聊天型 AI**：如 ChatGPT、Claude、DeepSeek 等。

5.  **通用助手底座（如 OpenHands、Hermes Agent）**：這類工具連接了大模型、本地工具、記憶系統與消息平台，就像一個小型「賈維斯」（Jarvis），只要給予授權，就能在系統後台幫你做很多事。

### 核心背景與基礎概念

什麼是 MPC（多方安全計算）？

-   傳統私鑰：如同開啟金庫的唯一實體鑰匙，一旦遺失或被竊，資產便徹底不保。
    
-   MPC 技術：將鑰匙「打碎成多個碎片（私鑰分片）」，由不同成員（如用戶手機、雲端伺服器）各自保管。
    
-   共同簽名：轉帳時，各方拿出各自的碎片進行「密碼學運算」，在不拼湊出完整鑰匙的前提下，共同完成交易簽名。
    

### BitForge 漏洞的本質

-   漏洞名稱：BitForge
    
-   受影響對象：幾乎所有主流開源的 GG18 與 GG20 MPC 協議實現。
    
-   問題根源：MPC 協議的安全建立在特定的數學假設（Paillier 同態加密）之上。
    
-   該漏洞利用了系統未嚴格檢查參與者金鑰合法性的漏洞，導致「安全地基」崩塌。
    

### 駭客攻擊步驟拆解

駭客若想竊取其他參與者（例如雲端伺服器）的私鑰碎片，會經過以下三個階段：

1.  第一步：準備階段（構造非法金鑰） 正常做法：使用非常巨大且複雜的兩個大質數 p, q 來生成數學保險箱的公鑰 N = p 乘 q。 駭客作弊：故意不用大質數，而是用很多「很小的質數」拼湊成公鑰 N = p1 乘 p2 乘 ... 乘 p16 乘 q。 結果：由於原協議只檢查了保險箱是否「無平方因子（Square-Free）」，駭客成功騙過系統混了進去。
    
2.  第二步：攻擊階段（在轉帳簽名時作弊） 駭客故意發起交易，並在簽名運算（MtA 協議）中，用特殊設計的數值代替隨機數。 正常的「零知識證明（Range Proof）」本該擋下這種惡意數值，但駭客利用第一步那個「偷工減料的保險箱」，成功偽造了假的證明，騙過其他參與者。 駭客在順利完成簽名的過程中，偷偷記錄下每次運算留下的數學線索。
    
3.  第三步：收尾階段（拼湊真相） 駭客只需要重複發起約 16 次轉帳簽名，就能收集到 16 個不同小因子對應的線索。 最終利用數學上的「中國剩餘定理（CRT）」，將這 16 個線索進行拼湊，就能精準推導出被攻擊方的私鑰碎片！ 最終後果：駭客擁有自己的碎片，又偷到了別人的碎片，即可合成完整私鑰，徹底掌控錢包。
    

### 防禦與修復方案（補上漏洞） 安全團隊（Fireblocks 與 Safeheron）提出的修復方案是從源頭掐斷攻擊，也就是在建立錢包時，強制進行「零件硬度檢測」。

### 在協議中額外添加了兩道零知識證明（ZKP）：

1.  Paillier Blum 模數證明：保證公鑰 N 最多只能由兩個質數組成，防止被拆成一堆小數字。
    
2.  無小因子證明：確保公鑰 N 中絕對沒有小於 2 的 256 次方的小質數因子。
    

-   有了這兩道防線，駭客在第一步提交「偷工減料的保險箱」時就會被當場抓包並拒絕。
    

### 如何檢測歷史數據是否曾被攻擊？

-   檢測原理：駭客攻擊的鐵證是歷史數據中一定存在「包含小因子的公鑰 N」。
    
-   操作方式：使用成熟的大整數分解算法工具，批量檢查過去所有 MPC 參與方提交的公鑰 N。
    
-   後續建議：若檢測出小因子，說明該錢包可能已被攻擊，應立即將資產轉移到已修復漏洞的新錢包中。

# 5/18(一)課程筆記

### 核心觀念：AI 時代的開發者定位

-   **基礎知識的不可取代性**：有了 AI，基礎知識反而更重要。AI 能大幅提升 Coding 與學習效率，放大個人能力，但**系統複雜度、錯誤成本與安全邊界並沒有改變**。
    
-   **人的價值定位**：人負責設計架構，AI 協助細化與執行；人必須擔任 AI 產出的「審查者」。開發者若缺乏紮實的基礎知識與架構能力，將無法判斷 AI 生成的程式碼是否正確（安全的前提是基礎紮實，否則都是空中樓閣）。
    
-   **結論**：我們要駕馭 AI（將其作為放大器），而非被 AI 驅動，最終的審核與安全責任仍在開發者手中。
    

### Web2 與 Web3 的融合與差異

-   **並非互斥，而是共生**：Web3 不是完全獨立於 Web2。多數 Web3 專案（如交易所 Binance、預測市場 Polymarket）仍重度依賴 Web2 的前端頁面與後端管理技術，Web2 的經驗可大量複用。
    
-   **底層信任機制的轉換（以支付系統為例）**：
    
    -   **Web2 (Bank Transfer)**：透過 API 交互，信任基礎在於 KYC、密碼權限與「中心化機構後端 (Institutional Backend)」。
        
    -   **Web3 (穩定幣支付)**：將銀行機構替換為 Blockchain，不同公鏈形同不同銀行。信任基礎轉移為：透明度 (Transparency)、不可篡改性 (Immutability)、共識機制 (PoW, PoS) 以及私鑰簽名。
        

### Web3 核心命題：安全與資產保護

在 Web3 中，涉及大量資金交互，「安全」不是事後附加的插件，而是貫穿產品生命週期的核心屬性。在 Web3，**對安全的理解力等同於資產保護力**。

1.  錢包與身份
    

-   **定義**：錢包在 Web3 中代表用戶的「身份」與「資產管理」憑證，用以證明用戶行為。
    
-   **私鑰與簽名**：私鑰是透過數學算法保證的大數，運作邏輯為`Message -> Input Private Key -> Output Signature`。
    
-   **錢包分類與託管**：
    
    -   類別：EOA (公私鑰對)、Smart Contract Wallet (智能合約錢包，如 Safe 多簽)、AA (帳戶抽象，結合 EOA 與智能帳戶)。
        
    -   託管方式：自託管、全託管 (如交易所)、混合託管 (Hybrid)。
        
-   **錢包安全防線**：嚴防私鑰外洩 (Private Key Leak)、防範簽名欺騙 (如禁用 `eth_sign`、提高簽名可視性)、防止權限濫用。
    

2.Web3 交易週期與關鍵參數

交易週期：**構造 -> 簽名 -> 廣播**

-   **Gas (交易手續費)**`Gas Fee = Gas Count * Gas Price` (受 EIP-1559 影響)。
    
-   **Nonce (確保交易冪等性，防重放攻擊)**：
    
-   ETH (Account 模型)：使用預設 Nonce 排序。
    
-   Tron/Solana：無 Nonce，透過 SDK 使用交易 Hash 確保。
    
-   BTC (UTXO 模型)：Cash 只能用一次，原生具備冪等性。
    
-   **Calldata**：鏈上交易的靈魂，可視為 Interpreter，執行端會自行解析交易 Data  
    

3.交易安全實踐

-   **交易模擬** ：在提交交易前驗證執行結果，且**必須在「簽名」前進行模擬**。
    
-   **Server 端安全**：資金量拆分以減少單點風險；利用可信任執行環境 (TEE) 保護私鑰，確保無任何明文暴露。
