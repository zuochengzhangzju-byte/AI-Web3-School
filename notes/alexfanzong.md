## Ethereum × AI Agent Economy — 学习笔记

Sophia 这场分享的起点是一个观察：过去十八个月，AI agent 已经从"可以问问题的聊天机器人"变成了真正的经济参与者——它们写代码、做决策、调用服务，甚至雇佣其他 agent。当 agent 开始参与经济，一个根本问题随之浮现：它们应该跑在什么样的基础设施上？

Web2 的支付与身份体系并不是为 agent 设计的。Agent 守不住密码，签不了法律合同，也无法在人类信誉体系里积累声誉。给 agent 一张信用卡让它买披萨，它可能乱花钱、买错商家、甚至买上千个。但如果给它的是一个限额 10 美元、指定商家的智能合约钱包，可编程货币本身就构成了行为约束——这是 agent 经济与传统支付分道扬镳的根本点：钱本身可以携带规则。

EF 的核心论点是，以太坊正在成为 AI agent 经济的协调与结算层。让它胜任这个角色的，是四条原生设计原则 CROPS：抗审查、开源免费、隐私、安全。这些属性恰好是 agent 间协作最需要的底层保障——必须中立、可验证、不可被任意切断。

具体落地的标准里，最值得关注的是两个。ERC-8004 为 agent 提供链上可验证身份、注册表与信誉评分，任何场景下都可以查询"这个 agent 是否可信"，且信誉绑定的是 agent 本身而非背后的人；ERC-8003（x402）基于 HTTP 402 状态码加稳定币结算，让 agent 之间可以直接付款，免去信用卡、API key、发票等传统流程。

EF 自己也在试点 Deep Funding，用 AI agent 协助分配公共物品资金。关键不是让 agent 决定钱给谁，而是人类设定价值标准、agent 扩展人类判断的规模——这是整场分享里最重要的治理姿态：机器扩展人类判断，而非替代人类决策。

对 Web3 × AI 的建设者来说，这场分享真正的信号是：agent 经济此刻还在早期，但今天选择的协议、信任机制、支付轨道，会随着 agent 活动规模化而被路径依赖锁死。Sophia 反复强调一句话——Ethereum for AI 最终意味着 Ethereum for Humans。如果 agent 经济跑在某一家公司的封闭基础设施上，被锁定的不仅是 agent，而是所有依赖 agent 的人。

# 2026-05-23

今天参加了上海 Bittensor AI Subnet Ideathon，10 小时，从早到晚，记录一下。

这次黑客松的主题是设计一个 Bittensor 子网。Bittensor 的核心理念是把 AI 能力拆成一个个垂直市场——每个子网就是一个市场，矿工生产智能商品，验证者审计质量，Yuma Consensus 聚合共识，再用代币激励分配奖励。目前生态已经有 128 个子网在运行，覆盖推理、训练、数据、金融预测等方向，Q1 真实调用收入超过 4300 万美金。

我提交的项目叫 LUO（Legal Uncertainty Oracle，多向罗盘），做的是跨辖区法律风险拓扑图。切入点很简单：法律 AI 最危险的问题不是"不知道"，而是模型编造一个不存在的法条或监管文件，格式做得跟真的一样，让你以为自己拿到了确定的答案。LUO 不输出法律答案，它惩罚伪造的确定。

具体机制是矿工从法律语料库里抽取证据，把每个法律主张绑定到具体文档。验证者从三个维度打分：引用是否真实存在、是否命中我预埋的伪造陷阱、结论有没有越过证据边界。我用 Tornado Cash 做了 demo 案例——同一个协议在美国、荷兰、瑞士、香港有四种完全不同的法律命运，传统法律意见书会把这种分歧压缩成一个结论，LUO 做相反的事，把分歧展开。

现场还有一个法律方向的项目 LegalMind，做的是单个 Case 的法律分析质量竞赛。跟 LUO 的区别是：他们奖励正确性，我们惩罚伪造的确定性；他们做单辖区深度分析，我们做跨辖区分歧地图。产品不同，客户不同。

今天最大的收获是跟 SN120 Affine 的 Nick 聊了子网运营的实战经验——防矿工作弊是每天最大的工作，评分函数设计决定了子网的生死。Cameron 也反复强调：评委最看重的是你的智能商品定义和反作弊机制，不是代码量。

一句话总结 LUO：Chainlink 告诉你价格在哪里，LUO 告诉你法律不确定性在哪里。

# 2026-05-22

**AI 缺的是可信， Web3 缺的是智能。**

AI 输出再强，落到金融、医疗、跨境合规这些高价值场景，第一个被问的永远是“凭什么信你”——模型黑箱、数据来源不可验证、推理过程不可追溯。而 Web3 这边正好相反：链上一切可验证、可审计、可追责，但智能合约本身是死的，只能 if-else，碰到任何需要“判断”的环节就抓瞎。

所以真正有意思的不是“用 AI 写 Solidity”这种工具层叠加，而是结构层的互补：把 AI 的判断能力，塞进链上可验证的容器里。

**对我自己的工作来说，落点很清晰：**

第一，AI Oracle 是 Programmable Compliance 的关键拼图。合规判断里有大量“非二元”问题——这份 KYC 材料是否真实、这笔交易是否疑似洗钱、这个对手方风险等级。这些事 if-else 写不出来，但如果让 AI 判断，输出又没法上链取信。zkML、TEE、可验证推理这些方向，本质都是在解“如何让 AI 的判断结果，能被智能合约信任”——这正是 ComplianceModule 真正能落地的前提。

第二，AI Agent 上链交易这件事，给跨境合规出了新题。Agent 自主调用合约、自动支付、跨链转账，传统 KYC 的“自然人/法人”框架直接失效。监管最终一定会追问：这个 Agent 是谁的代理？谁承担义务？这是 OfferLock 这类场景之外，Programmable Compliance 必须回答的第二个问题。

第三，工具层面，AI 已经彻底改变了合规工作的成本结构。法律条款解析、合规决策树提取、合约审计、跨法域比对——以前要团队做的事，现在我一个人加 Claude Code 就能跑通。这不是效率提升，是工种重构。

**一句话总结，** AI × Web3 的真正叙事，不是“AI 让 Web3 更好用”，而是“Web3 让 AI 终于可信”。前者是产品迭代，后者是范式转移。

# 2026-05-21

今天的讲座由交易所资深合约工程师 Elon 老师主讲，核心聚焦于高度中心化的 AI 与去中心化的 Web3 如何进行双向促进与范式融合。  
在 Web3 赋能 AI 维度，传统的 AI 算力和数据高度集中在少数巨头手中，容易受到政策与文化审查的限制。而以 Bittensor 为代表的 Web3 项目，正在打破这种垄断。它通过子网（Subnet）划分不同的 Agent 技能方向，并引入分布式算力质押机制。其商业设计的精妙之处在于“非链上计算、链上打分”：具体的数据处理和模型运行在链下各个分布式节点或集群中高速完成，而验证者节点仅在链上对服务质量进行评估打分，并据此分发代币激励，这为构建属于全世界的去中心化通用智能体底座提供了可能。  
在 AI 赋能 Web3 维度，两者的交叉孕育出了极具爆发力的应用场景。首要方向是 “意图感知（Intent-centric）与智能自主钱包”。传统的 Web3 入门门槛极高，而通过大模型理解用户的自然语言，将“帮我把 USD 换成 ETH 并定投”等语义意图直接转化为链上智能合约的自动执行，能让钱包从被动工具演变为主动的财产管理助手。其次，AI 在 链上行为分析与黑客攻击模拟审计 方面展现出巨大商业价值，通过实时监控全链数据进行地址标签画像，或通过 AI 自动化安全分析插件对目标交易进行风险评分，能够极大提升 UA 账户的资产安全性。  
核心启发：无论是大公司的宏观布局还是独立开发者的黑客松试水，未来的破圈爆款必然属于那些能够“降低 Web3 交互门槛”或“提供智能化主动风控”的 AI×Web3 交叉产品。在 AI 极大解放编程生产力的时代，精准切入应用场景的产品设计能力，将比单纯的代码编写更具商业壁垒。

**1\. 资产自托管：如何提高安全性、降低私钥复杂度？** 方向是把"单点私钥"拆成"可恢复的权限系统"。技术上靠 ERC-4337 账户抽象 + MPC/Passkey + 社交恢复，让签名权、恢复权、日常操作权分离；合规上对应"自托管不等于无人负责"——前端、Relayer、Paymaster 都是潜在受监管节点（参考 SEC Staff Statement 对 DeFi 前端的处理逻辑）。安全性提升的本质，是把密码学难题转化为可治理的权限结构。

**2\. 没有中心化机构/税收时，公共基础设施谁来维护？有税收又如何分配？** 答案不是"要不要税"，而是"税基怎么编程"。协议费、MEV 再分配、Sequencer 收入、RWA 利差，都是链上原生税基。分配则通过 Public Goods Funding（Gitcoin QF、Retro PGF）+ DAO 国库 + SBT 政治权重实现。这正是我"双轨治理"框架——SBT 决定谁有资格分配，Unit 决定分多少——回应的问题：去中心化不是去财政，而是把财政逻辑写进合约。

**3\. 没有审查且隐私很强时，如何界定并治理有害信息/黑产？** 治理点必然从"内容层"上移到"交互层"和"出入金层"。链上隐私（ZK、混币）保护的是默认状态，但 Travel Rule、KYT、地址画像、合规路由（ComplianceRouter）可以在资金进出可监管节点时介入。核心是承认"协议中立 + 边界合规"：协议本身不审查，但与法币、RWA、机构资产交互的路径必须可编程合规。隐私不是豁免，是默认设置。

**4\. 去中心化协作下，如何实现公平、可信的分配？** 公平不可能靠"投票多数"，必须靠机制设计。可信分配三要素：贡献可度量（链上行为、SBT、Attestation）、规则可验证（合约即合同）、争议可救济（Kleros、Optimistic Oracle、链下仲裁衔接）。我在 LXDAO 框架里提的方案是 SBT 锁定政治权利、Unit 锁定财务权利，分离"who decides"和"who gets paid"，避免金权合一导致的治理俘获——这才是 Programmable Compliance 在分配层的落地。

# 2026-05-19
<!-- DAILY_CHECKIN_2026-05-19_START -->





在本次 AI×Web3 School 的分享课程中，Draken 老师深入拆解了从“聊天机器人”到“智能体（Agent）”的范式转移。传统的 AI 只是问答式的聊天助手，而真正的 Agent 核心运作机制在于理解目标→调用外部工具→观察执行状态→自动自我修复的闭环。  
课程中横向对比了目前市面上的主流 AI 生态阵营：

-   **开发利器**：如 `Claude Code`，作为深度集成 AI 的 IDE 或 CLI 工具，是开发者在终端环境进行高效编程交互的直接选择。
    
-   **通用底座**：以 `Hermes` 为代表的通用助手底座。相较于深度定制的 `OpenInterpreter`，`Hermes` 对新手更为友好。其核心超能力在于**记忆（Memory）持久化机制**与**工作流固化（Skills）**。它能在日常交互中沉淀固定的任务流程，遇到同类任务时自动调用已有 Skill 达成流程化自动执行。
    

从商业与可落地的博弈视角来看，通用 Agent 的最大价值在于其多端接入能力。通过配置消息网关，将底座模型（如 Claude 或 DeepSeek）无缝对接到微信、Telegram 等消息平台。这不仅打破了传统的终端操作壁垒，更让移动端的定时任务、实时新闻简报和个性化学习计划自动生成成为可能。  
核心启发：在 AI 时代，未来的超级个体不应局限于代码堆砌，而应学会将大模型能力、本地工具与记忆流组合，将固定任务封装为自动化 Skill，借由高频社交网关释放生产力，构建属于自己的高效商业护城河。

今晚听了 Web3SecTc 老师的第一场分享会，主题是《AI 时代 Web3 开发者需要具备的基础知识和架构能力》。tc 老师是安全行业出身，最早做二进制病毒分析，目前在香港某 Web3 公司担任技术专家，所以整场分享里"安全视角"贯穿始终，跟我之前听过的偏业务向、偏叙事向的 Web3 分享都很不一样。

老师的讲法非常工程师范——没有上来抛概念，而是用一个最简单的支付系统作为切入点，从最朴素的 Web2 模型开始一步步推演，让没接触过 Web3 的人也能跟得上。整个分享被分成四个 part：常见疑问、系统设计、变与不变、AI 时代架构师应有的能力模型。中间穿插了大量安全设计的考量，把钱包、交易、签名、模拟、服务端安全这些重点概念用一个故事线串起来，对我这种法律背景出身、技术基础不算扎实的人来说，体感非常友好。

最受启发的部分是老师讲解"人与 AI 协作边界"的那段——他没有把 AI 描绘成万能助手，也没有渲染焦虑，而是冷静地指出 AI 改变了什么、没改变什么。听的时候我一直在对照自己最近用 AI 写合约、做架构设计的体验，确实有不少地方击中了我，让我重新审视了自己当前的工作方式。

Q&A 环节也很值得听，提问者背景各异，从在校生到转行者到正在 build 项目的开发者都有，老师的回答都很坦诚——不会的就说不会，能给建议的会给出非常具体的路径。我自己也从其中一位提问者关于 AI Agent 支付场景的问题里得到了一些产品上的反思。

整体下来两个小时干货密度很高，是值得回看二刷的分享。下来要把笔记整理一下，把里面提到的几个工具和开源项目都过一遍。
