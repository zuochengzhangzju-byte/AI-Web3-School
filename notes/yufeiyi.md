\## 学习内容

\### 模块：LLM（大语言模型）

**来源：** \[Handbook LLM\]([https://aiweb3.school/zh/handbook/ai/llm/](https://aiweb3.school/zh/handbook/ai/llm/))

\#### 核心认知

\> LLM 生成的是概率上合理的输出，而不是天然可信的事实。

LLM 本质上是一个\*\*超高阶的自动补全引擎\*\*——它不是在找"正确答案"，而是每一步只在问"下一个词最可能是什么"，走完所有步骤后，输出看起来像一个"答案"。

**关键的推导过程：**

1\. LLM 没有"我不知道"这个选项 → 它必须输出 token

2\. 遇到没见过的东西 → 强行拼一个概率上合理的序列

3\. 这不是"它选择编" → 是它\*\*本来就没有在找答案\*\*

4\. 所以解决方案不是"让它别编"（架构决定的）→ 而是靠外部校验

\#### Temperature 实验记录

同一个 prompt "用三句话解释什么是区块链"：

| 模式 | 输出风格 | 特点 |

|------|---------|------|

| T=0（确定） | "区块链是分布式的数字账本。数据以区块形式链接存储。不可篡改是核心特性。" | 教科书式、高密度、每次一致 |

| T=0.7（平衡） | "区块链是一个去中心化的账本系统。每个区块包含交易数据，用密码学链接。" | 自然、类人 |

| T=1.5（随机） | "想象一个大家都可查看的公共账本，记录永不消失！" | 有创意、可能跑偏 |

**结论：** 分类/提取/JSON 输出 → 用 T=0。剧情/NPC 对话 → 可用高温度。

\#### 关键概念总结

| 概念 | 一句话理解 |

|------|-----------|

| **Token** | 基本单位，中文 1-2 token，代码更贵 |

| **Temperature** | 0=确定，1+=随机 |

| **Hallucination** | 架构决定的必然行为——没有"不知道"选项 |

| **Prompt** | 接口设计，不是魔法咒语 |

| **Instruction** | 四段式：任务目标 / 可用输入 / 禁止行为 / 输出格式 |

| **Structured Output** | JSON/schema，让输出可被代码检查和校验 |

| **CoT** | 一步步推理，减少错误 |

| **Prompt Injection** | 靠代码层防，不靠 prompt 防 |

\#### Prompt 核心原则

**第一性原理：** Prompt 是软约束，不是安全边界。真正的边界必须由代码、权限、校验和审计来承担。

**关键区分：** "解释" vs "执行"

\- 研究助手可以整理资料，不能假装结论已验证

\- 代码助手可以生成 patch，不能默认已通过测试

\- 交易解释器可以解释风险，不能擅自替用户确认

\## 今日核心思考

\### 1. AI 时代什么最值钱？

\> **AI 时代 = 你的决断 × 你的判断质量 × AI 的执行速度**

| 能力 | AI 能做吗 | 人类优势 |

|------|----------|---------|

| 记忆 | ✅ 完胜人类 | ❌ 不必再卷 |

| 谋划 | ✅ 生成大量候选方案 | ❌ 优势缩小 |

| 决断 | ❌ 不能承担后果 | ✅ **人类最后的领地** |

**重新定义"执行力"：**

\- 以前的执行力：从 0 开始动手写

\- 现在的执行力：判断 AI 给的方案对不对 → 改 → 再判断 → 交付

执行力的核心变成了"判断质量的执行密度"——你有多快判断 AI 的产出有没有坑。

\### 2. 小说崩盘与 LLM 幻觉——同一个问题

长篇网文写到中后期崩盘，原因和 LLM 产生幻觉\*\*一模一样\*\*：

\> 系统只关注"当前这一步怎么生成才顺"，不关注"我之前承诺过什么"。

| 小说崩盘 | LLM 幻觉 |

|---------|----------|

| 主角第 100 章金丹，第 300 章又说筑基 | 先输出"Alice 余额 3 ETH"，后面又说"转完还剩 2 ETH" |

| 配角前面死了后面又活了 | 同一个 API 两次解释不一致 |

| 作者忘了前面的伏笔 | 模型混入旧版本文档 |

**解决方案也相同：** 把全局状态从生成系统内部移出来，放到外部可信存储。

\### 3. Web3 能保证不可篡改——但真正的问题是"存什么"

链上存储（NFT/Token）保证不可篡改 → ✅ 已实现。

但真正的问题是：

\> **AI 自己判断不了"这个信息需要锁死"还是"这只是临场发挥"。**

对 Web3+AI 游戏的实际分层：

\`\`\`

NPC 对话 → LLM 生成（不需要上链）

玩家资产（装备、金币） → 链上 NFT/Token ✅

任务进度 → 链下数据库（快速读写）

重要节点（Boss 首杀、关键选择） → 链上哈希存证 ✅

\`\`\`

\### 4. 【核心洞察】小说是"变化过程"，不是"静态设定"

当前主流思路认为 AI 写小说崩盘的原因是"记不住设定"→ 解决方案是"数据库存设定"。

但这个思路有盲区：

\> **小说的本质是变化过程，不是事实快照。**

甄嬛传可以用一句话概括：一个官家女子宫斗成为太后。

红楼梦：四大家族衰亡史。

这两个过程中，\*\*中间状态在不断变化\*\*：

\- 第 50 章：甄嬛爱皇帝 ✅

\- 第 150 章：甄嬛恨皇帝 ✅（不是矛盾，是成长）

如果用"锁死事实"的思路去做：

\- 第 50 章锁进数据库"对皇帝感情：爱"

\- 第 150 章想写恨 → 矛盾

**问题不是 AI 记错了，而是人物成长了。**

\#### 真正无人解决的问题

\> **如何让 AI 始终知道"当前状态是通往终点的第几步"，而不是"第几步之前承诺过什么"？**

这比"记住设定"深一层：

\- 理解\*\*变化弧线\*\*（arc），而不是一张事实表

\- 判断"这一步变化是否在允许的轨迹上"

\- 区分"角色成长"👌 vs "矛盾吃书"💩

\#### 对应到 AI 概念

| 你的洞察 | 对应概念 |

|---------|---------|

| 小说是变化过程 | **State** + **State Transition** |

| 需要知道"到终点的第几步" | **Planning** — 目标拆解 |

| 如何判断"成长"还是"崩坏" | **Evaluation** |

| 50章爱→150章恨不矛盾 | 用"关系状态机"而不是"关系字符串"建模 |

\#### 对这个问题的 Hackathon 方向思考

这个洞察完全可以直接变成一个项目方向：

\- 一个"变化轨迹管理系统"（用链上状态机管 arc，AI 填充内容）

\- 核心问题：\*\*什么该锁、什么该变、变到哪一步了\*\*——目前全部是人的判断

\### 5. 最终的认知闭环

\`\`\`

LLM 是概率模型

→ 它不知道自己在说什么，只是选下一个最顺眼的词

→ 所以它天然会编造（hallucination）

→ 所以它需要外部状态管理（数据库/链上）

→ 但 AI 自己判断不了"什么该存、什么该变"

→ 这个判断（决断）只能由人来做

→ 所以 LLM 是推理层，人类是决策层

\`\`\`

**AI 时代，决断力 > 谋划力 > 记忆力。**

\### 6. 延伸论证：LLM 拥抱"多数"——真理为什么有时在少数人手里

这个结论不是空话。从 LLM 的底层机制可以直接推导出来：

\> **LLM 的"知识"来自训练数据的统计分布。它输出的是"大多数人认为什么是对的"，而不是"客观上什么是真的"。**

利弗莫尔做空 1929 年大崩盘的案例完美暴露了这个局限：

| | 1929 年的主流 | 利弗莫尔 |

|---|---|---|

| 判断 | "股市永远涨" | "泡沫即将破裂" |

| 数据支持 | 95% 的人这么说 | 极少数 |

| LLM 如果训练在当时 | 输出"涨"（概率最高） | 输出不了"崩"（概率太低） |

| 谁对了？ | ❌ 错 | ✅ 对 |

LLM 不是故意选错的——它\*\*没有"主流有可能是错的"这个认知框架\*\*。它的架构决定了它只能选"统计上最顺眼的"。

这不仅是 LLM 的局限，是所有"统计学习"方法的共同盲区：

| 方法 | 依赖 | 盲区 |

|------|------|------|

| LLM | 训练数据分布 | 统计上的少数（即使是真理） |

| 搜索引擎 | 链接权重 | 没人写过的新问题 |

| 量化模型 | 历史模式 | 市场结构的根本变化 |

| 专家系统 | 编码规则 | 规则之外的创新 |

**这些方法都有一个共同点：输出基于过去，而真理有时在未来。**

\#### 认知链的最后一环

\`\`\`

LLM 是概率模型

→ 它选"统计上最顺眼的"而不是"客观上真的"

→ 所以多数人认为对的它就输出对的

→ 但真理有时在少数人手里

→ 所以 LLM 在"主流是错的"场景必然失效

→ 这种"识别主流错误并反向行动"的能力

恰恰是利弗莫尔、索罗斯、芒格说的那种判断力

→ 而判断力 → 人类最后的领地

\`\`\`

**"少数人掌握真理"不是玄学，是这个系统架构的必然推论——统计模型天生滞后于结构性的认知突破。**

\### 7. 练剑三境的隐喻

用中国古人的练剑境界来理解模型能力的演化：

| 境界 | 练剑 | 模型 |

|------|------|------|

| 第一境 | 手中有剑，心中有剑 | 模型刚训练完，每一步有意识地选 token |

| 第二境 | 心中无剑，手中有剑 | 经微调后，回答自然流畅，看起来像"懂了" |

| 第三境 | 手中无剑，心中无剑 | **只有人能到**——可以不输出、可以不判断、可以选择不做 |

模型永远卡在第二境——它看起来圆融如意，但本质上还是"必须生成 token"。第三境是"选择不做的能力"，这是概率模型从架构上做不到的。

\### 8. 终极类比：Prompt 的软边界 ≈ 这个世界上没有人不可收买

如果你理解了"Prompt 的边界是概率的，会随输入漂移"——你会发现它和一个朴素的人类社会规律完全同构：

| 人心世界 | LLM 世界 |

|---------|---------|

| 每个人都有个"忠诚度"基线 | 每个输出都有个"默认概率" |

| 出价不够高 → 不动摇 | Prompt 画了软边界 → 不越界 |

| 出价够高 → 忠诚度重新校准 | 输入够强 → 概率分布被重新分布 |

| 反复收买 → 底线越来越低 | 连续的对抗输入 → 边界持续漂移 |

| 没有不可收买的人，只有出不起的价 | 没有绝对安全的 prompt，只有不够强的攻击 |

**在概率驱动的系统里，一切约束都是可谈判的。**

所以"怎么防"的逻辑也和现实世界一模一样：

| 现实世界 | LLM 世界 |

|---------|---------|

| 核心资产不放人手里 → 保险柜 | 硬边界（代码 schema 校验） |

| 敏感操作双人签字 | Human-in-the-loop |

| 审计追责、事后查处 | 日志 + 追溯 |

| 不给守卫太多权限 | Session Key 限权 |

**没有任何一个系统靠"守卫的忠诚度"来保证安全——靠的是制度设计。**

对应的技术翻译：

\`\`\`

❌ "我写了一段超强的 prompt，它一定不会泄露信息"

\= "我加倍给了守卫工资，他一定不会被收买"

✅ "LLM 只能做只读操作，写入必须过代码层校验"

\= "核心金库的钥匙不在守卫身上，在保险柜里"

\`\`\`

**一句话：不要在概率世界里解决需要逻辑世界才能解决的问题。**
