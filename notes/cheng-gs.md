今天我主要完成了 **模块 A：AI × Web3 方向扫描与主方向选择** 的学习和初步选题。

首先，我快速梳理了 AI × Web3 的几个主要交叉方向，包括 Payment / Commerce、Identity / Reputation、Wallet / Permission、Privacy / Security、Dev Tooling / Agent Workflow 和 Governance / Coordination。我理解到，这一阶段的重点不是同时做很多项目，而是先看清不同方向分别在解决什么问题、适合什么类型的学员，以及它们更可能落在产品、工具、协议、风控模型还是研究层。

然后，我明确了 **Agent DeFi Execution** 的定位。它不是一个新的基础方向，而是一个 sponsor-defined applied path，主要是把 Payment / Commerce、Wallet / Permission 和 Privacy / Security 放到 DeFi 链上执行场景中集中检验。它可以作为 Cobo 相关 Hackathon / workshop 的优先应用路径。

在主方向选择上，我初步选择了两个自己感兴趣的方向：

1.  **Privacy / Security / Sovereignty**
    
2.  **Dev Tooling / Agent Workflow**
    

对于 **Privacy / Security / Sovereignty**，我认为真实用户是那些希望使用 AI agent 提升效率，但又不能接受敏感数据、权限、资产或身份失控的 Web3 用户和团队。这个方向的问题在于，agent 一旦接触钱包、API key、私钥、治理权限或敏感信息，就会带来 prompt injection、tool abuse、权限滥用和数据泄露等风险。如果没有 AI，很难实时、低成本地识别这些复杂风险；如果没有 Web3，这个问题又会缺少资产、身份、链上权限和可验证执行这些关键场景。因此，这个方向更适合做 risk model、security tooling 或 research memo。

对于 **Dev Tooling / Agent Workflow**，我认为真实用户是 Web3 builder，包括智能合约开发者、Hackathon 参赛者、协议团队工程师、DevRel 和安全研究人员。这个方向关注的是 AI 能否真正改善 Web3 开发流程，例如合约阅读、交易解释、docs-to-agent、部署助手、测试脚本生成和 repo 自动维护。如果没有 AI，Web3 开发流程中的信息整合、代码理解、错误解释和上下文切换会非常低效；如果没有 Web3，这个方向就只是普通开发者工具，缺少链上状态、智能合约、交易执行和协议组合这些独特场景。因此，这个方向更适合做 developer tooling 或 product demo。

目前我的初步判断是：如果要快速做出 Hackathon demo，**Dev Tooling / Agent Workflow** 会更容易落地；如果要做长期研究或安全框架，**Privacy / Security / Sovereignty** 更有结构性价值。

我也发现这两个方向其实可以结合起来，形成一个更聚焦的主线：

**Secure Dev Tooling for Web3 Agents**

也就是做一个面向 Web3 builder 的安全型开发者 agent。它不仅能解释合约、交易和文档，还可以标注权限风险、敏感数据风险和 agent 执行风险。这个方向既有 developer tooling 的可展示性，也能承接 privacy / security 的长期价值。

# **Week 1 总结**

这一周对我来说，最重要的不是学了多少新名词，而是第一次把 AI 和 Web3 放到同一条实际可执行的流程里理解了。

我的背景是 **AI 比较熟，Web3 还是新手**，所以这周我最明显的学习策略就是先补 Web3 基础，再把这些基础和我已经熟悉的 AI workflow、Agent 思维连接起来。过去我对钱包、签名、交易、Gas、合约这些词有大概印象，但这一周之后，我开始能把它们放进一条完整链路里理解，而不是把它们当成彼此孤立的概念。

在 AI 这一侧，我不是单纯体验了一个工具，而是把 **Codex 当成自己的 Learning Agent** 来用。我用它帮我初始化了学习仓库、生成学习计划、整理概念卡片、生成 daily note、写运行记录、写提交材料，也开始尝试让它参与更接近执行层的任务，比如根据自然语言修改 Solidity 合约草稿、生成部署前说明和记录模板。这个过程让我更清楚地理解了 prompt、workflow、agent、tool use 和 human-in-the-loop 之间的区别。对我来说，这周一个很关键的收获是：**AI 很适合解释、整理、起草和辅助执行前准备，但一旦接近真实签名、授权、部署和转账，就必须明确把人工确认放回流程中心。**

在 Web3 这一侧，这周我完成了几件真正有“手感”的事情。我创建了测试钱包，明确区分了 **地址、助记词、私钥** 分别是什么，也真正理解了为什么助记词和私钥绝对不能泄露。我在 **Sepolia** 上完成了一笔真实测试交易，并在区块浏览器里查看了交易哈希、状态、Gas、区块高度这些信息。这让我第一次不是抽象地知道“链上会有交易”，而是能从浏览器记录里看到一笔链上操作到底长什么样。除此之外，我还在 **Remix** 上部署了一个最小合约 SimpleStorage，并完成了最基础的读写练习。对我来说，这一步很重要，因为它让我从“知道智能合约是什么”真正迈到“我已经亲手部署并调用过一个合约”。

这一周我也开始理解 **账户结构和权限边界**。过去我会把钱包想得很像 Web2 登录，但这一周后我更清楚了：钱包不是简单登录入口，而是控制权和风险入口。EOA、智能账户、多签账户看起来都像“账户”，但它们解决的是完全不同的控制权分配问题。我现在对“谁能发起交易、谁能批准、谁承担风险”这件事有了更清楚的认识，这对后面理解 AI Agent 和链上执行的关系特别重要。

如果说这一周最有代表性的实践成果，我觉得有两个。第一个是我把自己的学习过程沉淀成了一个 **公开 GitHub repo**，让它不只是笔记仓库，而是一个可以持续积累 proof-of-work 的工作区。第二个是我从零做出了一个 **AI × Web3 助手 MVP**。第一阶段它只是一个“Remix + AI 助手页面”，负责解释链上操作、提示风险、生成记录模板；第二阶段我开始把它推进到更完整的流程，让用户输入自然语言修改需求后，AI 生成 Solidity 草稿，再经过人工审查、浏览器内编译、连接 MetaMask、部署到测试网，最后自动整理交易哈希和部署记录。这个项目对我来说，不只是一个 demo，更像是我这一周对“AI 和 Web3 怎么真正结合”最直接的回答。

这一周我也遇到过问题。比如 GitHub CLI 的登录和权限问题、WCB Learning 内容需要人工确认、AI 输出不能直接照单全收、合约修改草稿必须人工复核。这些问题反而让我更明确了一件事：**真正可靠的 AI × Web3 流程，不是让 AI 自动做完所有事，而是明确哪些事可以交给 AI，哪些事必须由人负责。**

\# AI × Web3 助手 Stage 2 实现说明

这份文档说明的是：当前这个网页在 Stage 2 到底做了什么，已经打通了哪些环节，还保留了哪些人工确认节点，以及它目前的边界和限制。

\## 一、Stage 2 的目标

Stage 1 解决的是：

\- AI 帮助解释链上操作

\- AI 生成 Remix 操作步骤

\- AI 帮助整理交易记录

但 Stage 1 还没有真正把“修改合约 -> 编译 -> 钱包确认 -> 测试网上链”这条链打通。

所以 Stage 2 的目标是：

**让用户输入自然语言修改需求后，网页可以生成修改后的 Solidity 草稿，并在人工审查通过后，继续完成浏览器内编译、连接 MetaMask、部署到测试网、返回交易哈希和合约地址。**

\## 二、Stage 2 新增了什么

\### 1. 保留原有的链上交互说明模式

网页仍然保留了原来的“链上交互说明”模式，用于：

\- 解释函数调用

\- 判断只读 / 写入 / 部署

\- 输出风险提醒

\- 输出执行前检查清单

\- 输出执行后 Markdown 模板

这部分主要还是服务于学习、解释和记录。

\### 2. 新增“修改并部署合约”模式

这是 Stage 2 最核心的新增能力。

用户现在可以输入：

\- 目标网络

\- 合约名称

\- 自然语言修改需求

\- 原始 Solidity 合约代码

\- 构造函数参数

网页会生成：

\- 修改摘要

\- 改动清单

\- 人工复核清单

\- 风险提醒

\- 修改后的 Solidity 草稿

然后在同一个页面继续进入部署流程。

\## 三、现在完整的全链路流程是什么

Stage 2 现在打通的是这条链：

`用户输入自然语言需求 -> AI 生成 Solidity 草稿 -> 用户人工审查 -> 浏览器内编译 -> 用户连接 MetaMask -> 切换到目标测试网 -> 用户在 MetaMask 人工确认部署 -> 页面返回交易哈希和合约地址 -> 页面生成部署记录`

这条链路已经比 Stage 1 更接近“可直接执行”的产品原型。

\## 四、网页内部具体做了什么

\### 1. AI 生成合约修改草稿

网页通过本地后端调用 DeepSeek API，根据用户的自然语言需求和原始 Solidity 合约代码，返回：

\- 修改后的完整 Solidity 草稿

\- 改动说明

\- 风险提醒

\- 人工复核清单

也就是说，AI 现在不只是解释任务，而是已经开始参与“代码修改草稿生成”。

\### 2. 加入人工审查闸门

在 Stage 2 里，AI 生成代码后，网页不会直接自动部署。

页面新增了一个明确的人工确认步骤：

\- 用户必须勾选“我已经人工检查了修改后的 Solidity 代码、访问控制、函数副作用和构造函数参数”

没有勾选，页面不会继续编译和部署。

这一步的意义是：

\- 把“人工审查”从口头提醒变成真正的流程闸门

\- 明确区分 AI 草稿和用户批准后的执行内容

\### 3. 浏览器内编译 Solidity

Stage 2 在前端页面中接入了浏览器内编译能力。

做法是：

\- 页面加载浏览器端 `solc`

\- 把 AI 生成的 Solidity 草稿作为输入

\- 在浏览器里直接编译

\- 提取 ABI 和 bytecode

这意味着用户不需要先切去 Remix 才能知道合约能不能编译。

\### 4. 连接 MetaMask

Stage 2 页面现在已经支持：

\- 检测浏览器中是否有 `window.ethereum`

\- 连接 MetaMask

\- 请求账户访问

\- 获取当前 signer

也就是说，页面不再只是一个静态说明板，而是已经能进入钱包交互准备阶段。

\### 5. 自动切换或添加目标网络

当前页面会根据用户选择的网络尝试：

\- 切换到 `Sepolia`

\- 或切换到 `Base Sepolia`

\- 如果本地钱包未添加网络，则尝试提示添加

这样做的目的，是减少用户在钱包里手动切换网络的成本。

\### 6. 发起部署交易

在满足以下条件后：

\- 用户已人工勾选审查确认

\- 编译成功

\- MetaMask 已连接

页面会：

\- 用 `ethers.ContractFactory`

\- 基于已编译出的 ABI 和 bytecode

\- 使用用户当前 signer

\- 发起部署交易

这一步意味着 Stage 2 已经真正进入链上执行边界。

\### 7. 返回部署结果

部署发起后，页面会进一步：

\- 获取部署交易哈希

\- 等待交易确认

\- 返回部署后的合约地址

\- 生成区块浏览器链接

\- 展示区块高度等信息

这让用户不需要手动再去整理这些最关键的链上结果。

\### 8. 自动生成部署记录 Markdown

部署成功后，页面会自动生成一份部署记录，包含：

\- 网络

\- 合约名称

\- 用户修改需求

\- 构造函数参数

\- 人工审查说明

\- 交易哈希

\- 浏览器链接

\- 合约地址

\- 修改后的 Solidity 草稿

这一步的意义是把“执行结果”直接沉淀成可复用的 proof-of-work 材料。

\## 五、哪些地方仍然必须人工确认

虽然 Stage 2 已经打通到上链，但它仍然是一个\*\*受限助手\*\*，而不是自动化代理。

以下步骤仍然必须人工确认：

\### 1. 人工阅读 AI 生成的 Solidity 草稿

AI 修改代码不等于代码就是正确的。

用户仍然要自己检查：

\- 访问控制是否正确

\- 状态变量是否合理

\- 函数副作用是否可接受

\- 构造函数参数是否正确

\### 2. 人工勾选审查确认

只有勾选之后，页面才允许继续编译和部署。

\### 3. 人工在 MetaMask 中确认交易

页面可以发起部署请求，但真正的签名和批准仍然发生在 MetaMask 里，由用户自己确认。

\### 4. 人工判断是否真的继续执行

即使网页已经准备好了交易，用户仍然可以在钱包弹窗阶段拒绝继续执行。

\## 六、它现在还没有做什么

虽然 Stage 2 已经打通了更多链路，但它依然不是完整产品。

当前还没有做的事情包括：

\- 页面内直接读取已部署合约状态

\- 页面内直接调用已部署合约写函数

\- 对 AI 修改的 Solidity 做自动测试

\- 多文件 Solidity 项目支持

\- 完整构造函数复杂参数支持

\- 多网络复杂部署配置

\- 更强的部署失败恢复逻辑

\- 更完整的日志系统和 tracing

所以它现在更适合：

\- 单文件合约

\- 最小测试网部署

\- 学习和 demo 演示

而不是复杂生产级部署场景。

\## 七、它的安全边界仍然是什么

Stage 2 虽然更强，但安全边界没有变：

\- 不接触私钥

\- 不接触助记词

\- 不接触 `.env` 内容

\- 不自动越过 MetaMask

\- 不自动跳过人工审查

\- 不把 AI 结果当作最终权威

真正的链上结果仍然以：

\- 钱包确认内容

\- 交易哈希

\- 区块浏览器回执

为准。

\## 八、总结

如果用一句话总结 Stage 2 做了什么，那就是：

**它把“自然语言修改合约 -> AI 生成草稿 -> 人工审查 -> 编译 -> MetaMask 确认 -> 测试网上链 -> 自动记录结果”这条最小全链路真正接起来了。**

-   它现在已经不是单纯的说明工具，而是一个带明确人工闸门的受限 Web3 执行助手。

# Remix + AI Web3 助手网页介绍

## 这个网页是做什么的

这个网页是我当前 AI × Web3 项目的第一阶段 MVP，也就是一个**面向 Remix 测试网练习的受限 AI 助手页面**。

它的目标不是替用户自动完成链上操作，而是帮助用户在真正去 Remix 和 MetaMask 执行之前，先把这次操作理解清楚、准备清楚、记录清楚。

换句话说，这个网页负责的是：

-   解释
    
-   提醒
    
-   整理
    
-   生成草稿
    

而不是：

-   自动签名
    
-   自动部署
    
-   自动转账
    
-   自动写链上状态
    

## 为什么要做这个网页

在 AI × Web3 的学习和实践里，一个很常见的问题是：

-   用户知道大概要去 Remix 点哪里
    
-   但不确定这一步到底是读取、写入、授权还是部署
    
-   也不确定钱包弹窗为什么出现
    
-   更不知道执行后应该记录哪些材料，才能形成可验证的 proof-of-work
    

所以这个网页的意义是把一条原本分散的流程先理顺：

`用户输入目标 -> AI 生成说明 -> 用户去 Remix 操作 -> 钱包人工确认 -> 测试网执行 -> 用户回填结果 -> AI 整理记录`

它解决的不是“自动执行”的问题，而是“在执行前后减少理解成本和记录成本”的问题。

## 这个网页现在能做什么

当前这个网页主要支持两类能力。

### 1\. 链上交互说明

用户输入：

-   目标网络
    
-   合约名称
    
-   合约地址
    
-   函数名
    
-   参数
    
-   操作类型
    
-   钱包类型
    
-   用户目标描述
    

网页会生成：

-   这次操作的解释
    
-   这是只读、写入还是部署
    
-   是否需要钱包确认
    
-   是否会消耗 Gas
    
-   Remix 操作步骤
    
-   风险提醒
    
-   执行前检查清单
    
-   执行后 Markdown 记录模板
    

这部分的作用是：帮助用户在真正点击 Remix 和 MetaMask 之前，先知道自己要做什么、为什么做、哪里要小心。

### 2\. 自然语言修改合约

用户输入：

-   合约名称
    
-   自然语言修改需求
    
-   原始 Solidity 合约代码
    

网页会生成：

-   修改摘要
    
-   改动清单
    
-   风险提醒
    
-   人工复核清单
    
-   Remix 下一步
    
-   修改后的 Solidity 草稿
    

这部分的作用是：让 AI 先根据自然语言需求给出一版修改草稿，再由人把草稿拿去 Remix 编译、检查和决定是否部署。

## 这个网页和 DeepSeek API 的关系

这个网页现在已经支持通过本地后端接入 DeepSeek API。

整体结构是：

`浏览器页面 -> 本地后端 -> DeepSeek API`

这样设计的原因是：

-   API Key 不会暴露在前端页面里
    
-   API Key 只保存在本地 `.env.local`
    
-   前端页面不直接接触密钥
    

所以这个网页已经不是纯静态模板，而是一个：

-   前端负责收集输入和展示结果
    
-   本地后端负责读取环境变量并调用模型 API
    

的最小可运行系统。

## 这个网页的安全边界

这个网页是一个**受限 Web3 助手**，所以它有明确边界。

### 它可以做的事

-   解释函数和链上动作
    
-   判断只读 / 写入 / 部署的区别
    
-   生成操作说明
    
-   生成执行前检查清单
    
-   生成执行后记录模板
    
-   根据自然语言需求改写 Solidity 草稿
    

### 它不能做的事

-   不能接触私钥
    
-   不能接触助记词
    
-   不能接触钱包恢复信息
    
-   不能替用户签名
    
-   不能替用户点击 MetaMask 确认
    
-   不能绕过人工确认自动授权
    
-   不能自动转账
    
-   不能自动部署到链上
    

这条边界非常重要，因为它决定了这个项目现在的定位是：

**AI 辅助理解和准备链上操作，而不是自动控制链上执行。**

## 这个网页适合什么场景

它最适合这些场景：

-   学习测试网交易和合约交互
    
-   使用 Remix 做最小合约练习
    
-   在执行前确认风险和步骤
    
-   在执行后整理 proof-of-work
    
-   用自然语言先生成一版合约修改草稿
    

特别适合像我现在这样的阶段：

-   已经开始做测试网和 Remix 实践
    
-   想把 AI 和 Web3 结合起来
    
-   但又不希望一开始就做一个过度自动化、风险过高的 Agent
    

## 这个网页目前的限制

虽然它已经能运行，但它仍然是第一阶段 MVP，所以限制也很明确：

-   不直接连接 MetaMask
    
-   不直接读取链上状态
    
-   不自动获取交易哈希
    
-   不自动部署合约
    
-   不自动执行写入交易
    
-   不自动保存链上结果到仓库
    

另外，合约修改草稿虽然能由 AI 生成，但仍然必须经过：

-   人工阅读
    
-   Remix 编译
    
-   人工检查访问控制和副作用
    
-   人工决定是否测试或部署
    

## 我为什么先做这个版本

因为这个版本是打通全链路的最快方式。

如果一开始就做：

-   前端页面
    
-   钱包连接
    
-   `wagmi`
    
-   `viem`
    
-   页面内直接读写
    
-   状态管理
    
-   错误处理
    

那工程复杂度会明显上升，可能在真正验证需求之前就被技术细节拖住。

而现在这个第一阶段版本先验证的是：

-   AI 的解释有没有帮助
    
-   风险提醒有没有价值
    
-   记录模板是否真的节省整理时间
    
-   用户是否更容易理解链上操作边界
    

只有这些答案是肯定的，第二阶段的产品化才值得做。

## 下一步如何升级

这个网页的下一步升级方向已经比较明确：

### 第二阶段目标

把“解释、执行、验证记录”尽量收进同一个网页里。

### 可能的技术方向

-   Next.js
    
-   wagmi
    
-   viem
    
-   页面内连接 MetaMask
    
-   页面内直接读函数
    
-   页面内发起写入交易
    
-   自动拿到交易哈希
    
-   自动生成浏览器链接和记录
    

### 但仍然保留的边界

即使升级到第二阶段，也仍然要坚持：

-   AI 不接触私钥
    
-   钱包确认必须人工完成
    
-   授权、转账、部署、写入不能绕过人工确认
    

## 总结

如果用一句话总结这个网页的作用，那就是：

**它是一个面向 Remix 测试网练习的受限 AI 助手，负责帮助用户理解、准备和记录链上操作，但不替用户执行高风险链上动作。**

这也是我当前把 AI 和 Web3 结合起来的最小可行路径。
