**今天學到的 3 個重點**

**① Prompt 是接口設計，不是咒語**

它不是「怎麼問 AI」，而是把**任務目標、輸入邊界、輸出格式、安全規則**寫成可執行的溝通協議。好的 prompt 不是讓模型更自信，而是讓模型在**不確定的時候停下來**。

**② Instruction 拆四段最實用**

寫 prompt 時應該包含：任務目標 → 可用輸入 → 禁止行為 → 輸出格式／失敗格式。尤其要區分「解釋」和「執行」，模型可以分析但不能擅自確認。

**③ Few-shot + Structured Output = 生產力**

Few-shot 用範例教風格，Structured Output 用 JSON Schema 強制輸出的機器可讀性，讓後續程式碼能自動檢查、拒絕、回歸測試。

### 《上下文（Context）》學習筆記

**1\. 核心認知（第一性原理）**

-   **Context 不只是文本拼接，而是資訊治理**：Context 是模型在單次請求中能看見並受其影響的資訊空間。系統設計的難點不在於塞入多少內容，而是要清楚劃分系統規則、用戶目標、歷史狀態、工具結果與外部文件。
    
-   **決定資訊的「身份」與「生命週期」**：模型只能基於它看見的上下文行動。系統必須為每類資訊標註來源、時效、權限和可信度，並決定資訊何時該過期退出，否則模型可能會把「不可信的網頁內容」當作「系統規定的指令」來執行。
    

**2\. 四大關鍵知識節點**

-   **Context Window（上下文窗口）**：模型一次能處理的最大範圍。窗口大不代表能完美利用細節，盲目塞入合約源碼或交易日誌反而會讓模型失焦或引用過期資訊。關鍵狀態應直接給予，長篇文件則按需提取。
    
-   **Context Engineering（上下文工程）**：設計資訊進入模型的方式。這包含排序、裁剪、隔離不可信內容等，目標是讓模型在「正確的資訊層級」工作，其內容應涵蓋任務狀態、工具返回結果、可信數據源以及系統禁止事項等。
    
-   **Memory（記憶）**：跨請求保留的資訊（如用戶偏好或歷史任務），雖能提升操作體驗但伴隨隱性風險。記住一個原則：**所有涉及身份、權限、資產或外部副作用的記憶，都不能替代即時授權，必須重新綁定當前會話來確認**。
    
-   **Knowledge Base（知識庫）**：用來解決模型知識過期問題的可檢索外部資料。必須嚴格維護文件的來源、更新時間、適用協議版本等 metadata，否則只會把舊錯誤穩定地送進模型。
    

**3\. AI x Web3 的系統實踐原則**

-   **Context 是連接 AI 推理與 Web3 狀態的入口**：AI 負責推理，Web3 負責狀態和結算；而 Context 決定了模型究竟是基於用戶幻想、過期文件，還是「可驗證的鏈上事實」來做決策。
    
-   **上下文分層設計**：一個可靠的智能體（Agent）上下文必須明確分層，包含：
    
    -   **指令層**：系統規則、禁止事項。
        
    -   **任務層**：用戶目標與本次參數。
        
    -   **事實層**：鏈上狀態、工具結果、模擬（simulation）結果。
        
    -   **知識層**：外部標準與歷史案例。
        
    -   **記憶層**：用戶偏好。 層級越清晰，後續的權限控制、Prompt 注入防護與安全審計就越容易執行。

### 《提示詞（Prompt）》學習筆記

**1\. 核心認知（第一性原理）**

-   **Prompt 是軟約束，而非安全邊界**：Prompt 的本質是你與模型之間包含任務目標、輸入邊界與安全規則的溝通協議，但它無法保證模型永遠按規則行動。真正的系統邊界必須由程式碼、權限、校驗和人工審計來承擔。
    
-   **知止而後能定**：好的 Prompt 不是為了讓模型變得更自信，而是要教導模型「在合適的時候停下來」，避免它自行補完缺失的資訊、編造 API 或越權確認。
    

**2\. 四大關鍵知識節點**

-   **Instruction（指令）**：需清楚定義角色、任務目標、可用輸入、禁止行為以及失敗時的處理格式。在實務中，必須嚴格區分模型的「解釋」與「執行」權限（例如：可解釋交易風險，但不能擅自替用戶確認）。
    
-   **Few-shot（少樣本提示）**：透過提供正反範例讓模型模仿判斷方式與輸出格式。要注意，這些範例不是一次寫完就放著的素材，而是需要隨著業務規則調整、與評估（Eval）一起持續維護的測試資產。
    
-   **Structured Output（結構化輸出）**：要求模型以固定結構（如 JSON object 或 schema 欄位）返回結果。這不是為了排版美觀，而是為了讓後續的程式碼能夠有效地檢查、拒絕、記錄及進行回歸測試。
    
-   **Prompt Injection（提示詞注入）**：攻擊者可能透過惡意文件或輸入，讓模型忽略原始規則、洩漏資訊或調用危險工具。防禦的核心不能只靠在提示詞寫「不要被注入」，而是必須將外部內容視為不可信數據，並對敏感動作強制進行參數校驗與人工確認。
    

**3\. AI x Web3 的系統實踐原則**

-   **指令分層與防護鏈路**：系統規則、開發者規則與用戶目標不能混在一起。Prompt 僅負責定義任務和輸出格式，對於高風險動作（如執行支付、簽名、調用外部工具），絕對不能只靠 Prompt 攔截，必須串聯「Code 校驗 -> Guard 模擬檢查 -> Human check 人工確認」的完整安全防護鏈路。

# **Web3 運行原理筆記**

1.  **Web3 是新一代網路架構**  
    建立在區塊鏈上，強調「去中心化」與使用者擁有自己的資料與資產。
    
2.  **區塊鏈（Blockchain）是公開帳本**  
    所有交易都會被記錄、驗證，且難以被竄改。
    
3.  **區塊（Block）會串成鏈（Chain）**  
    每個區塊都包含前一個區塊資訊，因此形成 Blockchain。
    
4.  **Hash 讓資料具有不可竄改性**  
    資料只要改動，Hash 就會改變，因此能快速發現異常。
    
5.  **Smart Contract（智能合約）是自動執行程式**  
    鏈上執行的程式，可在特定條件成立時自動完成交易，不需中介機構。
    
6.  **Wallet（錢包）是 Web3 身份核心**  
    用來管理加密資產、登入 DApp 與驗證身份。
    
7.  **Gas Fee 是區塊鏈手續費**  
    使用區塊鏈功能（轉帳、Mint NFT 等）都需要支付 Gas。
    
8.  **DApp 是去中心化應用程式**  
    應用程式直接與區塊鏈互動，而非依賴單一公司伺服器。
    
9.  **Token 是 Web3 的經濟基礎**  
    可作為支付工具、治理投票、會員資格或數位資產。
    
10.  **Web3 核心精神是 Ownership（所有權）**  
     使用者真正擁有自己的帳號、資產與數位身份。

## 今日進度

**Large Language Models explained briefly:** [https://www.youtube.com/watch?v=LPZh9BOjkQs](https://www.youtube.com/watch?v=LPZh9BOjkQs)

這部影片深入淺出地解釋了大語言模型（LLM）的底層運作機制。以下是筆記精華總結：

### 1\. 核心運作原理：機率與文字接龍

-   **本質是預測下一個字**：當你與聊天機器人互動時，系統其實是在進行類似「完成電影劇本」文字接龍的任務。  
    LLM 是一個複雜的數學函數，它會根據已有的上下文，預測出下一個出現的字詞。
    
-   **機率分佈而非絕對確定**：模型不會給出單一確定的答案，而是為所有可能的下一個字分配「機率」。  
    為了讓生成的文字看起來更自然，系統通常會隨機選擇一些機率較小（但依然合理）的字，這也是為什麼輸入相同的提示詞，每次得到的回答可能都不一樣。
    

### 2\. 訓練的兩個關鍵階段

-   **預訓練（Pre-training）**：模型會處理海量的網路文本（例如，GPT-3 的訓練資料量相當於人類不眠不休閱讀超過 2600 年）。一開始，模型的數千億個「參數（Parameters）」是隨機的，只會輸出亂碼。透過**反向傳播（Backpropagation**）演算法，模型會將自己的預測與真實文本比對，不斷微調參數，使其預測越來越準確。這階段需要極度龐大的運算力。
    
-   **人類回饋強化學習（RLHF）**：僅僅會「自動補全網頁文字」並不能成為一個好的 AI 助手。因此模型必須經過第二階段訓練，由人類介入標註哪些預測是無用或有問題的，藉此進一步調整參數，讓模型的回答更符合使用者的偏好與期望。
    

### 3\. Transformer 架構的底層突破

-   **平行處理能力**：2017 年引入的 Transformer 模型打破了過去逐字處理的限制，它能一次性、平行地吸收所有文字內容，這也讓高度依賴 GPU 平行運算的訓練成為可能。
    
-   **數字編碼與注意力機制（Attention）**：模型會先將每個字轉換為一長串數字向量。Transformer 最獨特的是「注意力機制」，它允許這些代表文字的數字互相交流，**根據周圍的上下文來精煉其含義**（例如，根據上下文區分「bank」是指銀行還是河岸）。
    
-   **前饋神經網路（Feed-forward neural network）**：與注意力機制搭配運作，賦予模型更大的容量去記憶和儲存訓練期間學到的語言模式。這兩種操作反覆迭代，最終將最後一個向量轉化為下一個字的機率預測。
    

### 4\. 湧現現象與不可解釋性

-   **黑盒子特性**：儘管研究人員設計了模型的架構與步驟，但模型最終展現出來的驚人流暢度與實用性，其實是數千億個參數在訓練中微調出來的\*\*「湧現現象（emergent phenomenon）」\*\*。
    
-   這導致了一個巨大的挑戰：由於參數極其龐大，**我們很難確切知道模型究竟為什麼會做出某個特定的預測**。這也呼應了先前章節提到的：LLM 是機率模型，不應直接將其視為絕對的真相源。

# 學習筆記

### Handbook《大語言模型（LLM）》學習筆記

**1\. 核心認知（第一性原理）**

-   **LLM 是機率模型，而非真相源**：模型輸出的是機率上合理的內容，而不是天然可信的事實。
    
-   **定位為推理層**：模型可用於理解、歸納、生成和規劃，但不能單獨承擔事實來源與權限判斷。關鍵事實必須來自資料庫、API 等外部可信系統。
    
-   **暴露不確定性**：當資訊不足或過期時，系統設計應讓其顯式降級，而不是讓模型編造順滑的答案。
    

**2\. 五大關鍵底層概念**

-   **Token（詞元）**：模型處理文本的基本單位。影響上下文容量與成本，需注意程式碼、JSON 和表格通常比普通文字消耗更多 Token。
    
-   **Embedding（嵌入）**：將內容映射為向量以衡量「語義接近度」，常用於檢索增強生成（RAG）。但語義相近不代表結論正確，不能替代來源校驗。
    
-   **Transformer 架構**：依靠注意力機制（Attention）在上下文中尋找模式，賦予模型強大的組合能力，但**沒有給予它事實的最終裁決權**。
    
-   **Hallucination（幻覺）**：模型會生成看似合理卻不真實的內容（如編造 API 或假資料）。防範幻覺不能只靠優化提示詞，必須依賴外部規則檢查與人工確認。
    
-   **Multimodal（多模態）**：使模型能讀懂圖表、截圖等真實工作介面，但關鍵判斷仍需回歸結構化的數據與可信來源。
    

**3\. AI x Web3 的系統實踐原則**

-   LLM 位於系統的「理解和生成層」，需與資料層、編排層、執行層和安全層配合。
    
-   **核心守則**：LLM 越靠近執行層，就越需要將其自然語言輸出轉化為「可驗證的物件」（如 schema、參數、日誌），以確保系統運作的安全與穩定。
    

**4\. 最小實踐：做一个“交易解释器”的最小版本。**
