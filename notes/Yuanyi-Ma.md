# 2026-05-29
<!-- DAILY_CHECKIN_2026-05-29_START -->

我今天调研了 agent 协作过程中，区块链的机会，会认为其中非常多的部分都有不信任的问题出现，需要区块链来提供可信支持，大概暂时的结果如下

1.  **承诺可验证**  
    我对 Agent 协作的理解是，协作不能只停留在“多个 Agent 互相聊天或分工执行”，而应该先让每个 Agent 的承诺变得清楚、可记录、可检查。比如某个 Agent 承诺完成调研、代码实现、审阅或内容整理，就需要明确任务范围、输入来源、完成标准、时间边界和失败处理方式。只有承诺可验证，后续协作才不会变成不可追责的黑箱调用。
    
2.  **交付可验证**  
    Agent 协作中的交付物也应该可验证。一个 Agent 完成任务后，不应只输出结论，还应该说明它做了哪些步骤、依据了哪些材料、修改了哪些文件、产生了哪些结果，以及哪些部分仍然不确定。这样其他 Agent 或人类才能继续接手、复核或二次加工。可验证交付的核心，是让协作从“相信 Agent 说自己完成了”变成“可以检查 Agent 到底完成了什么”。
    
3.  **验收可验证**  
    Agent 协作还需要可验证的验收机制。不同任务应该有不同验收标准：代码任务可以看测试和 diff，调研任务可以看来源和推理链，课程总结可以看覆盖率和重点提炼，链上任务则要看交易记录、权限边界和执行结果。验收标准越清楚，Agent 之间就越容易形成稳定协作，也越容易把失败经验沉淀为下一次协作的改进规则。

# 2026-05-28
<!-- DAILY_CHECKIN_2026-05-28_START -->


1.  今天看了 Web3Privacy Now 关于 Neo-Cypherpunk 和隐私文化的分享。最大的收获是，隐私不是某个工具或后期补丁，而是 builder 在产品设计阶段就要考虑的基础能力。不同用户的风险模型不同，因此隐私方案不应只有一个标准答案。
    
2.  讲者把旧 Cypherpunk 和 Neo-Cypherpunk 做了区分：旧模式更偏专家、个人防御和末日叙事；新模式更强调可参与、社区互助、渐进采用和快乐传播。这对隐私产品很重要，因为如果工具只服务极客，就很难真正影响普通用户。
    
3.  我最感兴趣的是 AI Agent 隐私问题。Agent 会交易、通信、调用工具和留下链上轨迹，但多数产品还没有把隐私作为默认设计。未来可以探索 private agent、private relay、可选择公开的 Agent 支付日志，以及基于 CROPS 原则的项目评分框架。

# 2026-05-27
<!-- DAILY_CHECKIN_2026-05-27_START -->



今天调研了 ai+web3 的项目，为黑客松做准备，具体如下

对 Virtuals 整体的结论：

**Virtuals 目前最成功的是“AI Agent 资产发行与交易市场”，而不是“Agent-to-Agent 服务经济”。**

它已经证明了三件事：

1.  **市场愿意为 AI Agent 叙事定价**  
    AIXBT、LUNA、VADER 这类项目说明，AI Agent 可以被包装成资产、发行 token、获得流动性和注意力。
    
2.  **它有一套相对真实的 Agent 商业协议基础设施**  
    ACP 有源码、有 Job 流程、有支付、交付、评价、x402 集成和链上任务数据，不是纯 PPT。
    
3.  **它正在从 launchpad 向 Agent 商业网络转型**  
    Revenue Network、ACP v2、ERC-8004、ERC-8183 都说明它想把重心从“发 Agent token”转向“Agent 真实提供服务并产生收入”。
    

但它还没有充分证明三件事：

1.  **ACP 上已经跑出大规模高价值应用**  
    当前最出圈的 AIXBT 更像发行层成功案例，不是 ACP 服务协议成功案例。
    
2.  **Agent 服务收入能支撑当前估值**  
    官方 Agent revenue 级别还很小，和几亿美元市值不匹配。
    
3.  **Agent 能力发现、质量评价、抗刷和可信交付已经成熟**  
    Job 数量、token 热度、aGDP 都不能直接等于真实生产力。
    

所以最终判断是：

**Virtuals 是 AI Agent × Web3 赛道里最有代表性的早期基础设施项目之一；它已经跑通了 Agent 资产化和市场注意力，但真正的 Agent 服务经济还在验证期。**

# 2026-05-21
<!-- DAILY_CHECKIN_2026-05-21_START -->








1.  今天学习了 Web3 的运行原理。一笔链上交易并不是简单“点确认然后上链”，而是从钱包构造交易开始，经由私钥签名、RPC 广播、进入 Mempool 排队，再由 Builder 排序打包、Validator 验证出块，最后才能通过区块浏览器查询结果。这个过程让我更清楚地看到，链上交互其实是一套完整的执行和验证链路。
    
2.  我比较有收获的一点是：Web3 的去中心化并不是绝对的。账户创建、节点网络和客户端实现可以很分散，但用户实际访问链的入口，比如 RPC、钱包默认配置和前端页面，仍然可能形成中心化风险。因此实际构建链上应用时，不能只说“链是去中心化的”，还要考虑多 RPC 回退、结果交叉验证和入口失效时的处理方案。
    
3.  另一个重要收获是智能合约“不可变”并不等于协议逻辑永远不会变化。很多 DeFi 协议通过代理合约实现升级，用户调用的可能一直是同一个 Proxy 地址，但背后的实现合约可以被替换。更稳健的协议通常会设置时间锁，让用户和安全研究者在升级生效前有观察、审查和撤出的窗口。
