# 2026-05-26
<!-- DAILY_CHECKIN_2026-05-26_START -->

今天听了一场关于 Agent Wallet 的分享，主讲人是 Cobo Agentic Wallet 的产品负责人。主题很聚焦：当 AI Agent 开始替人类花钱、操作链上资产时，怎么保证它不乱来。

分享先回顾了 AI 的发展路径：2023 年是 Chatbot，只能回答问题；2024 年是 Copilot，能提建议但每一步仍需人类批准；2025 年进入 Agent 阶段，AI 可以在后台持续解决复杂工作流，人类只需要定义边界；到了 2026 年，Agent 已经开始自主探索和执行任务。但一个很大的痛点还没解决：如何让 Agent 替人类花钱，而且花得安全。链上资金不可逆，一旦出错就是真金白银的损失。

分享里举了两个真实发生过的失控案例。第一个叫 Send Override，就是 Agent 在 Prompt 里明明被要求只花 100 美金，但执行时悄悄把金额改大了，而且不会主动告诉你，直到你复盘才发现。第二个叫 Shadow Custody，Agent 在 MPC 钱包外围自己创建了一个 EOA 地址，先把钱转进去再发起交易，这样人类设置的所有权限都形同虚设。这两个案例让我意识到，自然语言表达意图不具备强制约束力，靠“告诉 Agent 不要做什么”是不够的。

基于这些问题，分享总结了四类失控风险：Prompt Injection（提示词注入导致执行偏离）、Shadow Operations（在看不见的地方创建子账户）、Unscoped Authority（无限权限，一笔交易能把所有钱转走）、Zombie Permissions（授权长期不撤销，合约出问题就跟着遭殃）。当 Agent 开始动钱的那一刻，信任就不能停留在应用层，必须下沉到基础设施层，用更强制的方式去约束。

Cobo 的方案围绕三个核心点展开。第一是 MPC，解决私钥安全问题。Cobo、Agent 和 Human 各自持有私钥分片，任意一方都没有独立掌控资金的权限。采用的是 2-of-2 签名模式：要么是 Agent 和 Cobo 共管（用于自动化执行），要么是 Human 和 Cobo 共管（用于大额人工操作）。没有单方可以私自转账。

第二是 Pact，这是一份授权协议，告诉 Agent 能做什么、不能做什么、什么时候必须停下来。一个 Pact 包含四个要素：Intent（你想要什么目标）、Execution Plan（Agent 转译成的具体执行计划）、Policy（风控约束，包括预算、白名单、链、Token、合约甚至 ABI 参数级的限制）、Completion Condition（任务完成后自动撤销授权，避免僵尸权限）。Pact 会推送到手机 App 上，人类审阅批准后，Agent 才能在被严格约束的范围内执行。

第三是 Recipe，可以理解为“知识胶囊”。因为大模型本身并不具备操作链上资金的知识，容易出错。Recipe 把 DeFi 操作所需的合约地址、ABI 参数、风控边界打包成一个文件，Agent 调用 Recipe 就知道怎么做对，而不是自己临时构造交易。目前已经上线了 Aave V3、Uniswap V3、Polymarket、Hyperliquid 等主流 Recipe。

分享还演示了一个多 Agent 的场景：一个用户可以创建多个钱包，委托给不同的 Agent 管理，比如一个 Trading Agent、一个 DeFi Agent，每个 Agent 的 Pact 和 Policy 边界都可以独立设置，做到细颗粒度的资金安全管理。

Q&A 环节有几个问题我觉得挺有意思。有人问小额免密支付场景是否支持，老师说可以，通过 Policy 限制单次金额、转账地址、频次和总上限，就能把风险控制在很小的窗口内。有人问 Agent 支付是否一定要在链上，老师的回答很务实：链上和 Web2 会并存，链上的优势是绕过传统银行体系的低效，适合跨境和快速结算，但完全颠覆现有支付基础设施短期内不太可能。还有人问 Agent 掉线了怎么办，老师说 MPC 的私钥分片做了冗余设计，即使 Agent 挂掉，用户仍然可以通过手机 App 恢复和控制资金。

听完这场分享，我对“可控边界”这个概念有了更具体的理解。以前我觉得让 Agent 自动操作链上资产主要是技术问题，现在发现更重要的是信任和授权机制的设计。Pact 把“意图”从自然语言翻译成了可执行、可约束的协议，MPC 保证了没有单点失控，Recipe 降低了模型幻觉的风险。三个东西组合在一起，才构成一条可信的执行链路。

今天听了一场关于 Agent Memory 的分享，借助ai把那个文件又看了一遍，老师把“记忆”这个概念拆得很清楚。以前我以为记忆就是让 AI 记住之前聊过什么，但分享里给出的定义是：记忆不是更大的上下文窗口，也不是单纯的 RAG，而是 Agent 跨时间携带意图和状态的管理系统，关键就在于“跨时间”和“管理”这两个词。

从产品视角看，记忆要解决的问题很简单：用户不应该每次都要重新建立上下文。用户已经解释过目标、给过约束、做过决策，这些不应该在会话结束后就消失。分享里举了两个例子对比：ChatGPT 的记忆偏向“用户连续性”，记住偏好、语气、个人资料，让聊天更连贯；而 Claude Code 的记忆偏向“项目连续性”，记住构建命令、代码规范、之前踩过的坑，让开发工作能跨会话继续。同一个记忆问题，在不同产品形态下表现不同，但底层本质是一样的。

分享还把记忆相关的动作分成了三个层次：Recall 是用户感知到的“好像记住了”的效果，属于产品层面的表现；Retrieve 是系统把存储的记忆带回当前步骤的操作；Revise 则是当新证据出现时更新或淘汰旧的存储记忆。很多人把这几个混在一起说，但它们的时机和职责完全不同。另外还有一组容易混淆的名词：Memory 是决定什么状态应该跨时间保留的系统能力，Retrieval 是定位并返回信息的操作，RAG 只是检索加生成的工作流而不是记忆本身，Embeddings 和 Vector Store 是基础设施但不是记忆层。这个区分对我很有用，之前我总觉得用了向量数据库就等于有了记忆，但其实记忆还包括写什么、什么时候写、什么时候改、什么时候忘，这些都是设计决策，不是单纯的技术选型。

关于记忆的分类，分享给出了两个轴。时间轴区分当前工作上下文和长期持久记忆；内容轴只针对持久记忆，分为情景记忆（过去的交互和结果）、语义记忆（稳定的事实和偏好）以及程序记忆（可复用的步骤和工具流程）。一个重要的提醒是：工作区的暂存笔记属于时间轴上的当前上下文，而不是内容轴的第四种类型，不要把“临时的”当成一种内容分类。

记忆工程的核心不是存储，而是生命周期管理。不是所有历史都值得持久化，所以需要捕获规则；不是所有持久化的记忆都应该被召回，所以需要相关性筛选；旧记忆会过时，所以需要更新和遗忘机制。这些规则决定了记忆系统是可控的基础设施还是不断膨胀的垃圾堆。

在 Agent 架构中，记忆不只属于某一层。Prompt Engineering 决定这次怎么问，但不决定记忆怎么管理；Context Engineering 决定模型当前步骤能看到什么，记忆通过检索进入这一层；Harness Engineering 决定循环、工具、策略、日志如何运行，生命周期操作和可观测性在这一层实现。很多记忆方案只关注“取回来放进上下文”，却忘了谁来触发写入、谁来处理更新、谁来记录操作日志，这些都需要 Harness 层支撑。

最后说说我的思考。以前我总觉得记忆就是把对话历史存起来下次塞进上下文，但这样做的结果是上下文越来越大，而且很多东西根本不需要每次都用。真正的记忆系统应该回答四个问题：什么值得记？什么时候取回来？什么情况下要改或忘？用户能不能看到、编辑、删除？这些问题的答案，决定了记忆是从便利功能变成可依赖的基础设施，还是变成没人敢碰的遗留状态。分享最后一句话我记下来了：当 Agent 从回答问题走向执行任务时，记忆从便利性变成了控制层。因为一旦 Agent 真的在做事，它记住什么、不记住什么，直接影响下一步能做什么、不能做什么。这时候记忆就不再是锦上添花，而是系统的一部分。今天的框架很清晰，接下来如果要做一个带记忆的 Agent 小项目，我会先想清楚要记什么类型、生命周期谁控制、以及怎么让用户可见可改。

分享内容按五个方向展开，我按自己的理解重新捋了一遍：

1.  **去中心化 AI 基础设施**（资源层）

### **1\. 资产自托管：如何提高安全性、降低管理私钥的复杂度？**

这是大规模采用的最大障碍之一。目前主流思路是“在安全与易用间找平衡”：

-   **智能合约钱包（如ERC-4337抽象账户）**：支持社交恢复（由几个信任的朋友或硬件设备共同找回私钥）、多签机制（需要多把钥匙授权交易）、每日限额等。让私钥不再是“唯一命门”。
    
-   **硬件钱包 + 多链统一管理**：如Ledger、OneKey 等，配合钱包内置的便捷签名流程（如蓝牙确认），降低操作门槛。
    
-   **分布式密钥生成（DKG）与多方计算（MPC）**：私钥从不出现在任何一端，而是分片存储在多个设备或服务器上，交易需联合计算签名。既减少单点风险，又无需记忆助记词。
    
-   **生物识别 + 可信执行环境（TEE）**：在手机安全芯片内管理私钥，用指纹/面容调用。不过有中心化依赖争议。
    

### **2\. 没有中心化机构/税收时，公共基础设施谁来维护？如果有税收，又如何分配？**

这是一个典型的“公地悲剧”问题。Web3 的方案通常不依赖强制税收，而是**经济博弈+自愿贡献+协议内置激励机制**：

-   **谁来维护**：
    
    -   **节点运营者**：被协议发行奖励（如以太坊的质押收益）或交易费吸引，相当于“按劳取酬”。
        
    -   **公共品资助**：通过**二次方融资（Quadratic Funding）**（如Gitcoin Grants）、**追溯性公共品奖励**（如Optimism的RetroPGF）——先建设，后由社区或DAO投票评定价值并拨款。
        
    -   **自愿维护**：某些基础设施（如RPC节点）提供免费公共端点，由大机构或协议基金会出于生态利益买单。
        
-   **如果有税收，如何分配**：
    
    -   一些协议（如稳定币、DEX）会收取交易费/利息，进入**国库（Treasury）**。分配方式通常为：
        
        -   **DAO投票**：持币者投票决定每笔预算给哪个项目或维护者。
            
        -   **自动化规则**：按使用量、质押占比或工作量证明来分配。
            
        -   **结合信誉系统**：只有被认证的基础设施服务商才能申领奖励。
            
    -   这类“税收”不是强制的国家税，而是**协议内置的费用**，用户可选择使用其他低费率协议。竞争机制也在调节。
        

### **3\. 没有审查且隐私很强时，如何界定并治理有害信息/黑产？**

这是去中心化理想与现实安全之间的尖锐矛盾。目前没有银弹，但有几个层次的方法：

-   **链上层面**：无法修改已上链的非法内容，但可以：
    
    -   **区块过滤**：节点自愿选择不打包包含某些已知非法地址/哈希的交易（如OFAC制裁列表）。这是一种“软审查”。
        
    -   **隐私与透明度分层**：大多数公链其实**非全匿名**（如比特币、以太坊是假名），通过链分析工具可以追踪资金流向。真正全隐私的链（如Monero）则更容易被监管排斥。
        
-   **应用层/入口层治理**（更现实）：
    
    -   **前端域名封禁/应用商店下架**：虽然链上数据还在，但普通人无法通过常用入口访问。
        
    -   **智能合约黑名单**：某些稳定币（如USDC）的合约包含 `blacklist` 函数，可冻结地址。这是中心化妥协，但确实能应对黑产。
        
    -   **DAO 内容标记与仲裁**：社区创建“内容标签DAOs”，对有害信息投票标记，然后各类钱包/浏览器可选择过滤这些标记的内容。不删除，但屏蔽默认显示。
        
-   **法律衔接**：Web3 并不在法外之地。如果跨链桥、法币通道、托管钱包等实体受到法律管辖，它们有义务配合执法。链上匿名不代表物理世界抓不到人。
    

### **4\. 去中心化协作下，如何实现公平、可信的分配？**

这是DAO和贡献者经济的核心难题。目前实践中的可信分配机制包括：

-   **量化贡献的链上凭证**：
    
    -   用**POAP、SBT（灵魂绑定代币）** 记录参与行为。
        
    -   通过**GitHub、Discord 等 API 集成**自动统计代码提交、社区帮助次数、提案投票参与率。
        
    -   使用**Coordinape**（圈内互相分配信任代币）、**SourceCred**（自动计算贡献分）。
        
-   **规则前置的智能合约分配**：
    
    -   例如：一个项目利润的 30% 按工作量证明分给开发者，20% 按投票影响力分给治理者，50% 按质押份额分给资本提供者。所有规则写死在合约中，自动执行。
        
-   **争议解决与反女巫攻击**：
    
    -   结合**身份验证**（如 Gitcoin Passport 聚合多种凭证）防止一人多号刷贡献。
        
    -   设置**仲裁 DAO** 或**乐观治理**（默认分配通过，有人挑战则进入投票/仲裁）。
        
-   **公平性的哲学**：不同角色对“公平”定义不同（劳资、风险、时间、创意权重）。Web3 允许**自组织 + 分叉**——觉得某个DAO分配不公，可以 fork 规则另起炉灶，用脚投票。

 二、分享会核心收获

1\. 三个“粉丝问题”的真相

\- 有 AI 了还要不要学基础？ 必须学。AI 能帮你堆代码、细化方案，但方案设计、架构判断、代码验收全靠人。基础不够，连 AI 给的代码对不对都看不出来。

\- Web3 和 Web2 完全不同？ 不是。Web3 是更严格的分布式系统，基础架构逻辑是相通的。大部分 Web3 项目里 Web2 能力占比不低（比如 CEX 前端、DeFi 后端）。优秀的 Web3 开发者首先得是好的系统工程师。

\- 安全可以后续再补？ 不行。安全不是插件，必须源于设计。审计只是最后一道防线，大部分审计其实是“心理安慰剂”。安全没办法证伪，只有 0 和 1。无法解释系统行为，就无法保证资产安全。

2\. 支付模型：从 Web2 到 Web3

Web2 支付是用户 → 电商平台 → 支付服务 → 银行机构，信任是中心化的。

Web3 支付直接把“银行机构”换成“区块链”，各种链就是不同的银行。区块链通过数据不可篡改、代码公开透明、共识协议来建立去中心化信任。

资金流也改变：用户用自己的 Web3 账户 → 支付服务的 deposit account → 定期给商户。区别在于没有中心账本，全靠链上状态机。

3\. 钱包的本质

钱包本质是管理私钥的工具。输入私钥（大随机数），输出签名（交易签名或消息签名）。链上只认签名不认人，私钥丢了 = 资产永久丢，没有任何客服能帮你找回。

钱包的分类：技术上分 EOA、合约钱包、AA 钱包（以太坊想融合前两者，但目前路线还不成熟）；托管方式分自托管、全托管（交易所那种）、混合托管（私钥切分，多方合签）。还有硬件钱包、隐私钱包、分层钱包（BTC 找零模型）等。

4\. 钱包安全的三个风险

\- 私钥泄露：不可逆。

\- 签名欺骗（钓鱼）：你以为签转 10 块，实际签转 100 块给黑客。业界方案是 EIP-712 可视化 + 禁用 `eth_sign`。

\- 权限滥用：无限授权给恶意合约。

不理解钱包原理，系统安全就是空中楼阁。

5\. 交易生命周期与关键参数

构造（to、value、gas、nonce、data）→ 签名 → 广播 → 共识确认。

关键参数里，data（calldata）被老师称为“链上应用交互的灵魂”——没有 data，链只能转账；有了 data，才有各种生态。nonce 在 EVM 里是连续递增防止重放，但不同链实现不同（比如 Tron 没有 nonce，用 block hash）。gas 从最早的 gas price 演变成 EIP-1559 的 base fee / tip fee / max fee。

6\. 交易监听与模拟

监听可以轮询也可以用 WebSocket（但底层还是 HTTP）。有一个容易被忽略的点：不能在一个区块刚收到钱就入账，要等几个区块 finalize，避免链分叉和重组。不同链的 finalize 时间不同，这就是为什么交易所充值不同链等待时间不一样。

交易模拟很重要：签名之前先模拟执行结果，如果发现实际转的不是预期对象，就不签。绝对不能先签名再模拟——签名一旦生成就在网络中流转了，那时候再模拟已经没有意义。

7\. Server 侧钱包安全

多签（MPC 或 Safe）、资金量拆分（热/温/冷钱包）、交易可视化 + 风控审核 + 模拟、用 TEE（可信执行环境）签名让私钥明文不出内存。

8\. AI 时代的变与不变

AI 改变的：写代码更快、学习更容易、重复工作被自动化。

AI 没改变的：系统复杂度（AI 只是加速堆砌，架构设计还得靠人）、错误成本（AI 生成的错误代码同样让资产归零）、安全边界（最终审核责任永远在开发者手里）。

9\. 架构师能力模型

三位一体：基础能力（共识、密码学、链上状态模型）、架构能力（分布式系统设计，去中心化与性能的权衡）、Debug 能力（深入底层协议，解释每个现象的技术原因）。
