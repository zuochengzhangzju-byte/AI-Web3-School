# 2026-05-28
<!-- DAILY_CHECKIN_2026-05-28_START -->
今天学习的是 **Evaluation，也就是 AI Agent 的测试与持续改进**。

我的理解是，Agent 不能只看“这一次回答好不好”，而是要建立一套可以反复验证的评估体系。因为 Agent 往往不是单次回答，而是会调用工具、读取上下文、执行多步任务。如果没有 Evaluation，就很难判断系统到底有没有变好，也很难发现失败是来自模型、Prompt、工具、上下文，还是流程设计。

Evaluation 的核心是把模糊的“感觉还不错”变成可以检查的标准。比如一个 RAG 问答系统，不能只看答案是否流畅，而要检查：有没有引用来源、引用是否真的支持答案、文档里没有的问题有没有拒答、版本不确定的问题有没有提示核对版本。之前做 ethers v5 RAG 的时候，我就体会到这一点：如果没有测试用例，系统很容易被相似内容带偏，然后看起来像答对了，其实证据链不成立。

对 Agent 来说，Evaluation 可以分成几类。第一类是答案质量，比如是否准确、完整、清楚。第二类是证据质量，比如 sources 是否可靠，是否引用了正确文档。第三类是工具调用质量，比如是否调用了正确工具、参数是否正确、失败时有没有重试或停止。第四类是安全性，比如有没有在证据不足时编造答案，有没有越权执行高风险操作。第五类是任务完成度，比如 Agent 是否真的完成了用户目标，而不是只给建议。

我觉得 Evaluation 最重要的作用是帮助我们定位问题。如果 Agent 失败了，不能只说“模型不行”。更好的方式是拆开看：是用户目标不清楚？是 Prompt 缺少约束？是工具返回错误？是上下文里混入了无关信息？是检索结果不够准？还是 Agent 没有在关键步骤请求人工确认？只有能定位失败层级，才知道该怎么改系统。

结合 Web3 场景，Evaluation 更重要。因为链上操作涉及真实资产，不能让 Agent 随便猜。比如一个 Web3 Agent 在分析交易风险时，至少要评估：是否读取了正确链和正确地址，是否区分了查询和写入操作，是否说明了不确定性，是否避免直接建议用户签名高风险交易。如果涉及合约调用、授权、转账或部署，系统必须加入 human-in-the-loop。

今天比较大的收获是：Evaluation 不是项目最后才做的测试，而应该从一开始就设计进去。每一个 Agent 都应该有自己的测试集，包括正常问题、边界问题、文档不存在的问题、版本不确定的问题和高风险操作问题。这样每次改 Prompt、改工具、改 chunking、改 retrieval，都可以跑一遍测试，确认系统没有退步。

我现在对 Agent 系统的理解更完整了：Prompt 决定 Agent 怎么思考，Tools 决定 Agent 能做什么，MCP 决定工具如何接入，RAG 决定外部知识如何进入上下文，而 Evaluation 决定这个系统能不能被持续验证和改进。

今天最大的理解是：真正可靠的 Agent 不是“看起来很聪明”，而是能被测试、能被复盘、能在证据不足时拒答、能在高风险操作前停下来。Evaluation 是把 Agent 从 demo 变成产品的关键一步。

# 2026-05-27

今天学习的是 MCP，也就是 Model Context Protocol。我的理解是，MCP 解决的是“模型如何稳定、安全、标准化地连接外部工具和上下文”的问题。以前我们给 Agent 接工具，往往是每个项目自己写一套函数调用、API 接入和权限逻辑；MCP 更像是把这些外部能力抽象成一个统一协议，让模型可以用更标准的方式访问工具、资源和提示模板。

MCP 里最重要的概念有三个：Tools、Resources 和 Prompts。Tools 是模型可以调用的动作，比如查询数据库、读取文件、调用链上 RPC、搜索文档等；Resources 是模型可以读取的上下文，比如文档、代码文件、配置、数据库记录等；Prompts 是预设好的任务模板，可以帮助模型以固定方式完成某类任务。

我觉得 MCP 和 Agent 的关系很紧密。Agent 负责理解目标、规划步骤和推进任务；MCP 负责把外部能力规范地暴露给 Agent。这样 Agent 不需要直接知道每个系统的底层实现，只需要通过 MCP 调用对应能力。比如一个 Web3 Agent 可以通过 MCP 接入区块链 RPC、合约 ABI、交易查询工具、协议文档库和 GitHub 仓库，从而完成链上分析、合约解释、交易风险提示等任务。

今天比较重要的收获是：工具不是随便接给 Agent 就可以。一个好的工具系统需要清楚定义输入、输出、权限和失败情况。特别是在 Web3 场景里，查询类工具和写入类工具必须区分。查询余额、读取合约、分析交易可以相对自动化；但签名、授权、转账、部署合约这些高风险动作，必须有人类确认，不能让 Agent 自动执行。

我也意识到，MCP 的价值不仅是“让模型能调用更多工具”，而是让工具调用变得更可控、更可复用、更容易审计。对于 Agent 产品来说，这很关键。因为真正复杂的 Agent 系统不是一次回答，而是长任务、多工具、多上下文、多步骤的执行过程。如果没有统一协议，系统会很快变得混乱；有了 MCP，工具边界和上下文来源会更清楚。

结合之前做的 RAG 练习，我觉得 MCP 也可以作为文档问答系统的工具层。比如把 ethers v5 文档库作为 Resource，把检索函数作为 Tool，让 Agent 在回答问题前先查文档。如果文档没有证据，就拒答；如果问题涉及版本，就提示需要核对版本。这样 Agent 的回答就不是凭感觉，而是有证据链。

今天最大的理解是：MCP 是 Agent 系统里的“连接层”。它让模型、工具、文档、环境和外部系统之间有一个更标准的接口。对 AI × Web3 来说，MCP 很适合用来连接链上数据、钱包信息、合约工具和协议文档，但必须配合权限控制和 human-in-the-loop，才能真正安全地使用。

# 2026-05-26

今天学习了 OpenAI Agents SDK。我的理解是，Agents SDK 不是单纯调用一次大模型 API，而是把“模型、工具、上下文、handoff、guardrails、tracing”这些 Agent 系统中常见的组件组织成一个可运行的框架。它解决的问题是：当一个 AI 不只是回答问题，而是要连续执行任务、调用工具、转交给其他 Agent、记录执行过程时，我们需要一套更清晰的运行结构。

Agents SDK 里最核心的概念是 Agent。一个 Agent 可以理解为一个带有角色、指令和工具能力的 AI 执行单元。它通常包括：模型、instructions、tools、handoffs、guardrails 等配置。也就是说，Agent 不只是“一个 prompt”，而是一个有边界、有能力、有职责的任务执行者。

第二个重要概念是 Runner。Agent 本身定义“它是谁、能做什么”，Runner 负责真正运行这个 Agent。运行过程中，Runner 会处理模型调用、工具调用、handoff、guardrails 和 session 等流程。它相当于 Agent 的执行引擎。

第三个重要概念是 Tools。工具让 Agent 能连接外部世界，例如查询网页、读取文件、调用 API、执行函数、访问数据库等。没有工具时，Agent 主要是在生成文本；有工具后，Agent 才能基于真实环境做事。工具调用的关键不是“越多越好”，而是每个工具都要有清楚的输入、输出和权限边界。

第四个重要概念是 Handoffs。Handoff 指的是一个 Agent 可以把任务交给另一个更专业的 Agent。比如一个客服 Agent 判断用户的问题和退款有关，就可以 handoff 给退款 Agent；一个开发 Agent 遇到安全问题，也可以 handoff 给安全审计 Agent。Handoff 的意义是让复杂任务可以由多个专门角色协作，而不是让一个 Agent 什么都做。

第五个重要概念是 Guardrails。Guardrails 是安全检查和质量控制机制，可以在输入或输出阶段做验证。比如检查用户输入是否包含敏感信息，检查 Agent 输出是否符合格式，或者在高风险操作前阻止执行。对 Web3 场景来说，guardrails 特别重要，因为钱包签名、资产转账、授权、合约部署都可能造成真实损失。

第六个重要概念是 Tracing。Tracing 用来记录 Agent 运行过程中发生了什么，包括模型调用、工具调用、handoff、guardrail 检查等。它的价值是让 Agent 不再是黑箱。只看最终回答，很难知道 Agent 是怎么做出判断的；有 trace 之后，就可以复盘每一步，定位失败发生在 prompt、工具、上下文、权限还是流程设计上。

我觉得 Agents SDK 的核心价值是：它把 Agent 从“写一堆临时脚本和 prompt”变成了一个更工程化的系统。一个真正可用的 Agent 应该能被运行、观察、调试、限制和改进。SDK 提供的 Agent、Runner、Tools、Handoffs、Guardrails、Tracing，正好对应了这些需求。

结合 Web3 场景，我认为 Agents SDK 可以用来做链上数据分析 Agent、合约文档问答 Agent、交易解释 Agent、钱包风险提示 Agent 或开发者调试 Agent。但必须注意权限边界：查询链上数据可以自动化，生成交易参数可以辅助，真正涉及签名、授权、转账、部署合约时必须让用户确认。

今天最大的收获是：Agent 的重点不是“让 AI 自动做所有事情”，而是设计一个可控的任务执行系统。Agents SDK 帮我们把这个系统拆成了几个明确模块：谁来执行、能用什么工具、什么时候转交、怎样限制风险、如何记录过程。对于复杂任务来说，这比单纯写 prompt 更可靠，也更接近真实产品。

今天学习了 AI Agent 的核心概念。AI Agent 和普通 Chatbot 的区别不只是回答质量，而是系统结构不同。普通 Chatbot 通常是“用户输入 -> 模型回答 -> 结束”，而 Agent 更像是“用户目标 -> 理解任务 -> 制定计划 -> 调用工具 -> 观察结果 -> 更新状态 -> 继续执行 -> 输出结果”。

我理解到，一个 Agent 通常由几个部分组成：Goal、Context、Tools、Memory、Planning、Reflection 和 Guardrails。Goal 是任务目标，Context 是当前可用的信息，Tools 是 Agent 可以调用的外部能力，Memory 负责记录任务状态，Planning 负责拆解步骤，Reflection 负责根据结果调整方向，Guardrails 负责限制风险和错误行为。

Agent 真正重要的能力不是“自动化一切”，而是能在任务执行过程中使用工具、读取反馈、更新状态，并且在证据不足或风险较高时停下来请求人工确认。比如在 Web3 场景里，Agent 可以帮助查询链上数据、解释交易、分析合约风险、读取协议文档、生成交互脚本，但不应该自动完成私钥管理、钱包签名、资产转账、无限授权或主网部署。

我觉得比较合理的 Web3 Agent 设计是：Agent 负责分析、准备、模拟和解释，人类负责确认、签名和提交高风险操作。这样既能发挥 AI 的效率，又能保留关键权限在人手里。

今天最大的收获是：AI Agent 不是一个单次模型调用，而是一套可控、可追踪、可恢复、可验证的任务执行系统。好的 Agent 不只是会回答问题，还要知道什么时候调用工具、什么时候引用证据、什么时候承认不知道、什么时候必须让用户介入。

# 2026-05-19
<!-- DAILY_CHECKIN_2026-05-19_START -->






一次链上操作可以理解成：账户发起意图，钱包管理身份，签名证明授权，交易承载指令，Gas 支付执行成本，合约处理逻辑，测试网用于练习验证，区块浏览器负责查看结果。

**账户**

账户是链上的身份地址，类似一个公开的账号。别人可以通过地址给你转账、查看你的公开链上记录。账户本身不等于钱包，账户更像“身份”，钱包更像“管理身份的工具”。

**钱包**

钱包用来管理账户和私钥。它可以展示余额、连接应用、发起交易，也可以帮你完成签名。真正重要的是私钥或助记词，因为它代表对账户资产和操作权限的控制权。

**签名**

签名是用私钥对某个操作进行授权。它的作用是证明“这个操作确实由账户控制者同意”。签名不会直接暴露私钥，但能让链或应用验证你的授权是否有效。

**交易**

交易是提交到区块链上的操作包。比如转账、调用合约、铸造 NFT、授权代币等，本质上都是交易。交易会包含发送方、接收方、调用数据、金额、Gas 等信息。

**Gas**

Gas 是执行链上操作需要支付的手续费。因为区块链上的每一步计算、存储、合约调用都需要节点执行，所以用户需要为资源消耗付费。操作越复杂，通常 Gas 越高。

**合约**

合约是部署在链上的程序。用户通过交易调用合约函数，合约根据预先写好的规则执行逻辑，比如转账、兑换、质押、投票、铸造等。合约一旦部署，通常具有公开、可验证、难以随意修改的特点。

**测试网**

测试网是模拟真实区块链环境的练习场。它使用没有真实价值的测试币，让开发者和用户可以测试钱包连接、签名、交易、合约调用等流程，避免直接在主网上花真钱试错。

**区块浏览器**

区块浏览器是查看链上数据的工具。你可以用它查询账户余额、交易状态、Gas 消耗、合约地址、事件日志、区块信息等。交易提交后，如果想确认是否成功，通常会去区块浏览器查交易哈希。

一条完整的链上操作链通常是：

1\. 用户用钱包连接应用。

2\. 应用识别用户账户地址。

3\. 用户发起操作，比如转账或调用合约。

4\. 钱包弹出交易详情，让用户确认。

5\. 用户用私钥完成签名。

6\. 签名后的交易被广播到区块链网络。

7\. 节点验证签名、余额和 Gas 是否足够。

8\. 交易被打包进区块并执行。

9\. 如果调用了合约，合约按规则改变链上状态。

10\. 用户通过区块浏览器查看交易是否成功。

所以，链上操作不是单个动作，而是一条连续流程：

**账户代表你是谁，钱包帮你管理身份，签名证明你同意，交易把操作送上链，Gas 支付执行费用，合约完成业务逻辑，测试网用于低成本练习，区块浏览器负责追踪和验证结果。**
