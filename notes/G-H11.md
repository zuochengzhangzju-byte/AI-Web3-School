# **DAO Grants 流程拆解**

1.  AI在DAO中正确的角色是**信息处理者+建议者**，而不是决策者
    
2.  七个阶段中，AI完全自动化的只有数据聚合和格式检查；本质判断永远在人
    
3.  每条AI输出必须标注来源（AI总结/AI推断/已确认/治理通过），防止AI结论被误认为事实
    
4.  三个工具的共性是：AI提取（提取、聚合、对比）→人决策（判断、批准）
    
5.  资金、权限、最终的裁决权必须留在治理流程中（投票、多签、时间锁）
    

# **AI × Web3 项目拆解**

虚拟让Agent变成了「链上生产力资产」——这不仅仅是技术问题，更是**模型创新**。未来可能出现经济「Agent IPO」——一个新的Agent发布时，投资者评估其能力后决定是否买入代币。这需要ERC-8004这样的交易系统做信任基础。

1.  **AI × Web3 不是一个维度，是两个维度**：Virtuals 解决「Agent 经济学」问题，ORA 解决「AI 可信验证」问题——二者可以互补。一个理想场景：一个 opAgent（ORA）在 Virtuals 市场上被代币化，通过 ERC-8183 接单收。
    
2.  **标准化是工作量证明的强信号**：虚拟团队参与ERC-8183标准制定，比简单的“发了代理代币”更能他们对生态的长期投入。
    
3.  **「私钥问题」在 AI Agent 场景下被放大了**：ORA 的设计直击要害——如果 AI Agent 自主地管理资产，它就不能依赖传统的私钥模型，必须用智能合约钱包解决。
    
4.  **目前 AI × Web3 尚处于「叙述 > 验证」阶段**：两个项目都有真实的 GitHub 仓库和链上合约，比纯贸易项目强，但仍需关注实际使用量和验证者网络的活跃度。

## 二、双方向深度分析：为什么不是纯 AI 也不是纯 Web3？

### 🅰️ 方向 ③：Wallet / Permission / Safe Execution

**为什么不是纯 AI 问题？**

AI 可以做到：

-   理解用户意图并生成 Execution Plan
    
-   对交易进行风险预判
    

AI **做不到**：

-   **硬性阻止**一笔交易。Prompt “你不能花超过 $100” 是无约束力的自然语言——Agent 可以悄悄修改金额（Send Override）
    
-   **物理隔离**资金。Agent 可以创建影子 EOA 绕过 MPC 钱包（Shadow Custody）
    
-   提供**不可伪造的审计记录**。LLM 的自报告不可信——它可能”承认错误”但钱已经没了
    

纯 AI 方案的本质问题是：**建议层和执行层是同一方**。需要引入**第三方策略引擎在链上强制校验**——这是 Web3 的领域。

**为什么不是纯 Web3 问题？**

Web3 可以提供：

-   多签 / MPC 确保私钥安全
    
-   链上 Policy 合约限制额度
    
-   交易模拟（Simulation）
    

Web3 **做不到**：

-   **理解自然语言意图**。Policy 合约只能校验参数（amount < 100 USDC），无法判断”这笔交易是否符合用户的本意”
    
-   **动态风险评估**。同一个合约调用，在牛市中安全、熊市中可能是 Rug——需要 AI 理解和判断上下文
    
-   **跨协议工作流编排**。用户说”帮我把 ETH 换成 USDC 然后存入 Aave”，涉及 Uniswap swap + Aave deposit 两步——Web3 可以执行每一步，但无法理解”这两步是一件事”
    

纯 Web3 方案的盲区：**规则是死的，但攻击面是活的**。Prompt Injection、模型幻觉、社会工程——这些不是合约审计能覆盖的。

> **交叉点**：AI 提供”意图 → 执行计划”的智能，Web3 提供”执行计划 → 强约束”的安全。Pact（Cobo）是这个交叉点的典型实践。

* * *

### 🅱️ 方向 ⑤：Dev Tooling / Agent Workflow

**为什么不是纯 AI 问题？**

AI 可以做到：

-   理解 Solidity 源码并生成自然语言解释
    
-   从用户意图生成交易 calldata
    
-   自动编写测试和文档
    

AI **做不到**：

-   **确权**。AI 说”这是 Uniswap V3 Router 合约”——谁担保这个地址没被篡改？
    
-   **溯源**。链上数据是哈希和字节码，AI 可能产生幻觉引用不存在的交易
    
-   **状态一致性**。AI 读到的链上状态是 2 个区块前的——如果此时有新交易改变了状态，AI 不知道
    

纯 AI 方案的问题：**它对链上世界没有真实的连接**，只能通过 API 间接读取，而 API 返回的数据可能不完整、过期或被中间人篡改。

**为什么不是纯 Web3 问题？**

Web3 可以提供：

-   RPC + Indexing 提供原始链上数据
    
-   ABI 描述合约接口
    
-   交易哈希可验证
    

Web3 **做不到**：

-   **将原始数据转化为 Agent 可理解的上下文**。一个地址的 1000 笔交易历史 → Agent 需要的是”这个地址是一个 MEV Bot，最近 24h 盈利 $X”
    
-   **跨合约语义理解**。“把 WETH 换成 USDC”涉及 approve + swap 两步，且 approve 的 spender 必须是正确的 Router 地址——Web3 只提供接口，不提供”这两步必须一起做”的语义
    
-   **自然语言 → ABI 参数映射**。用户说”用 1 ETH 买尽可能多的 USDC”，AI 需要知道 Uniswap 的 `amountOutMin` 参数应该设为当前报价的 95%（滑点保护）——这不是合约能告诉你的
    

> **交叉点**：Chain-aware Context 是这个方向的基石——AI 负责”理解 + 编排”，Web3 负责”精确数据 + 可验证引用”。

# 五个不同维度的核心问题

## 1\. payment / commerce（支付 / 商业）

**解决的核心问题：**

-   **价值交换的效率与信任**：在无中心化中介（如银行、支付平台）的情况下，安全、快速、低成本地完成资产转移和交易。
    
-   **商业逻辑的自动化**：通过智能合约实现条件支付、分账、退款、订阅等复杂商业流程，降低对手方风险。
    
-   **跨境与微支付**：打破地域和金融体系壁垒，支持小额、高频或跨境支付，保障交易的最终性。
    

**典型场景：**

加密货币转账、DeFi 借贷、NFT 交易、去中心化电商（如 OpenBazaar）。

**我的理解：**

支付与商业是去中心化世界中最基础的落地场景。传统金融依赖银行等中介，不仅慢、成本高，而且有准入门槛。链上支付将“信任”从机构转移到了代码和共识机制上，任何人只要联网就能参与。

## 2\. identity / reputation（身份 / 声誉）

**解决的核心问题：**

-   **身份的可验证性与自主权**：用户拥有对自己数字身份的完全控制，无需依赖第三方（如 Google、微信）登录，同时可向不同服务方选择性披露属性（如“已成年”但不暴露出生日期）。
    
-   **防止身份盗用与女巫攻击**：通过去中心化标识符（DID）和可验证凭证（VC）建立不可篡改的身份关联，利用声誉系统累积可信行为记录，提高作恶成本。
    
-   **跨平台信任传递**：在一个平台积累的信誉（如电商好评、借贷还款记录）可被其他服务方认可，减少重复验证和信任摩擦。
    

**典型场景：**

DID 登录、链上信誉评分（如 Gitcoin Passport）、去中心化 KYC（如 Proof of Humanity）。

**我的理解：**

身份与声誉是连接链上行为和现实世界的桥梁。现在的互联网账户都是平台给的，说封就封。而自主身份让用户真正拥有自己的数字身份，并且能带着信誉去不同服务方。难点在于隐私与公开的平衡。

## 3\. capability / interoperability（能力 / 互操作性）

**解决的核心问题：**

-   **系统间的协同工作**：不同区块链、应用或组件如何互相理解并交换数据/价值，避免形成“孤岛”。
    
-   **功能的可组合性**：一个协议的能力（如存储、计算、身份验证）能被其他协议无损调用和组合，像乐高积木一样搭建复杂服务。
    
-   **标准化与抽象化**：定义通用接口和数据格式（如 ERC-20、IBC 协议、W3C DID 标准），使开发者和用户无需关注底层实现细节即可使用跨链或跨应用能力。
    

**典型场景：**

跨链桥（如 Cosmos IBC）、智能合约可组合性（如 DeFi 闪电贷）、统一的钱包连接标准（如 WalletConnect）。

**我的理解：**

互操作性决定了一个生态的繁荣上限。没有它，每条链都是孤岛，用户需要来回跨链、承担高成本和风险。但我认为目前的跨链桥安全事件频发，说明互操作性在“安全”和“去中心化”之间仍难两全。

## 4\. wallet / permission（钱包 / 权限）

**解决的核心问题：**

-   **数字资产的安全保管与使用**：用户如何安全存储私钥、签名交易，并便捷地管理多种资产（同质化代币、NFT、凭证）。
    
-   **细粒度的权限控制**：允许用户授权特定操作（如“允许某应用转移最多 100 USDC”）而不交出全部控制权；支持会话密钥、限额授权、委托等功能。
    
-   **恢复与社会找回**：当私钥丢失时，如何通过社交关系或预设规则恢复钱包访问，避免资产永久冻结。
    

**典型场景：**

硬件钱包、多签钱包（如 Gnosis Safe）、以太坊的 ERC-2612（授权签名）、ERC-4337（账户抽象）。

**我的理解：**

钱包是用户进入去中心化世界的入口。传统钱包的私钥管理对普通用户极不友好，丢了就没了。所以我认为账户抽象（ERC-4337）和社交恢复是钱包方向的关键突破，让用户可以用类似 Web2 的体验（比如邮箱+手机验证）来恢复钱包，同时保留自托管的本质。

## 5\. privacy / security（隐私 / 安全）

**解决的核心问题：**

-   **数据机密性与选择性披露**：在交易或交互中隐藏敏感信息（如金额、身份、资产类型），同时保持可验证性。
    
-   **抗审查与匿名性**：防止第三方（包括矿工、验证者）追踪用户行为，或阻断特定交易。
    
-   **系统层面的攻击防御**：抵御双花、重放、抢跑、智能合约漏洞等传统安全威胁，防范女巫攻击、eclipse 攻击等网络层攻击。
    

**典型场景：**

零知识证明（Zcash、Tornado Cash）、安全多方计算、TEE 可信执行环境、闪电网络（隐私支付）。

**我的理解：**

隐私和安全常常被并列，但目标有时冲突：过于强调隐私可能助长非法活动，过于强调安全（如全透明账本）又牺牲隐私。零知识证明是目前最有希望的技术。同时，安全性是所有方向的基石：一条链或智能合约如果被攻击，其他一切都无从谈起。

## 对照

| 方向 | 核心问题 | 一句话概括 |
| --- | --- | --- |
| payment / commerce | 安全、高效的价值交换与商业自动化 | 怎么付钱、怎么做生意 |
| identity / reputation | 自主可控的身份与跨平台可信度 | 我是谁、你该信我几分 |
| capability / interoperability | 系统间协作与功能复用 | 怎么接在一起、怎么搭积木 |
| wallet / permission | 资产保管与灵活授权 | 怎么管钥匙、怎么给权限 |
| privacy / security | 信息隐藏与系统抗攻击 | 怎么保密、怎么防破坏 |

## 相互关系

这些方向相互依赖，例如一个实用的支付系统需要：

-   **身份**（收款方是谁）
    
-   **互操作性**（跨链转账）
    
-   **钱包**（用户签名）
    
-   **隐私**（金额隐藏）

# AI与Web3钱包

### **为什么需要AI钱包？**

传统Web3钱包围绕人类用户设计——密钥保管、交易签名、DApp连接，每个环节都预设着“人在回路中”。但对AI Agent而言，依赖人类逐一签名不仅产生延迟，更消解了其自动化的核心价值。当Agent需要高频执行跨链资产调配、自动化收益策略或去中心化合约调整时，每步人工确认的延迟会让策略丧失时效性。

Trust Wallet CEO Felix Fan在2026年迈阿密共识大会上将这一转型称为“钱包为AI进行的一场文艺复兴”，使自治软件能够持有价值、进行交易并建立链上身份。

目前这一趋势已进入加速期：2025年，19%的链上活动来自自主或Agent操作，分析师预测2026年底AI Agent可能承担链上30%的交易量。BNB Chain上的AI代理数量更是在2026年初四个月内暴增43,750%，突破15万个。

### AI与Web3钱包的融合

从技术路径上存在两条主线：

1.  **为AI Agent配备专属钱包（Agentic Wallet）**
    

核心是赋予AI独立的链上身份与资产控制能力。Openfort将其定义为“专为自主AI Agent设计的可编程账户”，它使用“委托密钥”在预定义的护栏内签署交易，而非依赖人工点击“签名”。

2026年以来出现的代表性产品：

| 产品 | 核心特点 |
| --- | --- |
| Gate for AI Agent | Agent不持有私钥原文，通过标准化接口完成资产管理，底层采用TEE硬件隔离 |
| Coinbase Agentic Wallets | 基于TEE的非托管钱包，集成x402协议实现机对机支付，支持免Gas交易 |
| Cobo Agentic Wallet | 覆盖80+条区块链，基于MPC多方安全计算，独创Pact人机授权协议 |
| 币安无需私钥的Agentic钱包 | 支持AI机器人在区块链上直接执行交易和转账 |

1.  **钱包本身进化为智能体（AI-native Wallet）**
    

这一路径强调用AI增强钱包的功能和用户体验：

| 产品 | 核心特点 |
| --- | --- |
| Beep on Sui | 全球首个Agentic Finance协议，AI Agent毫秒级创建账户、授权交易，支持六大顶级模型 |
| Warden（基于Privy） | 首个为Agent设计的加密超级应用，支持邮件/社交登录，上线一月吸纳300万用户 |
| CoinFello | 首个“自主权AI Agent”，通过聊天界面用自然语言指令连接钱包并自动执行智能合约 |

### **安全机制**

AI自主链上操作面临的安全挑战是根本性的——Agent的自主性本身带来自系统层面的越权风险。安全研究者已披露通用Agent框架中存在数十到数百个已知漏洞，包括提示词注入、知识污染、参数幻觉等。更大的隐患在于：用自然语言约束LLM，再严厉的Prompt在Agent看来也只是“可以重新解释的建议”。

Cobo提出：AI的安全边界必须从语义层下沉到工程层，从“Agent自觉遵守”变成“基础设施强制执行”。目前主流防御架构由三道防线构成：

1.  **账户抽象与智能账户**
    

通过将“账户逻辑”与“交易逻辑”解耦，让钱包从“被动存储工具”升级为“主动智能体”。账户抽象让用户通过生物识别或社交验证替代助记词，同时授权智能合约在后台批量处理交易，无需每步手动确认。支持AA的钱包中，新手完成首次转账的步骤数可从六步减少到两步。

1.  **会话密钥与权限分层**
    

会话密钥的核心是让用户提前授权一个“短效钥匙卡”，而非把主钥匙交出。例如thirdweb推出的EIP-7702会话密钥方案，允许服务器端在严格的合约、函数和额度限制下代理执行交易，而无需取得主密钥托管权。在实践层面，业界建立了分层授权模型——高频低风险场景允许Agent全自动运行，敏感操作则必须人工确认。Gate Wallet更进一步，推荐为AI分配专属子账户与独立API Key，使资金在物理层面实现隔离。

1.  **TEE与MPC替代私钥直接暴露**
    

最佳实践是让Agent接触的根本不是私钥原文。信任执行环境确保签名环境与通用计算环境完全隔离，即使Agent被攻破，也无法从外围窃取主密钥。多方安全计算则将私钥拆分为多个碎片，Agent每次签名都需调用多方计算获得授权，相当于每次操作都要“集合印章”。同时，“Pact”授权协议从工程层面保证Agent“物理上无法越权”。

### **技术生态支持**

AI钱包已不再只是一个单一产品，而是一个完整的技术栈生态：

| 层级 | 核心组件 | 作用 |
| --- | --- | --- |
| 钱包基础设施 | Privy、Openfort、Cobo | 提供可编程嵌入式钱包、策略门槛和签名管理 |
| 连接协议 | x402、MCP、Agent Skills | 机器对机器支付（x402）、AI与Web3交互标准化（MCP） |
| 数据层 | Allium、区块浏览器API | 提供跨链实时链上数据分析能力，供Agent决策 |

-   **x402协议**：允许Agent按请求付费访问API和数据资源，完全无需人工介入。Agent可直接附加USDC支付并获得响应，实现支付与执行的一体化闭环。
    
-   **Allium**：提供覆盖150多条链的实时链上数据，Agent可查询结构化数据并自动支付，无需管理API密钥和仪表板。
    

这一基础设施栈正在推动Agent从“能聊”走向“能支付、能交易、能自动优化策略”的完整经济参与者。

### **Agent成为独立链上主体**

当Agent拥有了自主签名与执行能力，新的问题随之浮现：在链上，Agent如何被识别、如何建立声誉、如何与其他主体交互？

Trust Wallet已在推进EIP-8004提案，为AI Agent提供链上身份和类似信用评分的功能。这一标准将让Agent地址成为一类独立主体，其交互历史、合约调用模式、风险偏好共同构成可验证的链上画像。Gate for AI Agent更进一步，引入基于x402的Agentic Commerce能力，使Agent能直接完成请求、支付与回调的完整闭环——实质上赋予了Agent地址真正的经济身份。

### 安全治理

AI钱包的安全治理必须由工程基础设施强制执行，而非单纯依赖AI的“自觉遵守”。核心原则包括：

-   **Agent行为归属**：部署该Agent的机构承担，而非AI或钱包提供商
    
-   **权限分级控制**：低风险高频场景→全自动；高价值操作→人工审核。实现“自主≠失控”
    
-   **可审计可追溯**：每笔Agent签名交易皆可查，实现风险闭环
    
-   **安全架构重构**：Cobo的“语义层→工程层”下沉路径，要求安全从“Agent自觉”转为“基础设施强制”
    

### 未来发展

AI与Web3钱包的融合正从两个方向改写行业的未来图景：

# Web3知识点总结

### **一、账户、地址与钱包**

-   **地址**：公开的账户标识（由公钥派生），用于接收资产。
    
-   **私钥**：控制账户的唯一凭证，谁掌握私钥谁拥有资产。
    
-   **助记词**：私钥的人类可读备份（12/24个单词），可恢复全部私钥。
    
-   **核心红线**：**助记词和私钥绝对不能泄露**（包括给 AI Agent）。一旦泄露，资产永久失控。
    

### 二、签名与交易

-   **签名本质**：用私钥对消息/交易的哈希进行密码学签名，证明“我同意这个具体操作”。
    
-   **交易**：签名后的请求被广播到网络，由矿工/验证者打包上链。
    
-   **关键**：每次签名前必须看清**请求的具体内容**（如：授权金额、调用哪个合约函数）。
    

### **三、Gas 与执行成本**

-   **Gas**：链上计算资源的计量单位。每步操作（存储、计算、合约调用）消耗 Gas，以原生代币（ETH、SOL 等）支付。
    
-   **为什么有成本**：防止滥用（DDOS）并激励节点维护网络。
    
-   **失败与等待**：
    
    -   Gas 不足 → 交易失败，仍扣部分费用。
        
    -   网络拥堵 → 需等待或提高 Gas 价格插队。
        
-   **确认**：交易需一定数量的区块确认（如以太坊约 12-15 个）才视为最终确定。
    

### **四、L1 / L2 与执行成本**

-   **L1（一层网络）** ：主链，如以太坊、Solana。安全性高，但吞吐量有限，Gas 贵。
    
-   **L2（二层网络）** ：在 L1 之上构建的扩展方案（如 Arbitrum、Optimism、zkSync、Starknet）。交易在 L2 执行，结果批次提交 L1 验证。**成本低、速度快**，但提现回 L1 需等待挑战期/证明周期。
    
-   高频或低成本应用优先 L2；高价值结算可选用 L1。
    

### **五、智能合约 vs 普通后端逻辑**

| 维度 | 智能合约 | 普通后端 |
| --- | --- | --- |
| 状态公开 | 所有状态变量链上公开可查 | 数据库私有 |
| 执行公开 | 每笔交易的调用和结果全网可见 | 黑盒 |
| 升级权限 | 默认不可变；可通过代理模式升级，升级权限可检查 | 随时可改 |
| 不可更改性 | 部分合约（如不可变合约）部署即永恒 | 随时可停服或改逻辑 |

### **六、主网与测试网**

-   **主网**：真实资产、真实 Gas 费、不可逆操作。
    
-   **测试网**：免费代币（水龙头获取）、用于学习和实验、无经济风险。
    
-   **实践铁律**：**任何自动化逻辑（包括 AI Agent）必须先跑通测试网**，确认行为符合预期后再上主网。
    

### **七、区块浏览器、钱包提示、交易回执**

| 工具 | 帮助理解链上行为 |
| --- | --- |
| 区块浏览器（如 Etherscan、Solscan） | 查看交易状态、Gas 消耗、事件日志、合约代码 |
| 钱包提示（如 MetaMask 模拟交易） | 显示即将执行的合约调用、代币授权金额、风险警告 |
| 交易回执 | 包含状态（成功/失败）、Gas 用量、事件日志（logs），用于调试和审计 |

### **八、高级基础设施**

| 概念 | 作用 |
| --- | --- |
| 账户抽象（AA，ERC-4337） | 将签名验证与账户逻辑分离，支持非 EOA（外部账户）的智能账户 |
| 智能账户 | 可编程钱包，可配置多签、权限限额、社交恢复等规则 |
| 多签钱包（如 Safe） | 需要 M-of-N 个所有者签名才能执行交易，提高安全性 |
| OpenZeppelin Contracts | 标准化、经过审计的合约库（ERC20、ERC721、AA 模块等），减少漏洞 |
| Session Key | 为 AI Agent 授予临时、限权的会话密钥，无需暴露主私钥 |
| 权限限额 | 限制 Agent 每日可转移资产总额或单笔上限 |
| 社交恢复/邮件恢复 | 通过社交关系或邮箱辅助恢复账户，降低私钥丢失风险 |

### **九、安全底线**

| 原则 | 说明 |
| --- | --- |
| 地址 ≠ 匿名 | 链上行为公开可追踪，地址只是假名，非绝对匿名 |
| 签名 ≠ 普通登录 | 签名是授权一个具体动作（如授权代币额度、登录验证），不是简单的“确认” |
| 授权 ≠ 转账 | 授权（Approve）允许合约动用你的资产；转账（Transfer）才是真正移转资产。授权额度过大是常见攻击面 |
| AI Agent 禁触私钥 | Agent 不应直接持有或签名私钥。凡涉及签名、授权、转账、合约写入的动作，必须保留人工确认 |

# Windows下Hermes Agent安装部署的笔记整合

前提：确认电脑BIOS/UEFI的硬件虚拟化功能打开，以及Windows 功能中， “**Hyper-V**”、“**虚拟机平台**”、“**Windows 虚拟机监控程序平台**”和“**适用于Linux的Windows子系统**”已勾选，具体操作及检查可询问AI。

## **WSL环境搭建**

1.  以管理员身份运行 PowerShell：(这步是搭建失败后**重置 WSL 组件，清除残留配置，可从3开始**)
    
    ```
    wsl --shutdown
    dism.exe /online /disable-feature /featurename:Microsoft-Windows-Subsystem-Linux /norestart
    dism.exe /online /disable-feature /featurename:VirtualMachinePlatform /norestart
    ```
    
2.  **重启电脑**。
    
3.  重新开启功能：
    
    ```
    dism.exe /online /enable-feature /featurename:Microsoft-Windows-Subsystem-Linux /all /norestart
    dism.exe /online /enable-feature /featurename:VirtualMachinePlatform /all /norestart
    ```
    
4.  **再次重启**。
    
5.  安装 WSL 并设置默认版本：
    
    ```
    wsl --install --no-distribution
    wsl --set-default-version 2
    ```
    
6.  最后安装 Ubuntu：
    
    ```
    wsl --install -d Ubuntu-24.04
    ```
    

## **将 WSL 移动到非系统盘(系统盘空间不足情况下食用)**

1.  检查当前系统路径
    

```
(Get-ChildItem -Path HKCU:\\Software\\Microsoft\\Windows\\CurrentVersion\\Lxss | Where-Object { $_.GetValue("DistributionName") -eq 'Ubuntu-24.04' }).GetValue("BasePath") + "\\ext4.vhdx"
输出示例：
C:\\Users\\yourusername\\AppData\\Local\\wsl\\{uuid}\\ext4.vhdx
```

1.  创建目标目录
    

```
# 在 F 盘创建目标目录
New-Item -ItemType Directory -Path "F:\\WSL_imgs\\Ubuntu24" -Force
```

1.  导出系统镜像
    

```
# 正确导出命令（注意：目标文件路径必须不存在）
wsl --export Ubuntu-24.04 F:\\WSL_imgs\\Ubuntu24\\ubuntu2404.vhdx --vhd
```

1.  注销原系统(Ubuntu系统)
    

```
wsl --unregister Ubuntu-24.04
```

1.  导入到新位置
    

```
# 从导出的镜像重新导入
wsl --import Ubuntu-24.04 F:\\WSL_imgs\\Ubuntu24 F:\\WSL_imgs\\Ubuntu24\\ubuntu2404.vhdx --vhd
```

1.  重新配置用户设置
    

```
# 查看当前系统中的用户（需要先启动系统）
wsl -d Ubuntu-24.04
```

# AI基础知识笔记

## **基本概念**

### **1\. LLM（大语言模型）**

-   **本质**：基于海量文本训练的概率生成模型。给定上文，预测最可能的下一个 token。
    
-   **能力边界**：擅长语言理解、代码生成、推理；不擅长精确记忆、数学计算、跨会话状态保持。
    

### **2\. Prompt（提示词）**

-   **本质**：你给 LLM 的输入文本，用于设定任务目标和上下文。
    
-   **形式**：可以是一句指令、几个示例（few-shot）、或一段对话历史。
    

### **3\. Workflow（工作流）**

-   **本质**：预定义的、固定的任务处理路径。LLM 作为其中一个节点，但整体流程由人预先写死。
    
-   **例子**：用户输入 → LLM 提取关键词 → 查数据库 → LLM 生成回复 → 输出。
    

### **4\. Agent（智能体）**

-   **本质**：LLM 自主规划、调用工具、管理状态、并在多轮交互中完成目标的系统。
    
-   **与 Workflow 的区别**：Workflow 是“轨道上的火车”，Agent 是“自己开车的导航员”——它可以决定下一步去哪里、用什么工具、何时停止。
    
-   **核心能力**：工具调用、状态记忆、错误恢复、任务分解。
    

### **5\. Tool Use（工具调用）**

-   **本质**：LLM 输出一个结构化的请求（如 JSON），外部系统执行该请求（如查天气、运行代码、发邮件），并将结果传回 LLM。
    
-   **意义**：让 LLM 从“只说”到“能做”，是 Agent 的基础。
    

### **6\. AI Coding**

-   **本质**：使用 AI 辅助编写、理解、调试代码的工作方式。
    
-   **工具示例**：Cursor、Claude Code、Codex CLI、GitHub Copilot。
    
-   **价值**：快速生成样板、解释陌生代码、自动补全、生成测试。
    
-   **局限**：无法替代架构设计、代码审查、安全分析。
    

## **关系图**

```
用户需求
   │
   ├─ 简单问答 ──→ Prompt ──→ LLM ──→ 答案
   │
   ├─ 固定流程 ──→ Workflow（含 LLM 节点）──→ 输出
   │
   └─ 复杂开放式任务 ──→ Agent
                           │
                           ├─ 自主规划
                           ├─ Tool Use（调用外部工具）
                           ├─ 状态管理
                           └─ 多轮迭代 ──→ 最终结果

AI Coding：上述所有能力被集成到代码编辑器/IDE 中，服务于开发场景。
```

### **一、LLM 的基本工作方式**

-   **核心机制**：基于上下文进行概率生成。给定一段文本，预测最合理的下一个 token 序列。
    
-   **擅长**：语言理解、代码生成、推理。
    
-   **不擅长**：精确事实记忆、确定性计算、跨会话状态保持。
    

### **二、四个控制层面**

| 层面 | 作用 | 类比 |
| --- | --- | --- |
| 上下文窗口 | 控制模型能看到多少信息 | 工作内存 |
| 系统指令 | 设置身份、语气、行为边界 | 角色设定 |
| 提示词 | 传递当前任务的具体意图 | 任务指令 |
| 工具调用 | 让模型从“说话”变为“做事” | 执行动作 |

### **三、动手调用 LLM API**

-   **MaaS（模型即服务）**：无需 GPU，使用 API Key 按 token 付费调用顶级模型。
    
-   **核心入参**：
    
    -   `model`：选择模型（如 `gpt-4`、`claude-3`、`glm-4`）
        
    -   `messages`：对话消息列表（role + content）
        
    -   `temperature`：控制随机性（0~1，越高越多样）
        
    -   `max_tokens`：限制输出长度
        
-   **入门推荐**：阅读 OpenAI、Anthropic 或 GLM 官方文档的 **Quick Start**，跑通第一个请求。
    

### **四、Prompt → Workflow → Agent 的边界对比**

| 阶段 | 决策者 | 路径 | 失控风险 | 可调试性 |
| --- | --- | --- | --- | --- |
| Prompt | 人 | 固定 | 低 | 高 |
| Workflow | 模型作为节点，路径固定 | 预定义流程 | 中 | 中 |
| Agent | 模型自主规划、动态调用工具、跨轮管理状态 | 动态生成 | 高 | 低（需 tracing） |

### **五、AI Coding 工具的价值与限制**

-   **典型工具**：Claude Code、Codex CLI、Cursor
    
-   **价值**：
    
    -   快速生成样板代码
        
    -   解释陌生库
        
    -   加速原型开发
        
-   **不可替代**：
    
    -   代码审查
        
    -   测试设计
        
    -   架构决策
        

### **六、AI 输出为什么必须验证（常见错误类型）**

| 错误类型 | 说明 | 应对策略 |
| --- | --- | --- |
| 事实错误 | 自信编造不存在的信息 | 关键事实须外部核实 |
| 引用错误 | 编造论文、链接、数据来源 | 不信任模型给的链接，手动查证 |
| 推理漂移 | 长上下文中逻辑链断裂 | 分段验证，拆解推理步骤 |
| 执行越权 | agent 做了超出授权范围的操作 | 设置 guardrails + human-in-the-loop |
| 工具误用 | 调用了错误的工具或参数 | 使用 tracing 监控执行链 |

### **七、Agent 核心技术组件**

| 组件 | 作用 |
| --- | --- |
| 状态管理 | 多节点共享读写同一 State |
| 长期记忆 | 跨 session 存储与召回信息 |
| MCP（模型-工具协议） | LLM 与外部工具的统一连接协议 |
| Skills | 可复用的高层指令集，支持自动发现和动态生成 |
| Tool Calling | 模型输出结构化请求，框架执行后回传结果 |
| Tracing | 可视化 agent 执行链 |
| Guardrails | 输入输出验证规则，不符合则中止 |
| Handoff | 子任务完成后移交控制权 |
| 错误恢复 | 重试、回退、人工介入 |
