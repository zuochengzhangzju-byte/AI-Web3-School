# 氛围编程（Vibe Coding）

**Vibe Coding 本质**：不是甩需求给 AI 等结果，而是**人 + AI 协作迭代**：人定方向、边界、验收；AI 做搜索、生成、改代码、跑命令、解错误、写 PR。

## 为什么要学

AI 能读改代码、跑测试、解释错误、生成 PR，**显著提速开发**；但速度提升后，关键变成：**清晰描述任务、审查结果、控制改动范围、验证无新 bug**。

误区：不是 “凭感觉写代码”，反而要**更严格管理仓库、任务、上下文、测试、提交边界**；AI 提速，不替人做工程判断。

## 核心：第一性原理

AI 编程核心不是自动写代码，而是**缩短工程反馈循环**（搜索→理解→改→测→改错）。

关键前提：**必须有测试、审查、版本控制**，否则速度 = 混乱。

四大原则：

1.  **任务要小**：越具体、边界越清，AI 产出越可控、易审查。
    
2.  **上下文要准**：给对文件、约束、错误日志，比长篇需求更重要。
    
3.  **验证要硬**：跑测试、类型检查、构建、截图 / 日志，不靠 “看起来对”。
    
4.  **人控决策**：AI 辅助，人负责工程判断。
    

## 知识节点（分层）

### 1\. Vibe Coding（初级）

-   分工：人 = 目标 / 边界 / 验收；AI = 搜索 / 生成 / 修改 / 执行。
    
-   适用：搭原型、修小 bug、补测试、写脚本、重构局部、解释陌生代码。
    
-   不适用：**无边界重写整个项目**（易改错、引入不合规抽象）。
    

### 2\. Claude Code / Codex CLI（初级）

-   作用：把 AI 接入本地仓库 / 终端 / 测试，能读、改、跑命令、解测试输出。
    
-   安全关键：明确**可改文件、可跑命令、依赖 / 网络权限、人工确认节点**。
    

### 3\. 模型 / API 配置（中级）

-   不止选强模型，还要管：模型选择、API Key、上下文窗口、工具权限、成本、日志、团队复用配置。
    
-   常见坑：本地 / CI 模型不一致、Key 泄漏、上下文过长、权限过大、输出不规范。
    
-   工具：**OpenCLI**（统一网站 / 浏览器 / 本地工具为 CLI，稳定权限与输出，复用登录态）。
    

### 4\. GitHub/gh CLI（中级）

-   作用：把 AI 改动纳入版本控制、Issue、PR、评审，**可追踪、可审查**。
    
-   原则：AI 做**局部小补丁**，少做大改动；每次改完查：git diff、文件列表、测试结果、敏感信息。
    

### 5\. CLI / 脚本（中级）

-   适用：一次性脚本、数据迁移、检查、批量修改。
    
-   风险：**读写范围确认、改外部状态先 dry run**；删文件 / 发请求 / 改库 / 调生产 API，**必须人工把关**。
    

### 6\. 仓库工作流（Repo Workflow，高级）

-   核心：AI 融入完整工程流程（Issue→分支→提交→测试→评审→合并→发布），**不绕过流程，参与流程**。
    
-   AI 可做：提炼任务、搜文件、最小补丁、跑测试、补文档、写 PR 说明；**质量最终靠流程保障**。

# Framework

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/AI-Web3-School/main/assets/Marina-asui/images/2026-05-30-1780124436481-image.png)

从底层思想 → 基础能力 → 编排框架 → 优化工具 → 模型底座 → 成品系统，形成一条完整链路：

text

```
第一性原理
├─ 知识节点（核心概念）
│  ├─ 学习型智能体 Learning Agent
│  └─ 工具调用 / 结构化输出 / 多智能体协作
├─ 通用编排层
│  ├─ LangChain：基础组件库
│  └─ LangGraph：状态机/复杂工作流
├─ 厂商专属层
│  └─ OpenAI Agents SDK：OpenAI 生态工程化框架
├─ 优化层
│  └─ DSPy：Prompt/推理链路自动优化
├─ 模型与自进化层
│  └─ Hermes：强工具调用模型 + 自进化 Agent
└─ 成品执行层
   └─ OpenClaw：开箱即用的本地多通道 Agent 网关
```

* * *

### 1\. 第一性原理

-   是整个体系的**底层逻辑**：
    
    -   不依赖 “现成框架 / 模型的黑箱魔法”，而是回到「问题本质」思考：
        
        -   这个 Agent 要解决什么问题？
            
        -   哪些部分是可变的？哪些是核心约束？
            
        -   模型能力、编排流程、优化目标，三者如何匹配？
            

### 2\. 知识节点

-   是支撑所有框架的**基础概念**：
    
    -   工具调用、结构化输出、记忆、多轮对话、评估指标、学习闭环等。
        
    -   后面的 LangGraph / DSPy / Hermes / OpenClaw，都是这些概念的不同实现方式。
        

* * *

### 3\. LangChain + LangGraph（通用编排层）

-   **LangChain**：把「模型 + 提示词 + 工具 + 检索」封装成可复用的组件，解决基础流程编排问题。
    
-   **LangGraph**：在 LangChain 基础上，引入「状态机 + 图」模型，解决 **复杂流程（分支 / 循环 / 断点）** 问题。
    
-   关系：LangGraph 是 LangChain 生态的进阶版，让你能写更工程化的 Agent 工作流。
    

### 4\. OpenAI Agents SDK（厂商专属层）

-   基于 OpenAI 模型的**工程化 Agent 框架**，原生支持多 Agent 交接、安全管控、链路追踪。
    
-   和 LangGraph 的区别：
    
    -   它只适配 OpenAI 模型，但把企业级 Agent 的通用工程问题做了标准化实现。
        

### 5\. DSPy（优化层）

-   解决「Prompt 工程」的痛点：用数据 + 指标驱动，自动优化提示词和推理链路。
    
-   不关心流程怎么编排（这是 LangGraph/SDK 的事），只关心「给定任务和数据，如何让模型效果更好」。
    

### 实战组合：

-   复杂流程：LangGraph 负责编排
    
-   效果优化：DSPy 负责优化关键模块的提示词 / 推理逻辑
    

* * *

### 6\. Hermes（模型 + 自进化层）

-   不是框架，而是**强工具调用的开源模型 + 自进化 Agent 生态**。
    
-   核心价值：模型本身就把「工具调用 / JSON 输出」做稳了，能大幅减少框架层的兜底成本。
    
-   和其他框架的关系：
    
    -   你可以用 Hermes 作为模型底座，搭配 LangGraph 或 OpenClaw 使用，提升 Agent 工具调用的稳定性。
        

### 7\. Learning Agent（思想层）

-   定义了 Agent 的「持续迭代闭环」：从反馈 / 日志中学习，通过评估再上线。
    
-   是所有 Agent 系统的长期目标：
    
    -   DSPy 是「离线学习」的实现方式（用数据优化）
        
    -   Hermes 自带「自进化技能系统」
        
    -   LangGraph/OpenAI SDK 可以搭建人工审核 + 迭代的闭环流程
        

* * *

### 8\. OpenClaw（成品执行层）

-   前面所有框架 / 模型，都是「开发者用的库 / 工具」，而 OpenClaw 是**装好就能用的成品 Agent 系统**。
    
-   它把「多通道接入 + 技能市场 + 本地执行 + 记忆」打包成一个网关，让你不用写代码就能跑 Agent。
    
-   它和前面所有概念的关系：
    
    -   底层可以接 Hermes、GPT、开源模型
        
    -   内置了工具调用、记忆、流程调度
        
    -   也可以搭配 Learning Agent 的思想，基于日志迭代技能配置

# AI 框架

AI 框架核心作用是规整模型、检索、部署等模块，解决 AI 应用工程化问题，而非简化代码，选错框架易导致项目迭代困难。

开发核心原则为先理清业务工作流、状态与异常路径，再按需选框架，避免逻辑迁就框架，优先选择可灵活替换组件的框架。

主流工具各有侧重：LangChain 适配快速搭建基础 AI 组件；LangGraph 适配多步、有状态的复杂任务流程；OpenAI Agents SDK 主打可组合的智能体工程流程；DSPy 依托数据指标优化提示词流程；Hermes 侧重工具调用与结构化输出。

此外，学习型智能体需搭建评估闭环，禁止线上反馈直接变更系统，规避数据污染等问题。

# Agent

## 一、核心笔记总结

### 1\. 什么是 Agent？

-   **定义**：能围绕目标，持续调用工具、读取状态、调整步骤的 AI 系统。
    
-   **核心差异**：
    
    -   普通 LLM：只生成文本回答，风险低。
        
    -   Agent：能从 “会回答” 升级为 “能做事”，风险高（错误会变成真实的操作事故）。
        
-   **关键原则**：
    
    Agent 的重点不是 “像人”，而是**被约束的执行循环**—— 必须在明确权限、可审计流程里，把建议推进到行动。
    

* * *

### 2\. 为什么要学 Agent？

-   它能做 LLM 做不到的事：拆任务、查资料、调用 API、写代码、生成草稿、等待反馈再继续。
    
-   风险点也在这里：一旦能调用工具、写入系统、触发支付，错误就不再是 “答错”，而是 “做错”。
    
-   学习重点不是追逐框架，而是建立分工：
    
    -   模型：提出候选行动
        
    -   系统：限制行动空间
        
    -   用户：批准高风险操作
        

* * *

### 3\. 第一性原理：Agent 的本质

Agent 不是无限制的自主权，而是一个**被约束的执行循环**，缺一不可的五大要素：

1.  **目标**：要完成什么任务
    
2.  **工具**：能调用的外部能力
    
3.  **状态**：任务进度与上下文
    
4.  **权限**：能做什么、不能做什么
    
5.  **停止条件**：什么时候该停下来
    

关键细节：

-   只有模型 + 工具，只是 “会说话的脚本执行器”，不是真正可用的 Agent。
    
-   工具按风险分级：只读（查天气）< 写入（改配置）< 高风险（支付 / 删除数据）。
    
-   状态必须外置、可查、可审计，不能只藏在模型上下文里。
    
-   停止条件要明确：目标达成、预算用完、信息不足、风险越界、用户拒绝，都要能停下。
    

* * *

### 4\. 四大核心知识节点详解

（1）Tool Use（工具调用）

-   定义：Agent 调用外部能力（搜索、数据库、API、代码执行、邮件、支付接口等）。
    
-   核心风险：Prompt Injection、模型误判，风险等级随操作类型指数级上升。
    
-   工具设计必须明确 6 项规则：
    
    1.  输入 schema（参数格式校验）
        
    2.  权限范围（能访问哪些资源）
        
    3.  是否只读（区分读写风险）
        
    4.  是否有外部副作用（是否会造成不可逆影响）
        
    5.  调用前后如何记录（日志审计）
        
    6.  哪些调用需要人工确认（高风险操作）
        

（2）Planning（任务规划）

-   定义：把大目标拆成可执行的步骤，比如 “分析 DAO 提案” 拆成 读取提案→查历史讨论→总结争议→检查投票机制→生成风险清单。
    
-   关键提醒：计划只是**候选路线**，不是授权。越靠近高风险动作，越需要系统规则检查。
    
-   好的计划要暴露的信息：
    
    -   每一步需要的工具
        
    -   每一步是读还是写操作
        
    -   可自动执行 / 需用户确认的步骤
        
    -   失败重试机制
        
    -   任务完成的验证方式
        

（3）State（状态管理）

-   定义：Agent 当前任务的完整状态，包括用户目标、已完成步骤、工具返回结果、错误、预算、确认记录、最终输出。
    
-   核心要求：
    
    -   不能只存在 Prompt 历史里，必须外置、可查询、可恢复、可审计。
        
    -   生产系统中，还要记录环境、版本、关键参数、工具调用结果、确认请求和撤销事件。
        
-   解决的问题：回答 “Agent 为什么调用这个工具？”“它拿到用户确认了吗？”“哪一步偏离了目标？”
    

（4）Reflection（自我反思）

-   定义：让 Agent 检查中间结果，发现信息不足、工具失败、计划不合理，修正下一步行动。
    
-   作用：提升复杂任务的质量。
    
-   限制：不能替代外部验证，尤其是写入、授权、支付这类高风险动作，自我反思只能作为辅助诊断，不能作为最终安全判断。
    

（5）Multi-Agent（多智能体协作）

-   定义：多个 Agent 分工协作，比如 研究 Agent 读资料、开发 Agent 写代码、安全 Agent 做风险审查、执行 Agent 调用工具。
    
-   适用场景：复杂工作流。
    
-   潜在问题：上下文传递丢失、责任边界模糊、错误被当成事实、工具权限扩散。
    
-   设计前提：先确认多 Agent 是否真的能减少复杂度，而不是把流程拆得更乱。
    

* * *

## 二、完整实例：「论文润色与提交 Agent」

**任务目标**：帮用户润色一篇学术论文，并按期刊要求提交到投稿系统。

### 1\. 第一步：明确 Agent 的核心要素

-   **目标**：润色论文格式与语言，生成可提交版本，并完成期刊投稿。
    
-   **权限**：只能访问用户指定的论文文件和期刊投稿系统，不能修改其他文件，不能触发支付。
    
-   **停止条件**：润色完成并生成预览、用户确认投稿、系统提示投稿失败 / 超时，任一情况触发即停止。
    

* * *

### 2\. Planning（任务规划）

Agent 把目标拆成 5 步，同时标注每一步的风险与要求：

表格

| 步骤 | 操作 | 工具 | 读写类型 | 是否需要确认 |
| --- | --- | --- | --- | --- |
| 1 | 读取用户上传的论文文件 | 文件读取工具 | 只读 | 否 |
| 2 | 按期刊模板检查格式、润色语言 | 论文格式检查工具、语言润色工具 | 只读 + 写入（生成新文件） | 否（仅生成草稿） |
| 3 | 生成润色后的论文预览 | PDF 生成工具 | 写入（生成新文件） | 否 |
| 4 | 检查投稿系统的提交要求，生成投稿信息表单 | 投稿系统查询工具 | 只读 | 否 |
| 5 | 提交论文到期刊系统 | 投稿系统提交工具 | 写入（不可逆操作） | 是（必须用户确认） |

* * *

### 3\. Tool Use（工具调用，以关键工具为例）

我们以「投稿系统提交工具」为例，看它如何满足工具设计的 6 项规则：

1.  **输入 schema**（JSON Schema）
    
    json
    
    ```
    {
      "type": "object",
      "required": ["journal_id", "paper_pdf_path", "author_info"],
      "properties": {
        "journal_id": { "type": "string", "description": "目标期刊ID，仅支持用户预设的白名单期刊" },
        "paper_pdf_path": { "type": "string", "description": "润色后论文PDF路径，仅允许访问指定目录" },
        "author_info": {
          "type": "object",
          "properties": {
            "name": { "type": "string" },
            "email": { "type": "string", "format": "email" }
          }
        }
      }
    }
    ```
    
2.  **权限范围**：仅能访问用户指定的论文目录，仅能提交到预设的白名单期刊，无支付权限。
    
3.  **是否只读**：否，是写入操作，会在期刊系统创建投稿记录。
    
4.  **是否有外部副作用**：是，提交后会生成不可逆的投稿记录，影响用户投稿流程。
    
5.  **调用前后记录**：记录调用时间、参数、返回结果、用户确认状态，存入审计日志。
    
6.  **是否需要人工确认**：是，必须用户点击 “确认提交” 后才能执行。
    

* * *

### 4\. State（状态管理）

Agent 的状态不只是存在对话历史里，还会外置存储在数据库中，字段包括：

-   任务 ID、用户 ID、目标描述
    
-   当前步骤（如 “步骤 4：生成投稿表单”）
    
-   已完成步骤的工具调用结果（如 “格式检查：符合期刊要求；润色完成度：95%”）
    
-   错误记录（如 “投稿系统查询失败，重试中”）
    
-   用户确认记录（如 “步骤 5 提交操作，用户已在 2026-05-12 15:30 确认”）
    
-   预算与剩余调用次数（如 “API 调用次数：12/50”）
    

* * *

### 5\. Reflection（自我反思）

在每一步完成后，Agent 会进行自我检查：

-   步骤 1：读取论文后，检查文件是否损坏、格式是否支持（.docx/.pdf），如果损坏则停止并提示用户。
    
-   步骤 2：润色后，检查是否遗漏了期刊的格式要求（如参考文献格式、字数限制），如果不符合则重新润色。
    
-   步骤 4：生成投稿表单后，检查是否有必填项缺失，是否符合期刊要求，如果信息不全则提示用户补充。
    

* * *

### 6\. Multi-Agent 扩展（可选）

如果把这个任务升级为多 Agent 协作：

-   **格式检查 Agent**：专门调用格式检查工具，验证论文是否符合期刊模板。
    
-   **语言润色 Agent**：专门调用润色工具，优化论文语言表达。
    
-   **安全审查 Agent**：检查所有工具调用的权限和风险，确认投稿操作已获得用户授权。
    
-   **执行 Agent**：负责按计划调用工具，记录状态，处理重试和停止条件。

# RAG

## 一、核心总览

**定义**：检索增强生成（RAG）并非简单给大模型对接向量库，而是一套**完整证据链**：从外部知识库取资料→筛选校验→规范引用后交由模型作答，核心目标是解决大模型知识过时、上下文有限、回答无依据、生成幻觉等问题。

**核心价值**：让回答**可溯源、有版本、有边界**；缺失引用（citation）、时效性（freshness）的 RAG，只是将模型幻觉转移到检索环节。

**适用场景**：产品文档问答、代码助手、客服知识库、API/SDK 辅助工具、内部资料检索、链上协议解读等。

## 二、RAG 第一性原理：三层核心判断（系统可靠性关键）

RAG 的可靠性由全流程证据链决定，整套系统必须完成三次核心判断，任一环节出错都会输出错误答案：

1.  **文档切分**：如何拆分长文档为检索片段
    
2.  **内容筛选**：查询时选取哪些有效资料
    
3.  **生成输出**：规范引用原文 + 检索无结果时**拒答**（不编造内容）
    

配套三条基础原则：

-   检索结果仅为候选证据，需核验来源、时间、版本、适用范围；
    
-   回答关键内容必须可追溯至原文；
    
-   检索无有效信息时，主动拒答，禁止模型补全编造。
    

## 三、核心模块详解

### 1\. Chunking（文档切片）

-   **作用**：将长文档拆分为适合检索的最小单元
    
-   **痛点**：切片过小易断裂语义；切片过大增加噪声与 Token 成本
    
-   **最优方案（技术文档专用）**：按文档**结构切片**，而非固定字数
    
    拆分单元：标题、API 接口、函数说明、参数、标准条款、FAQ 问答、审计 / 变更记录
    
    强制绑定元数据（Metadata）：来源 URL、标题路径、更新时间、版本号
    

### 2\. Vector DB（向量数据库）

-   **作用**：存储文本向量化（Embedding）结果，基于语义相似度快速检索切片
    
-   **关键要求**：不只存向量，必须附带完整元数据（来源、版本、更新时间、可信度、是否弃用）；检索前先做元数据过滤，再做相似度排序
    
-   **误区**：向量相似≠内容正确，老旧文档、非官方内容即使语义匹配也不可采信
    

### 3\. Retriever（检索器）

-   **作用**：根据用户问题召回候选切片，支持向量检索、关键词检索、混合检索、图检索等
    
-   **进阶要求**：识别问题中的**版本、时间、环境、对象**等限定条件，结合元数据过滤，不做纯语义检索
    

### 4\. Rerank（重排序）

-   **作用**：对初筛的候选切片二次排序，优先展示**高相关、高权威、内容完整**的片段
    
-   **适用场景**：候选结果多、语义相似度接近、技术文档 / 治理问答等高严谨场景
    
-   **取舍**：会增加延迟与成本，简单 FAQ 类场景可省略
    

### 5\. Citation（引用溯源）

-   **作用**：将回答内容与原始文档绑定，是答案可信度的核心保障
    
-   **必备溯源信息**：文档来源、是否为官方资料、版本 / 更新时间、区分原文内容与模型归纳内容、标注证据不足部分
    

### 6\. 拒答机制

检索结果完全无关、信息缺失 / 矛盾时，系统明确告知无法解答，杜绝模型编造。

* * *

## 四、全流程场景案例（串联所有概念）

### 场景设定

某 Web3 项目**v2.1 版本官方 API 文档**，搭建 RAG 问答系统，面向开发者解答接口问题；文档包含 API 接口、函数说明、FAQ、版本变更记录、审计记录。

### 用户问题

`项目v2.1版本中，transfer接口的调用限制是什么？旧版v1.0还能使用该接口吗？`

### 步骤 1：Chunking 文档切片

按**结构拆分**文档，生成独立切片，每片附带元数据：

1.  transfer 接口函数说明切片（URL、路径、更新时间、版本 v2.1）
    
2.  接口调用限制 FAQ 切片（官方来源、v2.1）
    
3.  版本变更 & 审计记录切片（标注 v1.0 接口下线时间、废弃状态）
    
    _不按字数乱切，保证函数、限制、版本记录语义完整_
    

### 步骤 2：Embedding + Vector DB 入库

所有切片转为向量存入向量库，同步保存元数据：版本、是否废弃、来源类型（官方 / 审计 / FAQ）、更新时间。

### 步骤 3：Retriever 检索

检索器识别问题包含**版本 v2.1、v1.0、transfer 接口**限定词，先通过元数据过滤，再做语义检索，召回 3 条相关候选切片。

### 步骤 4：Rerank 重排序

对候选切片二次排序：将「transfer 调用限制 FAQ」「v2.1 函数说明」「版本变更记录」排在最前，过滤无关闲聊内容。

### 步骤 5：生成回答 + Citation 引用 + 边界说明

模型基于筛选后的切片作答，并**标注引用来源**：

1.  调用限制内容 → 引自【官方 FAQ v2.1，2026-05 更新】
    
2.  v1.0 接口状态 → 引自【版本变更记录，v1.0 已废弃】
    
3.  明确区分：以上内容均为文档原文归纳，无模型自主推断
    

### 步骤 6：拒答机制（模拟兜底场景）

若用户询问文档中**未记载的合约底层逻辑**，检索无匹配切片，系统直接拒答：`未查询到相关资料，无法解答该问题`，不编造内容。

### 补充：Web3 落地约束

若开发者想直接根据回答发起链上调用，该 RAG 系统仅提供文档依据，**不执行签名操作**，需额外经过规则校验与人工复核。

# 上下文（Context）体系

## 一、核心定义

Context（上下文）是大模型单次请求中，**可见、可用、会受其影响的全部信息空间**。

核心难点不在于无限扩充内容，而在于**分类管理**：清晰划分系统规则、用户目标、历史状态、工具返回结果、外部文档五大类信息。

## 二、意义

1.  **能力边界限制**
    
    LLM 无法自动连接数据库、本地文件、接口 API、用户历史记录与项目文档，**仅依靠当前上下文生成回答**。一旦上下文缺失、错误、内容过期，即便模型性能优异，输出结果也会不可靠。
    
2.  **排查应用常见问题**
    
    多数 AI 应用故障，表面归因于 “模型不够智能”，本质是**上下文设计混乱**，典型错误场景：
    

-   将非正规网页内容当作系统强制指令
    
-   沿用旧版文档作为现行规则
    
-   把用户主观想法当成客观事实
    
-   上一轮任务状态残留，干扰下一轮任务执行
    

1.  **关联安全风险（工具 / Agent 场景）**
    

-   模型无法获取真实工具返回数据，只能主观猜测，输出偏差内容；
    
-   Agent 若不清楚实时状态、操作权限、预算限制，极易触发越权、违规操作。
    

> 举例：合约审计 Agent 缓存了旧版本权限配置，未刷新，继续按过期权限执行检测，引发判断错误。

## 三、第一性原理

1.  行动逻辑：模型**完全依据当前可见的上下文执行任务**。
    
2.  系统职责：由程序 / 人工决定三类规则 —— 哪些信息允许进入上下文、信息标注身份、过期信息如何清理退出。
    
3.  本质定性：上下文不是简单文本拼接，而是**信息治理工作**。
    
    要求：为每一条信息标注**来源、时效、权限、可信度**，避免模型混淆四类信息：用户口述内容、网络网页内容、链上查询数据、系统官方规则。
    

### 三大强制落地规范（结合实操）

**1\. 可信来源显式标注 + 分区存放**

将不同来源信息独立分区，明确可信度等级，模型按分区优先级处理。

**实操**

```
===== 【系统状态 - 高可信】=====
当前合约版本：v2.1，运行环境：主网
===== 【工具检索文档 - 高可信】=====
2026-05审计报告：v2.1无高危权限漏洞
===== 【用户输入 - 待验证】=====
用户：合约权限存在漏洞
===== 【历史日志 - 低可信/仅参考】=====
2026-03旧版v2.0存在权限越权问题
```

**2\. 上下文可刷新（禁止长期缓存）**

**管控对象**：系统状态、配置、权限、价格、软件版本、任务进度

**核心要求**：这类动态数据不能长期缓存复用，必须按需重新查询、实时更新。

**反面案例**：电商 AI 助手缓存商品旧价格，商家调价后仍沿用缓存数据，向用户播报错误价格；合约系统缓存旧版本配置，未刷新导致检测逻辑失效。

**3\. 记忆可撤销**

1.  定义：Memory（跨会话记忆）指长期留存的用户偏好、历史任务、常用配置等。
    
2.  规则：记忆仅用于优化使用体验，**严禁转化为隐藏权限、永久身份默认设定**。
    
3.  安全底线：历史授权≠当前授权，涉及身份、权限、资产、外部操作的行为，必须重新校验当前会话权限。
    
    **案例**：Agent 记住用户常用钱包地址，不可默认使用该地址执行转账；记住用户过往高风险操作偏好，不可自动放宽风险校验规则。
    

## 四、核心知识节点

### （一）Context Window（上下文窗口）

1.  定义：模型单次请求能够处理的**最大上下文容量**。窗口越大，可容纳资料越多，但不代表模型能精准利用全部内容。
    
2.  长窗口通病：信息过载，**看得见内容，但抓不住核心**。
    
    **案例**：将合约源码、多份审计报告、社区讨论、交易日志全部塞入窗口，无关信息分散模型注意力，同时易误引用过期片段。
    
3.  产品落地方案：不盲目堆砌内容，组合四类能力使用
    
    -   关键状态：直接结构化展示
        
    -   长文档：按需检索截取片段
        
    -   冗余内容：摘要压缩
        
    -   低可信内容：单独隔离、标注风险
        

### （二）Context Engineering（上下文工程）

1.  定义：设计信息进入模型的全流程，包含数据源筛选、内容排序、文本裁剪、来源标注、低可信内容隔离、高频动态数据定时重查。
    
2.  标准 Agent 上下文完整组成（不止 “问题 + 简单 JSON”）：
    
    -   当前任务状态
        
    -   工具调用返回结果
        
    -   相关日志 / 佐证材料
        
    -   可信数据来源说明
        
    -   外部校验结果
        
    -   用户原始诉求
        
    -   系统禁止操作规则、输出格式规范
        
3.  核心目标：**不填满窗口，而是让模型在正确的信息层级中工作**。
    

### （三）Memory（跨会话记忆）

1.  存储内容：跨多次请求持久化的信息，如用户偏好、历史任务记录、常用地址 / 配置、历史分析结果。
    
2.  优缺点：
    
    -   优点：优化交互体验，减少重复输入
        
    -   风险：易产生隐性权限、默认身份等安全隐患
        
3.  硬性规则：记忆**不能替代实时授权**，历史行为、过往许可，不代表当前有效。
    

### （四）Knowledge Base（知识库）

1.  定义：可检索的外部文档集合，包含产品手册、SDK 文档、代码注释、论坛内容、FAQ、内部运维手册等。
    
2.  核心作用：解决**模型原生知识过期**的问题。
    
3.  关键提醒：知识库不会自动保证内容正确，必须人工维护元数据，否则 RAG 会反复推送老旧错误内容。
    
4.  知识库必填元数据（开发者必备）+ 作用解析
    
    | 记录字段 | 作用说明 | 实操示例 |
    
    | ---- | ---- | ---- |
    
    | 文档来源 + URL | 内容溯源，区分官方 / 第三方 | 来源：合约官方文档；URL：[https://xxx.contract.com](https://xxx.contract.com) |
    
    | 最后更新时间 | 判断内容新旧，优先选用最新资料 | 更新时间：2026-05-20 |
    
    | 适用协议版本 | 限定文档生效版本，过滤旧版内容 | 适用版本：v2.1 |
    
    | 适用版本 / 运行环境 | 区分部署环境，精准匹配场景 | 运行环境：正式主网 |
    
    | 官方 / 第三方标识 | 划分可信度等级，排序上下文 | 官方文档 = 高可信，论坛讨论 = 仅参考 |
    
    | 是否需要人工复核 | 风险管控，可疑内容先校验再入库 | 第三方博客：需复核；官方公告：无需复核 |
    

## 五、全流程

1.  **文档预处理**：对知识库文档切块、生成向量、存入向量库，同步绑定全部元数据（版本、时间、来源等）。
    
2.  **用户发起提问**：将问题转为向量，语义检索匹配相关文档片段。
    
3.  **多层过滤**：依靠元数据过滤过期版本、废弃内容、无关信息。
    
4.  **信息加工**：核心数据结构化、长文本摘要压缩。
    
5.  **上下文组装**：按可信度分区、标注来源，拼接固定优先级规则（Prompt 指令）。
    
6.  **调用模型**：将完整上下文送入 LLM，模型按层级规则输出答案。
    
7.  **动态维护**：版本 / 权限 / 价格等数据变更时，主动刷新知识库与缓存，保证上下文实时有效。

# 学习笔记：OpenAI API 提示词管理机制 (Prompt Object)

## 1\. 核心概念：什么是 Prompt Object？

在调用 OpenAI API 时，传统的做法是将大段的提示词文本直接写死在代码里。而现在，OpenAI 提供了一套类似“动态网页渲染”的机制，允许开发者将“提示词撰写”**和**“代码开发”彻底分离。

## 2\. 我的核心疑点：大括号 `{{variable}}` 到底怎么用？

**疑问：** 为什么说“不需要每次都在代码里重新拼接一大串文本”？ **解答：** 过去写代码调用 AI 时，如果遇到变动的信息（如商品名、用户输入），必须用字符串拼接操作（例如 `let prompt = "写一段关于" + productName + "的文案";`）。当提示词长达几百字时，代码不仅极难阅读，而且每次修改提示词都需要重新编译和部署程序。

**现在的模板化思路：** 这就如同在 Web 开发中使用 PHP 或 JavaScript 渲染动态页面（比如通过 `<?php echo $var; ?>` 灌入数据）。

-   **后台管理：** 在 OpenAI 的工作台上写好模板，把动态变化的地方用 `{{变量名}}` 挖空（例如：`{{productName}}`）。保存后获取一个唯一的 ID（如 `prompt_888`）。
    
-   **前端/代码端：** 代码里不再出现提示词原文，只需要发送这个 ID 和对应的变量数据即可。
    

## 3\. 代码示例对比

❌ 过去的痛点做法（硬拼接）

代码繁琐，极难维护。

JavaScript

```
let productName = "复古风外套";
let targetAudience = "大学生";
// 提示词和代码高度耦合
let myPrompt = "你是一个专业的文案。请为一款名叫【" + productName + "】的商品写一段宣传语。目标受众是【" + targetAudience + "】，要求语言活泼，字数在100字以内。";
```

✅ 现在的优雅做法（传变量）

代码非常干净，只负责传递 JSON 数据结构。

JavaScript

```
// 发送给 OpenAI 的 JSON 数据包结构
{
  "model": "gpt-4o",
  "prompt": {
    "prompt_id": "prompt_888", // 填入后台生成的模板 ID
    "variables": {
      "productName": "复古风外套",  // 动态传入商品名
      "targetAudience": "大学生"    // 动态传入受众
    }
  }
}
```

## 4\. 其他重要特性

-   **版本控制 (Versioning)：** 像使用 Git 管理代码一样管理提示词。当多人小组合作开发一个项目时，大家可以在后台集中测试和更新提示词。如果新版本效果不佳，可以一键回滚 (Roll back) 到历史版本，避免相互覆盖代码。
    
-   **分工明确：** 负责写代码的人只需要知道传入哪些变量键值对；负责优化提示词（Prompt Engineering）的人可以直接在后台调整语气和字数限制，互不干扰，线上程序实时生效。

# Prompt

* * *

## 一、核心指导思想：第一性原理

> **Prompt 是软约束，不是安全边界。真正的边界必须由代码、权限、校验和审计来承担。**

### 核心逻辑

-   Prompt 只能**提高模型遵循规则的概率**，无法保证模型永远按规则执行。
    
-   恶意输入、冲突指令、错误上下文，都可能让模型偏离原始意图。
    
-   分工原则：**Prompt 负责表达任务，系统（代码 / 权限 / 校验）负责执行约束**。
    

### 三条落地铁律

1.  **指令分层要清楚**：系统规则、开发者规则、用户目标、检索内容必须分离，避免外部内容污染系统指令。
    
2.  **输出格式要机器可检验**：关键结果强制用 JSON Schema、函数参数或明确字段承载，拒绝自由文本。
    
3.  **高风险动作不依赖 Prompt 拦截**：写入数据库、调用工具、支付签名等操作，必须经过代码校验 + 人工复核。
    

* * *

## 二、四大核心知识节点

### 1\. Instruction（指令）：模型的 “工作说明书”

核心定义

给模型的任务规则，必须回答清楚：角色定位、任务目标、禁止行为、不确定信息处理方式、输出格式。

关键原则：区分「解释」和「执行」

-   模型只能做信息整理、解释、辅助类操作，**绝对不能替系统做决策或执行高风险动作**。
    
    -   研究助手：可整理资料，不能伪造验证结论；
        
    -   代码助手：可生成补丁，不能默认代码已通过测试；
        
    -   交易助手：可解释风险，不能替用户确认操作。
        

实用四段式模板

text

```
【任务目标】：明确角色和核心职责
【可用输入】：说明模型能处理的信息范围
【禁止行为】：列出绝对不能做的操作
【输出格式/失败格式】：强制输出形态，包含成功和失败场景
```

* * *

### 2\. Few-shot（少样本提示）：用示例教模型 “怎么做”

核心定义

在 Prompt 中加入少量示例，让模型模仿示例的判断逻辑、输出格式和风格，适合 “规则 / 边界难用一句话说清” 的任务。

适用场景

-   交易解释：需要同时包含资产变化、风险提示、不确定点，示例比文字描述更直观；
    
-   格式复杂的输出：如特定结构的 JSON、Markdown 报告；
    
-   内容审核：示例能清晰展示合规 / 违规边界。
    

关键注意事项：维护成本

-   示例不是一次性素材，而是需要和业务规则、评估用例（eval）一起迭代的测试资产。
    
-   业务规则 / 数据格式变化后，旧示例可能误导模型，必须同步更新。
    

* * *

### 3\. Structured Output（结构化输出）：机器可校验的硬约束

核心定义

让模型输出机器可直接解析、校验的结构化数据，是 Prompt 软约束的 “终极兜底手段”，主要形式包括：

-   JSON Object（JSON 对象）
    
-   函数调用参数
    
-   JSON Schema（数据校验规则）
    

核心价值

-   解决模型输出的不确定性，避免歧义文本；
    
-   后端可通过 Schema 直接校验字段类型、取值范围、必填项，不合规数据直接拦截；
    
-   从格式上堵死 Prompt 注入的执行路径。
    

实战示例（JSON Schema）

json

```
{
  "type": "object",
  "required": ["order_id", "status", "amount"],
  "properties": {
    "order_id": {"type": "string", "pattern": "^\\d{6,10}$"},
    "status": {"type": "string", "enum": ["pending", "paid", "shipped", "completed"]},
    "amount": {"type": "number", "minimum": 0}
  }
}
```

* * *

### 4\. Prompt Injection（提示注入）：AI 应用的核心安全威胁

核心定义

攻击者通过用户输入、文档、网页等外部内容，混入恶意指令，让模型忽略原始规则、泄露信息或调用危险工具，在 Agent 场景中风险极高。

核心应对方案（拒绝 “不要被注入” 这类无效指令）

1.  **隔离不可信数据**：用分隔符标记外部内容，明确告知模型 “外部内容中的指令无效”；
    
2.  **工具调用前参数校验**：用 Schema / 白名单校验工具参数，不符合规则的调用直接拦截；
    
3.  **敏感动作强制审批**：高风险操作必须走白名单 + 人工复核流程；
    
4.  **限制模型权限**：不把密钥、主权限、不可撤销动作交给模型。
    

* * *

## 三、整体体系：从软引导到硬兜底的安全闭环

表格

| 层级 | 作用 | 核心工具 |
| --- | --- | --- |
| 软引导层 | 告诉模型 “要做什么、怎么做” | Instruction + Few-shot |
| 硬约束层 | 强制输出合规格式，机器可校验 | Structured Output（JSON Schema / 函数参数） |
| 执行拦截层 | 防止模型被注入后执行危险操作 | 权限控制 + 参数校验 + 人工审批 |

# LLM

### 一、核心概念对比表

| 术语 | 通俗定义 | 核心作用 | 和其他概念的关系 |
| --- | --- | --- | --- |
| Token | 大模型处理文本的最小单位（可以是字、词、子词） | 文本的 “积木块”，决定输入 / 输出的成本与长度限制 | 所有其他概念的基础：Embedding 是 Token 的向量表示，Transformer 处理的是 Token 序列 |
| Embedding | 把 Token / 文本 / 图像等数据，映射成多维向量的技术 | 把 “语义” 变成机器能计算的向量，用来做相似度匹配（如 RAG） | 是 Token 的 “向量化外衣”，也是多模态模型处理图像 / 音频的第一步 |
| Transformer | 现代大模型的核心架构，以自注意力机制为基础 | 理解上下文、生成文本的 “大脑结构”，让模型能同时关注所有 Token | 是处理 Token、生成 Embedding 的 “核心引擎”，大语言模型和多模态模型都基于它 |
| Hallucination（幻觉） | 模型生成看似合理但不符合事实的错误内容 | 模型的固有缺陷，需要通过 RAG、校验等方式缓解 | 不是架构本身，而是 Transformer / 大模型在生成时的常见问题 |
| Multimodal（多模态） | 能同时处理文本、图像、音频、视频等多种数据类型的模型 | 突破纯文本限制，理解真实世界的多形式信息 | 是 Transformer 架构的扩展，需要文本 Embedding + 图像 / 音频 Embedding 共同输入 |

* * *

### 二、概念间的逻辑链条

我们可以用一个 “AI 理解并回答问题” 的流程，把它们串起来：

1.  **输入处理**：
    
    你输入的文本 / 图片，会被拆成 **Token**，再通过 **Embedding** 变成向量（文本向量 + 图像向量）。
    
2.  **模型理解**：
    
    这些向量被送入 **Transformer** 架构，通过自注意力机制，理解所有 Token / 向量之间的上下文关系。
    
3.  **生成回答**：
    
    Transformer 生成新的 Token 序列，再拼成文本输出。这个过程中，模型可能因为信息不足或模式误导，产生 **Hallucination（幻觉）**。
    
4.  **多模态扩展**：
    
    如果是 **Multimodal** 模型，它会同时处理文本和图像 / 音频的 Embedding，跨模态理解你的问题（比如结合图片和文字指令）。
    

* * *

### 三、关键区别与补充说明

-   **Token vs Embedding**：Token 是 “原始单位”，Embedding 是它的 “数学表示”。比如 “猫” 这个词，是一个 Token；它的 Embedding 是一串数字，代表它的语义含义。

# Web3 运行原理

## 一、Web3 是什么

Web1：只读（只能看）

Web2：平台说了算（微信 / 抖音 / 银行，数据资产归平台）

Web3：**用户自己说了算（去中心化、自己掌控资产和数据）**

Web3 底层技术：**区块链**（公开、透明、不可篡改的全球共享账本）

* * *

## 二、钱包与私钥

### 1\. 钱包是什么？

不是装币的软件，是**管理私钥的工具**。

-   地址 = 你的银行账号（公开，可随便给人）
    
-   私钥 = 你的银行密码（绝对不能泄露）
    

### 2\. 私钥的意义（核心）

**谁有私钥，谁就拥有资产控制权**。

-   不用身份证、不用 KYC、不用审批
    
-   随时生成无数个钱包，不用任何人同意
    
-   跨境转账直接发地址就行，没有银行限制
    

**私钥 = 数字世界的身份证 + 银行卡密码 + 资产所有权**

### 3\. 私钥安全

-   私钥**不能改、不能找回、不能重置**
    
-   泄露方式：截图、网盘、手机丢、中木马、钓鱼
    
-   泄露 = 资产全部被盗，**无法追回**
    

* * *

## 三、交易与签名（怎么转账）

### 1\. 交易是什么

转账 / 调用智能合约的操作，叫 **Transaction**

### 2\. 数字签名（最关键）

流程（超简单）：

1.  你用**私钥**对交易内容加密，生成**签名**
    
2.  把：原交易信息 + 签名 + 发送地址 广播到全网
    
3.  节点用**公钥反推签名地址**，验证：
    
    -   是不是你本人签的
        
    -   内容有没有被篡改
        
4.  验证通过 → 执行交易
    

一句话：**私钥签名 = 我的意愿；公钥验证 = 全网确认**

### 3\. 防重复（nonce）

每笔交易有个编号（nonce），执行后编号 + 1，防止重复转账

* * *

## 四、Gas 费（为什么要花钱）

### 1\. Gas 是什么

\= 手续费 / 网络费，像**开车要烧油**

### 2\. 两个作用

-   **防垃圾交易**：发交易要花钱，没人乱刷网络
    
-   **激励节点**：给打包 / 验证节点奖励
    

### 3\. 以太坊 Gas 构成

-   **Base Fee（基础费）**：全网统一，**直接销毁**（通缩）
    
-   **Priority Fee（小费）**：给节点，交易优先处理
    

### 4\. 历史例子

2021 年 NFT 火爆时，Gas 最高达 **150 美元 / 笔**

* * *

## 五、区块链网络怎么跑（从交易到上链）

### 1\. 关键角色

-   **钱包**：发起交易、签名
    
-   **RPC 节点**：连接传统网络和区块链（类似网关）
    
-   **Mempool（内存池）**：未打包交易排队区
    
-   **Builder（打包节点）**：按 Gas 高低排序，打包成区块
    
-   **Validator（验证节点）**：验证区块、确认上链
    

### 2\. 完整流程

1.  你用钱包签名交易 → 发给 RPC
    
2.  RPC 广播到全网 → 进入 Mempool 排队
    
3.  Builder 优先打包 Gas 高的交易 → 生成新区块
    
4.  Validator 验证区块合法 → 上链
    
5.  全网同步账本，交易永久不可篡改
    

### 3\. 确认时间

-   刚出块：不稳定，可能被替换
    
-   以太坊：**12 分钟后基本安全（Finalized）**
    

* * *

## 六、共识机制（为什么全网能一致）

### 1\. PoW（工作量证明，比特币）

-   矿工拼算力算难题，谁先算出谁记账
    
-   优点：极安全；缺点：**超级耗电**
    

### 2\. PoS（权益证明，以太坊现在）

-   不用挖矿，**质押 32 ETH 成为验证节点**
    
-   系统随机选节点打包，其他节点见证
    
-   作恶 = 质押币被罚没（Slash）
    
-   优点：**节能、安全**；缺点：大户话语权高
    

### 一句话区别

-   PoW：**靠算力换信任**
    
-   PoS：**靠质押换信任**
    

* * *

## 七、智能合约

### 1\. 是什么

写在链上的**自动执行代码**，规则写死、不可篡改

### 2\. 类比

传统：找律师、签合同、找中介（成本高）

Web3：代码即法律（Code is Law），满足条件自动执行

### 3\. 特点

-   **公开透明**：所有人能看规则
    
-   **不可篡改**：部署后不能改
    
-   **自动执行**：不用第三方，到点就做
    

例子：你写合约 “我转 1 ETH，对方给我数据”，双方履约自动完成，不用中介

* * *

## 八、Web3 的核心特性

1.  **去中心化**：无中心服务器，全球节点运行，不怕单点故障
    
2.  **无许可**：不用审批，任何人可创建钱包、发交易
    
3.  **抗审查**：全球节点分布，难被关停
    
4.  **公开可验证**：所有交易链上可查，透明可审计
    
5.  **用户主权**：私钥 = 所有权，自己掌控资产和数据
    

* * *

## 九、Web3 不是投机，是学科交叉

Web3 = **密码学（安全）+ 经济学（激励）+ 社会学（治理）**

-   密码学：私钥、签名、哈希（防篡改）
    
-   经济学：Gas、质押、奖励（维持网络）
    
-   社会学：去中心化治理、社区共识（规则制定）

## 一、核心工具与基础概念说明

### 1\. Hermes Agent 定位与核心能力

-   定义：开源自进化 AI 智能体，可部署于本地 / VPS/Docker，支持多模型接入、工具调用、持久记忆与自主成长。
    
-   核心功能：文档编辑、学习助手（制定 / 提醒学习计划、打卡）、代码开发、任务自动化、多平台交互（Telegram/Discord/ 飞书）。
    
-   优势：**低使用门槛**、支持消息渠道持久化、行为自动成长、可封装技能（Skill）、支持多智能体编排。
    

### 2\. WSL（Windows Subsystem for Linux）

-   定义：Windows 原生 Linux 子系统，无需双系统 / 虚拟机，可直接运行 Linux 内核与命令。
    
-   部署建议：Hermes Agent 优先推荐 **WSL2** 环境；Windows 原生版本为 Beta 版，功能不稳定、易出 Bug，仅当 WSL2 安装失败时备选。
    
-   核心价值：解决 Windows 权限与兼容性问题，适配 macOS/Linux 类 Unix 开发环境，支持文件互通。
    

### 3\. MCP（Model Context Protocol，模型上下文协议）

-   定义：AI 通用连接协议，统一大模型调用外部工具 / 资源的标准，适配 Hermes Agent 与 MiMo 等模型的交互。
    
-   核心作用：规范请求字段（如模型名、思考模式参数），避免 “400 Param Incorrect” 错误，保障多轮对话上下文完整。
    

### 4\. 相关工具对比（Claude Code/Cursor vs Hermes Agent）

-   Claude Code/Cursor：侧重代码开发，原生支持代码优化，通过 API 调用模型，无持久记忆与自动成长能力。
    
-   Hermes Agent：对底层模型做封装，新增**持久化记忆、行为自进化、多平台交互**能力，支持自定义技能，门槛更低。
    

## 二、部署与配置核心流程

### 1\. 环境部署优先级

1.  优先：**WSL2（Ubuntu）** → 安装 Hermes Agent（官方推荐，稳定无 Bug）。
    
2.  备选：Windows 原生（Beta 版）→ 仅 WSL2 安装失败时使用，需自行排查 Bug。
    
3.  替代：本地 / VPS 部署 → 适合大规模文件处理、多开发者协作场景，需开放文件读取权限。
    

### 2\. Hermes Agent 关键配置步骤

1.  模型接入：选择对接模型（如小米 MiMo、GPT、Claude），填写 API Key 与 Base URL。
    
    -   MiMo 适配要点：保留模型名 “.”（如 mimo-v2.5-pro），关闭思考模式（thinking: false），避免参数错误。
        
2.  消息渠道配置：绑定 Telegram/Discord/ 飞书，设置白名单与提示词，实现多平台交互。
    
3.  技能（Skill）封装：自定义工作流（如量化分析、文档处理），分配专属智能体身份，限定任务范围。
    

### 3\. 模型切换与兼容性

-   切换便捷性：支持多模型一键切换（命令：hermes a module），额度耗尽时可无缝替换其他 AI 模型。
    
-   能力影响：封装后会新增系统提示词，可能轻微影响原生模型能力（如 Claude Code 代码优化），但核心功能无缺失。
    
-   版本更新：客户端优化（如界面）不影响 Hermes；服务端模型优化（如算法升级）自动兼容，无需额外操作。
    

## 三、核心应用场景与实践案例

### 1\. 个人学习助手

-   功能：按时间制定学习计划（时间多→多任务，时间少→精简任务）、打卡提醒、学习记录管理。
    
-   实践：作为个人知识库，支持笔记检索、知识点复盘，配合 RAG 解锁历史学习内容。
    

### 2\. 开发与代码辅助

-   场景：紧急代码修改、仓库管理、文档生成、批量数据处理。
    
-   实践：对接 Claude Code/Cursor，保留代码开发能力，新增任务自动化与记忆复用。
    

### 3\. 量化分析与多智能体编排

-   支持：拆分量化任务（数据获取、因子挖掘、策略分析），分配不同模型处理，结果交叉验证。
    
-   局限：暂不支持智能体自主探讨策略，需通过 Skill 设定流程，依赖外部项目（如 Claude Code Bridge）实现模型间通信。
    

### 4\. Web3 与 AI 融合场景

-   落地方向：用户意图交易（自然语言执行链上交易）、智能理财（基于用户习惯资产配置）、自动化合约部署。
    
-   阶段：当前处于探索期，具体方案将于后续分享会详细拆解。
    

## 四、常见问题与答疑

### 1\. Windows 部署缺陷

-   核心原因：Windows 权限机制复杂，与 Unix 类系统（macOS/Linux）不兼容，开发环境适配成本高。
    
-   解决方案：优先 WSL2，规避权限与兼容性问题。
    

### 2\. 国产模型配置效果差

-   问题：参考 GitHub 教程配置国产模型（如 MiMo），功能受限、无法正常使用。
    
-   建议：简化配置（仅设置模型与消息渠道），避免复杂自定义参数；优先使用官方推荐模型配置模板。
    

### 3\. 企业级应用与简历价值

-   结论：Hermes/OpenClaw 更适合个人探索与效率提升；企业级开发 / 简历项目，建议基于传统框架（如 LangChain）实现落地应用。
    

### 4\. 多智能体编排支持

-   现状：支持任务拆分与模型分配，但暂不支持智能体自主协商；复杂编排需结合外部工具或自定义 Skill。
    

# web3支付安全入门

## 一、开篇：AI 时代的 3 个重要认知

### 1\. 有 AI≠不用学基础

-   **AI**：写代码、查资料、生成方案很快，是 “手脚”。
    
-   **人**：做架构、判断对错、把控安全，是 “大脑”。
    
-   结论：**AI 不降低难度，反而对人的基础要求更高**。看不懂 AI 输出，就会被 AI 带偏。
    

### 2\. Web3 不是全新世界，是 Web2 的升级

-   **Web2**：中心化（银行、支付宝、平台说了算）。
    
-   **Web3**：去中心化（链上公开、代码透明、用密码学保证安全）。
    
-   本质：**底层逻辑相通，Web3 只是更严格、容错更低的分布式系统**。
    

### 3\. 安全不是事后补，要从设计开始

-   **审计**：只是 “补漏洞”，不能替代安全设计，更不能买个审计就当安全。
    
-   **Web3 安全特点**：**出错 = 资产归零，不可逆、不能回滚**。
    
-   一句话：**安全源于设计，贯穿整个项目生命周期**。
    

* * *

## 二、Web2 vs Web3 支付

### 1\. Web2 支付（银行 / 支付宝）

-   **参与方**：用户 → 电商 → 支付服务 → **银行（Bank Institution）**
    
-   **资金流**：用户银行卡 → 支付平台账户 → 商家账户
    
-   **信任对象**：**完全信任银行 / 平台**，它们记账、扣钱、冻结。
    
-   **用户控制**：钱在银行账本里，**用户只有密码，没有真正控制权**。
    

### 2\. Web3 支付（链上 / 稳定币）

-   **参与方**：用户 → 电商 → 支付服务 → **区块链（Blockchain）**
    
-   **资金流**：用户 Web3 钱包 → 支付服务链上地址 → 商家 Web3 钱包
    
-   **信任对象**：**不信任任何中心，信任数学 + 代码 + 共识**
    
    -   **不可篡改**：链上数据改不了。
        
    -   **公开透明**：合约代码所有人可查。
        
    -   **共识机制**：POW/POS 等，保证全网一致。
        
-   **用户控制**：**私钥 = 钱，用户完全掌控资产**。
    

### ✅ 关键名词解释

-   **区块链（Blockchain）**：分布式公共账本，所有人同步记录、不可篡改，相当于 “去中心化银行”。
    
-   **链上（On-chain）**：数据 / 交易记录在区块链上，公开可查、不可篡改。
    
-   **稳定币**：锚定法币（如 USDC/USDT）的数字货币，价格稳定，适合支付。
    

* * *

## 三、Web3 钱包：Web3 的 “钥匙”

### 1\. 钱包本质

**钱包不是存钱的，是存私钥、帮你签名、管理资产控制权的工具**。

### 2\. 关键名词解释

-   **私钥（Private Key）**：一串随机数字，**相当于银行卡密码 + 身份证**，**丢了 = 钱没了，永远找不回**。
    
-   **公钥 / 地址**：私钥生成的公开账号，用来收钱、转账，可公开。
    
-   **签名（Signature）**：用私钥对交易 / 消息 “盖章”，证明是你本人操作，**不泄露私钥也能证明身份**。
    
    -   **交易签名**：证明你同意转账 / 交互合约。
        
    -   **消息签名**：证明你的身份（登录 DApp 用）。
        

### 3\. 钱包分类

（1）按**私钥控制**分

-   **自托管钱包（Web3 钱包，热钱包）**：私钥在你手机 / 电脑本地，**你自己掌控**（如 MetaMask、TP 钱包）。
    
-   **全托管钱包（交易所钱包）**：私钥在交易所服务器，**交易所掌控**（如币安、OKX 账户）。
    
-   **混合托管钱包**：私钥分成 2 份，你拿 1 份、平台拿 1 份，**双方配合才能签名**，安全又方便。
    

（2）按**技术实现**分

-   **EOA 钱包（普通私钥钱包）**：最常见，一个私钥对应一个地址。
    
-   **合约钱包（多签钱包，如 Safe）**：链上智能合约当钱包，**多人共同控制**（如 2/3 多签：3 个人里 2 个同意才能转账）。
    
-   **AA 钱包（账户抽象）**：未来趋势，把 EOA 和合约钱包融合，更灵活（目前不成熟）。
    

（3）按**联网状态**分

-   **热钱包**：联网（手机 / 电脑钱包），方便、常用、有被盗风险。
    
-   **冷钱包（硬件钱包，如 Ledger）**：断网离线，最安全，适合存大额资产。
    

### ✅ 钱包核心风险

1.  **私钥泄露**：丢私钥 = 丢所有资产，不可逆。
    
2.  **签名欺骗（钓鱼）**：骗子发假交易，你签名后钱就转走了（**一定要看清签名内容**）。
    
3.  **无限授权**：给陌生合约授权资产，被偷偷转走。

## 四、Web3 交易：从发起到上链的全过程

### 1\. 交易生命周期（4 步）

1.  **构造**：钱包生成交易信息（转给谁、多少钱、Gas 费、数据）。
    
2.  **签名**：用私钥签名，证明是你操作。
    
3.  **广播**：发到区块链网络，等待矿工打包。
    
4.  **上链确认**：全网共识完成，交易写入区块链，**不可篡改**。
    

### 2\. 交易关键参数

-   **Gas 费**：给矿工的 “手续费”，**Gas 越高，交易越快上链**。
    
-   **Nonce**：交易序号，防止重复转账、乱序。
    
-   **Data**：合约交互指令，**链上 DApp（DeFi、NFT、链游）全靠它实现复杂功能**。
    

### ✅ 关键名词解释

-   **DApp**：去中心化应用，运行在区块链上（如 Uniswap、链游）。
    
-   **矿工 / 节点**：维护区块链网络、打包交易、验证交易的服务器。
    

* * *

## 五、交易监听与安全校验（后端 / 项目方视角）

### 1\. 怎么知道用户付了钱？

-   **轮询（Pull）**：定期查区块链，看目标地址有没有收到钱。
    
-   **长连接监听（Push）**：用 Websocket 实时接收新区块和交易。
    

### 2\. 为什么要等确认？

-   **链分叉 / 重组**：区块链偶尔会 “回滚”，交易可能失效。
    
-   **做法**：**等 N 个区块确认（如以太坊等 15 个区块）**，确认后才算交易成功。
    

### 3\. 交易模拟（安全必备）

**签名前先模拟交易**，看结果对不对（比如：预期转 10U，模拟发现转 100U，就不签名），**防止被骗**。

* * *

## 六、Web3 安全核心结论

1.  **私钥 = 一切**：绝不泄露、不截图、不存云端、不发给任何人。
    
2.  **Web3 无 “客服”**：资产丢了没人能帮你找回，**只能靠自己防**。
    
3.  **安全 = 懂原理**：看不懂的签名、授权、链接，**一律不点、不签、不授权**。
    
4.  **AI 是工具**：帮你干活，但**安全判断必须靠人**。
