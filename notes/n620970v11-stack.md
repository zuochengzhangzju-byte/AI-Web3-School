# NFT 是什麼？從「非同質化」到鏈上所有權的入門概覽

## 摘要

NFT（Non-Fungible Token，非同質化代幣）在 2021 年前後引發了一波社會熱潮，但很多人對它的理解停留在「天價 JPEG」的印象。拋開炒作，NFT 本質上是一種用區塊鏈技術實現的「獨特性證明機制」，理解它的底層運作，才能判斷它的價值和局限在哪裡。

## 核心概念

### 同質化 vs 非同質化

理解 NFT 的起點，是先搞清楚「同質化（Fungible）」是什麼意思。

同質化代幣，就是每一顆都可以互換、等值的代幣——100 元換 100 元沒有差別，ETH 換 ETH 也沒有差別。以太坊上最常見的代幣標準 ERC-20，發行的就是這類同質化代幣。

非同質化代幣則相反：每一個都有獨立的身份識別碼（Token ID），兩兩之間不能直接置換，也不可以被分割成更小的單位交易。現實世界的類比是房屋產權、出生證明、或一幅畫——你的不等於我的，也無法切一半給別人。

### ERC-721：NFT 的標準協議

NFT 在以太坊上的實現，靠的是 ERC-721 這個代幣標準。它在 2018 年 1 月正式發布，有趣的是，它的共同提案人之一 Dieter Shirley，同時也是第一個大規模使用 NFT 的應用 CryptoKitties（加密貓）的技術總監。

ERC-721 的核心設計很簡單：每個 NFT 都有一個 `uint256` 的 Token ID，在同一份合約裡全域唯一。合約負責記錄「誰擁有哪個 Token ID」，也提供轉移所有權的標準函數。從外部看，NFT 的所有權轉移，本質上就是在智能合約的 mapping 裡改了一行紀錄`tokenId → 新地址`。

另一個值得知道的標準是 ERC-1155，它更有彈性，允許一份合約同時管理同質化和非同質化代幣，還支援批量轉移（一次交易可以移動多種代幣），特別適合遊戲道具這類應用場景。

### NFT 和圖片的關係：鏈上存的不是圖

這是很多人誤解的地方。絕大多數 NFT 並不把圖片本身存在區塊鏈上——鏈上空間太貴，根本存不起。

鏈上存的其實是一個指向 metadata 的 URI（可以理解成一個網址），metadata 是一份 JSON 檔，記錄這個 NFT 的名稱、描述和圖片連結，圖片則通常存放在 IPFS（去中心化儲存網路）或一般的中心化伺服器上。

這衍生出一個重要的風險：如果圖片放在中心化伺服器，伺服器關掉，NFT 對應的圖就消失了——鏈上記錄還在，但那個 URI 已經指向一片空白。這也是為什麼「鏈上 NFT（on-chain NFT）」特別被部分社群重視，因為資料完全存在鏈上，不依賴任何外部服務。

### NFT 能做什麼？

NFT 最直接的用途是「數位所有權證明」，但它的應用場景比藝術收藏更廣：

\- **數位藝術與收藏品**：BAYC（無聊猿）、CryptoPunks 是最知名的例子

\- **遊戲道具**：角色、裝備可以真正「屬於」玩家，可以跨平台交易

\- **門票與會員資格**：持有 NFT 即代表擁有特定活動或社群的入場資格

\- **創作者版稅**：智能合約可以設定每次二手交易時，原創作者自動收取一定比例的版稅，不需要任何中間人介入

## 觀察與反思

NFT 熱潮退去之後，很多人對它的評價變得極端負面，但我覺得把「炒作泡沫」和「底層機制」混為一談是不公平的。ERC-721 作為一種「唯一性證明」的技術，本身的設計邏輯是清晰的。問題在於，圖片不存在鏈上這件事讓「所有權」的意義大打折扣，加上早期的 NFT 市場大量充斥著投機行為，讓它的信譽受損。

真正讓我感興趣的問題是：當 NFT 被用來代表「真實世界資產（RWA）」時，鏈上的所有權紀錄和法律上的所有權如何對應？技術上可以轉移，但法律上承認嗎？這個邊界目前還很模糊。

## 延伸思考

\- IPFS 是怎麼運作的？它和一般的中心化伺服器相比，穩定性和去中心化程度如何？

\- ERC-721 和 ERC-1155 在實際開發上要怎麼選擇？

\- NFT 的版稅機制在不同交易平台之間為什麼無法統一強制執行？

# L1、L2、L3 是什麼？讀懂區塊鏈的分層邏輯

## **L1、L2、L3 是什麼？讀懂區塊鏈的分層邏輯**

在 Web3 的語境裡，「Layer」這個詞出現的頻率非常高，L1、L2、L3 各有各的定位。這些分層不是憑空設計的，而是為了解決一個根本的矛盾：區塊鏈不可能同時做到去中心化、安全、又快速便宜。搞懂為什麼要分層，才能理解每一層在做什麼。

## 核心概念

### 一切的起點：區塊鏈不可能三角

Vitalik Buterin 提出過一個概念，叫「區塊鏈三難困境（Blockchain Trilemma）」：去中心化（Decentralization）、安全性（Security）、可擴展性（Scalability）這三件事，以現有技術很難同時做到。最多只能在其中兩項做好，第三項必然要妥協。

以太坊主網選擇了去中心化和安全性，代價是速度慢、Gas 費高。這就是為什麼 Layer 2 會出現——它不是要取代以太坊，而是在旁邊幫它分擔計算壓力。

### Layer 1：地基

L1 就是區塊鏈本身——比特幣、以太坊、Solana 都是 L1。它負責最核心的事：交易驗證、共識機制、資料儲存、網路安全。所有的規則都在這裡定義，也是整個系統安全性的根基。

L1 的問題是吞吐量有限。以太坊主網大概每秒能處理 15–30 筆交易，遇到熱門項目上線或市場劇烈波動，就會堵塞、Gas 費暴漲。這不是以太坊「做錯了什麼」，而是它在設計上選擇了去中心化和安全性之後，必然面對的取捨。

### Layer 2：在主鏈旁邊跑

L2 的核心思路是：把大量的計算工作移到主鏈之外進行，只把最後結果（或證明）送回 L1 確認。這樣既可以大幅提升速度和降低費用，又能繼承 L1 的安全性。

目前以太坊生態最主流的 L2 技術叫 **Rollup**，又分兩種路線：

**Optimistic Rollup**（樂觀派）：預設交易都是合法的，先打包上鏈，若有人發現問題可以在挑戰期內提出「欺詐證明」。代表專案是 Arbitrum 和 Optimism。優點是對 EVM 相容性好、開發者遷移容易，缺點是從 L2 提款回 L1 要等挑戰期結束，通常需要約 7 天。

**ZK Rollup**（零知識派）：每批交易都附上一個數學證明（零知識證明），L1 可以立即驗證正確性。代表專案是 zkSync 和 StarkNet。優點是提款速度快、理論上安全性更高，缺點是技術複雜度高，對 EVM 的完整相容花了更長時間才追上。

兩者各有取捨，目前 Optimistic Rollup 生態更成熟、TVL 更高，ZK Rollup 則被許多人認為是更長遠的方向。

### Layer 3：更上面一層

L3 的概念比較新，也還在演化中。簡單說，L3 是建立在 L2 之上的再一層，目標是為特定應用或垂直場景做更深度的客製化——比如專門為遊戲、隱私計算、或跨鏈互操作性設計的鏈。

Arbitrum 的 Orbit 框架就允許開發者在 Arbitrum（L2）之上再建自己的 L3 鏈。這樣的設計讓不同的應用可以有自己的規則和費用結構，而不需要全部擠在同一條鏈上競爭資源。

Vitalik 自己也寫過文章討論 L3 的合理使用場景——他認為 L3 的價值不在於把 Rollup 再堆一層做通用擴容，而是在於提供「客製化功能」，例如更強的隱私保護，或是特定應用專屬的執行環境。

## 觀察與反思

我以前覺得 Layer 只是個行銷術語，看到 L2、L3 就覺得是在炒概念。真正理解之後才發現，這些分層其實是在認真解決一個工程上的根本矛盾——如果 L1 什麼都做，它就快不起來；如果什麼都往外推，又失去了安全保障。Rollup 的設計讓我覺得聰明的點在於：它沒有放棄 L1 的安全性，只是把計算搬出去，把「證明」留下來。

不太確定的地方是：L2 的安全性到底繼承了多少 L1？聽說側鏈（如早期的 Polygon PoS）並不算真正的 L2，因為它沒有把資料提交回以太坊。這條邊界想再研究清楚。

# EVM 是什麼？以太坊虛擬機與 Gas 機制的運作邏輯

## 摘要

以太坊之所以能執行智能合約，靠的是一個叫做 EVM（以太坊虛擬機）的執行環境。所有節點都跑著同一台 EVM，確保相同的交易在任何地方都得到相同的結果。而 Gas 機制則是讓這台虛擬機能持續運作、不被濫用的關鍵設計。

## 核心概念

### EVM 是什麼？

EVM（Ethereum Virtual Machine）是以太坊的智能合約執行引擎。開發者用 Solidity 寫的程式碼，會先被編譯成 EVM 位元組碼（bytecode），再部署到鏈上。每次有人呼叫合約，節點就啟動 EVM 去執行這段位元組碼，並把結果寫入區塊鏈。 EVM 在設計上是個沙盒（sandbox）：它和底層作業系統完全隔離，合約程式碼無法存取檔案系統、網路，也無法直接呼叫外部服務，只能和其他合約互動。這個設計讓 EVM 具備高度的可預測性——相同的輸入，一定產生相同的輸出，這也是去中心化共識能成立的前提。 EVM 是圖靈完備的，理論上可以執行任何可計算的邏輯。但之所以說「準圖靈完備（quasi-Turing complete）」，是因為 Gas 機制給計算加了一個上限，讓真正的無限迴圈在實際上不可能執行完。

### **Gas 機制：計算的油錢**

Gas 是以太坊用來衡量計算量的單位。每一條 EVM 指令都有固定的 Gas 消耗，執行越複雜的邏輯，消耗的 Gas 越多。 交易發送者在送出交易時，需要指定：

-   **Gas Limit**：這筆交易最多願意消耗多少 Gas
    
-   **Gas Price**：每單位 Gas 願意支付多少 ETH（EIP-1559 後拆成 base fee + priority fee） 實際費用 = Gas Used × Gas Price。如果執行到一半 Gas 耗盡，交易被強制終止，狀態回滾（當作沒發生），但已消耗的 Gas 不退還，因為節點已經做了計算工作。如果 Gas 有剩，多的會退回給發送者。 這個設計解決了一個根本問題：在一個公開的去中心化網路上，如果執行計算不要錢，惡意的人只要提交一個無窮迴圈合約就能讓全網癱瘓。Gas 讓每一次計算都有成本，讓這類攻擊在經濟上不合算。
    

### **EVM 和 JVM 的比較**

偶爾會看到有人拿 EVM 和 Java 的 JVM 比較。兩者確實有相似之處：都是把高階語言編譯成位元組碼，再在虛擬機上跑，目的都是讓程式碼跨平台執行。但 EVM 和一般的虛擬機（如 VirtualBox）是完全不同的東西——一般虛擬機是為了模擬整個作業系統，EVM 的目標只有一個：在去中心化環境下確定性地執行智能合約。

## **觀察與反思**

Gas 機制這個設計讓我覺得很聰明——它同時解決了「防止無限迴圈」和「補償節點運算成本」兩個問題，而且用市場機制（Gas Price 競價）來調節網路壅塞，不需要任何中心化的管理者介入。 目前還不太清楚的地方：EIP-1559 把 base fee 燃燒掉而不是給礦工，這對以太坊的通膨/通縮模型有什麼影響？另外，為什麼某些操作的 Gas 消耗特別高（比如寫入儲存），背後的設計邏輯是什麼?

# **MetaMask 彈出的簽名視窗，背後是什麼標準？EIP-191 與 EIP-712**

## **摘要**

用 MetaMask 操作 DApp 時，常常會跳出一個簽名請求——有時是一串看不懂的文字，有時又顯示得很清楚。這個差異其實來自不同的簽名標準：EIP-191 和 EIP-712。搞懂它們，也順帶搞懂了「重放攻擊」是怎麼一回事。

## **核心概念**

### **為什麼需要簽名標準？**

早期以太坊的訊息簽名沒有統一規範，有個安全問題：一段簽名資料，在格式上有可能被當成合法的以太坊交易來廣播。為了把「簽名一般訊息」和「簽名交易」明確區分開來，EIP-191 的做法是在訊息前面強制加上一個 `0x19` 前綴位元組。因為正常的 RLP 編碼交易不會以 `0x19` 開頭，這樣就能保證兩者永遠不會混淆。 `personal_sign` 就是 EIP-191 的其中一個版本（version `0x45`），也是最常見的形式。它在訊息前面自動加上 `"\x19Ethereum Signed Message:\n"` 加上訊息長度，再做雜湊簽名。

### **EIP-712：讓簽名內容可讀**

`personal_sign` 有個問題：使用者在 MetaMask 裡看到的只是一串雜湊或難讀的位元組，根本不知道自己在簽什麼，容易被釣魚攻擊。 EIP-712 改進了這點。它要求每個欄位都明確標示類型和值（例如 `to: addressamount: uint256`），MetaMask 就能把內容用人讀得懂的格式顯示出來。另外，EIP-712 還要求帶上一個「domain」結構，裡面包含應用名稱、版本、chain ID、合約地址等資訊。這樣做的用意是：同一份簽名，無法被拿去另一個不同應用或不同鏈上重複使用，也就解決了「重放攻擊」的問題。

### **重放攻擊是什麼？**

重放攻擊（replay attack）指的是：攻擊者攔截到一個合法的簽名，然後在另一個場景裡重新廣播它，讓它再次生效。最簡單的例子是：如果你在某個合約上簽了「授權轉帳 100 USDC」，而簽名裡沒有綁定合約地址和鏈資訊，攻擊者理論上可以把同樣的簽名拿去另一條鏈或另一個合約上重播。EIP-712 的 domain 設計就是為了堵住這個漏洞。

### **EIP-191 的三個版本**

EIP-191 定義了三種格式，以 version byte 區分`0x00` 用於指定目標驗證合約的場景`0x01` 就是 EIP-712 的結構化資料`0x45`（對應字母 `E`）則是 `personal_sign` 的常規訊息。

## **觀察與反思**

我之前每次在 MetaMask 按「簽名」都沒有仔細看內容，現在才意識到舊式的顯示介面真的是資安黑洞。EIP-712 的設計思路讓我印象深刻——透過讓人「看得懂」來增加安全性，比起純技術手段更直接。 不過我還沒搞清楚`ecrecover` 在合約裡驗證 EIP-712 簽名時，domain 的驗證是誰負責的？是合約自己去重算然後比對嗎？

# 以太坊的簽名？ECDSA 原理入門

## 摘要

在以太坊裡，每一筆交易都需要「簽名」才能被網路接受，而這背後靠的是一套叫做 ECDSA（橢圓曲線數位簽章演算法）的密碼學機制。搞懂它，才算真正理解「私鑰控制帳戶」這件事的底層邏輯。

## 核心概念

### 私鑰、公鑰、地址的關係

以太坊帳戶的起點是一把 256 位元的私鑰，本質上就是一個隨機大數。從私鑰出發，透過 ECDSA 使用的橢圓曲線（secp256k1）可以推導出公鑰；公鑰再經過 Keccak-256 雜湊，取最後 20 個位元組，就得到我們熟悉的錢包地址。 這條路只能單向走：私鑰 → 公鑰 → 地址。要從地址逆推回私鑰，以現有的計算能力幾乎不可能。這種「只能往一個方向走」的函數，密碼學上叫做「陷門函數（trapdoor function）」，是整個系統安全性的基礎。

### ECDSA 的簽名過程

所謂「簽名」，並不是把私鑰本身附在交易上，而是用私鑰對交易資料的雜湊值進行一次數學運算，產生兩個數字`r` 和 `s`（加上一個輔助的 `v` 值，用來方便後續恢復公鑰），合稱 `(v, r, s)`，這就是簽名的本體。 這個過程的精妙之處在於：簽名結果可以讓任何人用公鑰驗證「這筆交易確實由私鑰持有者授權」，但整個過程完全不需要揭露私鑰。以太坊甚至還有一個 `ecrecover` 函數，可以直接從簽名反推出簽名者的地址，這是鏈上驗證身份的常用技巧。

### 為什麼用 secp256k1？

secp256k1 是比特幣選用的橢圓曲線參數，以太坊沿用了它。這條曲線是一種 Koblitz 曲線，計算效率高，在大規模使用場景下有實際優勢。安全性方面，它的 256 位元密鑰長度，讓破解所需的計算量大到幾乎不可行（理論上要解「橢圓曲線離散對數問題」）。 值得一提的是，ECDSA 純粹是一個簽名演算法，不能拿來加密。這點和 RSA 不同——以太坊帳戶系統不做加密，只做「我是我」的身份證明。

## 觀察與反思

我一開始以為「簽名」就是某種密碼或密文，讀完之後才明白它其實是一組數學計算的結果。比較讓我困惑的地方是 `v` 這個值，資料說它是為了恢復公鑰用的，但具體怎麼用還沒有完全看懂，之後要再查。 另外，secp256k1 這個名字拆開來看倒是很有趣：sec 是標準組織縮寫、p 是素數域、256 是位元長度、k 是 Koblitz 曲線、1 是第一條。記這種命名邏輯比死背要好記多了。

# 密碼學：Web3 的信任地基

## 一、雜湊函數（Hash Function）

-   輸入改一個字，輸出完全不同（雪崩效應）
    
-   不可逆：從指紋推不回原始資料
    
-   以太坊用 **Keccak-256**，Bitcoin 用 **SHA-256**
    

### **在 Web3 的用途**：

-   每個區塊包含上一個區塊的雜湊 → 改一筆資料就會讓後面所有區塊的雜湊全部失效
    
-   錢包地址就是從公鑰雜湊出來的
    

## 二、非對稱加密（Asymmetric Cryptography）

-   用**私鑰簽名**，用**公鑰驗證**
    
-   兩者數學上綁定，但從公鑰推不出私鑰
    

### **在 Web3 的用途**：

-   你的錢包地址衍生自公鑰
    
-   發交易時用私鑰簽名，任何人都能用你的公鑰驗證「確實是你」
    
-   區塊鏈永遠不需要你的私鑰——它只需要看到你的簽名
    

## 三、橢圓曲線（Elliptic Curve Cryptography）

-   以太坊、Bitcoin 用 **secp256k1**
    
-   Solana、新一代協議多用 **Ed25519**（更快、更短）
    
-   256 bits 的私鑰能產生幾乎無限不重複的公鑰——碰撞機率趨近於零
    

**為什麼選橢圓曲線**：同樣的安全強度，金鑰比 RSA 短很多，適合鏈上的空間與計算限制。

## 四、Merkle Tree

-   改動任何一筆交易，Root Hash 就會變
    
-   **Merkle Proof**：只需提供一條從葉節點到根的路徑，就能證明某筆資料在樹裡——不需要揭露整棵樹
    

### **在 Web3 的用途**：

-   每個區塊的所有交易都壓進一個 Merkle Root
    
-   NFT 白名單（Allowlist）常用 Merkle Proof 驗資格，省 Gas 又安全
    

**學到的概念**：密碼學不是「加解密資料」，而是**建立可驗證的信任**。Web3 裡的「信任」跟傳統世界最大的差別是：任何人都能獨立驗證，不需要問第三方。

**準備採取的行動**：在 Etherscan 上找一筆自己的交易，把 `from` 地址、`input data`、`txHash` 這三個欄位對照今天學到的東西，試著說出每個欄位背後用了哪個密碼學原語。

**想繼續追問的問題**：零知識證明（ZKP）是怎麼做到「能證明我知道某件事，但完全不透露那件事是什麼」？這跟今天學的簽名驗證有什麼根本上的不同？

# 區塊鏈與私鑰錢包：信任是怎麼來的

## 摘要

學 Web3 卡關的第一道牆，是搞懂「為什麼這套東西可以信任」。這篇整理我對區塊鏈本質和錢包機制的理解 —— 信任不是來自某個機構背書，而是來自演算法、密碼學、跟一群人共同維護的紀錄。

## 區塊鏈：一群人一起記帳

區塊鏈最直覺的比喻就是**分散式帳本**：一群人各自拿著一本帳本，每隔一段時間把新的交易記下來、彼此核對、確認大家寫的都一樣。一旦寫下去就不能改 —— 因為要改的話，得同時去改所有人的帳本。

### 要成立這件事情，需要包含三個關鍵因素：

-   區塊與雜湊：每個區塊都包含前一個區塊的雜湊值，形成鏈式結構。改其中一筆，後面全部都會崩掉。
    
-   共識機制：PoW、PoS、PBFT 機制不同，但都在解同一個問題 —— 沒有中心化裁判時，怎麼讓陌生人對「事實是什麼」達成一致。
    
-   合約公開透明：規則寫在鏈上，所有人都能審視，不靠機構口頭承諾。
    

## 私鑰：你在鏈上的身分

鏈上沒有帳號密碼，只有**私鑰**。

私鑰本身就是一串隨機產生的數字（在 secp256k1 曲線上是 256 bits），不是用戶自己「想」一個，而是錢包透過隨機數產生器（RNG）幫你生成。特點如下：

-   私鑰**一旦生成就不會變**
    
-   區塊鏈**永遠不會、也不需要**向你索取私鑰
    
-   所有需要證明「你是你」的操作，都靠**用私鑰對訊息做簽名**完成
    

## 觀察與反思

傳統世界的信任靠機構，鏈上的信任靠數學跟程式碼。前者可以打電話客訴，後者只能自己負責。私鑰一旦弄丟或洩漏，沒有任何人能救你 —— 這既是 Web3 的自由，也是它最重的責任。
