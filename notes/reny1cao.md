# 2026-05-26
<!-- DAILY_CHECKIN_2026-05-26_START -->
今天听 Cobo Agentic Wallet 的分享时，我一开始觉得普通钱包和 agent wallet 好像差别没有那么大，都是在完成“付款 / 交易 / 授权”这件事。但往下想之后，发现区别不在最终动作，而在控制权和权限边界。

传统支付里的钱包，比如支付宝或银行卡，本质上是在平台、银行和清结算系统里完成转账，中间有支付机构、风控、客服、冻结、退款和争议处理。Web3 普通钱包不太一样，它更像是地址和私钥的签名入口。钱包本身不是中介，链上交易一旦被有效签名并执行，通常就很难像传统支付那样直接撤回。

所以 agent wallet 的重点也不是简单“给 agent 多开几个小钱包”，而是把人的完整钱包权限拆成更小、更明确、可限制、可过期、可撤销、可审计的权限，让 agent 只能在边界内行动。比如限制最多花多少钱、什么时候过期、能和哪些合约交互、能调用哪些函数、能用哪些资产，超过阈值就必须重新让人确认。

我现在理解，agent 可以自动做的事情不只是准备交易，还可以包括理解用户意图、模拟交易、检查风险、组合多步操作、记录执行结果；但越靠近转账、授权和合约写入，就越不能只依赖“它应该懂我的习惯”。真正可靠的 agent wallet，不是让 agent 拥有更大权限，而是让权限被拆得更细、风险被关在更小范围里、每一步都能被追踪。

这也接上了昨天对 long-term memory 的理解：agent 该记住的是偏好、规则、风险阈值、历史决策和操作证据，而不是私钥、助记词，也不是因为用户过去经常这样做就自动放行。对 AI x Web3 来说，wallet 的未来可能不是“AI 帮我点确认”，而是“AI 在明确授权和可审计边界内帮我执行低风险任务”。

# 2026-05-25
<!-- DAILY_CHECKIN_2026-05-25_START -->
今天对 agent 长期记忆的理解更具体了一点：它不是把聊天记录存得更久，而是让一个持续任务在不同时间点之间不断线。

一个真正有用的 agent，应该知道事情已经推进到哪里，之前为什么这样判断，哪些结论已经形成依据，哪些问题还没解决，下一步应该继续补概念、做实践，还是先检查风险。这样每一天不是重新开始，而是在前一天的基础上自然接上。

放到 Web3 场景里，长期记忆还不能只服务于效率，它必须记住安全边界。比如涉及钱包、签名和合约交互时，要持续记得检查当前网络、接收地址、授权额度、gas、nonce，以及 transaction data 到底在调用什么。agent 可以帮助解释交易、整理风险、保留 tx hash、合约地址、参数和操作前后的状态证据，但不能替人签名，也不能越过人的判断。

所以我现在更倾向于把 long-term memory 理解成三件事：状态系统、流程系统和审计系统。它让 agent 能长期跟进一件事，也让过程、结果和边界都能被追踪。对 AI x Web3 来说，这种记忆不只是让 agent 更聪明，而是让它更可靠、更谨慎，也更适合参与真实协作。

最后比较有感觉的是 milestone escrow 这个场景。它不是一次性交付和付款，而是把任务拆成几个阶段，资金先进入链上托管，worker 接受任务、提交结果，再由 client 或 PM 审核后释放付款。我看了一下本地的 kite-milestone-escrow 项目，里面的 ProjectEscrow 大概就是这样的思路：任务从 Created、Assigned、Accepted、InProgress、Submitted 到 Paid，一步步往前走，中间还有未接受过期、提交后超时自动释放、PM fee 上限这些现实协作里会遇到的规则。

这让我把 ERC-8183 想得更具体了一点。所谓 Agentic Commerce，并不是简单让 AI agent 自动收付款，而是要把任务、验收、托管、超时、评价和结算这些环节设计成可执行、可验证的流程。AI 可以参与拆 milestone、整理验收标准、解释合约状态，甚至提醒风险；但真正涉及付款、审批、取消和释放资金时，还是需要明确权限、人工确认和链上记录。

接下来我想继续把 Week 1 的基础任务补扎实，包括概念卡片和测试网交易；同时把 milestone escrow 当作 Week 2 的观察方向，看看它和 ERC-8183 / Agentic Commerce 能不能对应起来。

今天学习了「Web3 运行原理」。这节课把钱包、私钥、助记词、交易、签名、Gas、RPC、mempool、builder / validator、出块和区块浏览器串成了一条完整链路。最重要的理解是：资产并不在钱包里，钱包主要负责保存或调用私钥、构造交易和发起签名；真正的账本状态在链上，任何人都可以通过区块浏览器验证。

今天我重点整理了两个安全边界：第一，助记词是整组账户的根备份，泄露后会影响所有派生账户；单个私钥泄露通常影响对应账户。第二，交易不是简单点击按钮，而是对某个链上动作的授权，签名前要检查 to、value、data、gas、nonce 和 network。

Proof-of-work 是把“创建钱包流程”做成了一个 HTML 学习页，并补了一张更清晰的 graph，把助记词、seed、派生路径、私钥、地址、签名和广播上链的关系放在一张图里。下一步会基于这份笔记准备测试网交易，并用区块浏览器记录 transaction hash、gas、状态和确认信息。

# 2026-05-29
<!-- DAILY_CHECKIN_2026_05_29_START -->
这周我逐渐把 AI x Web3 从“AI 帮我操作钱包”理解成一个更具体的系统设计问题：agent 能理解意图、整理信息、准备交易，但真正难的是权限边界怎么设计。

普通钱包和 agent wallet 最终都可能是在完成交易，但 agent wallet 的关键不在“多了一个自动操作的人”，而在于人的完整权限被拆成更小、更明确、可过期、可撤销、可审计的授权。比如金额上限、时间范围、合约白名单、函数范围、资产范围，以及什么时候必须重新让人确认。

前几天关于隐私的内容也让我意识到，隐私不是最后加上的保护层，而是一开始就要进入产品设计的问题。一个 agent 如果要长期记住我的偏好和操作习惯，它应该记住的是规则、风险偏好和确认标准，而不是敏感信息本身。

所以我今天的阶段性收获是：AI x Web3 里真正有价值的 agent，不是越自动越好，而是能在清晰边界内帮人完成复杂流程。接下来我想继续把这个方向整理成一个 agent wallet 的权限策略和 threat model。
