# 2026-05-30
<!-- DAILY_CHECKIN_2026-05-30_START -->
Agentic Commerce（代理式商业）是电子商务和零售领域正在兴起的一种全新范式。它的核心在于：由 AI 智能体（AI Agents）代表消费者进行商品搜索、跨平台比价、谈判甚至自动完成支付交易。

过去，AI 在电商中扮演的是导购或客服角色（比如基于历史行为推荐商品，或回答物流问题）。而在 Agentic Commerce 中，AI 跨越了提供建议的边界，直接获得了行动权与交易权。

## 核心运作逻辑与演进

在传统模式下，用户必须自己打开 App、输入关键词、浏览几十个页面、对比价格、加入购物车并结账。而在 Agentic Commerce 模式下，复杂的消费旅程被极大地压缩，变成了：表达意图 + 设定约束条件 + 一次确认。

| 维度 | 传统电商 | 代理式商业 (Agentic Commerce) |
| 交互对象 | 人 vs 图形界面 (网页/App) | 人 vs 智能体，智能体 vs 商家系统 |
| 决策过程 | 用户自行搜索、对比、判断 | 智能体基于参数自主跨平台比价与权衡 |
| AI 的角色 | 辅助者（商品推荐、内容生成） | 行动者（自主下单、代办退换货） |
| 商家优化目标 | 吸引人类（SEO、视觉交互、流量） | 适配机器（API 接口、结构化数据） |

**运作流程示例：**

1.  **意图输入：** 你对 AI 说：“帮我订下个月中旬去大阪的机票和两晚私汤温泉酒店，预算 8000 元以内，不要红眼航班。”
    
2.  **自主研究与决策：** AI 会同时调用多家航司、OTA 平台和酒店的数据接口，自动发现“周三出发比周五省钱”，并在你的预算和约束内组合出最优解。
    
3.  **交易执行：** AI 生成一套结构化方案，你只需回复“确认”，AI 就会自动调用支付凭证，跑通扣款和订单锁定流程。
    

## 支撑 Agentic Commerce 的三大基石

要让 AI 真正能够独立完成交易，单靠大语言模型（LLM）是不够的，必须有底层的系统和协议支撑：

1.  **统一商业协议**
    
    如果每个电商网站的接口都不同，AI 就无法顺畅跨站购物。目前行业正在推动标准化的通信协议，例如 Google 参与主推的 **UCP（通用商业协议）** 或 **ACP（代理式商业协议）**。这些协议让商品数据、库存、购物车状态能以统一标准向智能体开放。
    
2.  **代理支付**
    
    让 AI 替你花钱，安全是底线。这催生了代币化支付机制（如 Shared Payment Token）。用户可以为 AI 生成一个临时虚拟凭证，限定“只能在某类商家使用”、“上限 1000 元”且“24 小时内有效”，避免 AI 幻觉误操作或遭遇欺诈导致资金流失。
    
3.  **对 Agent 友好的底层数据**
    
    传统的商品落地页是给人看的（充满炫酷的视觉和情绪化文案）。但在 Agentic Commerce 中，商家必须提供机器可读的结构化数据（如 [Schema.org](http://Schema.org) 标记），确保 AI 能准确无误地抓取库存、规格、运费、退换货政策等硬性指标。
    

## 对商业生态的颠覆性影响

-   **对消费者：** 购物从一种费时的“劳动”变成了真正的服务。信任机制将从“信任某个电商平台”逐渐转移到信任我的专属 AI 代理。
    
-   **对品牌与商家：** 流量分配逻辑将被彻底重写。过去的营销是做 SEO（搜索引擎优化），未来则是做 **GEO（生成式引擎优化）** 和 **AEO（智能体引擎优化）**。如果你的商品无法通过 API 被 AI 检索和结算，你的品牌就会从未来的数字世界中隐身。
    
-   **对平台方：** OTA（在线旅游）或大型电商平台，可能会部分退居幕后，从直面消费者的超级 App演变为面向无数 AI 智能体提供全球供应链和结算服务的底层基础设施。
    

## 关键挑战

Agentic Commerce 落地面临的最大风险不在于 AI 够不够聪明，而在于行动边界的设计。

在真实的交易场景中，真正的关键判断往往发生在临界点（比如遇到退改签政策冲突、或发现商品有未注明的瑕疵时）。优秀的 Agentic Commerce 系统必须具备因果授权能力——它需要知道在什么情况下可以自主执行，在什么情况下必须暂停行动，并将决策权交还给人类。毕竟，高效率的自动化如果不加上完善的回退和止损机制，就会放大整个商业系统的风险。

## AI 概念重新理解：Agent ≠ 聊天机器人

本周最大的认知更新：

AI Agent 不是更聪明的聊天框。 它是一个可以：规划步骤 → 调用工具 → 执行任务 → 验证结果的系统。

但它有一条硬边界—— 不能碰私钥，不能自动签名，不能绕过人工确认。

这条边界不是技术限制，是安全设计。

## Web3 概念重新理解：Gas 和签名

签名这件事我以前理解错了。

它不是"输密码，而是： 用私钥对这笔交易的数据做数学证明， 证明是我授权的这笔交易，我认可这个内容。

Gas 也不只是手续费。 它是链上计算资源的定价机制，防止无限循环和垃圾交易。

每一笔链上写操作都需要两样东西：签名 + Gas。 读操作（view 函数）两样都不需要。

AI Agent 可以帮你准备一笔链上交易， 但能不能发起它？

Ritual 解决的问题非常具体：**"我怎么知道这个 DeFi 协议用的 AI 风险模型是它声称的那个？"** 这在传统 Web2 中完全依赖信任，但在链上金融场景中不可接受。Ritual 的路径——密码学证明 + 经济惩罚双重保障——是目前我见过最系统性的答案。

产品视角的洞见：**"可验证 AI 推理"本质上是在解决 AI 的信任问题，而不只是技术问题**。随着 AI Agent 开始控制链上资金（DeFi Agent、自动化合约执行），"这个 Agent 真的按照它声称的逻辑做出了决策"会变成核心安全需求。

**核心限制（技术诚实）：**

1.  **ZK 证明的算力开销巨大**：目前 ZKML 的证明生成比原始推理慢 10,000 倍（2025 年已从 1,000,000 倍改善）。这意味着 GPT-4 级别的大模型实际上无法走 ZK 路径，只有小模型（BERT 级别）才实用。
    
2.  **量化精度损失**：ZK 电路要求整数域运算，浮点数 ML 模型需要量化，这会带来精度损失，敏感场景下可能不可接受。
    
3.  **Ritual Chain 还未上线**：当前可验证的产品是 Infernet（Oracle 层），Ritual Chain 和 Symphony 共识仍是路线图中的内容，还没有主网可验证。
    

**未解疑问：**

-   EOVMT 模型（执行一次、验证多次）如何处理节点提交错误证明的情况？验证节点如何在不重新执行推理的情况下检测出来？
    
-   当 AI 模型本身被更新（版本迭代），链上的"我用了这个模型"如何锚定版本？模型哈希上链是唯一答案吗？

## 横向对比与总结

| 维度 | Bittensor | Ritual |
| --- | --- | --- |
| 核心定位 | 去中心化 AI 市场（竞争出质量） | 可验证 AI 推理（密码学保证真实性） |
| AI 可验证方式 | 经济激励（Validator 评分，非密码学） | ZK 证明 / TEE / 乐观验证（密码学） |
| Web3 角色 | 奖励结算账本 + 去中心化治理 | 智能合约集成 + 经济安全（EigenLayer） |
| 最大风险 | Validator 串谋、治理中心化 | ZK 算力瓶颈、产品尚未完全生产可用 |
| PoW 可验证性 | 链上权重记录（弱）；推理本身链下不可验证 | 密码学证明上链（强）；Infernet 有 GitHub 可运行 Demo |
| PM 视角关注点 | 谁在为 AI 质量背书？激励机制是否 hold | AI + 链上金融的风险合规场景；Agent 经济时代的信任基础设施 |

**最终判断**： 这两个项目代表了 AI × Web3 的两种截然不同的路径——Bittensor 是**市场机制路径**（用代币博弈代替中心化质量评估），Ritual 是**密码学路径**（用数学证明代替信任）。两者都没有完全解决各自的核心难题，但方向都是真实的工程问题，不是纯叙事。

对于 Hackathon 项目而言，Ritual 的 Infernet 是更容易作为构建基础设施的工具层——它有可运行的 SDK 和清晰的合约集成接口；Bittensor 的 Subnet 生态更适合作为 AI 模型的分发市场来利用，而不是底层基础设施。

Cobo Agentic Wallet 是一款专门为 **AI Agent（人工智能智能体）** 设计的加密钱包。 在过去，如果你想让 AI 帮你自动交易或管理资产，你要么得手动审批它的每一笔操作（效率极低），要么得直接把钱包私钥交给 AI（安全风险极高，AI 可能会产生幻觉或被黑客攻击）。

Cobo 通过引入 **可强制执行的自治**解决了这个问题。AI 没有私钥，而是通过一个名为 **Pact（契约）** 的框架，在人类设定好的边界内（如：预算上限、允许交易的代币、白名单合约等）自主运行。后端的 MPC（多方安全计算）技术会在密码学层面上确保 AI 绝对无法越界。

负责这样一款融合了 AI、区块链和密码学硬核技术的产品，PM 的工作内容与传统的互联网产品经理有很大不同，主要体现在以下几个维度：

1\. 平衡“AI 自治”与“人类控制”

PM 需要设计一套既不阻碍 AI 高效执行，又能让用户绝对放心的产品机制。

-   **Pact 框架设计：** 设计用户与 AI 之间的“契约”流程。例如，如何让用户在 App 里直观地输入意图（Intent），审核 AI 生成的执行计划和策略限制（Policies），并最终授权。
    
-   **紧急熔断机制：** 规划并完善“一键冻结（Kill Switch）”功能，确保用户在发现 AI 行为异常时，能通过手机 App 瞬间撤销所有授权。
    

2\. 定义开发者与 C 端用户的双端体验

CAW 既是一个底层基础设施，也有面向用户的移动端 App（目前已在 iOS 商店上线）。

-   **开发者体验（DX）：** 优化 Python 和 TypeScript SDK 的接入流程，确保 AI 开发者（比如使用 LangChain、CrewAI 的开发者）能顺畅地为他们的 Agent 集成钱包功能。
    
-   **移动端体验（UX）：** 设计 iOS App 的交互，让普通用户能直观地看到授权请求、资金动向以及 AI 被拦截的越权操作记录（Audit Trail）。
    

3\. 将硬核的 MPC 技术“翻译”为用户价值

CAW 的底层使用了 2/2 门限的 MPC（多方安全计算）技术——用户/Cobo 各拿一个私钥分片，或者 AI/Cobo 各拿一个。PM 的核心能力在于，不需要让用户懂什么是“私钥分片”，只需要让他们明白：“哪怕 AI 被黑客劫持了，它也偷不走你的钱，因为只有在规则内 Cobo 的服务器才会配合签名。”

**Claude Code** 是 Anthropic 推出的 AI 编程 Agent（目前 Beta），核心特点是它直接运行在你的**终端**里，而不是浏览器 IDE。

### **它能做什么**

| 能力 | 说明 |
| --- | --- |
| 读整个代码库 | 自动理解项目结构，不需要你手动粘贴代码 |
| 直接操作文件 | 创建、修改、删除文件，不需要复制粘贴 |
| 执行命令 | 跑测试、安装依赖、Git 操作，你确认后直接执行 |
| 多步骤 Agentic | 可以拆解复杂任务，自主规划多步执行 |
| Web 搜索 | 可以联网查文档、API 参考 |

### **与普通 AI 助手的区别**

普通 Chat AI: 你 → 描述问题 → AI 给代码 → 你手动改文件

Claude Code: 你 → 描述任务 → AI 直接改文件 + 跑命令 → 你确认

**Vibe Coding** 是 2025 年初 Andrej Karpathy（前 OpenAI）提出的词，描述一种新的人机协作编程方式：

> _"你完全沉浸在 Vibe 里，用自然语言表达意图，AI 生成代码，你不深究细节——只要能跑就行。"_

### **核心思路**

传统编程思路: 需求 → 设计 → 手写代码 → 调试 → 交付

Vibe Coding: 需求 → 自然语言描述 → AI 生成 → 迭代反馈 → 交付

你的工作从"**写代码**"变成了"**描述意图 + 验收结果**"。

### **三个关键特征**

**1\. 意图驱动，不是语法驱动**

\# 不是这样想：

"我要写一个 for 循环遍历数组..."

\# 而是这样想：

"给每个用户发一封欢迎邮件，主题包含他们的名字"

**2\. 持续对话式迭代** 不是一次生成完美代码，而是像改 PRD 一样来回打磨：

-   "这个按钮颜色改成深一点"
    
-   "加一个 loading 状态"
    
-   "报错信息对用户来说太技术性了，改成人话"
    

**3\. 人负责方向和验收，AI 负责实现**

你负责： 知道要做什么、判断做得对不对、决定继续还是回滚

AI负责： 知道怎么做、处理样板代码、记住所有 API 细节

## 1\. 地址 (Address)

**一句话解释：**

地址是你在区块链上的门牌号，是一串公开的字符串，任何人都可以向它转账或查询余额，但无法凭地址本身控制里面的资产。

**具体例子：**

`0xAbCd...1234` 就是一个以太坊地址。你把这个地址发给朋友，他就能给你转 ETH——就像把银行账号告诉别人让他打款一样。

**安全提醒：**

地址是公开的，泄露没关系；但注意转账时要逐字核对地址，区块链转账一旦发出无法撤回。黑客常用地址污染手法，发一笔小额交易让你的历史记录里出现一个与你地址高度相似的假地址，引诱你下次复制错。

* * *

## 2\. 钱包 (Wallet)

**一句话解释：**

钱包不存储币，它存储的是你的私钥，相当于一把可以证明这些链上资产是我的的万能钥匙；资产本身永远在链上。

**具体例子：**

MetaMask 是最常见的浏览器钱包插件。你用它连接 Uniswap 时，网站请求签名，你点确认——MetaMask 在背后用私钥生成签名，但私钥从未离开你的设备。

**安全提醒：**

钱包被盗本质是私钥/助记词泄露，不是 App 被入侵。如果一个网站或人要求你输入助记词，立刻关闭页面，100% 是骗局。

* * *

## 3\. 助记词 (Seed Phrase / Mnemonic)

**一句话解释：**

助记词是一组 12 或 24 个英文单词，是钱包的总根密码，可以从它派生出你所有的私钥和地址——谁拥有助记词，谁就完全控制你的钱包。

**具体例子：**

`abandon ability able about above...`（示意，非真实）

如果你的手机丢失，只要在新设备上输入同一组助记词，就能完整恢复所有账户和余额。

**安全提醒：**

助记词**只写在纸上**，锁进保险柜；绝不截图、绝不存云盘（iCloud/Google Drive）、绝不发给任何人。任何客服或官方要求你提供助记词都是诈骗。

* * *

## 4\. 私钥 (Private Key)

**一句话解释：**

私钥是一串 256 位的随机数，是签署每一笔交易的数字签章印章，由助记词派生，比助记词更底层也更难记。

**具体例子：**

`0x4c0883a69102937d6231471b5dbb6e...`（示意）

当你在 MetaMask 点发送时，钱包在幕后用私钥对交易内容做数学运算，生成一个唯一签名附在交易里，矿工/验证者验证后才会执行。

**安全提醒：**

私钥≈助记词，同等危险。正常使用时你永远不需要手动接触私钥；如果某个 DApp 或工具让你粘贴私钥，立刻停止操作。

* * *

## 5\. 签名 (Signature)

**一句话解释：**

签名是用私钥对一段数据做的数学盖章，可以让任何人用对应的公钥/地址验证这条消息确实是这个账户发出的，而不暴露私钥本身。

**具体例子：**

登录 OpenSea 时，网站让你签名一条消息；这个操作不花 Gas、不上链，只是证明你是这个钱包的主人——类似在合同上按指纹。

**安全提醒：**

签名≠无害。某些恶意合约会构造SetApprovalForAll等危险内容伪装成普通签名请求，签了等于授权它转走你所有 NFT。永远阅读签名内容，不认识的字段立刻拒绝。

* * *

## 6\. 交易 (Transaction)

**一句话解释：**

交易是你向区块链发出的一条指令，可以是转账、调用合约、部署合约等，每笔交易都需要私钥签名、支付 Gas 费，并被永久记录在链上。

**具体例子：**

你在测试网向朋友转 0.01 ETH：MetaMask 弹出确认框，显示收款方、金额和 Gas 费；你点确认 → 交易被广播到网络 → 矿工打包进区块 → 12 秒后状态变为"成功"，区块浏览器可查。

**安全提醒：**

交易一旦上链，任何人无法修改或撤销，包括官方和开发者。发送前务必核对地址、金额和合约调用内容，测试网上多练习再上主网。

* * *

## 7\. Gas

**一句话解释：**

Gas 是以太坊上执行任何操作的计算燃料费，用 ETH 支付，防止垃圾交易刷屏，也补偿验证者的计算资源。

**具体例子：**

一笔普通转账消耗 21,000 Gas；调用复杂合约可能消耗几十万 Gas。如果你设置的 Gas Limit 太低，交易会失败（out of gas），但已花掉的 Gas 不退还。

**安全提醒：**

Gas 费由网络拥堵程度动态决定，高峰期可能是平时的 10 倍以上。在测试网练习时 Gas 是免费的假 ETH，但习惯确认步骤，上主网才不会因慌乱点错导致损失。

* * *

## 8\. 智能合约 (Smart Contract)

**一句话解释：**

智能合约是部署在区块链上的程序，条件满足时自动执行，没有任何中间人可以干预——代码就是规则，规则就是法律。

**具体例子：**

你在 Uniswap 用 ETH 换 USDC：你的 ETH 打入流动性池合约，合约自动按公式计算，转出对应 USDC 给你——整个过程无需 Uniswap 公司的员工操作，毫秒完成。

**安全提醒：**

代码即法律是双刃剑：合约里的 bug 无法修改（除非有升级机制），黑客可以利用漏洞永久抽走资金。交互陌生合约前，先查审计报告或在测试网验证行为。

* * *

## 9\. 测试网 (Testnet)

**一句话解释：**

测试网是以太坊（或其他链）的沙盒环境，规则和主网完全一样，但代币没有真实价值，专门用于开发者和学习者试验，不怕犯错。

**具体例子：**

Sepolia 是目前以太坊最主要的测试网。你可以从 Faucet（水龙头网站）免费领取测试 ETH，然后部署合约、发送交易、测试 DApp，不花一分真钱。

**安全提醒：**

测试网的代币没有价值，任何声称"测试网 ETH 可以兑换真钱"的项目都是骗局。此外，测试网偶尔会重置，不要依赖它保存重要数据。

* * *

## 10\. 区块浏览器 (Block Explorer)

**一句话解释：**

区块浏览器是链上数据的搜索引擎，你可以用它查询任意地址的余额、任意交易的状态、任意合约的代码和调用记录——区块链的公开账本可视化界面。

**具体例子：**

在 Etherscan ([etherscan.io](http://etherscan.io/)) 搜索你的交易哈希 `0xabc...`，可以看到：发送方、接收方、金额、Gas 费、区块高度、时间戳，以及"Success / Failed"状态——这是链上验证的核心工具。

**安全提醒：**

Etherscan 只是只读展示工具，它本身不能操作你的资产。但要注意假冒 Etherscan 的钓鱼网站（域名细微差别如 `etherscan.com` vs `etherscan.io`），别在这类网站上连接钱包或授权操作。

Web3 的底层逻辑是对传统金融中介模式的彻底重构。它通过密码学手段将资产控制权从银行、平台等中介手中剥离，重新赋予个人。

**1.1 传统金融账户与 Web3 钱包地址的深度对比**

| 维度 | 传统金融账户 | Web3 钱包地址 |
| --- | --- | --- |
| 身份绑定 | 必须执行 KYC，绑定真实身份信息。 | 无需许可，通过算法自主生成。 |
| 审批机制 | 机构拥有最终解释权，开户与交易需审核。 | 算法即审批，符合数学签名逻辑即可执行。 |
| 所有权控制 | 机构可执行冻结、限制或划扣。 | 私钥持有者绝对控制，资产存在于链上而非钱包内。 |
| 通用性 | 账套封闭，不同银行间无法通过同一密码互通。 | 遵循统一协议，助记词可在不同客户端通用。 |

**1.2 从私钥派生到地址生成的数学逻辑**

数字主权的实现源于严密的派生关系。Bruce 老师详细拆解了这一过程的社会学与技术意义：

-   **私钥 ：** 个人主权的终极起点。它是控制地址资产的唯一钥匙。由于私钥本质上是极长的随机数，社会学管理门槛极高且不可更改。
    
-   **助记词 ：** 私钥的备份与派生系统。一个助记词通过特定路径可派生无限个私钥。需注意，助记词泄露即意味着该路径下所有资产主权的永久丧失。
    
-   **地址 的技术细节：** 地址是由原始公钥经过哈希计算派生而来的。以太坊地址是通过截取原始长公钥的最后 20 字节（40 个十六进制字符）生成的。由于哈希碰撞概率极低，这种截取在保证安全性的同时提升了可用性。
    

**1.3 信任磨损的消除**

智能合约是以太坊等 Web3 平台的核心驱动。它是一段部署在链上、以字节码形式存在的程序。当用户发起交易触达合约地址时，代码会在以太坊虚拟机（EVM）中自动执行。这种机制将信任的对象从人和中介转向了代码共识，极大地降低了律师、政府等第三方参与带来的信任磨损。

**2\. 技术运行机制**

在去中心化网络中，交易的执行是一场关于效率、安全与经济博弈的复杂过程。

**2.1 交易流动的六大阶段身份认证：** 用户用私钥对交易（如转账 1.5 ETH）进行数字签名。

1.  **授权：** 网络节点通过算法校验签名的合法性。
    
2.  **传播：** 交易通过 RPC 接口广播。**警示：** 除非运行自建节点，否则用户必须依赖 Infura 或 Alchemy 等中心化 RPC 提供商，这存在隐私被追踪或服务被拒绝的潜在风险。
    
3.  **排序 ：** 这里引入 Bruce 老师的汽车车位类比：一个区块就像一辆座位有限的汽车。Builder（构建者）作为检票员，会根据乘客支付的车票（GAS 费）高低来安排座位和排序。
    
4.  **执行：** 验证者运行 EVM 跑通交易逻辑，更新全网状态。
    
5.  **最终确认：** 交易正式打包进区块并被后续区块引用。
    

**2.2 GAS 费的双重作用与 EIP-1559 机制**

GAS 费不仅是防止垃圾交易攻击的屏障，更是网络资源配置的经济杠杆。在现行机制下：

-   **基础费 ：** 根据网络拥堵情况自动调整，并直接销毁（实现资产通缩）。
    
-   **优先费 ：** 归属于验证者，作为激励其优先处理交易的报酬。
    

**2.3 最终性与安全博弈**

以太坊交易通常需要等待约 12 分钟 才能达到 Finalized（最终确认）状态。这是因为每个新块都会包含前一个块的 Parent Root Hash。随着链的延伸，篡改历史区块的成本呈指数级上升。等待 12 分钟是确保交易在数学和经济层面达到“不可撤销”的行业安全标准。

在 Web2 中，应用程序的后端逻辑和数据库都由单一公司（如 Google、腾讯、亚马逊）控制的集中式服务器托管。而在 Web3 中，核心逻辑和数据被分布在全球的去中心化网络（区块链）上。

## Web3 的四大核心组件

一个完整的 Web3 应用程序（通常称为 **DApp**，去中心化应用）的架构通常由以下几个部分组成：

### 1\. 前端

这部分与 Web2 几乎没有区别。开发者依然使用 HTML、CSS、JavaScript（如 React 或 Vue）来构建用户界面。浏览器依然是访问 Web3 应用的窗口。

### 2\. 加密钱包

在 Web3 中，不再需要通过邮箱注册账号或设置密码，而是通过加密钱包（如 MetaMask）来登录。

-   **原理：** 钱包包含一对“公钥”和“私钥”。公钥就像银行账号，任何人都可以看到；私钥则是密码，用于签署交易并证明身份。
    
-   **作用：** 当用户在 DApp 上进行任何涉及数据更改或资产转移的操作时，前端会弹出一个请求，要求用户用钱包进行数字签名以授权该操作。
    

### 3\. 智能合约

Web2 的后端代码运行在中心化服务器上，而 Web3 的后端是由智能合约构成的。

-   **原理：** 智能合约是部署在区块链（如 Ethereum）上的一段自动执行的代码。它包含了“如果发生 X，则执行 Y”的业务逻辑。
    
-   **特点：** 代码一旦部署就无法轻易篡改，且完全开源透明。它不依赖任何中心化服务器来运行，而是由区块链网络中的无数个节点共同执行和验证。
    

### 4\. 区块链

Web2 使用集中式数据库（如 MySQL、MongoDB）来存储个人信息、帖子或余额，管理员可以随时修改或删除这些数据。

-   **原理：** Web3 使用区块链作为底层状态机或数据库。区块链是一个公开的、分布式的账本。
    
-   **特点：** 数据被打包成区块并加上加密时间戳，按顺序链接在一起。由于数据分布在网络的所有节点上，任何人（包括应用的开发者）都无法单方面篡改资产或历史记录。

钱包在 Web3 中不仅是资产载体，更是身份的唯一验证器。钱包架构的选择直接定义了系统的安全性底色。

**钱包分类与账户抽象**

-   **EOA 钱包 (Externally Owned Account):** 由私钥直接控制。风险极高，私钥丢失或泄露即意味着资产永久丧失。
    
-   **合约钱包 (Smart Contract Wallet):** 如 Gnosis Safe。本质是运行在链上的程序，天然支持多签逻辑。
    
-   **AA 账户抽象 (Account Abstraction):** 未来账户体系的终点，致力于在 EVM 层面融合 EOA 与合约钱包的优势，提升用户交互体验。
    

**MPC 与合约多签的博弈**

-   **MPC (多方计算) 多签：** 具有极强的技术美感。它将私钥切片化（例如将私钥 10 切分为 7、8、9 的分片），每个参与者持有一片。最核心的架构优势在于：**私钥在全生命周期中（生成与签名）从未以完整形式存在过**，从而彻底消除了单点泄露风险。
    
-   **合约多签：** 在链上通过透明逻辑（如 2/3 签名）判定执行，适合治理与机构资产公开透明管理。
    

**TEE 与远程证明**

针对服务端私钥管理，**TEE（可信执行环境）** 提供了硬件隔离的最后一道防线。在 TEE 中，私钥明文仅在独立内存中运行，输入加密，输出仅为签名结果。结合 **远程证明（Attestation）** 技术，可以确保签名逻辑未被篡改。

**生命周期、模拟与风险防控**

Web3 交易的确定性是由复杂的生命周期和共识机制支撑的，理解这些细节是构建稳健架构的前提。

**交易模型：UTXO vs. Account**

-   **UTXO 模型 (BTC):** 类似于100 元纸币模型。你支付 1 元，系统会消耗这 100 元纸币，并找回 99 元到你的新地址。这种模型天然支持并行处理，且自带幂等性。
    
-   **账户模型 (ETH):** 类似于银行卡余额。每一笔交易必须由一个递增的 **Nonce** 来维护顺序。
    

**关键参数与跨链差异**

-   **Gas Fee 结构:** 在 EIP-1559 协议下，Gas 由 Base Fee（燃烧）和 **Tip Fee（支付给 Proposer 的激励）** 组成。
    
-   **Nonce 与幂等性:** 以太坊依赖 Nonce 的连续递增；而像 **TRON** 这样的链虽然兼容 EVM，但其实际上并不依赖传统 Nonce，而是通过 SDK 计算 **Ref Block Hash（引用块哈希）** 来确保交易不被重复解析。
    
-   **Finality (终局性) 与链重组:** 开发者必须警惕不同链的确认延迟。例如，**Polygon** 作为以太坊侧链，其最终确认需要通过Checkpointing打回到 L1，这导致其在交易所充值时通常需要比以太坊更多的确认层数。
    

**交易模拟**

在 Web3中，**交易模拟是架构中的必选项**。在用户签名之前，通过模拟环境可视化资产流向（例如：检查一个预期的 1 USDC 授权是否在底层变成了 500 USDC 的转出）。**模拟必须在签名动作之前进行。** 任何已签名的交易信息在系统中流转都存在被恶意广播的风险。

### 1\. Prompt（提示词）

**定义**：Prompt 是人类与 AI 模型交互的“指令”或“输入”。它告诉 AI 你希望它做什么、扮演什么角色以及输出什么格式。

**核心作用**：

-   **引导生成**：通过明确的上下文和约束，控制模型的输出质量。
    
-   **提示词工程（Prompt Engineering）**：一门通过系统化设计、优化和迭代输入文本，来激发大模型最佳表现的技术。这包括 Few-shot（少样本提示）、Chain-of-Thought（思维链）等高级技巧。
    

### 2\. Context Window（上下文窗口）

**定义**：模型在生成单个回复时，能够同时“记住”并处理的最大文本信息量（通常以 Token 为单位计算）。

**核心作用**：

-   **AI 的“短期记忆”**：你可以把它想象成 AI 的工作台。工作台越大，能同时摊开参考的资料（如长文档、多轮对话历史）就越多。
    
-   **约束与边界**：如果输入的文本或历史对话超出了上下文窗口的上限，最前面的信息就会被“遗忘”或截断。现代模型的上下文窗口正在不断扩大（例如从几千 Token 扩展到 100万甚至 200万 Token）。
    

### 3\. Workflow（工作流）

**定义**：预先定义好的一系列步骤或处理逻辑，将大模型的能力与传统代码逻辑结合起来，以完成复杂任务。

**核心作用**：

-   **流程编排**：它通常是高度结构化和确定性的。比如：步骤A（提取文本摘要）$\\rightarrow$ 步骤B（翻译成英文）$\\rightarrow$ 步骤C（发送邮件）。
    
-   **常见模式**：包含提示链（Prompt Chaining）、路由分发（Routing）、并行处理（Parallelization）等。工作流的设计者是**人**，AI 只是在既定轨道上执行任务的“节点”。
    

### 4\. Agent（智能体）

**定义**：具备自主感知、推理决策和执行能力的 AI 实体。与只能被动回答问题或按预设 Workflow 运行的系统不同，Agent 具有高度的自主性。

**核心作用**：

-   **自主决策**：给定一个高层级的模糊目标（如“帮我调研一下这只股票并写一份报告”），Agent 可以自己拆解任务、制定计划、选择工具，并根据环境的反馈（如网页报错或搜索结果）动态调整策略，直到完成任务。
    
-   **构成要素**：通常由大脑（LLM）、记忆（上下文与外部存储）、规划能力（Planning）和工具（Tools）四部分组成。
    

### 5\. Tool Use（工具调用 / Function Calling）

**定义**：赋予大语言模型与外部世界交互的能力，使其打破“只懂文本输出”的局限。

**核心作用**：

-   **扩展能力边界**：大模型的知识受限于训练数据且无法直接操作现实系统。通过 Tool Use，模型可以“决定”在什么时候调用什么工具（如：调用搜索引擎获取最新新闻、调用计算器进行复杂数学运算、调用 API 查询天气、执行 Python 代码等）。
    
-   **闭环执行**：这是 Agent 能够产生实际行动（Action）的核心机制。
    

### 6\. AI Coding（AI 编程）

**定义**：利用大模型能力来辅助、加速甚至自动化软件开发的整个生命周期。

**核心作用**：

-   **从辅助到主导**：早期形态是代码补全（如 GitHub Copilot）；现在已经演进为具备 Agent 能力的开发环境和工具（如 Cursor、Claude Code、Google Antigravity 等）。
    
-   **应用场景**：AI 可以跨越整个代码库理解上下文（Context Window），自主规划修改方案（Agent Workflow），调用本地终端执行命令和测试（Tool Use），最终自动修复 Bug、重构代码或开发新功能。

## 一、 什么是 AI Agent？

**AI Agent（人工智能智能体）** 是一种能够感知环境、进行决策并采取行动以实现特定目标的软件系统。 在当下的大模型时代，AI Agent 通常是指**以大语言模型（LLM）为大脑**，具备自主规划任务、拥有记忆机制，并能调用外部工具的智能系统。

**核心概念对比：**

-   **LLM（大语言模型）：** 一个超级聪明的“大脑”，懂得多，但被动。你问一句，它答一句。
    
-   **Copilot（副驾驶）：** 协助人类工作的助手，需要人类频繁介入、引导和拍板。
    
-   **Agent（主驾驶/智能体）：** 具备高度自主性的执行者。给定一个目标后，它能自行拆解任务、克服困难、调用资源并交付最终结果。
    

* * *

## 二、 AI Agent 的四大核心组件 (架构)

一个成熟的 AI Agent 系统通常由四个核心部分组成：

### 1\. 大脑

大语言模型是 Agent 的中枢神经，负责提供自然语言理解、逻辑推理、决策判断和内容生成能力。Agent 的“智商”上限很大程度上取决于底层 LLM 的推理能力。

### 2\. 规划

面对复杂的现实任务，Agent 需要像人一样将大目标拆解为可执行的小步骤。

-   **任务分解：** 将宏大目标分解为多个更小、易管理的子任务。
    
-   **反思与自我纠错 ：** 在执行过程中，Agent 能够评估当前结果。如果发现某条路走不通，它会自我纠正，从错误中学习并调整策略（例如 Reflexion 框架）。
    
-   **高级推理思维：**
    
    -   **思维链 (CoT - Chain of Thought)：** 引导模型“一步一步地思考”，将复杂逻辑过程显性化。
        
    -   **思维树 (ToT - Tree of Thoughts)：** 在任务的每一步探索多种可能的推理路径，从而找到最优解。
        

### 3\. 记忆

为了保持上下文连贯并从过往经验中学习，Agent 必须具备记忆机制：

-   **短期记忆 ：** 类似于人脑的工作记忆，通常指当前对话的上下文窗口。
    
-   **长期记忆 ：** 通常通过外部的向量数据库（Vector Database）实现。Agent 可以从中快速检索几天前甚至几个月前学到的信息、文档或历史操作记录。
    

### 4\. 工具使用

LLM 本身是被“困”在训练数据里的，而“工具”就是它与物理或数字世界交互的手脚。

-   **常用工具：** 网页搜索（获取实时资讯）、代码解释器（运行代码、处理数据）、计算器、API 接口（发送邮件、查询天气、操作企业数据库等）。
    
-   **前沿发展：** 行业内正在推广 **MCP（模型上下文协议）** 和 **Agent Skills**。MCP 标准化了 AI 调用外部数据和系统的方式，而 Agent Skills 则像是给 Agent 安装打包好的“能力扩展包”（比如专门处理 PDF 文件的技能），进一步降低了开发门槛。
    

* * *

## 三、 AI Agent 的核心工作流

Agent 的执行过程通常是一个动态的循环。目前最主流的运行框架之一是 **ReAct**（Reason + Act，即推理+行动）：

1.  **目标下达：** 用户输入：“帮我调研一下目前市面上排名前五的防水跑步鞋，并写一份优缺点报告。”
    
2.  **思考与规划：** Agent 大脑开始运转：“我需要先去搜索引擎查找 2026 年最新的跑鞋评测，提取前五名，然后分别搜索它们的优缺，最后汇总写报告。”
    
3.  **行动 ：** Agent 调用搜索引擎工具，输入搜索词。
    
4.  **观察 ：** 提取并阅读搜索返回的结果。
    
5.  **循环/迭代：** Agent 评估收集到的信息是否足够。如果不够，调整搜索策略继续执行；如果足够，则进入总结阶段，输出最终报告。
    

* * *

## 四、 典型的应用场景

-   **自动化办公与企业协同：** 通过接入飞书、Slack 等办公软件的 CLI（命令行工具），Agent 可以直接读取群聊消息、查询日历、自动新建文档甚至发邮件。它不再只是给出“建议”，而是把工作真正“落到实处”。
    
-   **程序员的得力助手 ：** 例如 AutoGPT 或深度集成在 Cursor 等 IDE 中的 Agent。你只需用自然语言描述需求，它就能自行阅读代码库、编写代码、运行测试甚至修复 Bug。
    
-   **多智能体协作 ：** 著名的“斯坦福虚拟小镇”实验中，25 个 AI 角色拥有各自的性格、记忆和目标。它们自主社交、举办派对、竞选市长，展现了惊人的社会涌现能力。在企业中，这也意味着可以构建“产品经理 Agent”和“程序员 Agent”相互协作的工作流。
    
-   **高级客户服务：** 突破传统的死板问答。现代客服 Agent 能在理解客户情绪的同时，自动查询企业 ERP 系统的订单状态，并自主决定是否为用户执行退款或补偿操作。
    

* * *

## 五、 当前面临的挑战

尽管 AI Agent 潜力巨大，被视为 AI 发展的下一代核心，但目前仍面临一些挑战：

1.  **幻觉与无限循环：** 如果 LLM 推理出错，Agent 可能会偏离轨道，或者在某一个错误步骤里反复尝试（死循环）。
    
2.  **安全性与权限管理：** 赋予 Agent 越高的自主操作权（例如转账、删除文件），一旦遭到恶意提示词注入攻击，造成的破坏就越大。
    
3.  **上下文与效率限制：** 尽管大模型的上下文窗口在变大，但处理极长链路的复杂任务时，仍容易遗忘早期指令，且 API 调用的延迟和成本依然较高。
