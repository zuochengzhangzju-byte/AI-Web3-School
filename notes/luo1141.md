2026年 AI Agent × Web3 融合图谱：从身份到结算的全面打通

1.  身份识别与能力协商 (Identity & Capability) 核心痛点： AI Agent 要想真正协同工作，必须先解决“你是谁”以及“你能干什么”的问题。 AI 侧的诉求： Agent 之间需要动态发现彼此、协商任务，并执行复杂的多步骤协作（A2A，Agent-to-Agent）。 Web3 侧的基建： 链上身份标准： MetaMsk、Coinbase 和 Google 等联合推进了 ERC-8004（目前处于 Draft 阶段），它以 ERC-721 NFT 的形式为 Agent 提供链上身份。值得注意的是，该标准专门预留了 x402 支付证明槽位，身份与支付天然互补。 接口与连接标准： Anthropic 出品的 MCP（Model Context Protocol） 已成为 Agent连接工具的绝对标准（支持 Python/JS/Rust）。截至 2026 年 5 月，GitHub 上已有 564 个 MCP 相关仓库，其中偏向 Web3 的热门服务包括 armor-crypto-mcp (182⭐) 和 solana-mcp (159⭐) 等。 多方推进： 谷歌正在推进的 A2A 项目与 MCP 形成互补，共同解决 Agent 间的协作难题。
    
2.  钱包、权限表达与安全防线 (Wallet & Permission) 核心痛点： 拥有了身份的 Agent，在链上进行交互时，如何保证资产安全并防止“自主越权”？ AI 侧的诉求： 需要钱包具备异常检测、误触预防以及必要时的人工确认（Human-in-the-loop）机制。 Web3 侧的基建： 从私钥到代码规则： 行业正加速从 EOA（普通钱包）转向 Smart Account（智能账户）。通过 ERC-6900（模块化账户抽象） 和 ERC-7732（推进中的智能合约钱包接口），权限表达从冰冷的“私钥控制”变成了灵活的“代码规则”。 会话密钥（Session Key）： 多签龙头 Safe 正在探索专门给 AI Agent 授权的方案——允许其在有限权限、限时范围内自动交互，且支持自动撤销。 重大安全警示（2025-2026）： 提示词注入（Prompt Injection） 已成为 AI 钱包的最大威胁。攻击者通过在对话或数据中注入恶意指令，污染 AI 的 tool call，导致 Agent 无法区分“用户真实意图”与“恶意注入指令”。2025 年已有多起实验性 Agent 被诱导转移资产的案例（多为研究性质，暂无大额损失）。目前行业正依赖 Rebuff 等工具来专门拦截此类注入攻击。
    
3.  机器支付与商业结算 (Payment & Commerce) 核心痛点： 身份和安全问题解决后，Agent 需要具备独立自主的经济能力，无需人类反复确认。 AI 侧的诉求： AI Agent 能够自动做出商业决策，为了买 API、买数据或买算力自动触发支付。 Web3 侧的基建： 机器支付标准： 生产级的 x402 状态码（HTTP 402） 专为机器支付而生，其生态联盟极为豪华，包括 Stripe、Visa、MasterCard、Coinbase、Google Cloud、AWS 等 23 家巨头。 结算层稳定币化： 全面采用 USDC 结算。目前 Stripe 的 USDC 收款功能已上线，且 Cloudflare Workers 实现了原生支持。结合 EIP-3009，Agent 可以实现完全免 Gas 费（Gasless）的资金转账。
    
4.  隐私保护与数据防泄露 (Privacy & Security) 核心痛点： 机器在疯狂跑数据的同时，如何避免暴露用户的敏感隐私，尤其是在天然公开透明的区块链上？ AI 侧的诉求： 在推理和交互过程中，识别并隔离敏感信息，防御注入攻击，保护数据不外泄。 Web3 侧的基建： ZK + AI： 零知识证明正在成为热点方向。其核心价值在于：向外界证明“AI 模型确实不折不扣地执行了某段代码”，但同时“绝不暴露模型的权重或核心隐私数据”。 TEE（可信执行环境）： 高度依赖硬件的隐私计算方案。目前主要依靠 Intel SGX 和 ARM TrustZone，在硬件级别隔离出一个安全区域来执行 AI 推理。
    
5.  开发者工具与公链原生集成 (Dev Tooling) 核心痛点： 怎样降低门槛，让传统的 AI 开发者能无缝调用 Web3 的数据和合约？ AI 侧的诉求： 期望用自然语言查询链上数据、自动生成 Solidity 代码，或让 AI 辅助做合约审计。 Web3 侧的基建： AI 编码与审计： Cursor AI、GitHub Copilot、ChainGPT 等工具对 Solidity 的支持已非常成熟；同时，Slither、Mythril、Certora 和 CertiK 等安全巨头也推出了 AI 增强型的智能合约审计工具。 公链原生 AI 集成（2025-2026 核心趋势）： 主流 L1/L2 都在推出官方 MCP Server 或 AI SDK。 Solana： 推出了 AI Agent 原生程序，生态内有多个 AI+MCP 集成（如 solana-mcp）。 Base： Coinbase 官方在 2025 年极力推进 Base AI 开发者工具的集成。 NEAR： 提供 near-mcp

1.我学到的核心概念、方法与案例核心概念是LLM，简单讲就是基于海量数据预测下一个词的概率模型，比如给它一段话它能续写，但常见误区是把它当成绝对准确的数据库或逻辑推导器，它其实是会胡说八道的，这叫幻觉。 2.另一个是prompt，就是输入给模型的指令，例子是代码审查时用结构化角色，误区在于觉得它是魔法，其实模型本身能力不行的话怎么写提示词都没用。 3.Context Window是上下文窗口，指模型一次能处理的Token上限，比如把整个仓库代码塞给200k窗口的模型去分析，但误区是觉得窗口越大越好，其实存在迷失在中间的现象，而且长了很贵又很慢。 4.Workflow非常关键，是用确定性工程架构约束概率模型的手段。例子比如一个翻译流：初译-术语比对-润色-输出。常见误区是盲目追求全自动Agent而轻视工作流，实际上在严肃场景必须用硬编码逻辑把步骤写死。 5.Agent就是智能体，LLM做大脑，有记忆、规划和工具调用能力，比如自动监控链上大额转账并报警。误区是觉得现在能完全无人值守，其实步骤超过5到10步错误率就断崖式下跌，必须有Human-in-the-loop。 6.Tool Use是工具调用，模型判断需要外部能力时输出JSON参数。例子是问实时天气或币价时它生成API参数，系统去执行。误区是以为模型自己去运行了工具，其实真正的执行权在系统代码手里，必须做好参数校验防止被注入攻击。 7.一个我准备采取的行动是既然上次明确了以后在云服务器上部署Hermes省去本地折腾，我准备这周就在云端用hermes skills install命令装上常用skills，不让Agent瞎跑，而是用Workflow的思路限制它的执行路径。 8.一个我还想继续追问的问题是当Agent大规模使用Function Calling跟数据库或者链上环境交互时，工业界到底怎么做Guardrails拦截

1\. 我学到的核心概念、方法与案例核心概念：AI 时代对开发者的“反向高要求”与系统的“复杂度守恒”这次分享纠正了我之前的一个误区。以前觉得有了 AI 辅助编写代码，开发者就可以减少对底层原理的钻研。但会上提出的观点是，AI 时代对人的要求反而更高了。AI 确实代替了人的“手和脚”，极大地减少了编码和日常查阅的精力消耗，但它没有改变系统的复杂度。人依然需要负责顶层设计，而 AI 编码质量的关键在于开发者能否进行“自我验证”。如果开发者缺乏扎实的基础知识和架构能力，就根本无法判断 AI 生成的代码是否存在逻辑漏洞或方向偏差。更现实的是，如果代码最终出现安全风险，担责的依然是开发者自己。因此，必须确保 AI 在可控范围内，对所有输出进行严格的审核与验收。架构融合：Web3 项目中的“Web2 含量”与支付系统升级案例Web3 并不是与 Web2 完全割裂的独立体系。事实上，绝大多数 Web3 项目（包括币安、Bybit 等交易所）的 Web2 能力占比相当高。前端页面、后端状态管理决定了用户体验，而区块链只负责最核心的“价值流转”和“共识”那一层。要在 Web3 走得远，Web2 的功底必须稳固。分享中关于 Web3 支付系统设计的案例非常具有启发性：它先梳理了 Web2 支付链路（用户 $\\rightarrow$ 电商 $\\rightarrow$ 支付服务 $\\rightarrow$ 银行），然后展示如何将其升级为支持稳定币的 Web3 支付——核心在于用“区块链”替换“银行机构”，并引入“链上监听服务（Listener）”来处理交易确认。这让抽象的请求流和资金流变得非常具体。安全设计与钱包矩阵：安全应该“源于设计”，贯穿整个生命周期，审计只是某个时间点的第三方检查，不能代替安全本身。会上强调：“如果你无法解释系统的行为，就无法保证资产的安全。”此外，会上系统梳理了钱包的分类，从控制逻辑上区分了 EOA 钱包、合约钱包、AA（账户抽象）钱包、多签钱包，从物理和隐私上区分了硬件钱包、隐私钱包、分层钱包，并明确了私钥作为所有权凭证在“构造-签名-广播”生命周期中的核心地位。2. 一个我准备采取的行动行动计划：拆解并动手实现一个“Web3 链上监听服务（Listener）”原型既然 Web3 支付系统的核心演进在于将传统银行替换为区块链与链上监听，我准备在课后基于 Web2 的基础，使用 Node.js 或 Go 语言结合 ethers.js / web3.js，动手写一个简单的链上交易监听脚本。具体步骤包括：部署或选择一个测试网上的稳定币合约，编写 Listener 去实时捕获该合约的 Transfer 事件，并在本地 Web2 数据库中更新模拟电商订单的状态。通过这个实际编码行动，把讲座中提到的“请求流与资金流的对接”彻底吃透，并顺便验证 AI 在构造此类监听服务时的代码质量，练习自己的“审核与验收”能力。3. 一个我还想继续追问的问题继续追问：在追求极致用户体验的 AA（账户抽象）钱包或 Web3 支付场景下，如何平衡“高 Web2 含量”带来的中心化安全风险？会上提到 Web2 的前端和状态管理决定了体验，且 AA 钱包等技术也是为了优化用户门槛。然而，Web2 含量越高（例如依赖中心化的 Relayer 转发交易、依赖中心化的服务器展示链上资产状态），意味着引入的中心化信任假设和攻击面就越多。我想继续追问：在实际的工业级大厂架构中 prioritize 体验的同时，有哪些成熟的行业标准或密码学方案（如 ZK 证明、多方安全计算 MPC），能让这些负责“体验”的 Web2 组件即便遭到黑客篡改，也绝对无法威胁到用户链上资产的资产安全（即如何做到“不可解释的系统行为绝对无法发生”）？

五大方向构成 Agent 经济栈

1\. 去中心化 AI 基础设施（资源层）

核心问题：AI 生产要素（算力、存储、数据、模型推理、服务）怎么被组织。

关键理解：主流路径为链下计算，链上激励/结算/验证/服务发现；中心化云是平台化资源市场，而Bittensor、Render、Akash是更开放的资源市场实验；关键难点在于质量验证、反作弊、稳定性、价格效率及企业级SLA。

项目案例：

\- Bittensor（激励网络）：Subnet 定义任务 → Miner 提供服务 → Validator 评分 → 链上共识 → 优质者获奖励

\- Render Network（GPU）：闲置 GPU 进入开放市场

\- Akash（云计算）：去中心化云计算市场

\- Filecoin / Arweave（存储）：AI 数据资产长期存储

\- [Fetch.ai](http://Fetch.ai) / ASI（服务网络）：AI agent 之间交易服务

2\. AI Agent + 钱包（行动层）

AI 拥有链上身份、资产账户、支付能力和执行权限。案例：Coinbase AgentKit

3\. AI 链上分析工具（认知层）

把链上数据（地址、资金流、项目风险）翻译成人能理解的判断。案例：Arkham

4\. 智能钱包和安全助手（用户保护层）

在签名/授权/转账前，帮助识别真实后果。案例：Blockaid — 交易模拟 + 欺诈识别

5\. 交易所和 DeFi 智能风控（机构风控层）

用模型、规则、链上监控发现异常交易、攻击和系统性风险。案例：Chainalysis

技术栈分层

底层资源（算力/存储、数据/模型） → 经济账户（钱包/身份、支付/权限） → 认知工具（链上分析、资金流解释） → 安全防线（欺诈识别、签名前保护） → 系统风控（KYT/预警、攻击监控）

核心观点

\- 资源：AI负责模型推理、数据处理；Web3负责算力/存储市场、激励分发。

\- 账户：AI负责理解意图、决策；Web3负责身份认证、支付、权限。

\- 认知：AI负责分析链上数据；Web3负责可验证数据来源。

\- 安全：AI负责风险识别；Web3负责透明审计。

\- 风控：AI负责异常检测；Web3负责链上监控。

1.  Web3 的核心本质并不是单纯的软件技术更新，而是互联网规则和控制权的改变。通过今天的课程，我发现它其实是三门学科的结合：利用密码学来确保数据的真实性与所有权，利用 Gas 费和惩罚机制等经济手段来规范用户的行为，最终在社会学层面实现大家共同制定规则的去中心化管理。它相当于把原本由大公司掌控的信任，变成了由底层代码和数学公式来保证。
    
2.  准备亲自动手验证一下公链的数据开放性。结合课堂上讲到的主流区块链“公开账本和伪匿名”的特点，以及不同应用之间可以互相调用的特性，我打算利用课后时间，自己找几个公开的区块链数据接口，写一个简单的程序来读取和分析一段链上的交易记录。通过亲眼看到资金是如何在各个不同的平台之间流转的，来真正搞懂它的公开可验证性，同时也感受一下数据完全公开所带来的隐私问题。
    
3.  面对应用之间互相嵌套带来的连带风险，开发者应该如何做好防范？ 课程中提到，区块链上的各种程序和接口可以像积木一样互相拼接和调用，但这也会带来很大的安全隐患——一旦最底层的那个基础程序出现漏洞或遭到攻击，上面所有依赖它的应用都会跟着一起倒霉。对于资源有限的个人开发者或者小团队来说，在设计自己的程序时，应该怎么用最简单、最有效的方法来做安全隔离？有没有什么现成的警报或切断机制，能确保在别人的代码出问题时，自己的程序不会受到牵连？

今天这场分享的信息密度很大，涵盖的议题比较多，从开发心态到具体技术架构，再到链上交互的细节，很多观点需要记下来反复琢磨。

AI与开发者的关系，这个话题上的一些观点，纠正了我之前的一些想法。之前有人会觉得有了AI写代码，人可以少学原理，但会上tc老师提出的观点是，AI时代对人的要求反而更高了。AI改变了开发的方式，确实写代码更快，学习新东西也更容易，它极大减少了精力消耗，相当于代替了人的手和脚去执行任务。但AI没有改变的是系统的复杂度。人依然需要负责设计方案，AI负责编码执行，而AI编码质量的关键在于它能够进行自我验证。如果开发者自己缺乏基础知识和架构能力，就根本不知道AI生成的代码对不对，方向有没有偏。一个很现实的问题是责任归属，如果AI生成的代码最终出现安全风险，担责的还是开发者自己。所以开发者必须确保AI在可控范围内，对所有输出进行审核和验收。

关于Web3与Web2的关系，Web3并不是与Web2完全独立的体系，大多数Web3项目中，Web2的能力占比相当高。像币安、Bybit这类交易所，虽然核心业务涉及加密资产，但实际上它们的Web2含量非常高。没有前端页面和后端状态管理，项目就无法提供良好的用户体验。区块链只是其中负责价值流转和共识的那一层，不能割裂地去看。要在Web3方向走得长远，Web2的功底必须稳固。

安全设计的理念是反复被强调的重点。有一个原则贯穿始终，安全应该源于设计，贯穿项目整个生命周期，而不是事后补救。审计只是锦上添花，是某个时间点的第三方检查，不能等同于安全本身。Web3项目因为直接涉及资金交互，安全性更加敏感。会上提到一个很根本的衡量标准，如果你无法解释系统的行为，就无法保证资产的安全。这其实也对开发者提出了更高的要求，对每一个状态变化和资金流向都必须有清晰的理解。

Web3支付系统设计的例子，用一个具体案例串联了很多概念。先梳理了Web2支付流程里各个参与方之间的交互，包括用户、电商平台、支付服务、银行机构这条链路。接着讲解怎么把这个系统升级为支持稳定币的Web3支付，核心区别在于把银行机构替换为区块链，同时增加链上监听服务来处理链上交易的确认。这个案例让抽象的概念落到实际架构层面，理解了请求流和资金流之后，再去想象链上部分的对接，会清晰很多。

钱包的分类在这次分享里讲得比较系统。钱包是Web3中身份和资产管理的核心工具，形式上分了很多种。按类别记下来，EOA钱包是私钥直接控制的外部账户，合约钱包把逻辑写在智能合约里，AA钱包做账户抽象以改善体验，多签钱包适合多人共管，硬件钱包把私钥隔离在物理设备里，隐私钱包隐藏交易细节，分层钱包从一个种子派生大量子密钥。托管方式也有自托管、全托管和混合托管之分，差别在于私钥归谁管。用什么类型的钱包，取决于安全需求和使用场景。

私钥是所有权凭证，不能暴露，签名用私钥完成授权而不泄露私钥本身，任何签名前都必须仔细核对内容。交易从构造、签名到广播上链的整个生命周期，每一步都依赖于私钥安全、合约透明和网络共识。
