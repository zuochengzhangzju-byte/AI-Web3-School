-   **Agent Memory 的概念与重要性**：
    
    -   **概念定义**：agent memory 是一个跨越时间的意图和状态的管理系统，有面向人类用户和大语言模型的部分，还有自身 runtime。
        
    -   **重要性体现**：在 ToC 产品中，如基于深圳小智 AI 硬件的产品，若没有 memory，约 45% 的用户会流失；在软件工程领域，有助于解决项目一致性问题，实现长时段运行。
        
-   **产品案例分析**：
    
    -   **ChatGPT**：有 save the memory 和 chat history 两层 memory，需保存用户偏好、有意义的上下文并让用户控制上下文和 memory；在不同时间节点有版本更新，如 2024 年 2 月 13 号发布 beta 测版的 member，2024 年 9 月 5 号向付费和企业级用户推出，2025 年 4 月 10 号 plus 版本中 Chat history 和 self - save memory 显示。
        
    -   **Code**：早期未上线 memory 系统，因软件过程有既有准则和最佳实践约束大模型；主要目的是解决工程中项目推进和长时间运行、避免代码漂移问题；2025 年开始有较多迭代，推出 memory 功能。
        
    -   **小龙虾**：记忆扁平化，执行链路难，失败概率高；虽有移动办公潜力，但因 memory 架构没做好，在受追捧后暴露出巨大痛点。
        
-   **Agent Memory 的技术栈**：
    
    -   **面向领域**：有面向传统数据库、增强、散文工程或散文管理三个面向，涉及 memory 的 owner、书写和检索等不同领域解决方案。
        
    -   **层级与工程**：memory layer 是一个层级，memory engineering 是整个 history engineer 中重要一环，处理不好可能导致失败。
        
    -   **Benchmark**：市面上至少有四种关于 agent memory 的 Benchmark，作者更看重召回指标，因为只有解决记忆召回问题，用户才能真切感受 agent。
        
-   **记忆分类与处理**：
    
    -   **分类方式**：从时间维度分为 working memory、short term memory、long term memory（有的分两层）；从类型维度分为语义记忆（记住陈述中的事实）、事件记忆（关于用户经历的记忆）、程序记忆（agent 的技能、workflow）。
        
    -   **处理问题**：涉及记忆的检索、提取、摘要、压缩、整合、解压等功能学问题。
        
-   **相关概念辨析**：
    
    -   **提示词工程**：与裸模交互，通过参数和切分形式约束提示词，清晰传达意图。
        
    -   **上下文工程**：提供更多上下文窗口，让大模型看到更多内容。
        
    -   **Mori**：对应上下文层，决定人类意图、agent 空间调用、人类知识和 agent 状态。
        
-   **生态情况**：
    
    -   **开源架构**：如 Memcached 等开源架构，不同公司有不同技术路线看法。
        
    -   **创业公司目标**：所有 memory 层的创业公司都想做成通用的 memories，但目前技术和项目处于观望状态。
        
-   **任务**
    
    -   **架构搜索任务**：在开源社区换关键词搜索，查找与己方团队相似且可借鉴的服务架构，同时关注监控信息是否有可学习之处。

x402 规定了 Agent 怎么付款，但没规定：

-   人给 Agent 授权多少额度？在哪里设定？
    
-   Agent 付了什么人能看到？多清楚？
    
-   Agent 要超限了，怎么冒泡问人？
    
-   出错了，"收据"（Signed Receipt）怎么变成人能读懂的纠纷证据？
    

这四个问题是纯 UX 问题，不是协议问题。协议开发者不擅长，评委最关注。

## **最关键的一个扩展：**`upto` **scheme**

这是你要重点看的。不是 `exact`（固定价格），而是：

> **买方预授权一个上限，卖方按实际用量收费。**

像打表计费的出租车——你上车前说"最多给 $10"，下车按实际里程付。

这意味着 Agent 在执行任务过程中，可以动态消耗服务，不用每次问人。但"$10 上限"是谁设的？在哪设的？能不能针对不同类型的服务设不同上限？能不能临时提额？**这是你作为 UX 设计师要设计的核心界面。**

## **还有一个宝：Bazaar（发现层）**

x402 文档里有个 Bazaar 扩展——Agent 怎么"找到"可用的付费服务。机器可读的服务目录，像 App Store 但给 Agent 看的。

**产品想象**：一个 Agent，预算 $5，目标是"帮我研究某个链上项目"。它自主去 Bazaar 找合适的数据服务，比价，付最便宜的，完成任务，给你一张人话收据。人只做一件事：设初始预算。

这就是 Agentic Commerce 的完整闭环，4 周能做出来的最小版本。

![Snipaste_2026-05-20_17-31-15.png](https://raw.githubusercontent.com/IntensiveCoLearning/AI-Web3-School/main/assets/lh1564803535-code/images/2026-05-20-1779282340893-Snipaste_2026-05-20_17-31-15.png)

ERC-8004 讲的是 Agent 身份和信誉。你带一个具体问题去：

> **"Agent 历史上在 x402 上付过多少钱、付给谁，这个记录能不能作为信誉依据？一个'花钱靠谱'的 Agent 和一个新 Agent，人给他们的预授权上限应该不同吗？"**

这个问题把 x402（支付）和 ERC-8004（信誉）连起来，是 Agentic Commerce 里最有意思的交叉点。课上能问出这个，你在组队时就有了清晰的产品角度。
