# **EOA、智能账户、多签账户：权限与安全边界比较**

理解三类账户的核心，就是回答三个问题：**谁能发起交易？谁能批准？谁承担风险？**

## **1\. 横向对比表**

| 维度 | EOA（外部拥有账户） | 智能账户（Smart Account，如 ERC-4337） | 多签账户（Multi‑Sig，如 Gnosis Safe） |
| --- | --- | --- | --- |
| 谁持有控制权 | 单一私钥持有者 | 合约规则决定（可以是用户、设备、或特定条件） | 多个签署方（例如 3/5） |
| 谁可以发起交易 | 私钥持有者（单方） | 用户、或者自动化策略触发 | 任何一个有签署权限的成员可以发起提案，但未批准前不会执行 |
| 是否支持多人确认 | ❌ 不支持 | ✅ 支持（通过社交恢复或授权逻辑实现） | ✅ 核心功能（必须达到阈值） |
| 是否支持恢复、限额、自动化策略 | ❌ 不支持（私钥丢失 = 账户丢失） | ✅ 支持：每日限额、会话密钥、两步恢复、自动定期支付 | ✅ 有限支持：可编程限额和条件，但通常需要链上提案-批准流程 |
| 典型使用场景 | 个人新用户、临时钱包、高频小额、低价值交互 | 高频DeFi交互、游戏内交易、需要账户恢复的用户 | DAO金库、团队资金管理、高净值个人冷钱包交互 |
| 主要风险点 | 私钥单点故障：被盗或丢失则账户永久失控 | 合约漏洞风险；恢复机制若被攻击可能接管账户 | 合谋风险（多数签署方作恶）；治理僵局（成员失联）；gas 成本较高 |

* * *

## **2\. 谁来发起交易？**

-   **EOA**：只有私钥持有者。发起即执行，没有“待确认”状态。
    
-   **智能账户**：可设置为“用户手动发起”或“预设条件自动发起”（如每周末定投、余额低于阈值自动补充）。发起交易后，可根据策略无需人工二次确认（例如用会话密钥）。
    
-   **多签账户**：**任何人**（包括非签名者）都可以发起一个交易提案，但只有达到阈值数量的签名者**人工确认**后交易才会执行。
    

> **哪些操作必须人工确认？**
> 
> -   EOA：每一笔交易（因为一次签名即代表批准）。
>     
> -   智能账户：取决于策略。修改恢复条件、转移大额资产、改变限额 → 通常必须人工确认；小额的重复动作可委托给自动化。
>     
> -   多签账户：确认交易提案（每一笔执行前的最后批准必须由人类签名者确认，不能绕过阈值）。
>     

* * *

## **3\. 谁承担风险？**

-   **EOA**：完全由私钥持有者承担。私钥泄露 → 资产全失；没有客服或恢复机制。
    
-   **智能账户**：风险分散在合约逻辑和用户设置的恢复条件中。若设置了社交恢复（比如 3 个监护人），则监护人的可靠性成为新风险。若合约有漏洞，也可能被批量攻击。
    
-   **多签账户**：风险由签名者集合共同承担。需要防止多数签署方作恶（例如 3 人中的 2 人串通盗取资金）。同时，若签名者丢失所有设备且无备份，账户也可能僵死（虽然可设计备用恢复方案，但非默认）。
    

* * *

## **4\. 一句话总结安全边界**

-   **EOA**：安全边界就是私钥本身。一旦越过（被盗），没有第二道防线。
    
-   **智能账户**：安全边界是**合约规则 + 恢复机制**。可以动态调整，但复杂度增加攻击面。
    
-   **多签账户**：安全边界是**投票门槛**（如 n/m）。攻击者需要控制至少 m 个独立签名者，但门槛越高日常操作越繁琐。
    

* * *

## **5\. 选择建议**

-   **初次使用区块链、资金 < $500** → EOA（但要小心备份助记词）。
    
-   **需要账户恢复、或让孩子/企业员工有限使用资金** → 智能账户（配合 session key）。
    
-   **持有 DAO 金库、团队运营、个人高净值冷钱包** → 多签账户（至少 2/3 或 3/5）。

**1\. 什么是加密货币？**

-   **定义**：由**区块链账本**保障的**交换媒介**。
    
-   **核心创新**：无需依赖可信第三方（如银行）来维护交易账本。
    
-   **第一个加密货币**：**比特币**（2009年由中本聪创造）。
    

**2\. 什么是以太币 (ETH)？**

-   **注意**：常被混淆为“以太坊价格”，但以太坊是平台，ETH是它的原生加密货币。
    
-   **核心用途**：
    
    -   支付以太坊网络的**交易手续费（燃气费/Gas）**。
        
    -   合并（The Merge）后，用于主网的**验证**和**提议区块**。
        
    -   其他用途：DeFi 贷款抵押品、NFT 市场计价单位、商品/服务支付等。
        

**3\. 以太坊与去中心化应用 (dapp)**

-   以太坊允许开发者创建 **dapp**，共享有限的计算资源池。
    
-   **ETH 的作用**：为计算能力定价，防止资源被滥用（恶意程序耗尽ETH后会自动终止）。
    

**4\. 铸造 (Minting) 与销毁 (Burning)**

| 操作 | 方向 | 如何发生 | 目的/效果 |
| --- | --- | --- | --- |
| 铸造 | 创造新ETH | 协议奖励给验证者（提议区块、检查点等） | 激励验证者维护网络安全 |
| 销毁 | 永久移除ETH | 每笔交易的基础燃气费被燃烧 | 抑制区块生产者操纵交易，简化费用估算 |

-   **网络高需求时**：销毁的ETH可能 > 铸造的ETH → 抵消发行（通缩压力）。
    

**5\. 以太币的面额 (重点：Wei 和 Gwei)**

| 面额 | 含义 | 等于多少 ETH | 常见用法 |
| --- | --- | --- | --- |
| Wei | 最小单位 | 10⁻¹⁸ ETH | 技术实现、黄皮书计算 |
| Gwei | Giga-wei (十亿 Wei) | 10⁻⁹ ETH | 表示燃气费（人类可读） |

**6\. 转移与查询以太币**

-   **转移**：每笔交易用 `value` 字段指定发送金额（以Wei计）。发送到智能合约时，合约代码可用这些ETH支付Gas。
    
-   **查询余额**：通过账户的 `balance` 字段（以Wei计）。
    
-   **常用工具**：**Etherscan**、**Blockscout**（网页浏览器）、钱包或直接向节点发送请求。

**SHA-256哈希函数原理简述：**

1.  **填充（Padding）**：在输入数据末尾添加一个"1"，再添加足够多的"0"，使数据长度满足 `l + 1 + k ≡ 448 mod 512`，最后附加一个64位的数据长度表示。这样总长度成为512的整数倍。
    
2.  **分块（Block Decomposition）**：将填充后的数据按512位分成若干个消息块。
    
3.  **消息扩展（Message Expansion）**：将每个512位的消息块扩展为64个32位的字（W₀到W₆₃），用于后续压缩循环。
    
4.  **压缩函数（Compression Function）**：对每个块进行64轮迭代运算。每轮使用以下位运算：
    
    -   `Ch(x,y,z)` = 按位选择
        
    -   `Maj(x,y,z)` = 按位多数投票
        
    -   `Σ₀`、`Σ₁`、`σ₀`、`σ₁` = 循环移位和异或运算
        
5.  **输出**：所有块处理完毕后，8个32位的寄存器中的值拼接成最终的256位哈希值。
    

> 整个过程是**确定性的**：相同输入永远产生相同输出。但具有**单向性**：从输出无法反推输入

### **一、SHA‑256 哈希函数**

SHA‑256 是一种密码学哈希函数，具有以下核心性质：

1.  **输入任意长度**，输出固定长度（256 位，通常表示为 64 位十六进制数）。
    
2.  **确定性**：相同输入必产生相同输出。
    
3.  **雪崩效应**：输入极微小变化（如一个字符），输出哈希值会发生不可预测的巨大变化。
    
4.  **不可逆性**：无法从哈希值反推出原始输入。
    
5.  **抗碰撞性**：难以找到两个不同输入产生相同哈希。
    

> 实际应用示例：
> 
> -   输入 `"Anders"` → 哈希以 `19ea` 开头
>     
> -   输入空字符串 → 哈希为 `e3b0c442...`
>     
> -   输入任意其他内容 → 哈希完全不同
>     

哈希函数是区块链的**基础构件**，用于唯一标识数据内容。

* * *

### **二、区块的构成**

一个区块通常包含以下字段：

| 字段 | 含义 |
| --- | --- |
| 区块号（Index） | 区块在链中的顺序编号 |
| 随机数（Nonce） | 用于工作量证明的可调整整数 |
| 数据（Data） | 实际存储的内容（交易信息、文本等） |
| 哈希（Hash） | 上述所有字段的 SHA‑256 计算结果 |
| 前一区块哈希（Prev Hash） | 指向前一个区块的哈希值（链式结构的关键） |

区块的哈希值 = `SHA‑256(区块号 + Nonce + 数据 + 前一区块哈希)`

* * *

### **三、挖矿与工作量证明**

**挖矿**是指通过不断尝试 Nonce 值，使区块哈希满足特定难度条件（如哈希值前若干位为零）的过程。

-   难度条件示例：哈希值以 `0000` 开头。
    
-   由于哈希结果不可预测，只能通过**暴力枚举** Nonce 来寻找有效解。
    
-   一旦找到满足条件的 Nonce，该区块即被视为 **“有效”或“已签名”**。
    

> 示例：在演示参数下，Nonce = 59396 可使区块哈希以四个零开头。

工作量证明机制使得**生成有效区块需要消耗计算资源**，从而防止随意篡改。

* * *

### **四、区块链的结构与防篡改原理**

区块链是由多个区块**按时间顺序**组成的链表结构，其核心特征包括：

1.  **每个区块保存前一个区块的哈希值**（通过 `Prev Hash` 字段）。
    
2.  **首个区块**（创世区块）的 `Prev Hash` 为全零。
    

**防篡改机制：**

-   若**修改任意区块中的数据**，该区块的哈希值将发生变化。
    
-   由于后续区块中存储的是**原哈希**作为 `Prev Hash`，哈希不匹配将导致**从被修改区块开始，后续所有区块失效**。
    
-   要使修改后的链重新有效，必须**重新挖矿该区块及之后的所有区块**。
    
-   区块链越长，重新挖矿所需的总计算量越大，**篡改成本呈指数级上升**。
    

> 结论：区块链提供了一种**随时间累积的、计算上难以篡改的历史记录**。

* * *

### **五、分布式区块链与共识**

在实际系统中，区块链通常以**分布式网络**的形式存在：

-   网络中多个节点（A、B、C…）各自保存完整的区块链副本。
    
-   节点之间通过**比较区块哈希**（尤其是最新区块哈希）来验证一致性。
    
-   **共识规则**：当多数节点的链版本一致时，该版本被视为有效；少数不一致的节点将被覆盖或同步。
    

> 优势：
> 
> -   无需逐块验证，仅通过哈希比对即可快速发现篡改。
>     
> -   单点失败或恶意修改无法影响整个网络。
>     

* * *

### **六、基于区块链的简单加密货币**

将区块中的“数据”字段替换为**交易记录**，即可构建一个基本的加密货币系统。

**交易示例：**

text

```
Darcy → Bingley   $25.00
Elizabeth → Jane  $4.27
```

**重要设计原则：**

-   系统**不保存账户余额**，只保存**转账历史**。
    
-   验证某人是否有足够余额：**回溯所有相关交易**，计算其净收入。
    

**防双重支付与防篡改：**

-   修改任何历史交易 → 该区块哈希改变 → 破坏后续链条 → 与其他节点副本不一致 → 被共识拒绝。
    

* * *

### **七、Coinbase 交易（新币发行）**

为了在系统中引入初始资金，每个区块可包含一笔特殊的 **Coinbase 交易**：

-   作用：**凭空创建新币**，奖励给特定地址（如矿工或系统指定账户）。
    
-   示例：`"Coinbase → Anders $100"`
    
-   规则：通常每个区块仅允许一笔 Coinbase 交易；新币发行总量受控。
    

> 没有 Coinbase 交易，则无人拥有初始资金，系统无法启动交易。

* * *

### **八、总结：区块链的核心价值**

| 特性 | 说明 |
| --- | --- |
| 去中心化 | 无单一控制节点，所有副本平等 |
| 防篡改 | 哈希链 + 工作量证明 + 多数共识 |
| 可追溯 | 完整历史记录，可用于审计 |
| 适用场景 | 加密货币、供应链、存证、投票等 |

> 一句话定义：  
> **区块链是一种通过哈希链和工作量证明实现不可篡改、去中心化共识的分布式账本技术。**

L1与L2的核心区别在于安全性与执行成本的权衡。Layer 1如以太坊主网提供最高安全性但Gas费昂贵，Layer 2如Arbitrum和Optimism将交易批量处理后提交到L1，从而将单笔执行成本降低10到100倍。开发中应根据操作价值选择：高频或低价值操作适合L2，高价值结算才需要L1的直接安全性。

智能合约与普通后端逻辑有本质不同。普通后端的状态存储在私有数据库中，执行逻辑是黑盒运行且可以随时修改代码。而智能合约的状态完全公开存储在链上，任何人都可以读取；其字节码公开可验证，升级需要通过代理模式或合约迁移，部分合约如Uniswap V2甚至完全不可更改；权限控制通过onlyOwner等modifier在链上公开检查。这意味着智能合约更适合作为可审计、可组合的价值承载逻辑，而非复杂多变的业务系统。

主网与测试网必须区分使用。主网使用真实资金进行生产环境交易，任何错误操作都会导致真实损失。测试网如Sepolia可以通过水龙头免费领取测试代币，适合学习、原型开发和自动化测试。黄金规则是所有智能合约和AI代理行为都必须在测试网完整验证通过后，才能部署到主网。

区块浏览器、钱包提示和交易回执是理解链上行为的三个关键工具。区块浏览器如Etherscan可以查看地址余额、交易状态、内部调用和事件日志，重点关注To地址、Value金额、Gas Used消耗量、Logs日志和Status状态码。钱包如MetaMask会在签名前模拟交易结果，提示风险比如授权过多或调用未知合约。交易回执包含transactionHash、logs、gasUsed和status字段，其中status为1表示成功，0表示失败。如果交易失败，先查看gasUsed是否接近区块上限，再检查logs中的revert原因，多数问题出在授权不足或断言失败。

账户抽象、智能账户、多签、Safe、ERC-4337和OpenZeppelin Contracts共同构成了AI与Web3融合的关键基础设施。ERC-4337账户抽象将签名验证与执行解耦，支持非外部账户发起交易，使AI可以拥有智能账户而无需直接管理私钥。Safe多签钱包需要多个密钥共同批准才能执行交易，例如2-of-3模式，AI可以作为其中一个签名者但不能单独控制资金。OpenZeppelin Contracts提供了AccessControl、Ownable、ReentrancyGuard等经过实战检验的标准模块，降低了AI编写合约的风险。这些组件之所以成为关键基础设施，是因为它们解决了三个核心问题：AI不能安全持有私钥所以需要智能账户加会话密钥，AI行为需要受到限制所以需要限额和白名单，误操作或攻击后需要可恢复性所以需要社交恢复和多签控制。

从传统钱包到AI原生账户的演进核心是恢复机制、授权模型和安全边界的重构。传统的外部账户钱包依赖私钥或助记词作为唯一控制凭证，丢失即永久失去资产，完全不适合AI或自动化系统，因为一旦私钥泄露AI就可以无限操作。智能账户引入了多项关键能力：社交恢复允许监护人联名恢复账户而无需助记词；邮件恢复通过链下身份验证加链上时间锁实现恢复；会话密钥可以授予临时密钥特定权限，比如一周内每天最多转账0.01 ETH；权限限额可以按时间、次数、金额或调用的合约方法来限制操作；人工确认机制确保关键操作必须获得人工签名授权才能执行。这套体系让AI代理能够在受限、可监控、可恢复的边界内自主行动，真正实现AI与区块链的安全融合。

3个关于web3应用落地的想法

核心想法是病历上链，通过患者签名来控制谁可以查看病历。基于这个逻辑，目前想到了三个比较落地的应用场景：

## **我的Web3学习笔记：从技术基石到个人主权**

### **一、核心理念：个人主权**

-   **定义**：个人完全掌控自己的资产、身份和数据，无需依赖第三方机构。
    
-   **体现**：持币即拥有，数据归己，操作无需许可。
    
-   **代价**：**私钥丢了就是真的丢了，没有任何客服能帮你找回。** 这是自由的成本。
    

### **二、经济模型：Gas费**

-   **本质**：在区块链上执行计算或存储数据所需支付的燃料费，以原生代币（如ETH）支付。
    
-   **关键作用**：**有效抵御DDoS攻击与垃圾交易**。因为发送每笔交易都有成本，所以网络很难被无效请求堵塞。
    

### **三、技术基石：从共识到验证**

**1\. 共识机制：账本由谁写？**

-   **PoW（工作量证明）——以比特币为代表**
    
    -   **逻辑**：算力竞争，谁先算出难题谁有权写账本。
        
    -   **特点**：耗能极高，但**去中心化程度极好**，攻击成本与设备、电力成本直接挂钩。
        
-   **PoS（权益证明）——以以太坊为代表**
    
    -   **逻辑**：质押代币成为验证者，按质押比例获得记账权。
        
    -   **特点**：节能高效，但有人认为**富者更易主导**，存在一定中心化风险。
        
-   **逻辑提炼（CZ的观点）**：无论技术发展到哪一步，**加密总是比解密容易**。这是保障安全的基本不对称性。
    

**2\. 关键组件：验证器（Validator）**

-   **角色定位**：PoS网络中负责打包交易、验证区块、维护网络安全的核心节点。
    
-   **与Gas的关系**：用户支付Gas给验证器作为劳动报酬。
    
-   **与你笔记中**`validiter`**的关联**：你记下的`validiter`是Rust语言中用于**数据验证**的工具库，它负责逻辑层面的“条件检查”；而共识机制中的验证器负责**账本真实性**的检查。一个管“对不对”，一个管“真不假”。
    

### **四、未来挑战：量子计算的威胁与对策**

-   **风险**：量子计算机未来可能轻松破解目前保障比特币、以太坊安全的椭圆曲线算法，从而**反向推导出私钥**。
    
-   **对策——PQC（抗量子密码学）**
    
    -   **核心思路**：用**新的数学难题**取代旧的（如大数分解）。
        
    -   **无需更换硬件**：只需**升级软件算法**，例如改用基于**格密码学**的算法。
        
    -   **当前状态**：全球机构和项目正在逐步推进PQC标准化，属于未雨绸缪的中长期升级。
        

### **五、社会实验：智能合约的意义**

-   **传统世界（人治）**：我们信任机构和人，但规则可以临时更改，解释权不透明。
    
-   **Web3世界（代码+共识）**：
    
    1.  **规则可见**：代码开源，写在区块链上，任谁都可以查看。
        
    2.  **难以篡改**：规则部署后，几乎没有临时修改的空间。
        
    3.  **信任转移**：从信任“人/机构”转向信任“公开代码”和“多方共识”。

1、account：

在区块链上记录资产和状态的实体，分为外部账户和合约账户，外部账户EOA由私钥控制，合约账户由代码控制

在MetaMask创建的账户，可用来接受别人的转账ETH或交易所体现，是外部账户；USDC代币合约，记录了所有地址的USDC余额和转帐逻辑，是合约账户

提醒：删掉钱包不等于删掉账户，账户在链上，只要私钥在账户即可恢复

2、adress

账户的唯一标识符（通常以 `0x` 开头的 42 位字符串），类似银行账号

`0xd8dA6BF26964aF9D7eEd9e03E53415D37aA96045`（Vitalik 的知名地址）

提醒：账户只能用来收款，不可凭地址控制或动用资产

3、private key

私钥，一个 64 位十六进制随机数，用来签名交易、证明账户所有权

`0x4c0883a69102937d623b1476c…` 一个私钥对应一个地址的唯一控制权

提醒：私钥等于账户的完全控制权。常见骗局：假“矿工验证”、假“空投领取”要求你粘贴私钥，不可相信

4、wallet

用来管理私钥和地址、发起交易的工具（软件如 MetaMask，硬件如 Ledger）

用 Rabby Wallet 在测试网上发送 1 枚 Sepolia ETH

提醒：钱包不能存钱，钱在链上，钱包只是钥匙串

5.signature

用私钥对一段消息生成的密码学证明，证明你同意而不泄露私钥

在MetaMask转账，需要用私钥进行签名，签名后的交易广播到以太坊网络，矿工验证签名有效后，才把交易打包上链

提醒：要看内容再签名，不要随意签名，可能导致NFT被盗

6.seed phase

助记词，一组 12–24 个单词，可推导出所有私钥，是钱包的主备份

初始化钱包时得到 `abandon art baby …` 共 12 词，用它在另一台设备恢复同一账户

提醒：不要截图助记词，不要保存在电脑中，最好是写下来藏起来，谁有了助记词就可以拥有你的所有资产

7.transaction

交易，改变链上状态的操作，需签名且广播到网络

从A地址向B地址发送0.2ETH

提醒：交易一旦上链不可更改，发送到错误地址可能会永久丢失

8、testnet

测试网，模拟主网环境，代币无真实价值，用于模拟测试

Sepolia、Goerli（以太坊测试网），领取测试 ETH 来练习调用合约

提醒：测试网的私钥也不可随意泄露

9、gas

以太坊类网络中执行交易或合约计算的手续费，用 ETH 支付

发送 ETH 通常消耗 21000 gas；复杂合约调用需要几十万 gas

提醒：gas 不足交易会失败，但已消耗的 gas 不退。测试网 gas 免费（领水），但主网 gas 波动大。

10、block explorer

区块链的搜索引擎，查询交易、地址、合约等链上公开数据

Etherscan（以太坊）、Sepolia Etherscan（测试网）

验证交易是否成功、检查对方地址是否有可疑行为

11、smart contract

智能合约，部署在链上、自动执行的代码，无主观意志，所有人都可调用

Uniswap 的 V3 池子合约，自动完成代币兑换

合约不可升级是误解——很多合约通过代理模式升级，用户需留意是否有“管理员后门”

12、EOA:外部账户，私钥控制的普通账户

Smart account：合约逻辑控制，如可编程钱包（Safe 的模块化账户）

multisig：多签账户，需 M 个签名中的 N 个才可执行交易，通常用合约实现

一个 2/3 多签钱包：3 个人持有私钥，至少 2 人批准才能转出资金

提醒：多签能防止单点私钥被盗，但签名界面仍有盲签风险（成员可能集体被骗）

私钥和助记词等于账户的完全控制权。一旦泄露，攻击者可无需任何额外确认直接转走你所有资产，且无法撤销。常见陷阱包括：假客服要求"验证钱包"、钓鱼网站诱导粘贴私钥、在GitHub等公开仓库误上传助记词文件。签名是使用私钥对消息的密码学证明，但一次恶意盲签（不看消息内容就确认）可导致攻击者获得转账授权、批准NFT挂单骗局，或登录钓鱼网站。授权（approve）则是允许某合约操作你的代币，即使你不主动转账。典型骗局是假冒Uniswap界面让你无限授权（unlimited approve），之后合约瞬间吸走所有USDT或代币。核心防御原则：私钥和助记词永不触网（不复制、不输入、不拍照、不存云端）；签名前务必读懂消息内容（使用硬件钱包或解析工具如WalletGuard）

1、LLM

大语言模型，基于海量文本训练、能理解和生成自然语言的深度学习模型

chat-gpt，deepseek

提醒：LLM 不是知识库，也不具备真正的“理解”或“推理能力”——它是在做概率性的下一个词预测，可能编造事实

2、prompt

提示词，给 LLM 的输入文本或指令，用来引导模型生成期望的输出

用 Python 写一个读取 CSV 文件并计算平均值的函数，其中python、读取csv文件、计算平均值、写函数，都是提示词

提醒：提示词并非越多越好，过度约束反而会局限答案的生成

3、workflow

工作流，将任务拆解为多个步骤，由LLM按照顺序调用不同组件完成

一个“论文润色 workflow”：步骤1提取论点 → 步骤2检查逻辑漏洞 → 步骤3优化句式 → 步骤4输出最终版本，每一步调用不同 prompt

注意：workflow 不是让 LLM 一次做完一切，而是通过流程工程降低复杂度和错误率；但过度拆分也会增加延迟和复杂成本

4、agent

智能体，能自主规划、调用工具（如搜索、代码执行）、循环迭代直至完成目标的 LLM 驱动系统

给 Agent “帮我查北京天气并发送到我的 Slack”——它会：调用天气 API → 获取数据 → 调用 Slack API → 发送消息，无需每一步都由我决策

5、tool use

工具调用，LLM 或 Agent 通过生成特殊格式的输出来触发外部函数/API 执行，并将结果返回继续推理。

如，你说“2 的 10 次方加 5 等于多少”，模型不直接算，而是调用 `calculator(2**10)` 工具得到 1024，再加 5 输出 1029

提醒：tool use 不是模型“主动学会用工具”，而是工程上预先定义好工具描述和调用格式；模型可能虚构工具、调用错误参数或过度调用，需加验证层

6、tracing

追踪，记录 LLM 应用的每一次调用链条（输入、输出、中间步骤、token 消耗、延迟），用于调试、评估和优化

你写了一个 Agent 查询数据库，tracing 系统会显示：prompt→调用 search\_tool→返回结果→第二次 prompt→最终输出，并标明每步耗时

提醒：tracing 不是仅记录日志，而是需要结构化（如 OpenTelemetry）以支持搜索和可视化；缺失 tracing 时，Agent 出错几乎无法调试
