# **AI Agent 的最后一公里：从会聊天，到能授权地花钱**

**问题不在会不会付，而在该不该付**

授权：Agent 请求付费时，必须确认这是用户真实意图，谁给了支付权限。

责任：买错、超预算或结果异常时，系统必须说明谁下单、谁执行、谁验收、谁仲裁。

预算：必须限制金额、支付对象、时间窗口和使用次数，不能把一次请求扩成长期权限。

结论：真正的难点不是支付通道本身，而是把支付动作放进被授权、可控、可追溯的闭环。

**统一支付入口与策略层**

x402：服务端返回 402 Payment Required，声明资源价格、收款方和付款要求，Agent 付费后清算结算。

CAW / Pact：限制代币、金额、时间窗、次数和审批阈值，让支付凭据过流程，解决该不该付、最多付多少。

审计：request、pact、payment、delivery 留下可回溯记录，确保错误可追溯、争议可仲裁。

一句话：x402 解决怎么付，CAW / Pact 解决该不该付、最多能付多少，以及出了问题去哪里查。

**Harness 实操准备**

Agency（感知、推理、行动的能力）来自模型训练，而非外部代码编排。

一个可工作的 Agent 产品 = 模型（司机）+ Harness（车辆）。

本教程教授如何构建 Harness。

一个 Agent 请求付费 API 时，是先触发 x402 再经 Pact 校验，还是 Pact 先生成授权凭据再触发 x402 结算？

## **From Copilots to Autonomous Agents**

聊天机器人：AI回答问题，人类完成工作。

副驾驶：AI建议行动，人类批准每一步。

代理：AI执行工作流，人类定义范围。

自主代理：AI可以持续以机器速度转移资金。技术已就绪，但信任基础设施尚未就绪。

## **真实世界的警示**

静默篡改：代理被提示不要修改金额，但仍在中途静默执行了修改。仪表盘显示成功，后端无痕迹。文字表达意图，但不能强制执行权限。

影子托管：代理在所有者托管边界之外创建了新钱包并转移资金。无策略引擎、审计追踪或终止开关。拥有原始钱包访问权限的代理没有架构理由停留在边界内。

## **当代理转移资金时，信任成为基础设施**

提示注入：对抗性输入可引导代理执行未授权操作。

无范围权限：宽泛的钱包访问权限将单次漏洞利用变成全面暴露。

影子操作：代理可在所有者视野之外创建账户和执行路径。

僵尸权限：过期的密钥和永不过期的特权成为持久攻击面。

## **控制如何维持**

当前默认做法：给代理钱包，希望不出问题。

实际需要：给代理有范围的授权，并将执行绑定到授权上。不要给代理钱包，给它们“契约”（Pacts）。

## **Cobo Agentic Wallet 的三个行业首创**

MPC for AI Agents：私钥在用户、代理和Cobo之间分割，任何一方都无法单独转移资金。

Pact 授权协议：结构化委托，包含意图策略和自动到期机制，由基础设施强制执行。

Recipe 技能层：用于真实链上任务的已验证执行路径，不会产生幻觉的合约地址或ABI参数。

## **MPC 钱包的数学保证**

代理 + Cobo 组（2/2 阈值）：代理密钥分片 + Cobo 密钥分片共同签名。

人类 + Cobo 组（2/2 阈值）：人类密钥分片 + Cobo 密钥分片共同签名。

任何一方都不能单独转移资金。只有人类所有者可以完全恢复私钥。

## **Pact：由基础设施强制执行的人类-代理授权**

意图：代理被授权执行的具体任务或目标。

执行计划：代理完成工作的透明、可审查的路线图。

策略：预算、审批、白名单、代币/合约约束等由钱包强制执行的护栏。

完成条件：自动终止契约的条件（时间到、预算花完、工作完成）。

## **Pact 的生命周期**

阶段1：用户用自然语言陈述意图。

阶段2：代理起草契约（意图、执行计划、策略、完成条件）。

阶段3：用户在移动应用中审查和批准（查看计划、策略和风险，然后批准、调整或拒绝）。

阶段4：钱包在策略下执行，每笔交易都对照契约进行检查。完成后自动结束，密钥自行撤销。

## **Agent Memory Is a Cross-Time Intent-State Management System**

用户的意图起初是不完整的，只是大致方向。

通过澄清、决策、工具结果和中间产物，模糊的请求逐渐转化为可工作的状态。

记忆的作用是将这种状态跨时间传递，让下一轮或下一次会话能够继续。

## **Restart vs Resume**

好的记忆能减少用户重复解释的工作。

空白重启：系统每次从头开始，用户需要再次说明目标、约束和决策。

记忆状态：偏好、决策、任务历史得以保留。

恢复的工作：在需要时带回相关上下文，使下一次交互能从上一次有用的步骤继续。

## **ChatGPT 的记忆：用户连续性**

记住稳定偏好（语气、个人资料细节、重复习惯）。

引用有用的历史上下文，改善回复而不必让用户重述。

用户可查看、删除、暂停、重置记忆。

## **Claude Code 的记忆：项目连续性**

需要构建、测试等可重复命令。

记住代码库的规范、架构模式、本地工作流程。

记住从失败尝试、已解决决策和已知约束中获得的经验教训。

## **Claude Code 记忆的演进**

手动阶段：记忆以显式笔记和文件形式存在。

管理阶段：项目记忆可导入和编辑。

自动阶段：记录和召回成为产品行为，系统自行承载状态。

## **产品记忆随自主性变化**

聊天机器人：优化用户连续性，记住偏好，减少重复。

编码代理：优化项目连续性，保留项目规则，恢复未完成工作。

随着自主性提高，记忆需要选择性带回正确的状态，而非简单积累历史。

## **记忆不仅仅是表面连续性**

用户撰写的记忆：通过设置、编辑、命令或项目笔记直接创建或确认。

代理撰写的记忆：代理从决策、观察、结果中自行产生的记忆。

所有记忆都需要生命周期控制（选择、修订、遗忘）。

## **写、检索、修订**

撰写：记忆可由用户、文件或代理写入。

检索：仅在当前步骤需要时返回存储的记忆。

修订：当新证据出现时，存储的记忆需要变更（更新、合并、淘汰）。

## **回忆、检索、修订**

回忆：用户看到的产品“记得”的表面效果。

检索：系统将记忆载入活动上下文的具体操作。

修订：随时间改变已存储记忆的操作。

## **记忆不是检索**

检索是定位并返回相关信息或状态的操作。

RAG 是检索加生成的工作流。

记忆决定什么状态应该跨时间保留，并在未来影响行为。

## **五个术语，五个工作**

记忆：跨时间保留有用状态的系统能力。

检索：在需要时定位并返回相关信息的操作。

RAG：使用检索结果支持生成的工作流。

嵌入：将内容转化为可搜索的向量形式。

向量存储：存储和搜索向量的基础设施。

## **Agent Memory 的两个轴**

时间轴：当前工作上下文 vs. 跨时间的持久记忆。

内容轴：在持久记忆中，区分情景性（episodic）、语义性（semantic）、程序性（procedural）记忆。

工作暂存区位于时间轴的活动上下文中，不是第四种内容类型。

## **记忆工程是生命周期**

选择性捕获：不是所有历史都值得持久化。

选择性检索：不是所有持久化记忆都应该返回。

持续修订：旧记忆会过时或被取代，系统必须支持更新和遗忘。

## **Agent 栈中的记忆**

提示工程：决定“这次如何提问”。

上下文工程：决定“模型看到什么”，检索和记忆在此处喂入实时上下文。

框架工程：决定“系统如何运行”，包含循环、工具、策略、日志和生命周期操作。

记忆跨越多个层次，既影响模型所见，也影响系统行为。

## **记忆成为基础设施**

连续性最初是便利功能（减少重复）。

当代理在工具和流程中行动时，记忆开始塑造系统能安全执行的下一步。

当系统依赖记忆来实现连续性、协调和受控的未来行为时，记忆就成为基础设施。

私钥即主权 → 签名即授权 → 交易即指令 → 区块即账本 → 共识即规则。核心“资产不在钱包里”，钱包只保管私钥和构造签名，余额、NFT、合约状态全在链上账本。这意味着用户真正拥有的不是资产本身，而是“能移动账本上某条记录的私钥”。传统应用里拥有数据，Web3 里只拥有一把钥匙，数据在公开账本上谁都能看，但只有能改属于自己的那部分。

私钥体系的三层结构：助记词→ 私钥→ 地址。一个助记词可通过派生路径生成无数个私钥/地址，这就是一张助记词管理多链多账户的原理。私钥不是普通密码，丢了没人能重置，泄露等于交账户。

安全铁律：不截图、不网盘、不发人、不复制到不可信环境——剪贴板也不安全，恶意软件能直接替换地址。

交易的完整生命周期：用户视角的“转账”在技术上是包含六要素的数据包——你要做的事（转账/mint/投票/授权）、Gas费（为计算和存储付费）、Nonce（账户的第几笔交易，防止重放，也可用来加速/取消交易）、签名。

签名过程：私钥在本地签名后广播到网络（私钥永不离开本地），任何人可用公钥验证签名确实由该私钥授权，但几乎无法伪造。从发送到确认的路径：Wallet本地签名 → RPC/Node接收并传播 → Mempool（待打包交易池）排队 → Builder排序 → Validator出块/证明 → Block可查询。

确认需要等待的原因是：区块像账本新页，后续区块越多，推翻历史越难；Ethereum PoS 中最终确认finality约需 12.8 分钟2 个 epoch。Gas 的三重作用：挡垃圾交让占用公共资源有成本、给出块者激励（base fee 销毁，priority fee 给验证者）、让资源有价格，谁占用谁付费，免费资源最易被滥用。

共识机制：PoW（比特币）靠算力竞争记账——解一道很难但答案好验证的题，谁先解出谁记账；PoS（以太坊）靠质押——验证者质押 ETH，系统在活跃验证者中伪随机选取提议者和证明者，作恶或严重失职会被罚没。PoW 更“无准入”（有算力就能参与），但耗能且慢；PoS 更节能，但初始质押门槛可能影响去中心化程度。

之前学的 Hermes Agent 可以在这个体系里找到具体应用场景，解析到账后触发 Telegram 推送；或者用 Agent 定期调用 RPC 读取 mempool 里的 pending 交易，分析 Gas 价格分布来做成本优化。这比“问 ChatGPT 怎么写”更落地的原因是：真正理解了 Nonce、Gas、签名这些底层机制，Agent 调用 RPC 时才知道返回的字段什么意思、出错时怎么排查。

AI Agent 的本质进化与核心机制

AI 从“告诉我怎么做”（问答交互）向“直接帮我做”（任务执行）。AI Agent 的核心运行机制包含四个闭环步骤：理解目标、调用工具（终端命令、文件读写等）、观察结果、出错时自动修正。区别于聊天型 AI（ChatGPT、DeepSeek）和编辑器 AI 助手（Cursor、Copilot），终端型 Agent（如 Claude Code、Codex、Hermes）能够直接进入项目目录，读取修改代码、运行测试、修复 Bug，扮演“住在命令行里的自动化工程师”角色。

Hermes Agent 的定位与核心能力

Hermes 属于通用 Agent 系统，被定位为“AI 的基础基建”，而非单一用途工具。它的六大核心能力包括：多模型支持（灵活接入 GPT、Opus 等）、工具调用（文件、搜索、终端）、Cron 定时任务（自动监控/摘要）、Memory 记忆（记住偏好与工作习惯）、Gateway 消息网关（直接接入 Telegram 等）、Skills 技能（沉淀固定任务流程）。演示中我了解到了自动重试机制（API 失败后自动 retry）、终端交互命令，还有与 Telegram bot 的集成通过 @mention 触发任务。下一步我的想法是试用 Hermes Agent 完成一个具体开发任务可以是配置 Telegram bot 作为消息网关，让它自动读取项目目录、执行一个脚本并返回结果。

今日看昨天的回放

有了AI还需要学基础吗？人负责设计方案、评估合理性、验收结果；AI协助编写、检查和测试。基础不足就无法判断AI输出的质量。

Web3与Web2是完全不同的世界吗？Web3是更严格的工程系统，大部分项目仍大量使用Web2能力。优秀Web3开发者首先必须是优秀的系统工程师。

安全可以后续再考虑？安全不是插件，而是设计核心。审计只是“心理安慰”，安全需要从设计阶段就深度嵌入。

从Web2支付到Web3支付：Web2模型是电商平台 → 支付服务 → 银行机构（信任权威）。引入稳定币支付-将“银行机构”替换为“区块链”，请求流和资金流基本不变。

区块链的定位：去中心化账本 + 全局共享状态机。数据不可篡改、代码公开透明、共识协议（PoW/PoS/DPoS）决定信任模型。

钱包的本质：用户主权体现——私钥管理（绝对控制权）、签名能力（唯一合法身份）。通往Web3的唯一凭证。

钱包的安全风险：私钥泄露（不可逆）、签名欺骗（诱导签署恶意交易）、权限滥用（授权恶意合约）。

交易关键参数：Gas

动力源gas fee = gas count × gas price，以原生币支付。EIP-1559引入base fee/tip fee/max fee。

交易关键参数：Nonce

顺序守护者，防止重放攻击和乱序执行。不同链实现有差异，如Tron无nonce但兼容EVM。

链上监听与确认：支付服务通过pull方式同步链上数据，解析区块中的交易和事件，需等待足够多的区块确认（防分叉/重组）。

交易模拟Simulate：提交前验证执行结果，预测交易成败，避免资产损失。“如果解释不了一笔交易为什么失败，就不算真正理解Web3。”

服务端钱包安全：权限拆分MPC、Safe等多签方案+ 完善的Web2安全流程与风控 + Web3规则加固。大资金账号必须用多签。

AI改变了什么：效率革命，即Coding更快（AI辅助生成、多Agent编程）、学习更容易（全天候知识获取）、基础自动化（部署、初步审计）。

AI没改变什么：工程本质的坚守，系统复杂度（AI加速堆砌但架构设计仍是核心）、错误成本（AI错误同样导致资产损失，Web3容错为零）、安全边界（判断系统怎么设计靠人脑）。

AI是放大器，放大了架构思维与工程素养。当代码变得廉价，对系统全貌的洞察力变得前所未有的昂贵。驾驭AI的前提是成为系统底层逻辑的深度掌控者。架构能力（设计、实现、维护复杂系统）、Debug能力（深入底层协议）、基础能力（编程概念、分布式原理、密码学）、框架能力（Web3框架及AI工具）。
