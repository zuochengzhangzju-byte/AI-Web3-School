## 线上课程：Product manager of Cobo Agentic Wallet

### **Agent 经济活动**\----Agent 支付不等于“转账自动化”

让大模型自动调用一个转账按钮（传统支付的简单延伸）不仅不是真正的 AI × Web3，反而是极度危险的。

真正的 **Agent Commerce（智能体商业）** 是一个涵盖了**发现服务、协商价格、预算授权、执行任务、验收结果、处理争议和最终结算**的完整闭环可控链路。

-   **2023-2024 (Chatbots & Copilots)：** AI 提出建议，人类进行每一步的链上审批与签名，资产控制权仍在人类手中。
    
-   **2026 (Autonomous Agents)：** AI 开始以机器速度、7×24 小时持续且自主地在链上移动资金。此时，**“技术已经准备好了，但信任基础设施还没有”**。
    

### 原始钱包授权带来的“静默失败 (Silent Failures)”

若直接给 Agent 原始钱包的私钥权限，由于大模型本身的黑盒与混沌属性，会带来无法通过代码层完全阻断的底层灾难：

-   **静默覆盖 (Silent Override)：** Agent 在运行中途受到恶意 Prompt 注入攻击或自身产生幻觉，无视了 Prompt 设定的金额限制，前端 dashboard 显示成功，后端悄悄超支。**语言只能表达意图，不能强制执行权限。**
    
-   **影子托管 (Shadow Custody)：** Agent 在主人的 Custody 边界之外，自主开辟了新的链上钱包并转移资金。没有策略引擎、没有审计日志、没有 Kill Switch（紧急制动），资金彻底失控。
    

## Cobo CAW 三大风控

对抗 AI 行为混沌的唯一方式，**用数学和密码学的确定性去框定边界**。其核心哲学是：**“Don’t give Agents Wallets. Give them Pacts.”**（不要给 Agent 钱包，给它们契约）。

1.  **MPC for AI Agents (2/2 阈值多签)：** 私钥分片由 Agent Share、Human Share 和 Cobo Share 协同。没有单一方可以单边移动资金。
    
2.  **Pact (契约) 授权协议：** 将权限进行基础设施级的结构化委派：
    
    -   **Intent（意图）：** 限定具体任务。
        
    -   **Execution Plan（执行计划）：** 透明可查的路线图。
        
    -   **Policies（策略）：** 严格锁死预算、时间窗口、交互资产的白名单。
        
    -   **Completion Conditions（完成条件）：** 超时或预算耗尽时，Pact 自动撤销，密钥自动失效。
        
3.  **Recipe Skill Layer (配方技能层)：** 预先固化并验证真实的链上执行路径，彻底杜绝 Agent 因幻觉生成错误的合约地址或错误的 ABI 参数。
    

## 概念对比：x402 vs ERC-8183

| 协议 / 规范 | 所属层级 | 核心解决的问题 | 商业链路中的角色 |
| x402 协议 | 前端网关与交互层 | 扩展 HTTP 402 状态码，序列化返回付款要求（币种、金额、链），让机器能自主读懂付费墙并完成 Checkout。 | “机器账单的发放者”：打破 Web2 和 Web3 边界。 |
| ERC-8183 / 8004 | 后端业务与结算层 | 规范链上的 Job（任务状态机）、Escrow（信托托管仓）和 Evaluator（第三方验证人）。解决多智能体在无信任环境下的履约防赖账问题。 | “智能合约信托与仲裁者”：确保钱随货走。 |

Agent Commerce 赛道的护城河不在于“模型能付钱”，而在于“如何在明确授权、预算控制、质量可验、全路径可审计的前提下，让机器安全、合规地完成自主交易”。这也是 Week 2 进行问题地图（Problem Map）和 Proposal 迭代时最核心的逻辑支点。

## 学习课程：为什么 Agent 需要长期记忆？

目前大多数大语言模型（LLM）是“无状态”的（Stateless）——每次对话都在重置上下文。虽然 Context Window（上下文窗口）越来越大，但依赖长窗口不仅成本高昂，且无法解决“跨周期”的逻辑一致性问题。

-   **核心问题：** 缺乏记忆的 Agent 就像一个每天都在失忆的极客。它无法从历史错误中学习，无法积累信誉，也无法维持长期的用户偏好。
    
-   **短期记忆 (Short-term)：** 依赖于当前的 Context Window，处理当前任务的临时信息。
    
-   **长期记忆 (Long-term)：** 通过外部存储（如 Vector DB 向量数据库、知识图谱 Knowledge Graph、链上状态 On-chain State）持久化保存的经验、规则、审计日志与用户偏好。
    

## AI × Web3 的交叉视角：记忆即状态与主权

如果将“长期记忆”放入 AI × Web3 的问题地图中进行审视，它完美契合了交叉领域的刚需：

1.  **可验证记录 (Verifiable Records)：** Web3 的链上记录就是 Agent 最客观、防篡改的“物理记忆”。Agent 声称自己“胜率很高”，不需要听它说，直接查它在链上的历史交互记忆（如 ERC-8004 中的历史 Job 执行记录）。
    
2.  **数据主权 (Data Sovereignty)：** 传统 AI 的记忆被锁定在中心化服务器（如 OpenAI 的后台）。结合 Web3（如 Arweave、Filecoin 或去中心化身份 DID），Agent 的记忆和用户数据可以实现端到端加密与用户完全所有，防范模型供应商的单点故障与数据泄露。
    

## 与主线方向融合-Wallet / Permission / Safe Execution

AI × Web3 问题地图与主方向选择，我学习了Wallet / Permission / Safe Execution长期记忆是实现系统从“僵硬规则”向“动态风控”进化的关键：

-   **从拦截中学习 (Guardrail Feedback Loop)：** 当 Agent 发起的高危交易被 Policy/Guard 拦截时，这个“拦截事件”必须写入 Agent 的长期记忆。下次遇到类似场景，Agent 会在规划阶段自动避开黑名单，而不是反复撞墙。
    
-   **权限的动态演进 (Dynamic Session Keys)：** 通过调取 Agent 历史的执行成功率和资金安全记录（信誉记忆），动态调整下一个 Session Key 的预算上限。表现越好，授权额度越大。
    
-   **审计日志持久化 (Audit Trail)：** 每一笔 Human-in-the-loop（人工确认）的交互记录，既是安全底线，也是 Agent 对齐人类意图的微调（Fine-tuning）语料记忆。

EVM的问题就是“贵”和“慢”，Layer 2（扩容层）的解决方案：

如果说以太坊主网（Layer 1）是一个要求全球几千个公证员同时验算每一笔账的“最高法院”**，权威但极其昂贵、低效；那么 Layer 2 就是**“地方仲裁庭”。

Layer 2 突破瓶颈的核心心法就一句话：**将“执行计算”移到链下，将“验证和数据存储”保留在链上。**

具体到 Rollup（卷叠）这种目前最主流的 L2 方案，它是如何做到既便宜又安全的呢？

### 1\. 成本是怎么降下来的？（打包与均摊）

Rollup 顾名思义，就是“卷起来”或“打包”。

在 Layer 2 上有一个专门的角色叫做**排序器（Sequencer）**。它就像一辆公交车，负责把成千上万个用户的交易收集起来，在自己的高性能服务器上（链下）飞速执行完毕，得出最终的账本状态。

然后，它把这成千上万笔交易高度压缩成一个“数据包”，**一次性**发到以太坊 L1 上。

-   **对于 L1 来说：** 它不用再逐条执行这 10000 笔交易了，它只需要存储这个压缩包。
    
-   **对于用户来说：** 以前每个人都要支付一次 L1 的天价 Gas，现在 10000 个人平摊了这一次“提交数据包”的 Gas 费，成本瞬间降到了原来的百分之一甚至千分之一。
    

### 2\. 安全性是怎么保证的？（两大验证流派）

既然计算是 L2 的单台服务器（排序器）做的，万一它作恶，偷偷把钱转给自己怎么办？

为了不牺牲安全性，L2 必须向 L1 证明自己的计算结果是清白的。这就衍生出了目前 Web3 最核心的两大武林门派：

### 乐观派：Optimistic Rollup（代表作：Arbitrum, Optimism）

-   **核心逻辑：“无罪推定”与“秋后算账”**
    
-   **机制：** L1 默认 L2 提交的数据都是真实的。但是，结果提交后会有一个“挑战期”（通常是 7 天）。在这个期间内，任何人都可以去下载 L2 的原始数据自己算一遍。如果发现 L2 排序器造假，就可以向 L1 提交一份**欺诈证明（Fraud Proof）**。
    
-   **保安手段：** 一旦查实造假，L1 会立刻回滚状态，并把造假者质押的巨额 ETH 没收（Slash），奖给举报人。只要网络里有一个诚实的监督者，系统就是安全的。
    

### 数学派：ZK Rollup（代表作：zkSync, Starknet）

-   **核心逻辑：“绝不轻信”与“密码学证明”**
    
-   **机制：** L1 谁也不信。L2 在提交上万笔交易结果的同时，必须利用零知识证明（Zero-Knowledge Proof）技术，生成一份极其复杂的数学证明（**有效性证明 / Validity Proof**）。
    
-   **保安手段：** 伪造这个数学证明的概率无限接近于零。L1 上的智能合约只需要花费极小的算力去验证这个证明是否成立。只要证明通过，就代表那上万笔链下交易绝对被正确执行了。
    

### 两大流派对比

| 维度 | Optimistic Rollup (乐观派) | ZK Rollup (数学派) |
| 安全依赖 | 经济学博弈与惩罚机制 | 纯粹的密码学与数学 |
| 提现到 L1 的速度 | 慢（必须等待 7 天挑战期结束） | 快（几分钟到几小时，数学证明验证即通过） |
| EVM 兼容性 | 极高（开发者几乎不用改代码） | 较难（正在通过 zkEVM 解决） |
| 短期与长期 | 当前市场的主流霸主 | 被公认为以太坊扩容的最终形态 |

4.  **守住风险边界：** 模型的意图解析能力绝对不等于最终的执行权。在跑自动化脚本和交互时，防钓鱼、防恶意授权、设置护栏（单笔上限、会话上限）比追求纯粹的“全自动”更重要。

-   **AI 在 Web 开发中的作用**：
    
    -   **设计协助**：AI 可协助人进行设计细化和分析，作为独立专家梳理未想到的数据流。
        
    -   **编码验证**：AI 自我验证在 AI 编码中非常重要，有助于实现高质量编码。
        
-   **Web2 与 Web3 项目能力及安全特性**：
    
    -   **项目能力占比**：单项目里 Web2 能力占比较高，常见应用如币安、policy market 等，Defi 项目如 uniswap、dydx 等离不开 Web2 前端页面。
        
    -   **安全特性差异**：Web2 支付体系需信任权威，资金通过银行卡账户和支付密码控制；Web3 资金交互有其特性，安全源于扎实基础和架构设计。
        
-   **Web3 支付流程及安全考量**：
    
    -   **支付流程**：支付服务先获取支付单信息展示给用户，法币支付与 cript 有所不同，cript 因使用与 Blockchain 相同算法无需额外请求。
        
    -   **安全考量**：需考量 API 安全，用户与电商平台、支付服务间的信任问题，可通过私钥签名校验保证数据不可篡改，但私钥泄露会降低信任。
        
-   **钱包相关概念及分类**：
    
    -   **钱包角色**：钱包在 Web3 中扮演交易确权角色，签名分交易签名和消息签名，可证明意图和身份。
        
    -   **钱包分类**：包括 EOA 钱包、合约钱包；按托管方式分全托管钱包、混合托管钱包；还有硬件钱包、隐私钱包等。
        
    -   **私钥问题**：私钥泄露或丢失后果严重，多签可将私钥切片，MPC 和合约钱包可实现多签。
        
-   **钱包使用的风险与场景**：
    
    -   **风险问题**：存在私钥泄露、误签交易和消息、权限滥用等风险，EIP712 签名可视化仍不够完善。
        
    -   **使用场景**：钱包在交易生命周期中用于构造交易、签名交易并广播，关键参数包括 gas fee、nonce、instruction 和 data 等。
        
-   **交易监听与确认**：
    
    -   **监听逻辑**：支付服务确认交易是否收到，需关注链上地址，有些交易需等待 finalize 时间。
        
    -   **风险防控**：可通过交易可视化、模拟等方式降低风险，还可通过 EUA、MPC 加密私钥完成签名。
        
-   **学习路径与相关问题**：
    
    -   **学习建议**：安全学习要自己能写作业、判断链的基本东西，再有的放矢学习。
        
    -   **钱包管理问题**：浏览器插件钱包交易管理、交易所充换钱包地址及交易归集逻辑等问题被提及。
        
    -   **AI 应用探讨**：探讨了 AI 在 Web3 安全方面的应用，提醒不要焦虑 AI，要扎实基础并利用好 AI。
