# 我今天第一次真的感觉到，这一周不是在做很多分散的任务，而是在逼自己把同一个问题讲清楚

今天如果只看表面，其实是一个很容易写成流水账的日子。

我从白天一直做到晚上，中间补了很多 Week 2 的东西：支付、身份、权限、threat model、治理协作、总交付，最后晚上还去上麦分享。如果按“我今天干了什么”来写，这篇笔记大概会很快变成一串文件名、一串文档标题，甚至写着写着自己都会觉得“今天就是在补作业”。但我现在回头看，今天最重要的感受反而不是“我做了很多”，而是：**我第一次真切地感觉到，这一周所有看起来分散的东西，其实一直在逼我回答同一个问题。**

那个问题就是：**当 AI Agent 不再只是回答问题，而是开始替人调用服务、准备交易、甚至尝试花钱时，我们到底该怎么给它边界？**

我觉得我这周前半段其实一直在从不同方向撞到这个问题。

5/25 装 Hermes 的时候，我第一次有了“Agent 真的进我电脑里跑起来了”的感觉。那天最强的体感不是“我会装一个工具了”，而是“一个东西开始长期存在于我的机器和我的日常里”。它不再是网页上一个临时对话框，而是会占着我电脑里的进程、会接收消息、会消耗我的资源、会对我的环境产生持续影响。那时候我还只是朦朦胧胧地感觉到：**Agent 一旦开始属于我，我就得先想它能被放到哪里。**

5/26 看 Cobo 那场直播的时候，这种感觉第一次被压成了更具体的问题。因为一旦 Agent 开始碰到钱包、预算、链上动作，事情就完全不一样了。你不能再用那种很松的说法，说“它大概帮我做一点操作”。你必须说得非常具体：这次能做什么、最多能花多少、在什么时间窗内有效、失败了怎么办、超出范围谁拦。也就是从那一天开始，我越来越确定，**Agent 的能力可以继续往上加，但权限一定要往下压得更细。**

5/27 Peter Farbey 讲隐私的时候，这个问题又被补上了另外一半。前一天我还主要在想“权限要不要给、预算怎么限、动作怎么卡”，第二天突然意识到：**就算授权边界已经想清楚了，也不代表用户已经被保护好了。** 一个 Agent 可以只花很少的钱，不代表它就没有暴露太多上下文；它可以只执行白名单里的动作，不代表它没有把你的行为轨迹、你的偏好、你的交互路径过度带出去。那一刻我开始更明确地意识到，“权限”和“隐私”其实不是两个完全分开的题，它们更像是同一个产品问题的两面。

所以今天当我继续写这些 Week 2 文档时，我脑子里其实已经没有把它们当成独立任务看了。我反而越来越清楚，它们像是在帮我从不同角度把同一件事讲完整。支付那部分在回答：Agent 如果真的要花钱，支付入口怎么被打开？权限策略那部分在回答：它的钱包能力怎么被限定在任务、对象、额度和时间窗里？threat model 在回答：就算边界写出来了，工具污染、上下文污染、外部依赖、错误输入还是会怎么把这条链路击穿？治理协作那部分又提醒我：AI 就算很擅长总结，也不能把“总结”直接偷换成“决定”。

今天我觉得最具体的一个变化，是我对“支付”这件事终于没那么浪漫化了。

以前我一想到 AI x Web3 里的 Agent 支付，会很自然地觉得这件事很酷，好像重点在于：未来 Agent 终于可以自己买服务、自己调用资源、自己执行链上动作了。但今天越写越发现，真正重要的不是“它终于会付钱了”，而是：**它凭什么付？它是按谁的意图在付？它最多能付多少？它付错了之后谁负责？**

x402 这个东西对我最大的价值，就是它逼我把这个问题看得足够具体。因为一旦你真的去想那个流程，就会发现它不是一句“Agent 自动支付”能带过去的。它更像这样：Agent 去请求一个资源，服务端不直接给结果，而是先返回 402，告诉它“这里需要付款，价格是多少，收款方是谁”。这个动作本身就很关键，因为它第一次把“AI Agent 的经济行为”从一种想象，变成了一个程序真的必须处理的条件分支。可一旦它具体了，我也更明显地看到它的边界：**x402 主要解决的是“怎么付”，它不自动解决“该不该付”。**

而“该不该付”这件事，今天我终于觉得自己能说清楚了。那是 CAW / Pact 这一层该解决的问题。不是负责生成结果，也不是负责收钱，而是负责把这笔钱限定在一个明确可解释的边界里。只能付给谁、最多付多少、多久内有效、超过阈值是不是要人工确认、出问题之后有没有日志和审计记录能回看。也就是说，支付动作本身可以越来越低摩擦，但**授权、预算、时间窗、可追责记录不能一起被自动抹平。**

我觉得这也是我今晚上麦的时候，最明显感受到的一件事：如果不把这个问题收拢，你一讲就会散。因为我这周做的东西太多了，真的很容易变成“我做了 A、B、C、D、F、G、总交付、进阶设计……”这种报菜名式分享。可我真正想讲给别人听的，其实根本不是“我这周做了很多文档”，而是：**我这周终于把一个问题讲明白了。**

我最后在分享里一直抓住不放的，其实就是三句话：

**draft 不是 decision。**

**simulation 不是 permission。**

**prepare 不是 execute。**

我现在特别喜欢这三句话，不是因为它们听起来像一句总结，而是因为它们真的像三条线，把这一周所有零散判断都穿起来了。Agent 可以帮我起草，可以帮我解释，可以帮我模拟，可以帮我检查，但它不能因为“已经帮我准备得很充分”，就自动获得下一步的决定权。以前我比较容易把“Agent 更强”理解成“Agent 替我做得更多”；现在我越来越觉得，真正成熟的方向可能刚好相反：**不是先让它做更多，而是先把它每一步到底能做到哪里讲清楚。**

今天还有一个特别实在的变化，就是我第一次觉得，这一周的东西开始不像几份笔记，而像一个真的项目了。因为我现在已经能很自然地说出，如果继续往下做一个 hackathon demo，我会做什么：我会做一个叫 AgentPay Guard 的东西。用户先给 Agent 一个很小的预算，比如 0.1 USDC；Agent 去请求一个付费的链上风险分析 API，如果对方返回 402，它先检查价格是不是在预算里；如果在预算里，就完成支付；如果不在，就把确认权还给用户。支付成功之后拿回结果，同时把这次调用、花费、返回内容和审计记录都展示出来。这个想法现在对我来说第一次不只是“听起来很酷”，而像一个我已经能说明白它为什么存在、它解决什么问题、它的边界在哪里的小项目。

所以如果要我给今天留一句最像结论的话，我会写这个：

**我终于不再只是分别理解 Agent、钱包、支付和权限，而是开始把它们当成同一个问题来思考了。**

而当这几个东西被真正放进同一条链路里之后，AI x Web3 对我来说也终于不再像一个概念拼贴，而开始像一个值得继续往下做、也值得对外讲清楚的方向。

## 我开始觉得，好的 Agent 产品卖点不该只是更能干，而是更会默认保护人

这两天连续听完 Cobo 讲 Agentic Wallet、Peter Farbey 讲隐私之后，我今天脑子里一直卡着一件事：**以前我会把“权限”和“隐私”当成两个分开的题，现在越来越觉得它们其实是同一个产品问题的两面。**

前几天我更在意的是，Agent 如果开始碰钱包、预算、链上动作，到底该怎么授权、怎么限额、怎么撤回。这个问题已经够硬了。但听完 Peter 之后，我今天更明确地意识到：**就算授权边界设计清楚了，也不代表用户已经被保护好了。** 一个 Agent 可以只花 10 美元，不代表它就没有过度暴露信息；它可以只执行白名单里的动作，不代表它和谁通信、暴露了哪些上下文、把多少行为轨迹留在外面，这些问题就自动消失了。

这让我开始重新想“一个更好的 Agent”到底好在哪里。以前我比较容易把“更好”理解成更强：会更多工具、接更多模型、自动化更多步骤、替我做更多事。可这两天的输入叠在一起之后，我今天反而越来越觉得，**未来真正有区分度的 Agent，未必是最能干的那个，而是最会克制的那个。** 它不是默认多拿权限、多看数据、多做动作，而是默认少拿一点、少暴露一点、少替我决定一点；真的需要放权的时候，再把边界讲清楚，把授权给具体，把撤回做简单。

我觉得这个判断对 AI x Web3 特别重要。因为 Web3 很容易让人先想到支付、钱包、交易，但如果只看到“能执行”，其实只看到了能力那一半。另一半应该是：**能不能把授权、可见性、撤回、审计这些保护层也一起设计出来。** 不然 Agent 只是比以前更会做事，不一定比以前更值得信任。

所以如果要我写下今天最确定的一句收获，我会写这个：**好的 Agent，不该默认更开放，而该默认更保护人。**

能做很多事当然重要，但在真正碰到钱、身份和长期上下文之前，先学会克制，可能才是它更成熟的开始。

今天这场 Peter Farbey 的分享，真的把我对“隐私”这件事的感觉改掉了不少。

我以前一听到隐私，脑子里会自动把它想得很重，好像这是一套特别 hardcore 的生活方式：你得完全不用大厂、完全隐藏自己、完全活成一个 cypherpunk，不然就不算认真。但 Peter 今天讲得很明白，隐私根本不是 0 和 100，也不是“要么全做、要么别做”。它更像一个刻度。你可以从一个很小的选择开始，比如换一个浏览器、换一个聊天工具，或者哪怕只是先知道自己在把什么数据交出去。这个说法对我特别有用，因为它一下子把隐私从一种“太难了先算了”的东西，变成了一种“可以慢慢往前走”的实践。

我觉得这点对我们这种在做 AI x Web3 的人尤其重要。因为这两天我脑子里一直在想 Agent 的边界问题。昨天听 Cobo 的时候，我更强烈地感觉到：Agent 一旦要碰钱包、预算、链上动作，权限一定要先卡好。结果今天 Peter 又补了另一层：**就算权限边界想清楚了，隐私边界也不等于自动解决。**

一个 Agent 能不能替你做事，和一个 Agent 默认会暴露多少信息，其实是两回事。它跟谁通信、通信是不是默认可见、交易是不是默认裸奔、什么时候该透明、什么时候该先保护用户不被观察、不被盯上、不被诈骗，这些都是产品一开始就该想的，不是等出事了再补。

今天我印象最深的一段，其实是他回答 AI Agent 那个问题的时候。他的意思很直白：现在大家都在疯狂做 Agent，但大多数人根本还没认真想过隐私。这句话对我有点像提醒，也有点像鼓励。提醒是：这个坑真的现在还没人认真填。鼓励是：那这反而可能就是我们这种 builder 现在最值得做的地方。不是再多拼一个炫 demo，而是去想，未来如果 Agent 会长期运行、会彼此通信、会替人执行交易，那它是不是应该默认带一层 privacy layer，而不是默认什么都暴露，出了问题再补洞。

我还挺喜欢他讲 Neo-Cypherpunk 的方式。不是那种高高在上的“你不够纯”，也不是老派隐私圈那种很沉重、很末日的感觉。他讲的是一种更松一点、更开放一点的方式：你不用一下子变成最懂密码学的人，也不用活成全套隐私人格，你只要开始做一些更有意识的选择，一点点把主动权拿回来就行了。我觉得这个说法很适合现在的我。因为我本来就不是从“隐私极客”这个身份进入这个领域的，我更像是先从 Agent 和 Web3 的交叉问题进来，然后慢慢意识到，原来隐私不是边角料，它其实就在核心位置。

所以如果要我总结今天最明确的一句收获，我会写这个：**隐私不是一套吓人的高标准，而是一层应该尽量默认给到人的保护。**

而且这层保护，不只是给人，也应该给以后越来越能行动的 Agent。

# 真正难的不是把 Agent 装起来，而是决定它能被放到哪里

昨天我终于把 Hermes Agent 装到自己电脑上，第一次有了“Agent 真在我机器上跑”的感觉。到今天晚上看完 Cobo Agentic Wallet PM 这场直播，我突然发现，装起来其实只是开始，真正难的问题根本不是“怎么把 Agent 跑起来”，而是“跑起来之后，哪些权限可以给它，哪些绝对不能默认放出去”。

这两天的感受连在一起特别明显。Hermes 让我第一次从“文档里的 Agent”走到“进程里的 Agent”——它会占内存、会连 Telegram、会长期存在。可一旦它不再只是帮我总结信息，而是开始碰支付、钱包、链上动作，问题立刻变了。今天这场 Cobo 直播对我最大的提醒，不是某个产品细节，而是一个很硬的边界感：**Agent 的能力可以越来越强，但权限必须越来越具体。** 不是抽象地说“它可以帮我付钱”，而是要明确到“它能为哪件事付、最多付多少、在什么时间窗内、失败了谁来接手、超出范围谁来拦”。

这件事刚好也反过来照到了我昨天的 Hermes 安装体验。昨天我最深的感受是“Agent 真的开始属于我了”；今天我更确定，**Agent 一旦开始属于我，我就必须先学会限制它，而不是先放大它。** 以前我对 AI Agent 的想象更偏“能帮我做更多事”，现在开始变成“我能不能给它一个足够清楚、足够可审计、足够容易撤回的任务边界”。这比多接一个模型、多装一个插件更关键。

所以我今天的认知更新，不是“又学了一个钱包方案”，而是把本周主线想明白了一点：**AI × Web3 真正有意思的地方，不是让 Agent 获得支付能力，而是让支付能力被约束在可验证的授权里。** 从这个角度看，Hermes 昨天解决的是“Agent 作为进程存在”的问题，Cobo 今天推进的是“Agent 碰到钱时该怎么被限制”的问题。前者让我第一次有了“Agent 的我”，后者让我第一次认真意识到，“Agent 的我”背后必须有一套清楚的 permission strategy。

**Agent 的「我」第一次成立——从读懂 install 到进程占着我电脑**

5/20 那天我在残酷共学里写过一句给自己的预言：「对 Agent 真正的判断，要在它真正占着我电脑的端口、消耗着我申请的 key、回复着我发的消息之后才能给。在那之前我所有的『理解』都是文档级别的，是『Agent 的概念』，不是『Agent 的我』。」当时这话写得有点漂亮但我自己知道它是个 IOU——欠自己的、还没兑现的认知。今晚兑现了。

实装跨了两个晚上的边界，从 5/24 23:18 备份 `.zshrc` 开始，到 5/25 凌晨 Telegram bot 回我第一句话结束。中间最让我意外的不是 Hermes 本身，是**装一个 Agent 跟我以为的「跟教程走」完全不是一回事**。我跟的是 0xUSDHG 同学那份指南——之前读了三遍、笔记列了关键命令、自以为路径很清晰。真正开始装，第一步 `brew install python@3.11` 就炸了：post-install 报 `pyexpat` 模块找不到符号 `_XML_SetAllocTrackerActivationThreshold`。根因翻到最后，是 macOS 26 Tahoe 系统的 libexpat 跟 Homebrew 编译这版 Python 时链的 expat header 不匹配。一个我从来没听说过的兼容性 bug，挡在 install guide 第 1 步前面。**这种细节没人能提前在指南里写**——它需要你的系统恰好是这个版本组合才会撞上。绕过去的方案也是当场摸出来的：卸 brew python，让 Nous 官方 `install.sh` 用 uv 装 portable Python（uv 装的 Python 是 static-linked，不碰系统 libexpat）。这一步让我第一次理解一件事——**Agent 装在「我的机器」上，意味着它要跟我的系统状态、版本组合、过往装过的工具周旋**，不是"复制粘贴"能搞定的。

第二件意外是关于「红线」的元层面反思。装之前我先开 plan mode 把整套流程写成了一份 plan 文件，里面在 launchd 那一步明确写了「故意不做后台常驻」**，理由是「不留长期攻击面，对齐 Handbook 的『不放它在能动钱的位置』红线」。结果** `hermes setup gateway` **wizard 走到最后一步问** `Install the gateway as a launchd service? [Y/n]:`**，默认 Y，我手指就按下去了。装完才意识到——**plan 里的红线被 wizard 的默认值悄悄绕过了。我先把 launchd 卸了一次`launchctl unload + rm plist`），又坐下来重新权衡：白名单只放自己一个 ID + Docker 沙箱跑工具 + `.env` chmod 600 三层兜底，承受 7×24 后台跑这个偏离应该够。最后选了接受偏离，重新装回 launchd。**重要的不是最后选了哪边，是这件事让我看到一个真实的工程规律——红线如果只写在 plan 文件里，不会自动传染到 wizard 操作那一刻**。下次写 plan，关键操作点旁边要加一条「执行提示」标红 wizard 默认值是什么、跟 plan 冲突时手动选 N。

收尾时 Telegram 那边发了句「测试」，bot 回了。那一刻我盯着屏幕想，5/20 写的那句「Agent 的我」其实是个进程哲学问题——一个 Agent 是不是「你的」，不看它叫什么名字，看它**有没有真的占着你电脑里一个 PID、消耗着你账号下的 token、在你白名单守护下接你手机推送**。Hermes 今晚有了。**装一个 Agent，不是把 install guide 跑完，是把你的系统、你的红线、你的注意力，都让给它一部分**。

# 了解 Hermes Agent 和不同类型的 Agent

今天主要了解了 **Hermes Agent**，也顺便对比了一下 **Claude Code** 和 **Claude Cowork**。

我之前对 Agent 的理解比较笼统，感觉只要能帮我干活就都叫 Agent。今天看下来发现，其实不同 Agent 的定位差别挺大。

**Hermes Agent** 更像是一个可以长期运行的个人自动化助手。它可以部署在本地或者 VPS 上，通过 Telegram、CLI 等方式使用，还可以记住项目上下文、沉淀 skills、做定时任务。放到 Web3 场景里，我觉得它很适合做链上监控、数据分析、定时报告、自动跑脚本这些事情。

**Claude Code** 更偏开发场景。它主要是在终端或者代码项目里帮开发者写代码、改 bug、跑测试、看报错、处理 Git 流程。比如我做 Solidity、Hardhat、React DApp 项目时，这类 Agent 会比较直接提升开发效率。

**Claude Cowork** 更像是办公和资料整理类 Agent。它适合处理文档、合同、报告、简历、项目资料整理这些任务，不是专门写代码的。

所以我今天的理解是：

**Claude Code 是写代码的 Agent，Claude Cowork 是整理资料的 Agent，Hermes Agent 是长期运行和自动化任务的 Agent。**

这也让我更理解 AI × Web3 里 Agent 的价值。它不只是聊天，而是可以结合工具、钱包、链上数据和权限控制，帮助用户完成真实任务。

不过 Web3 场景里的 Agent 也要特别注意安全边界。比如它可以帮用户分析数据、生成交易建议，但如果要真正发起交易，就必须考虑钱包授权、用户确认、额度限制和风险提示。

今天的收获是：  
**Agent 的核心不是会聊天，而是能在明确权限和工具范围内，把任务真正执行下去。**
